CH1 · 문제와 비전지식 사일로의 비용

지식은 있는데, 찾을 수 없다

대기업 제조 현장의 데이터는 N개 시스템에 흩어져 있고, 통합 비용은 지금 사람의 시간으로 지불되고 있습니다.

문제의 구조

N개 시스템, 0개의 공통 입구

같은 부품이 ERP에선 자재코드, MES에선 설비 파라미터, 메일에선 별명으로 불립니다. 데이터가 아니라 연결이 없는 것이 문제의 본질입니다.

발상의 전환

사람을 돌아다니게 하지 말고, 지식을 한곳에 검증해 쌓는다

지식은 원본 시스템이 아니라 검증된 그래프에 있다 — 원본은 ETL 대상일 뿐, 질의 시점에는 접근하지 않습니다.

핵심 명제AI는 노가다(추출·라우팅·재시도)를 하고, 사람은 판단(진성/가성·코멘트)만 하고, 지식은 자산이 되어 그래프에 누적됩니다.
보안 관점

폐쇄망이라는 전제

공정 데이터·설계 변경·수율 노하우는 외부로 나가면 안 되는 자산입니다. 그래서 처음부터 폐쇄망 서빙 가능한 오픈웨이트 LLM(gpt-oss·Llama·Qwen 등)을 전제로 설계합니다 — 2장에서 개념도로, 3장에서 플랫폼·모델 선택 기준으로 해부합니다.

CH1 · 문제와 비전4-Layer 아키텍처 조감

전체 조감 — 4개 계층, 2개의 흐름

사용자(L3)부터 원본 시스템(L0)까지. 질의 흐름은 1.4초에 끝나고, 동기화 흐름은 백그라운드에서 사용자 모르게 돕니다.

A. Query Flow — 사용자가 보는 것

질문 → 플랫폼이 Skill 선택 → LLM이 SPARQL 생성 → KG 조회 + 벡터 보강 → 자연어 답 + 출처. 원본 시스템은 건드리지 않습니다.

B. Sync Flow — 사용자가 못 보는 것

원본 변경분(CDC) → Kafka → LLM 추출 → Staging KG → HITL 검증 → Production KG. 검증을 통과한 지식만 답변에 쓰입니다.

CH2 · 폐쇄망 아키텍처구축 개념도 — 한 장

구축 개념도 — 모델은 들어오고, 데이터는 나가지 않는다

경계가 먼저입니다. 오픈웨이트 모델만 DMZ 반입 절차를 거쳐 들어오고, 그 뒤의 모든 질의 — ① 질문 → ② 라우팅 → ③ MCP tool 호출 → ④ RAG 검색 → ⑤ 근거 답변 — 는 폐쇄망 안에서 완결됩니다. 상자는 전부 역할(자리)이고 괄호의 제품명은 구성 예시 조합 A일 뿐 — 어떤 벤더 조합을 골라도 원리는 동일합니다(무엇을 고를지는 3장).

CH2 · 폐쇄망 아키텍처반입 · 서빙 · 사이징

폐쇄망 구성 원칙 — 왜 이렇게까지 하나

"편한 외부 API 한 줄"이 금지인 이유부터, 단계별 GPU 사이징까지 — 개념도의 각 상자를 운영 언어로 풉니다.

왜 외부 API 금지인가

프롬프트 자체가 유출 경로

"협력사 C가 납품한 Sensor_X의 불량 이력은?" — 이 질문 한 줄에 협력사명·부품명·품질 이력이 담겨 외부 서버로 나갑니다. 그래서 from openai import OpenAI는 코드 리뷰에서 즉시 반려 — 사내 서빙 LLM 엔드포인트만 허용합니다.

모델 반입 — 오픈웨이트라서 가능

오픈 모델은 가중치 파일을 통째로 들여온다

gpt-oss·Llama·Qwen 등 폐쇄망 서빙 가능한 오픈 LLM이 후보(선택 기준은 3장). 이 덱의 예시는 개발 gpt-oss:20b(Ollama, RTX 4090 1장), 운영 gpt-oss:120b(vLLM, H100). DMZ에서 보안 검토 → 해시 검증 → 승인 후 반입하며, 이후 추론은 인터넷 없이 돕니다. 엔드포인트·모델명은 환경변수로만 참조(절대 규칙 #7) — 반입·이관 때 코드가 안 바뀝니다.

권한과 감사 — 플랫폼의 몫

플랫폼 레이어(예: watsonx Orchestrate)가 거버넌스를 맡는다

누가 어떤 Skill을 호출할 수 있는지(SSO 권한), 언제 무엇을 호출했는지(감사 로그)를 플랫폼 계층에서 통제합니다 — 이 "자리"는 Bedrock Agents·Vertex Agent Builder로도 채울 수 있습니다(3장 비교). 트레이싱은 LangSmith self-hosted — 관측 도구까지 폐쇄망 안에 둡니다.

CH2 · 폐쇄망 아키텍처핵심 정리

핵심 정리 — 2장 정리

핵심 요약
  • 경계가 먼저 — 사외망 / DMZ / 폐쇄망. 모델 파일만 일방향 반입, 데이터는 절대 반출 금지.

CH3 · 플랫폼·모델 전략결론 — 분리해서 고른다

그런데 왜 자체 모델이 존재하나 — 그리고 결론

자체 모델의 가치는 벤치마크 점수 밖에 있습니다. 공정하게 반론을 보고 나면, 결론은 하나로 수렴합니다 — 플랫폼과 모델을 분리해서 고른다.

반론 ① 법적 보호 (인덤니티)

IBM은 Granite 산출물의 제3자 IP 침해에 무상한(uncapped) 배상을 제공 — MS·Google·AWS도 자사 관리형 모델에 저작권 배상을 제공(범위·상한 상이). HF에서 직접 받은 모델에는 이 보호가 없습니다 — 조달·법무 심사를 뚫는 열쇠.

반론 ② 초저가 티어

Granite 4.0 Micro $0.02/1M 입력 · Nova Micro $0.035/1M 입력 — 분류·요약·추출 같은 대량 단순 태스크에선 프론티어 점수가 무의미하고 단가가 전부입니다.

반론 ③ 거버넌스·데이터 큐레이션

Granite는 학습 데이터 출처 공개 + watsonx.governance 통합, Nova는 Bedrock Guardrails 네이티브 — 감사 대응이 계약서와 함께 옵니다. 규제 산업에선 이것이 점수 몇 점보다 무겁습니다.

반론 ④ 컨텍스트·파인튜닝

Granite 4.1은 512K 컨텍스트, 플랫폼 내 파인튜닝 파이프라인 제공 — "우리 문서 전체를 넣고 우리 용어로 미세조정"이 계약 하나로 해결되는 경로입니다.

결론 — 분리 쇼핑

플랫폼은 "자리", 모델은 "부품"

플랫폼은 오케스트레이션·거버넌스·감사 레이어로 고르고(폐쇄망 재현이 1차 필터), 모델은 폐쇄망에 올릴 수 있는 최고 성능 오픈 모델을 독립적으로 고릅니다. 벤치마크에서 진 자체 모델을 플랫폼 때문에 억지로 쓸 이유가 없습니다 — 플랫폼 대부분이 이미 BYO-model을 공식 지원하기 때문입니다.

BYO-model 증거 — "대결"이 아니라 "흡수"

Bedrock Custom Model Import(Llama·Mistral 계열) · Vertex Model Garden 200+(Qwen·DeepSeek 포함) · Azure 카탈로그 11,000+ · watsonx BYOM(HF safetensors 업로드) · Databricks FM API(Llama 4·Qwen3 호스팅) · Snowflake Cortex(llama4·deepseek-r1을 SQL 함수로) — 플랫폼들 스스로 오픈 모델을 1급으로 흡수했습니다.

이 덱의 스택watsonx Orchestrate(자리) + gpt-oss(부품)는 이 원리의 한 인스턴스일 뿐 — 내일 부품이 Qwen·Llama로 바뀌어도, 자리가 Vertex로 바뀌어도 2장의 아키텍처와 4~6장의 설계는 그대로 갑니다.

CH4 · 지식그래프트리플 · TBox/ABox

지식의 최소 단위 — 주어·서술어·목적어

"Sensor_X는 협력사 C가 공급했다"라는 문장 하나가 트리플 하나. 스키마(TBox)가 규칙을, 인스턴스(ABox)가 사실을 담습니다.

트리플 = 문장 하나

주어-서술어-목적어. "Sensor_X는(S) CompanyC가 공급했다(P-O)". RDB의 행이 아니라 화살표로 저장하므로, 몇 단계를 건너뛰는 질문에 강합니다.

TBox vs ABox

TBox=클래스·속성·제약(스키마), ABox=실제 트리플(데이터). 도메인이 확장되면 TBox만 갱신 — 1장의 "변경을 한 층에 가두기"가 여기서도 반복됩니다.

표준 온톨로지 재사용 (절대 규칙 #6)

직접 만들기 전에 가져다 씁니다 — IOF(제조·설비), SOSA/SSN(센서·스마트팩토리), PROV-O(출처·필수), BFO(최상위)·SKOS(분류). JPMorgan이 금융 표준 FIBO를 쓴 것과 같은 원리입니다.

함정 #1온톨로지 과설계 — 핵심 클래스 10~20개로 시작하세요. 온톨로지는 완성하는 것이 아니라 진화시키는 것입니다.
CH4 · 지식그래프PROV-O — 절대 규칙 #1

출처 없는 트리플은 가치가 절반 이하다

가장 흔한 실패는 "기술 검증은 됐는데 사람이 안 쓴다" — 원인은 거의 항상 답변 신뢰성 부족이고, 처방은 모든 트리플에 PROV-O를 박는 것입니다.

# ABox — HITL 통과한 인스턴스 (Turtle) :Sensor_X a :Component ; :suppliedBy :CompanyC ; :requiresApprovalFrom :Engineer_Kim ; # PROV-O: 출처 메타데이터 — 필수 prov:wasAttributedTo :Engineer_Kim ; prov:generatedAtTime "2026-03-15T11:23:00Z"^^xsd:dateTime ; rdfs:comment "온도 보정 회로 문제로 확인됨" .
# 적재 코드 규칙 — 트리플만 넣는 코드는 PoC도 반려 def commit_triple(subj, pred, obj, *, verifier, comment=None): kg.add(subj, pred, obj) kg.add(subj, "prov:wasAttributedTo", verifier) kg.add(subj, "prov:generatedAtTime", now_utc()) if comment: kg.add(subj, "rdfs:comment", comment) # ❌ kg.add(subj, pred, obj) # 출처 없음 — 금지
① 신뢰 — 답변에 얼굴이 붙는다

"이 답은 김 책임이 3월 15일에 검증했고, 코멘트는 '온도 보정 회로 문제'" — 출처가 붙는 순간 답변은 검색 결과가 아니라 동료의 확인이 됩니다. 사용률을 가르는 건 정확도보다 이것입니다.

② 재검증 방지 — 한 번 판정, 영구 재사용

진성 판정된 사실은 다시 검증하지 않습니다. 검증 이력이 트리플에 붙어 있으므로 같은 질문에 같은 검증 비용을 두 번 내지 않습니다.

③ 암묵지의 명시지화

rdfs:comment에 박힌 베테랑의 한 줄이 신입에게 그대로 전수됩니다. 퇴직과 함께 사라지던 노하우가 그래프의 자산이 되는 지점입니다.

보안 관점PROV-O는 감사(audit) 관점에서도 필수 — "이 사실을 누가 언제 승인했나"에 즉답할 수 있어야 폐쇄망 시스템의 책임 소재가 성립합니다.

CH5 · RAG 파이프라인하이브리드 검색

벡터만도, 그래프만도 아니다 — 하이브리드

"협력사 C 부품이 들어간 제품은?"은 SPARQL이, "EV 증착 압력 이상 대응 절차는?"은 벡터 검색이 정답입니다. 질문 유형이 경로를 정합니다.

청킹 실예시

8D 보고서 12쪽 → 문단 경계 기준 ~500자 청크 40개. 표·이미지는 Unstructured.io로 별도 추출. 청크가 너무 크면 노이즈가, 너무 작으면 맥락이 죽습니다 — 검색 품질의 8할이 여기서 갈립니다.

임베딩 실예시

"압력 이상"과 "chamber pressure drift"는 글자가 다르지만 bge-m3의 1024차원 공간에선 이웃입니다. 한·영 혼용 문서가 많은 제조 현장에서 한글 최적화 임베딩(bge-m3, nomic-embed-text)을 고르는 이유입니다.

함정 #4벡터만 쓰면 "협력사 C 부품 목록" 같은 정형 질문에 빠뜨림이 생기고, 그래프만 쓰면 서술형 노하우를 못 찾습니다. 라우팅하는 하이브리드가 필수입니다.
보안 관점임베딩 모델도 폐쇄망 안에서 돕니다 — 문서를 외부 임베딩 API에 보내는 것 역시 유출입니다.
CH5 · RAG 파이프라인SPARQL self-correction

LLM이 쓴 쿼리를 그대로 실행하지 않는다

사내 LLM(예: gpt-oss)이 생성한 SPARQL은 파서 검증 → 실행 → 에러를 힌트로 재생성의 루프(최대 3회)를 통과해야 합니다 — 절대 규칙 #3.

# 표준 패턴 — safe_sparql_execute def safe_sparql_execute(query, max_retries=3): for attempt in range(max_retries): if not is_valid_sparql(query): query = llm.fix(query, error="syntax") continue try: return fuseki.run(query) except SparqlError as e: query = llm.fix(query, error=str(e)) raise RuntimeError("max retries exceeded")
함정 #5 — LLM 과신

LLM은 그럴듯한 오답 쿼리를 자신 있게 씁니다. 검증 없는 실행은 틀린 답을 "검증된 답"처럼 포장하는 최악의 실패 — 파서가 첫 방어선, 에러 피드백 루프가 두 번째입니다.

왜 temperature=0인가

추출·쿼리 생성은 창의성이 아니라 재현성이 목표입니다. 같은 질문엔 같은 쿼리 — 디버깅과 품질 평가가 가능해집니다.

CH5 · RAG 파이프라인핵심 정리

핵심 정리 — 5장 정리

핵심 요약
  • 질문 유형이 경로를 정한다 — 정형은 SPARQL, 비정형은 벡터(top-k), 결합이 기본형.
  • 청킹(~500자·문단 경계)과 한글 최적화 임베딩(예: bge-m3)이 검색 품질의 8할.
  • LLM 생성 쿼리는 파서 검증 + 에러 피드백 루프(≤3회) 없이 실행 금지.

CH6 · 에이전트·MCPSkill ≡ Tool ≡ MCP tool

같은 개념, 세 개의 이름

"LLM이 호출하는 액션 단위"를 오케스트레이션 플랫폼(예: watsonx)은 Skill, 에이전트 프레임워크(예: LangChain)는 Tool, 개방 표준에선 MCP tool이라 부릅니다. 용어에 속지 않는 것이 절반입니다.

왜 MCP가 표준이 되는가

도구 연결이 벤더 중립이 되기 때문입니다. 에이전트 프레임워크를 갈아타도(LangChain→다른 것) MES·ERP 커넥터는 그대로 — 2장 개념도의 MCP 서버 열이 교체 가능한 부품인 이유입니다.

docstring = 라우팅 판단 근거 (절대 규칙 #5)

LLM은 docstring을 읽고 어느 도구를 쓸지 정합니다. "검증된 지식 그래프에 SPARQL을 실행한다"처럼 한국어로 명확히 — docstring이 모호하면 라우팅이 무작위가 됩니다. 부작용(DB 쓰기·메일 발송)도 반드시 명시합니다.

플랫폼 통합 경로오픈소스 에이전트를 기업 플랫폼에 1급 시민으로 등록하는 다리(예: watsonx의 Agent Connect) — 플랫폼 거버넌스 아래에서 오픈소스 에이전트를 돌리는 하이브리드 구성이며, 주요 플랫폼마다 유사한 등록 경로가 있습니다.
보안 관점도구는 곧 권한입니다. Action 도구(메일·티켓)는 Knowledge 도구보다 호출 권한을 좁게 — 카탈로그 단위로 통제합니다.
CH6 · 에이전트·MCP구축 레시피 5단계

RAG + MCP 에이전트 구축 레시피

2장의 개념도를 코드로 내리는 순서 — 도구 정의부터 감사 로그까지 다섯 단계.

1

도구 정의 — @tool + 한국어 docstring

sparql_query · vector_search · commit_triple · send_verify_email — docstring이 곧 라우팅 근거이므로 목적·입력·반환·부작용을 명시.

2

MCP 서버 등록 — 사내 시스템 커넥터

MES·ERP·문서관리·메일별 MCP 서버를 세우고 오케스트레이션 플랫폼의 도구 카탈로그에 등재(예: watsonx Skill Catalog). 호출 권한을 카탈로그에서 부여.

3

에이전트 라우팅 — LLM이 도구 선택

사내 LLM이 질문을 보고 docstring 기반으로 도구를 고름. 정형→SPARQL, 비정형→벡터(5장의 하이브리드가 여기서 자동화).

4

HITL 승인 — 쓰기는 사람을 거친다

KG에 쓰는 액션(commit_triple)은 interrupt()로 멈추고 담당 엔지니어의 진성/가성 판정을 기다림 — 다음 슬라이드.

5

감사 로그 — 전 호출 추적

셀프호스팅 트레이싱(예: LangSmith) + 플랫폼 감사 로그 + PROV-O. "누가·언제·어떤 도구로·무엇을"에 항상 즉답 가능.

# ① 도구 정의 — docstring이 라우팅 근거 # 구현 예시(LangChain) — 스택 선택에 따라 대체 가능 from langchain.tools import tool @tool def sparql_query(query: str) -> str: """검증된 지식 그래프에 SPARQL SELECT를 실행해 JSON 결과를 반환한다. 부작용 없음(읽기 전용).""" if not is_valid_sparql(query): # 5장 루프 return error_with_hint(query) return fuseki_client.run(query) @tool def send_verify_email(to: str, payload: dict) -> str: """담당 엔지니어에게 검증 요청 메일을 보낸다. 부작용: 사내 SMTP 발송 — Action 권한 필요.""" return orchestrate.call_skill("send_verify_email", to=to, payload=payload)
LLM 초기화 — 구현 예시ChatOllama(model=os.getenv("LLM_MODEL"), temperature=0, base_url=os.getenv("LLM_ENDPOINT")) — ChatOllama는 Ollama 서빙 기준 예시로, 서빙 스택에 따라 해당 클라이언트로 대체합니다. 모델명·엔드포인트는 환경변수만(절대 규칙 #7) — 하드코딩하면 PoC→Pilot 이관 때 전부 터집니다.
순서의 의미①→⑤는 의존 순서이기도 합니다 — 도구 없이 라우팅 없고, HITL 없이 쓰기 없고, 로그 없이 운영 없다.
CH6 · 에이전트·MCPHITL — PostgresSaver

HITL 큐레이션 — 며칠을 잠들어도 깨어나는 워크플로우

검증 요청 메일에 엔지니어가 답하기까지 며칠~몇 주. 그 사이 서버가 재시작돼도 상태가 살아 있어야 하므로 — PostgresSaver가 절대 규칙 #4입니다.

# 체크포인트 — 상태 영속화 # 구현 예시(LangGraph) — 원칙은 "영속 체크포인트" from langgraph.checkpoint.postgres \ import PostgresSaver app = g.compile( checkpointer=PostgresSaver .from_conn_string(DSN)) # ❌ MemorySaver — 재시작하면 # "3주째 기다리던 검증"이 증발
왜 MemorySaver 금지인가

HITL은 며칠~몇 주짜리 트랜잭션입니다. 메모리 체크포인트는 배포·재시작 한 번에 전부 소실 — 검증 대기 중이던 수십 건이 사라지고, 엔지니어에게 같은 메일이 다시 갑니다. 신뢰가 무너지는 지름길입니다.

함정 #2검증자 피로 — 하루 5건 이하로 배치하고 우선순위를 매기세요. 검증 UI·빈도 설계가 추출 정확도보다 성패를 더 좌우합니다 (검증 1건 ≤90초 목표).

CH7 · 로드맵·안전망PoC → Pilot → Scale → Settle

로드맵 — 관문을 통과해야 다음 단계

Siemens의 교훈: 한 번에 전사 통합을 시도하지 않는다. 한 도메인에서 한 사이클을 완주하고, 숫자로 관문을 넘을 때만 확장합니다.

PHASE 1 · PoC — M1-2

한 도메인, 한 사이클 완주

문서 10~50건 · 핵심 클래스 10~20개 · 추출→검증→적재→질의 풀 사이클 1회.

예시 스택: Fuseki+Pellet · Qdrant · gpt-oss:20b(Ollama) · RTX 4090 ×1 · 라이선스 비용 0.

관문: 추출 정확도 ≥70% · 검증 1건 ≤90초
PHASE 2 · Pilot — M3-6

같은 도메인 전체 + 인접 1-2개

검증 배치 운영(일 5건 이하) · Learning Store 가동 · 트레이싱 품질 리뷰(예: LangSmith) 월간 거버넌스화.

예시 스택: GraphDB Free · gpt-oss:120b(vLLM) · H100 ×1.

관문: 가성률 30%→10% · 누적 트리플 월 1만+
PHASE 3 · Scale — M7-12

도메인 확장 + 자동 동기화

CDC(예: Debezium+Kafka) 상시 가동 · 추론·SHACL 검증 본격화 · MCP 커넥터 확대.

예시 스택: GraphDB EE 또는 Stardog(HA·OBDA) · H100 ×4 멀티노드.

관문: NL 응답 정확도 ≥90% · 원본 접속률 감소
PHASE 4 · Settle — M13+

UI 없는 일상 업무로 정착

Slack·메일에서 묻는 것이 기본 동선 · 신입 온보딩의 1차 정보원이 그래프 · 지식 자산이 복리로 증가.

관문: 온보딩 기간 30%↓
STORE 예시Fuseki (무료)→GraphDB Free→GraphDB EE/StardogGPU4090 → H100 → H100×4원칙관문 통과 전 지출 금지
벤치마크Siemens(점진 확장) · Bosch(OBDA로 분석 효율 70%↑) · AstraZeneca(LLM+KG 결합) · JPMorgan(표준 온톨로지 FIBO) — 공통점은 작게 시작해 검증하며 키웠다는 것.
CH7 · 로드맵·안전망10대 절대 규칙

10대 절대 규칙 — 전부 "당한 뒤에" 만들어진 것들

코드 리뷰에서 무조건 걸리는 열 가지. 규칙보다 "왜"를 기억하세요.

RULE 01

PROV-O 출처 필수

출처 없는 답은 아무도 안 믿는다 — 사용률이 무너지는 1번 원인.

RULE 02

사내 서빙 LLM만

외부 API 호출 = 프롬프트째 데이터 유출. gpt-oss는 한 예일 뿐, 원칙은 "가중치가 사내에".

RULE 03

SPARQL 검증 루프

LLM은 자신 있게 틀린다 — 파서 검증 + self-correction ≤3회.

RULE 04

PostgresSaver

HITL은 몇 주짜리 트랜잭션. 메모리 체크포인트는 재시작에 증발.

RULE 05

docstring 필수

LLM 라우팅의 유일한 판단 근거. 모호하면 라우팅이 복불복.

RULE 06

표준 온톨로지 우선

IOF·SOSA·PROV-O 재사용 — 신규 클래스 남발은 과설계 함정.

RULE 07

환경변수만

엔드포인트·모델명 하드코딩은 PoC→Pilot 이관 때 전부 터진다.

RULE 08

파라미터 바인딩

f-string으로 값 삽입 = SQL 인젝션 초대장. %s · :name · ? 만.

RULE 09

NULL 처리 명시

NOT IN에 NULL이 섞이면 결과가 통째로 빈다 — 조용한 오답.

RULE 10

월별 파티션 동적

테이블명 YYYYMM은 strftime으로 — 하드코딩은 매월 1일에 죽는다.

한 줄가장 흔한 실패는 "기술 검증은 됐는데 사람이 안 쓴다" — 원인은 거의 항상 답변 신뢰성 부족이고, 처방은 규칙 01(PROV-O 동봉)입니다.
CH7 · 로드맵·안전망핵심 정리 · 최종

핵심 정리 — 전체 코스 마무리

이 덱의 여섯 문장
  • 지식은 원본이 아니라 검증된 그래프에 있다.
  • 모델은 들어오고 데이터는 나가지 않는다 (폐쇄망).
  • 플랫폼은 자리, 모델은 부품 — 분리해서 고른다(BYO-model).
  • 정형은 SPARQL, 비정형은 벡터 — 하이브리드 + 검증 루프.
  • 도구는 하나의 개념(Skill≡Tool≡MCP), 쓰기는 HITL.
  • 관문을 넘어야 확장 — PoC 한 사이클이 모든 것의 시작.
내일 할 일

가장 작은 완주 — PoC 체크리스트

① 도메인 하나·문서 10건 선정 ② 핵심 클래스 10개 TBox ③ 오픈 LLM + KG 스토어 + 벡터DB(예: gpt-oss:20b·Fuseki·Qdrant)를 Docker Compose로 ④ 추출→검증 메일→진성 판정→PROV-O 적재→질의 응답까지 한 바퀴. 이 한 바퀴가 모든 확장의 증거가 됩니다.

  1. PoC 관문 — 추출 정확도 ≥70% · 검증 1건 ≤90초.
  2. 값이 쿼리 구조로 해석되는 인젝션 위험 + 따옴표·NULL 처리 버그 — 파라미터 바인딩(%s, :name, ?)만 허용.
  3. 진성 재사용(재검증 불필요) · 가성은 학습 신호 · 코멘트로 암묵지 전수 — 지식이 자산으로 복리 증가.
