합성 사례: 후보 검토 회의

설명을 위해 단순화한 가상 사례입니다. 모델 후보표를 화면에 띄우자 첫 행에는 높은 leaderboard 점수와 “32B”만 적혀 있고, license 원문 링크와 정확한 repository revision은 비어 있습니다. 반입 담당자는 파일 목록을 물었고, 운영 담당자는 24 GB GPU에서 어느 context 길이까지 버티는지 묻지만 표에서는 답을 찾을 수 없습니다.

처음에는 성능이 높은 두 모델만 서버에 올려 비교하려 했습니다. 그런데 한 후보는 영리 사용 조건을 확정하지 못했고, 다른 후보는 tokenizer와 quantized artifact의 출처가 빠져 있었습니다. 점수 차이보다 운영 후보로 성립하는지부터 확인해야 한다는 쪽으로 검토 순서가 뒤집힙니다.

따라서 이 글의 기술 권고는 release별 license와 사용 목적, 반입 artifact, 실제 VRAM 측정을 먼저 채우는 것입니다. 어느 칸도 모델 이름만으로 추정하지 않습니다. 조건이 바뀌는 라이선스와 runtime 정보는 확인일을 남겨야 하며, 이 표는 법률 자문이나 용량 보증을 대신하지 않습니다.

바로 적용할 기준

모델 후보표에는 벤치마크 점수보다 먼저 라이선스, 허용 용도, 반입 파일 목록, VRAM 요구량, 업데이트 경로를 적으세요. 이 다섯 항목이 통과하지 못하면 성능 비교를 시작하지 않는 편이 현장 검토 시간을 아껴줍니다.

가중치 파일을 내려받을 수 있다는 사실과 해당 릴리스를 영리 환경에서 운영할 수 있다는 판단은 별개입니다. 폐쇄망 반입 목록에는 정확한 model release, license 원문과 확인일, weight·tokenizer·runtime 파일, 예상 VRAM 조건을 함께 적습니다. 이 글의 모델·라이선스·VRAM 표는 2026년 7월 작성 시점의 검토 예시이며, 실제 도입 때 배포자가 제공한 해당 릴리스의 문서를 다시 확인해야 합니다.

BYO 모델: “오픈”이 실제로 주는 것

BYO model 방식은 승인된 weight와 runtime을 폐쇄망으로 반입해 사내 GPU에서 서빙합니다. 외부 API 의존성을 줄일 수 있지만 tokenizer, container image, Python package, model card까지 반입·업데이트 절차에 포함해야 합니다. Telemetry와 package download 경로가 남아 있지 않은지도 배포 시험에서 확인합니다.

"Open weight"는 parameter 파일 접근을 뜻할 뿐 허용 용도까지 같다는 의미는 아닙니다. 같은 model family 안에서도 release별 license와 acceptable use 조건이 달라질 수 있습니다. 후보표에는 계열명만 쓰지 말고 정확한 repository와 revision을 기록합니다.

라이선스가 먼저다: 벤치마크보다 앞서는 필터

법무 검토 전에는 다음 항목을 기술 담당자가 먼저 정리합니다. 아래 내용은 법률 자문이 아니라 release 원문 확인을 위한 질문 목록입니다.

  • 상업적 사용. 영리 회사의 해당 사용 사례가 허용되는가? Apache 2.0이나 MIT 계열이라도 bundled code와 dataset, 사용 정책에 별도 조건이 없는지 확인합니다.
  • 재배포와 이름. 파인튜닝 결과를 공유할 때 이름을 바꾸거나 저작권 고지를 포함해야 하는가? 예를 들어 Meta 라이선스는 파생 모델 이름에 Llama를 유지하고 고지 파일을 포함하도록 요구합니다.
  • 사용자 수 또는 사용 분야 제한. 일부 라이선스는 Llama의 월간 활성 사용자 7억 명 조항처럼 임계값을 두거나 특정 사용을 제한합니다. 내 사용 사례가 그 범위 안에 있는지 확인해야 합니다.
  • 오프라인 감사 가능성. 라이선스 텍스트는 가중치와 함께 반입되어야 하고 온프레미스 검토에서 확인 가능해야 합니다. 런타임에 인터넷에서 약관을 가져오는 방식은 폐쇄망에서는 통하지 않습니다.

EXAONE 릴리스별 라이선스 확인

EXAONE은 계열명만 보고 판단하지 말고 선택한 release의 license를 확인해야 합니다. 표의 2026년 7월 예시는 EXAONE 4.5 33B와 EXAONE AI Model License 1.2 - NC를 짝지은 것입니다. 상업·비상업 사용을 함께 허용하는 K-EXAONE처럼 다른 조건의 release도 있으므로 "EXAONE은 모두 NC"라고 분류해서는 안 됩니다. NC release의 연구·학술·교육 범위와 상업 계약 절차는 원문과 배포자 안내를 기준으로 판단합니다.

자가 학습과 대학 연구, 영리 FAB의 업무 배치는 사용 목적이 다릅니다. Weight가 정상적으로 load되고 답변이 나온다는 사실은 license 적합성을 검사하지 않습니다. POC 단계에서도 최종 사용 주체와 목적을 적고, 별도 허락이 필요한 release라면 계약 확인 전 운영 데이터와 연결하지 않습니다.

GPU 기준으로 크기 잡기: 어떤 모델이 어디에 올라가는가

다음에는 보유 GPU에서 목표 context와 동시성을 만족하는지 측정합니다. 4-bit weight는 FP16 weight보다 파일·가중치 메모리를 줄이지만 실제 VRAM은 quantization format, runtime buffer, KV cache, batch size, context 길이에 따라 달라집니다. 품질 손실도 task와 quantizer에 따라 달라지므로 RAG 평가셋에서 원본 또는 상위 precision과 비교합니다.

표 2. 2026년 7월 기준의 대략적인 weight VRAM 예시. Quantization 형식·runtime·KV cache·context·동시성에 따라 달라지므로 실제 artifact로 측정해야 합니다.
크기 FP16 대략치 Q4 대략치 실행 가능한 환경
7-8B ~16 GB ~5-6 GB 일반 소비자 GPU (16-24 GB)
14B ~28 GB ~10 GB RTX 3090 / 4080급
32B ~65 GB ~19-20 GB 24 GB는 빠듯, 48 GB는 여유
70B ~140 GB ~40 GB 1x 80 GB(Q4) 또는 멀티 GPU
120B MoE - 네이티브 MXFP4 ~1x 80 GB(gpt-oss-120b)

표의 숫자는 capacity 계획의 시작값일 뿐입니다. KV cache는 context와 동시 요청 수에 따라 늘고, 일부 runtime은 workspace와 graph capture 메모리를 추가로 씁니다. MoE는 token마다 활성화되는 parameter 수와 전체 weight 적재량이 다르므로 계산량이 줄어도 모든 weight를 위한 메모리가 필요할 수 있습니다. 선택한 runtime의 실제 artifact를 올려 peak VRAM, tokens/s, queue time, OOM 시 동작을 측정합니다.

팹을 위한 의사결정 경로

필터를 순서대로 적용하면 후보는 자연스럽게 줄어듭니다. 아래 흐름은 배포 절차서가 아니라 검토에서 기대하는 의사결정 순서를 보여주는 개념도입니다.

이 코드는 검토 순서를 설명하는 의사 코드입니다. License 문자열만으로 법적 허용 여부를 자동 판정하지 않으며, 정확한 repository·revision의 원문과 사용 정책을 담당 부서가 확인해야 합니다.

첫 파일럿의 model size는 보유 GPU와 질문 난이도에 따라 달라집니다. 14B-32B Q4가 단일 GPU 후보가 될 수 있지만 긴 context나 높은 동시성이 필요하면 더 작은 모델이 안정적일 수 있고, 24 GB에서도 32B는 runtime에 따라 여유가 부족할 수 있습니다. 70B나 MoE는 작은 모델 대비 평가 개선폭이 운영 비용을 정당화할 때 검토합니다. Weight와 함께 license 원문, revision, checksum, tokenizer, quantization 정보를 반입 기록에 남깁니다.

10분 후보 카드 실습

후보 하나를 골라 repository@revision, license 확인일, 허용 용도, weight·tokenizer·runtime 파일, quantization, 목표 context, 동시 요청 수, 보유 VRAM을 한 줄에 적어 보세요. License를 확정하지 못하면 “법무 확인 전 중단”, artifact checksum이 없으면 “반입 전 중단”, 실제 model load에서 OOM이 나면 “크기·context 재선정”으로 표시합니다. 빈칸이 남은 후보는 benchmark 대결에 올리지 않는 것이 이 실습의 통과 조건입니다.

검토 순서는 release별 license와 사용 목적, 반입 가능 artifact, 실제 GPU 측정, 현장 평가셋 순으로 둡니다. Leaderboard는 후보 탐색에는 유용하지만 SOP 근거 답변, 한영 혼합 용어, refusal, latency를 대신 평가하지 못합니다. License와 runtime 조건은 바뀔 수 있으므로 정기 review 때 같은 확인일과 revision을 갱신합니다.

후보 카드가 두세 개로 줄었다면 다음에는 표의 VRAM 추정치를 실제 요청으로 깨뜨려 볼 차례입니다. vLLM·Ollama 서빙 글에서 동일 모델과 prompt trace를 사용해 p95 지연, peak VRAM, OOM 복구를 비교할 수 있습니다.
