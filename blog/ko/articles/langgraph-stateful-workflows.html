LangGraph: 상태를 저장하고 재개 가능한 agent workflow

교대가 바뀌어도 남아야 할 상태

설명을 위해 단순화한 가상 사례입니다. 야간조가 SPC 편차 보고서 초안과 관련 계측값을 입력하고 공정 엔지니어의 승인을 기다리는 흐름을 가정해 보겠습니다. stateless chain에서는 승인 링크를 다음 날 열었을 때 초안의 근거와 반려 횟수가 사라져 같은 질문을 다시 실행했습니다. 화면에는 비슷한 새 초안이 나왔지만 전날 검토한 문서와 동일하다는 보장이 없었습니다. 그래서 보고서 ID, 근거 문서, 초안, 검토자, 결정, 반려 사유, 수정 지시, 재작성 횟수를 state에 넣고 승인 직전 checkpoint에서 멈추도록 바꿨습니다. 다만 checkpoint는 맥락을 보존할 뿐, 근거의 정확성이나 승인자의 권한을 대신 검증하지 않습니다.

그래프에 먼저 적을 것

기술 권고: 사람이 판단하는 단계에는 검토자 역할, 표시할 근거, 승인·반려 값, 반려 사유와 수정 지시, 반려 후 돌아갈 노드, 재개에 사용할 ID를 state로 적으세요. 이 정보가 없으면 일시정지는 가능해도 교대 인계와 재작성을 재현하기 어렵습니다.

일반 chain은 입력부터 출력까지 한 번에 흐릅니다. SPC 편차 보고서 초안, 레시피 변경 제안, 정비 티켓 종료처럼 중간에 사람이 판단해야 하는 작업은 다릅니다. 검토자가 근거를 보고 승인하거나 반려할 때까지 실행을 멈추고, 다음 교대가 같은 상태에서 이어받아야 합니다. LangGraph는 이처럼 분기와 일시정지, 재개가 있는 workflow를 명시적으로 구성합니다.

왜 chain이 아니라 graph인가

선형 chain은 A → B → C처럼 정해진 경로에 적합합니다. 승인 프로세스에서는 반려 시 초안 작성으로 돌아가고, 위험도가 낮으면 일부 검토를 건너뛰며, 다음 날에도 중단 지점에서 재개해야 합니다. LangGraph에서 node는 작업 단위, edge는 다음 단계로 가는 조건, checkpoint는 그 사이의 state를 보존합니다. 재개 가능성은 그래프 모양만으로 생기지 않으며, checkpointer와 안정적인 thread_id를 함께 구성해야 합니다.

Chain은 질문에 답합니다. Graph는 사람이 끼어드는 프로세스를 위치를 잃지 않고 실행합니다. — 기억해야 할 한 줄

아래 예시는 모두 로컬에서 실행됩니다. 폐쇄망 스택 안에서 Ollama 기반 온프레미스 LLM, 디스크에 저장된 모델 가중치, 로컬 checkpoint를 사용합니다. 그래프 실행을 위해 외부 네트워크를 호출하지 않습니다.

State, node, edge

State는 그래프를 따라 흐르는 타입이 지정된 dictionary입니다. 각 node는 state를 읽고 자신이 변경한 필드만 반환하며, LangGraph가 기존 값과 병합합니다. 아래처럼 TypedDict로 계약을 정의하면 어떤 정보가 checkpoint에 남는지 코드에서 확인할 수 있습니다. add_messages reducer는 메시지 목록을 덮어쓰지 않고 뒤에 추가합니다.

Node는 state를 입력으로 받는 함수입니다. 필요한 값을 읽고 변경한 필드만 dictionary로 반환합니다. 여기서 첫 번째 node는 온프레미스 모델에 편차 보고서 초안을 작성하게 하며, 공정 맥락은 message로 전달합니다.

Edge는 노드를 연결합니다. 일반 edge는 항상 한 노드에서 다음 노드로 이동합니다. START와 END는 그래프가 어디에서 시작하고 끝나는지 표시하는 특별한 상수입니다. 노드와 edge를 StateGraph builder에 등록한 뒤 compile()을 호출하면 실행 가능한 그래프가 됩니다.

조건 분기

검토 결과가 승인이면 commit으로 가고, 반려면 수정 정보를 남긴 채 초안 작성 단계로 돌아갑니다. 이 경로를 조건부 edge로 표현합니다. routing 함수는 state의 명시된 값만 읽어 다음 node 이름을 반환해야 하며, 자유 형식 모델 문장을 그대로 분기 조건으로 쓰지 않는 편이 안전합니다.

interrupt()로 사람 승인 기다리기

human_review node에서 interrupt()를 호출하면 그래프가 멈추고 UI·CLI·검토 대기열에 payload를 넘깁니다. 이 payload에는 초안만이 아니라 출처, 위험도, 요청된 결정값을 포함해야 합니다. state는 해당 지점에 checkpoint되고, 엔지니어 응답은 Command(resume=...)로 같은 실행에 전달됩니다. 그 값이 interrupt()의 반환값이 되어 node의 나머지 코드가 이어집니다. 재개할 때 노드는 처음부터 다시 실행되므로 interrupt() 앞에는 파일 저장, 외부 API 호출 같은 비멱등 부수 효과를 두지 마세요. 꼭 필요하다면 별도 노드로 분리하거나 멱등 키로 중복 실행을 막아야 합니다.

반려 응답은 사유와 수정 지시를 state에 저장하고, draft_report가 다음 호출에서 두 값을 프롬프트에 넣습니다. 따라서 재초안은 단순 재실행이 아니라 검토 의견을 반영한 수정본이 됩니다. 아래 실행 예제처럼 서버 인증 미들웨어가 확인한 principal만 reviewer_id를 넣게 하고, 필요한 역할이 없으면 재개 호출 자체를 거부해야 합니다. LangGraph의 interrupt()는 승인자 인증을 대신하지 않습니다.

Checkpoint로 재개 가능하게 만들기

Checkpoint는 node 실행 뒤의 전체 state를 저장합니다. 실행에 thread_id를 붙이면 몇 시간 뒤 같은 ID로 중단 지점을 찾을 수 있습니다. 개발 중에는 메모리 저장소로 흐름을 확인하고, 실제 배포에서는 SQLite나 Postgres checkpointer로 바꿉니다. 폐쇄망에서는 저장소와 백업도 내부에 두고, report ID와 thread_id의 매핑 및 보존 기간을 운영 규칙으로 정해야 합니다. 앞에서 human_review와 commit_report를 정의했으므로 이제 그래프를 실제 실행 순서대로 조립할 수 있습니다.

review loop 실행하기

첫 호출은 interrupt()가 발생할 때까지 진행되고 검토 payload를 반환합니다. 이후 검토할 때마다 Command로 같은 thread를 재개합니다. 아래 예시는 첫 초안을 반려해 수정본을 받은 뒤 다시 승인하므로 총 세 번 호출합니다. thread_id가 이 호출들을 하나의 지속 실행으로 묶어 줍니다.

구성 요소는 state, node 함수, 조건부 edge, checkpointer, interrupt()입니다. 이 뼈대는 레시피 변경 제안, 알람 분류, 교대 인계 요약, 편차 보고서 검토에 적용할 수 있습니다. 각 업무마다 승인 주체와 commit 권한은 따로 정해야 하며, 모델은 초안 작성과 제한된 routing만 맡습니다. 실행 기록과 checkpoint는 내부 네트워크에 남깁니다.

종이 한 장으로 하는 재개 tabletop

설명을 위해 단순화한 가상 실행입니다. 카드 세 장에 drafted, waiting_review, approved를 적고, 종이에는 report ID·근거 revision·reviewer·retry count를 씁니다. 아래 첫 두 항목은 현재 InMemorySaver 데모에서 같은 프로세스 안의 상태 전이만 확인합니다. 프로세스 재시작 복구는 영속 checkpointer와 별도 통합 시험이 있어야 통과로 인정합니다.

  • 승인 대기 중 같은 프로세스에서 동일 thread_id를 다시 조회합니다. 동일 초안과 근거 revision이 유지돼야 합니다.
  • 권한 없는 사용자가 resume를 요청합니다. state가 바뀌지 않는지 확인합니다. 운영 설계에서는 인증 gateway가 거부 이벤트를 별도 감사 저장소에 남겨야 합니다.
  • 검토자가 반려 사유를 남깁니다. retry count가 올라가고 수정본은 다시 waiting_review로 돌아가야 합니다.

어느 상황에서든 승인 전에 commit 카드로 이동하거나, 같은 thread의 근거 revision이 달라지거나, 반려 사유가 사라지면 이 데모를 통과시키지 않습니다. 운영 전에는 Postgres·SQLite 같은 영속 checkpointer로 프로세스를 실제 종료한 뒤 복구하고, 거부 감사 이벤트와 동시 resume 충돌도 시험합니다. 이 tabletop은 graph 전이와 기록 누락을 찾을 뿐 승인자의 도메인 판단을 검증하지는 않습니다.

LangGraph는 RAG 검색, 도구 호출, 온프레미스 LLM을 상태가 있는 프로세스로 묶습니다. 사람 판단을 기다리는 작업이라면 재개 가능성과 승인 지점을 처음부터 설계에 넣으세요. 다음 글인 ‘온프레미스 agent의 HITL과 MLOps’에서는 이 중단점을 영속 저장소·감사 로그·릴리스 게이트로 확장합니다. 그래프가 실행 순서를 보존해도 잘못된 원문, 과도한 권한, 만료된 승인은 별도로 막아야 합니다.
