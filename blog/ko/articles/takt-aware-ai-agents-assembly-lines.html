합성 장면: 답변이 도착했을 때는 다음 차체

Torque station의 cycle time이 길어지고 Andon이 켜진 뒤, Agent 요약은 몇 분 후 quality queue에 도착합니다. 내용은 맞지만 화면에는 이미 다음 VIN이 들어와 있고 line leader는 수동 escalation을 끝낸 상태입니다. 처음에는 답변 정확도 문제처럼 보이지만 실제로 빠진 입력은 대응 기한과 현재 line state입니다. 조립 라인에서 가능한 상황을 단순화한 가상 장면이며 특정 공장의 사건이 아닙니다.

설계 시작점

각 tool call 옆에 response class를 붙이세요. Immediate, Near-term, Planned stop, Engineering 중 하나를 고르고 owner와 만료 시각을 적습니다. 만료된 답변은 action 제안으로 쓰지 않고 새 line state를 조회하도록 하면 늦게 도착한 정답이 잘못된 행동으로 이어지는 일을 줄일 수 있습니다.

자동차 shop-floor Agent에는 시간 압박도 입력값입니다. 조립 라인에서는 다음 차체가 들어오기 전까지 남은 시간, Andon 상태, 승인자를 함께 봅니다. 아래 response class와 권한 구분은 기술 권고를 설명하는 제안 프레임워크입니다. 실제 적용 값은 사업장의 Control Plan, 안전 규정, 노사 절차, MES/QMS 권한, 승인 체계로 바꾸고 담당 부서가 검토합니다.

판단 질문

이벤트가 들어오면 세 가지를 확인합니다. 현재 takt 안에 사람이 확인해야 하는가, 다음 계획 정지까지 미뤄도 되는가, 생산 안정화 뒤 분석 과제로 넘길 수 있는가. 이 구분이 정해져야 Agent의 응답 시간과 권한을 시험할 수 있습니다.

Takt-aware의 의미

Takt time은 생산 수요에 맞춰 정한 목표 간격입니다. 아래의 60초 takt와 5분 대응 지연은 설명을 위해 단순화한 초기 예시값입니다. 대응 지연은 모든 이벤트가 아니라 지연 때문에 실제 라인이 멈춘 것으로 귀속된 비율만 손실에 반영합니다. 실제 영향은 buffer, 병렬 station, 고장 위치, 표준작업, 복구 방식에 따라 달라지므로 시뮬레이션 결과를 생산 계획이나 OEE 예측으로 사용하면 안 됩니다.

Takt-aware Agent는 모델이 혼자 빠르게 행동한다는 뜻이 아닙니다. action을 제안하기 전에 response time, risk, owner, approval path를 먼저 분류한다는 뜻입니다.

설명을 위해 단순화한 가상 계산이며 표시값은 초기 예시값입니다. Agent 지연 시간 중 라인 정지로 귀속된 비율만 차감하고, cycle time이 takt를 넘어서 생긴 손실과 분리해 표시합니다. buffer 회복, 휴식, starvation, 병렬 station은 제외하므로 실제 사업장 수치나 OEE 예측값으로 사용하면 안 됩니다.

잘못된 Agent 설계

잘못된 설계는 broad question에는 답하지만 station state, line cadence, authority를 모르는 Agent입니다. 이런 Agent는 "라인을 멈추고 영향 차량을 검사하세요"라고 말할 수 있습니다. 하지만 그 제안이 control plan에 이미 정의되어 있는지, line leader가 승인했는지, 의심 VIN 범위가 맞는지 모르면 위험합니다.

첫 production-safe 버전은 보수적이어야 합니다. 증거를 읽고, 요약하고, severity를 분류하고, owner를 찾고, action 초안을 만드는 역할부터 시작해야 합니다. 실제 write action은 gate 뒤에 둡니다.

현장 말투로 말하면

Agent가 line stop을 누구 승인으로, 어떤 증거 때문에, 어떻게 rollback할 수 있는지 설명하기 전에 line stop을 더 빨리 실행할 수 있다면 설계 순서가 거꾸로 된 겁니다.

Tool lane

Tool을 lane으로 나누면 Agent를 유용하게 만들면서도 무모해지지 않게 막을 수 있습니다.

승인 gate

자동차 Agent에는 막연한 human review가 아니라 이름 붙은 approval gate가 필요합니다. action에 따라 승인 owner가 달라집니다.

  • Line owner: takt, manpower, station blocking, Andon escalation.
  • Quality owner: containment, defect disposition, rework rule, customer risk.
  • Safety owner: robot zone, interlock, maintenance mode, human entry.
  • Maintenance or automation owner: robot program, fixture, tool calibration, PLC/robot recovery.
  • Process or manufacturing engineer: recurring pattern, control plan update, long-term corrective action.

Agent는 approval card를 만들어야 합니다. evidence, suspected scope, proposed action, affected VIN range, risk if delayed, rollback path, owner가 들어간 카드입니다. 자신감 있는 긴 문단보다 이런 카드가 현장에서는 훨씬 쓸모 있습니다.

Worked example: 1분 approval card

설명을 위한 가상 이벤트로 station T-220의 torque 결과 1건이 기준 밖이라고 놓습니다. Agent는 VIN과 원시 torque record를 읽고 response class를 Immediate로 분류하되, action은 "line stop"이 아니라 line leader 호출 + quality owner 확인 요청 초안으로 제한합니다. 카드에는 영향 VIN 1대, 다음 VIN 진입 전 확인할 항목, 지연 시 위험, 요청 취소 방법을 적습니다. 원시 기록이 없거나 station clock이 어긋나면 severity를 확정하지 않고 evidence missing으로 표시합니다. Safety interlock과 robot motion은 이 카드의 승인 범위 밖입니다.

시작 manifest

아래 manifest는 station delay와 quality signal이 함께 나타나는 상황을 설명하기 위한 가상 초안입니다. line·station 이름, tool 권한, hard block, 승인 필드는 실제 사업장의 안전 PLC와 작업 표준, 책임자 체계로 교체해야 합니다.

takt_aware_agent_manifest:
  scope:
    line: "final-assembly-1"
    stations: ["torque-220", "vision-310"]
    response_classes: ["immediate", "near_term", "planned_stop", "engineering"]
  read_tools:
    - mes_station_events
    - andon_history
    - torque_results
    - vision_results
    - work_instruction_search
    - control_plan_search
  draft_tools:
    - containment_note_draft
    - shift_handover_draft
    - maintenance_request_draft
  write_tools:
    - qms_issue_create
    - containment_assignment
  hard_blocks:
    - robot_motion_command
    - safety_bypass
    - line_restart
  approval_fields:
    - owner
    - evidence
    - affected_vin_range
    - risk_if_delayed
    - rollback_path

첫 번째 좋은 자동차 Agent는 자율성보다 훈련된 범위가 선명합니다. signal, evidence, owner, approved action 사이의 시간을 줄이되 라인의 권한 구조는 건드리지 않습니다. Genealogy가 불완전하면 VIN genealogy 글로 돌아가 영향 범위를 먼저 만들고, 잘못된 tool call을 멈추고 조사하는 절차가 필요하면 Agent 사고 대응 런북으로 이어가세요.

제안 프레임워크이며, 실제 사업장의 생산·품질·안전 규칙으로 교체해 검토해야 합니다.
