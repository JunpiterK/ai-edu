합성 장면: 겹치지 않는 두 압력 곡선

교대 인수인계 화면에 전날 run과 현재 run의 압력 곡선이 나란히 떠 있고, 현재 곡선만 오른쪽으로 밀려 있습니다. 처음에는 chamber drift처럼 보이지만 recipe step 경계를 켜자 안정화 구간이 늦게 시작했을 뿐 각 step 안의 모양은 거의 겹칩니다. 반대로 전체 run 시간은 같아도 plasma 구간만 떼면 차이가 드러날 수 있습니다. 여러 FDC 검토 상황을 합친 가상 장면이며 실제 장비 사건이 아닙니다.

파일럿 범위

첫 파일럿은 한 장비, 한 recipe family, 한두 개 fault mode로 제한하세요. 먼저 step alignment 전후의 곡선을 각각 저장하고, 판정이 뒤집힌 run을 engineer review 목록으로 만듭니다. 아래의 정상 run 20건과 이상 run 4건은 임계값 변화를 설명하기 위한 학습용 수치이며 실제 표본 규모의 권고가 아닙니다.

Fault Detection and Classification(FDC)은 run 중 신호를 기준 상태와 비교하고, 검토할 이상 유형을 분류하는 체계입니다. 기술적 권고는 원시 곡선보다 recipe step과 revision을 먼저 맞추고, 감지 결과를 SPC·metrology·정비 이력과 함께 해석하라는 것입니다. FDC 점수만으로 엔지니어의 hold·release 판단을 대신해서는 안 됩니다.

설비가 alarm 없이 run을 마쳐도 특정 step의 pressure, RF power, endpoint가 평소와 다를 수 있습니다. FDC는 이런 변화를 계측 결과가 나온 뒤에야 찾는 일을 줄이기 위한 조기 검토층입니다. 어느 편차를 관찰만 하고 어느 편차를 hold로 연결할지는 별도 운영 규칙으로 정합니다.

1분 모델

Trace data는 run 동안 시간순으로 쌓인 sensor 값입니다. Golden run은 정상으로 검토된 기준 집합이고, FDC는 현재 trace가 그 범위를 벗어났는지와 알려진 fault signature에 가까운지를 계산합니다.

FDC가 실제로 하는 일

공정 run은 하나의 평평한 time-series가 아닙니다. pump down, stabilize, gas flow, plasma strike, main etch, endpoint, purge, vent처럼 recipe step의 sequence입니다. step마다 기대되는 행동이 다릅니다. 어떤 step에서는 pressure spike가 정상이고, 다른 step에서는 이상일 수 있습니다.

처음 설계할 때는 수집, 정렬, 비교, 분류의 순서로 확인하면 빠진 조건을 찾기 쉽습니다.

  • Collect: 관련 sensor trace를 충분한 timestamp 품질로 수집합니다.
  • Align: trace를 recipe step, chamber, lot, wafer, product context에 맞춥니다.
  • Compare: 건강한 baseline이나 control band와의 거리를 봅니다.
  • Classify: 이상 pattern을 fault family와 engineering action에 연결합니다.

collection이 약하면 model은 noise를 봅니다. alignment가 약하면 틀린 순간끼리 비교합니다. baseline이 약하면 정상 변동도 다 수상해 보입니다. classification이 약하면 alarm은 뜨는데 힌트가 없습니다.

trace에 step alignment가 필요한 이유

두 etch run을 recipe start 이후 wall-clock time만으로 비교한다고 해봅시다. 한 run은 pressure 안정화가 조금 늦고, 다른 run은 plasma strike가 살짝 늦습니다. raw timestamp끼리 비교하면 process가 정상이어도 curve가 달라 보입니다. operation이 같은지 확인하지 않고 wafer 두 장을 비교하는 것과 비슷합니다.

Trace alignment 선택지.
Alignment 방식도움 되는 경우실패 모드
Run start time빠른 prototype, timing이 안정적인 단순 process.step drift가 가짜 차이를 만듭니다.
Recipe step첫 production 방식으로 가장 유용합니다.recipe revision 사이 step label 일관성이 필요합니다.
Event markerplasma strike, endpoint, valve transition, chamber state change 주변에서 유용합니다.event 누락이나 지연이 비교 위치를 밀 수 있습니다.

기술 판단: hold 근거의 최소 조건

recipe-step label이 없는 FDC chart는 어느 공정 구간에서 편차가 생겼는지 설명하기 어렵습니다. 탐색 화면으로는 볼 수 있지만 tool hold 근거로 올리려면 step 경계와 revision 표시가 최소 조건입니다.

Golden run과 fault signature

Golden run은 정상으로 확인된 reference set입니다. 포함 조건에는 product와 chamber 상태, recipe revision, metrology 허용 범위, alarm·정비 이력이 들어갑니다. 이 중 하나가 바뀌면 기존 baseline 적용을 잠시 멈추고 영향 범위를 다시 검토합니다.

Fault signature는 엔지니어가 이름 붙일 수 있는 반복적인 이상 pattern입니다. 예를 들면 이런 것들입니다.

  • plasma 전 pressure 안정화 시간이 평소보다 길다.
  • main etch step에서 RF reflected power가 오른다.
  • endpoint signal shape가 바뀌고 이후 CD drift가 보인다.
  • setpoint 변경 후 MFC response가 늦다.
  • chamber clean 이후 temperature recovery가 baseline보다 느리다.

알고리즘을 정하기 전에 숙련 엔지니어에게 어떤 trace 모양에서 추가 확인을 시작하는지 묻고, 그때 함께 보는 recipe step과 계측 결과를 기록합니다. 이 관찰 기준을 데이터로 재현할 수 있어야 fault signature 후보가 됩니다.

False alarm을 가볍게 넘길 수 없는 이유

False alarm은 모델 지표이면서 교대조의 검토 부하입니다. 정상 run을 반복해서 이상으로 표시하면 엔지니어가 실제 알람까지 늦게 보게 됩니다. 오탐 건수만 세지 말고 알람 한 건을 확인하는 데 걸린 시간과 severity별 처리 결과도 남겨야 합니다.

자주 나오는 false alarm 원인.
원인겉으로 보이는 모습개선
Mixed products정상 product 차이가 fault처럼 보입니다.product 또는 process family별 baseline을 둡니다.
Recipe revision drift정상 release change가 old limit을 건드립니다.limit을 recipe revision과 함께 versioning합니다.
Maintenance statepost-PM 행동이 abnormal처럼 보이지만 사실 예상 가능한 변화입니다.PM/chamber state를 context에 넣습니다.
Sensor noise시끄러운 tag 하나가 alert를 지배합니다.tag를 filter, validate, 또는 제거합니다.
Bad severity design모든 deviation이 urgent가 됩니다.watch, review, hold, stop level을 분리합니다.

실무 FDC는 "normal"과 "bad"만 있으면 안 됩니다. FAB에서는 action level이 중요합니다. watch signal은 note면 충분할 수 있습니다. review signal은 engineer 확인이 필요할 수 있습니다. hold signal은 release를 멈출 수 있습니다. stop signal은 드물어야 하고, 매우 조심스럽게 정당화되어야 합니다.

설명을 위해 만든 학습용 가상 데이터입니다. 정상으로 가정한 20건과 이상으로 가정한 4건을 golden run 대비 표준편차로 표현했으며, 건수·분포·임계값 모두 특정 회사나 설비의 측정값이 아닙니다.

실습 기록지

임계값을 낮은 값, 중간 값, 높은 값으로 한 번씩 옮기고 각 위치의 오탐 건수 / 이상 검출 / 교대당 예상 검토 시간을 적어보세요. 예를 들어 알람 한 건 검토에 3분이 든다고 가정하면 오탐 5건은 15분의 검토 부하입니다. 이 3분은 계산법을 보여주는 가상 가정입니다. 검출이 늘어도 교대조가 처리할 수 없거나 높은 임계값에서 놓치는 fault의 위해도가 크다면, 단일 threshold 출시는 보류하고 signal·product·action level을 나눠 다시 평가합니다.

첫 파일럿 체크리스트

첫 파일럿은 담당 엔지니어가 검토 기준을 설명할 수 있는 좁은 process와 fault family를 고릅니다. 여러 tool과 signal을 한꺼번에 넣으면 오탐 원인을 분리하기 어렵고 baseline 승인도 늦어집니다.

파일럿 모양

하나의 module과 tool type, 하나의 recipe family, 하나의 fault family, 하나의 action path를 문서로 고정합니다. 신호 검토 결과와 override 이유가 쌓인 뒤 다음 chamber로 넓힙니다.

  • Scope: tool type, chamber, recipe family, product family, target fault를 정합니다.
  • Data: trace tag, alarm code, recipe revision, metrology target, maintenance context를 나열합니다.
  • Baseline: inclusion/exclusion rule이 명확한 golden run을 고릅니다.
  • Labels: 알려진 fault example과 engineer-reviewed normal example을 모읍니다.
  • Action: deployment 전에 watch, review, hold, stop response를 정합니다.
  • Review: false alarm, miss, engineer override, saved time을 봅니다.

AI와 연결되는 지점

FDC는 rule, statistics, classical ML, deep learning, hybrid system을 모두 쓸 수 있습니다. AI 자체가 목적은 아닙니다. 목적은 controlled detection입니다. 이해 가능한 baseline에서 시작하고, 복잡도는 값어치를 할 때만 올리면 됩니다.

여기서 LLM/RAG가 companion이 될 수 있습니다. RAG는 감지된 pattern에 맞는 troubleshooting guide를 찾아줄 수 있습니다. Agent는 최근 alarm과 PM history를 read-only로 열어볼 수 있습니다. Knowledge graph는 chamber, recipe, lot, failure history를 연결할 수 있습니다. 하지만 이 모든 것은 FDC 기본기를 대체하지 않습니다. trace 품질, step alignment, golden run, fault signature, false-alarm discipline이 먼저입니다.

출시 판단에서는 감지율만 묻지 않습니다. 담당 엔지니어에게 recipe step과 근거 trace를 함께 보여주는지, 오탐 검토량을 감당할 수 있는지, hold와 stop 권한이 분리됐는지를 확인합니다. Sensor 교체, recipe revision, post-PM 상태처럼 baseline 의미가 바뀌는 이벤트는 자동 판정 중지와 재승인의 trigger로 둡니다. 원천 데이터 연결이 막힌 독자는 먼저 FAB 설비 데이터 101로 돌아가고, 감지 결과를 tool action으로 연결하려는 독자는 Agent 사고 대응 런북에서 중지·보존 조건을 이어서 확인하세요.

FDC 설계 원칙과 명시된 학습용 가상 데이터를 바탕으로 작성하고, 기술적 정확성을 검토했습니다.
