합성 사례: 프롬프트보다 먼저 그릴 선

설명을 위해 단순화한 가상 사례입니다. 검토 회의 화면에 “TOOL_A의 특정 recipe에서 반복된 alarm을 요약해 달라”는 프롬프트 초안이 떠 있습니다. 옆의 데이터 흐름도에는 cloud endpoint 화살표만 있고, prompt log의 보존 위치와 embedding telemetry 경로는 비어 있습니다.

처음에는 설비명과 수치만 가리면 충분하다고 보기 쉽습니다. 하지만 alarm 순서, recipe step, engineer comment의 조합 자체가 공정 맥락을 드러내고, vector DB나 관측 도구가 별도 outbound 연결을 만들 수도 있습니다. 프롬프트 한 칸이 아니라 검색·로그·telemetry를 포함한 전체 흐름을 봐야 판단이 바뀝니다.

이 합성 장면에서 도출하는 기술 권고는 데이터 등급, 허용 경로, 승인 주체를 먼저 표시하는 것입니다. 폐쇄망은 가능한 설계안 중 하나일 뿐이며, 외부 반출 금지와 승인된 반입의 구체 경계는 회사마다 다릅니다. 조직의 정책과 계약을 확인하기 전에는 실제 데이터를 연결하지 않습니다.

검토 전에 준비할 것

프롬프트, 검색 문서, 로그, embedding store, telemetry의 흐름을 한 장에 표시하세요. 여기에 데이터 등급, 승인 주체, 보존 기간, 계약상 처리 조건을 붙이면 법무·보안 담당자가 실제 쟁점을 확인할 수 있습니다.

제조 현장에서 LLM을 쓰려면 먼저 데이터의 이동 방향부터 정해야 합니다. 호스팅 API에 공정 질문을 던지는 작은 도우미는 데모로는 매끄럽습니다. 하지만 질문 안에 recipe 조건, alarm history, defect trend, engineer comment가 들어가는 순간 이야기가 달라집니다. 답을 좋게 만드는 바로 그 맥락이 밖으로 나가면 안 되는 정보이기 때문입니다.

증착 조건, 수율·불량 추이, 장비별 이상 징후는 조직에 따라 영업비밀이나 계약상 보호 정보로 분류될 수 있습니다. 외부 서비스 사용 가능 여부는 데이터 분류, 공급자 계약, 저장·학습 정책, 지역, 접근 통제 조건을 함께 검토해야 합니다. 폐쇄망 배치는 가능한 대응안 중 하나이며, 자동으로 법무·보안 승인을 보장하지 않습니다.

cloud LLM에 붙여넣을 때 실제로 나가는 것

"그냥 ChatGPT에 물어보자"는 말은 검색창을 쓰는 것처럼 가볍게 들립니다. 하지만 실제 현장 질문은 가볍지 않습니다. tool ID, recipe step, alarm sequence, defect count, 고객 조건, engineer 판단이 들어갑니다. 악의가 없어도 그 프롬프트는 통제 밖 인프라로 나가는 아웃바운드 전송입니다. 보안 검토에서 반복해서 확인해야 하는 노출은 다음과 같습니다.

  • 영업비밀이 프롬프트 안에 들어갑니다. 유용한 답을 얻으려면 실제 recipe, tool, defect pattern, 수치 조건을 설명해야 합니다. 답을 좋게 만드는 맥락이 바로 보내면 안 되는 맥락입니다.
  • 수율과 불량 데이터는 경쟁 정보입니다. defect rate 표나 excursion timeline은 공정이 어디에서 약한지 보여줍니다. 일부러 유출하지 않아도 "빠른 질문" 하나가 그대로 반출이 될 수 있습니다.
  • 계약상 제3자 제공 또는 처리에 해당할 수 있습니다. 고객 조건, 장비 로그, 공정 recipe를 외부 서비스에 입력하는 행위가 NDA나 위탁 처리 조항에 어떤 영향을 주는지는 계약 문구와 관할, 조직 정책에 따라 달라집니다. 법무·보안 담당자의 확인 없이 일반화하면 안 됩니다.
  • 클라우드 처리 위치와 접근 범위를 확인해야 합니다. 데이터가 머무는 region, 로그 보존, 공급자 운영자의 접근 가능성, 학습 사용 여부는 서비스 계약과 설정에 따라 다릅니다. 공급자 문서와 조직의 승인 조건을 대조해야 합니다.

의도와 관계없이 입력한 정보는 외부 서비스로 전송됩니다. 어떤 정보가 금지 대상인지, 승인된 enterprise 경로가 있는지, 어떤 기록을 남겨야 하는지는 조직 정책과 계약을 확인해야 합니다. 기술적 차단은 그 결정을 집행하는 수단이며 정책 자체를 대신하지 않습니다.

런타임에서 확인할 경로

Inference engine에 외부 호출이 없어도 vector DB, observability client, package telemetry가 DNS나 HTTP 연결을 시도할 수 있습니다. 폐쇄망 반입 전후에 tcpdump와 방화벽 로그로 각 프로세스의 outbound traffic을 확인하고, embedding·vector DB·analytics library의 telemetry 설정을 기록합니다. 허용할 통신과 차단할 통신은 조직 보안 정책에 맞춰 판정합니다.

질문을 보내기 전에 데이터 등급, 허용된 서비스, 계약 조건을 확인합니다.

대안: model은 들어오고 data는 남는다

외부 전송이 허용되지 않는 조직에서는 검증한 오픈 모델 가중치를 승인 절차로 반입해 내부 GPU에서 서빙하는 방안을 검토할 수 있습니다. 이 구성은 런타임의 외부 데이터 경로를 줄이지만, 모델 라이선스, 공급망 검사, 내부 접근 권한, 로그 보존 검토는 여전히 남습니다. 아래 세 부분은 검토용 기준 구조입니다.

  • 온프레미스 서빙. Llama, Qwen, gpt-oss 같은 오픈 모델을 vLLM 또는 Ollama 뒤에서 내부 GPU로 실행합니다. 가중치는 검증 후 한 번 반입하고, 런타임에 외부 호출이 없어야 합니다.
  • 폐쇄망 RAG. SOP, specification, 장비 로그, 품질 문서를 내부에서 embedding하고 색인화합니다. 답변은 내부 문서에 근거하지만 문서 자체는 밖으로 나가지 않습니다.
  • 내부 tool 연결. MES, historian, document store 같은 사내 시스템은 감사 가능한 tool을 통해 읽습니다. 권한, 로그, 승인 gate는 같은 경계 안에서 관리합니다.

오프라인 의존성 확인

인터넷 연결 환경에서 만든 코드는 모델 카드, tokenizer, Python wheel을 런타임에 추가로 내려받을 수 있습니다. 폐쇄망에서 시작이 멈추거나 긴 timeout이 발생하면 누락 의존성과 외부 접속 시도를 먼저 확인합니다. 가중치와 의존성을 승인 manifest에 포함해 사내 미러 또는 오프라인 cache로 반입하고, HF_HUB_OFFLINE=1 같은 오프라인 모드를 설정한 뒤 외부 경로가 차단된 staging 환경에서 재시험합니다. 반입 매체와 해시 검증 방식은 조직 절차를 따릅니다.

구성도에는 승인된 모델 반입 경로와 차단된 데이터 반출 경로를 따로 표시합니다. 질문, 검색 context, 답변, 로그의 저장 위치와 접근 주체도 같은 그림에 넣습니다. 이 자료를 바탕으로 보안 담당자가 경계 설정과 예외 통신을 검토할 수 있습니다.

설명을 위한 가상 시험입니다. 로컬 tokenizer cache를 비우고 기본 외부 경로를 차단한 상태에서 서비스를 시작합니다. 런타임이 model hub 접속을 시도하면 답변 화면이 정상이어도 파일럿을 보류합니다. 누락 tokenizer를 승인 manifest로 반입하고, 모델·GPU·동시 요청 조건을 기록한 부하 시험에서 packet capture를 반복합니다. 외부 연결 허용 목록과 합격 기준은 조직 보안 정책으로 확정해야 합니다.

air-gapped는 약하다는 뜻인가? 이제는 아니다

자주 듣는 질문은 "온프레미스로 가면 모델 성능이 떨어지는 것 아니냐"입니다. 답은 모델 이름만으로 정할 수 없습니다. 문서 기반 Q&A, 절차 검색, 로그 요약, 초안 작성처럼 범위가 좁은 업무라면 오픈 웨이트 모델도 평가 후보가 됩니다. Llama, Qwen, gpt-oss 같은 계열은 가중치를 내부에 반입해 운영할 수 있지만, 라이선스·언어·정확도·지연·GPU 조건을 조직의 평가셋으로 확인한 뒤 선택해야 합니다.

RAG를 붙였다는 사실도 우위를 보장하지 않습니다. 승인 문서가 최신이고 retrieval과 citation이 정확할 때에는 작은 로컬 모델이 업무 질문에 필요한 근거를 더 잘 제시할 수 있습니다. 반대로 문서 개정, ACL, chunking, 검색 평가가 틀리면 큰 모델보다 더 위험한 확신을 만들 수 있습니다. 같은 질문·문서·판정 기준으로 후보 구성을 비교하고, 통과한 범위만 운영 업무로 선언합니다.

VRAM 산정

모델 가중치가 GPU에 올라가도 긴 컨텍스트와 동시 요청이 붙으면 KV cache 때문에 OOM이 날 수 있습니다. 모델·GPU·양자화·입출력 길이·동시성을 기록한 부하 시험으로 피크 메모리를 측정하고, 부족하면 retrieval 청크 수, KV cache dtype, max model length, batch 설정을 조정합니다.

양자화 품질 확인

낮은 bit 양자화는 VRAM을 줄이지만 숫자 처리와 긴 문맥의 일관성에 영향을 줄 수 있습니다. Q4_K_M은 초기 후보 예시이며 보장값이 아닙니다. 승인된 문서와 질문 세트로 원본 또는 상위 정밀도 모델과 답변을 비교해 선택합니다.

엔지니어를 위한 1차 checklist

시작하기 위해 모든 것을 완벽히 해결할 필요는 없습니다. 하지만 파일럿 전에 아래 항목은 반드시 확인하세요. 대부분 실제 구축에서 한 번씩 발목을 잡는 지점입니다.

  • 모델 반입 절차와 라이선스. 가중치는 통제된 단방향 경로로 반입하고 해시를 검증합니다. 런타임에는 인터넷에서 아무것도 가져오지 않아야 합니다. 또한 "오픈 웨이트"가 곧 상업적 사용 허용을 뜻하지 않으므로 구축 전에 라이선스를 읽어야 합니다.
  • 실제 동시성 기준의 GPU sizing. headline parameter 수가 아니라 context length, 동시 요청, KV cache까지 계산하세요. 1인 데모와 팀 공유 서비스의 VRAM 요구량은 다릅니다.
  • 출구 없는 경계. 서빙 host, vector DB, embedding runtime, logging agent가 외부로 나갈 경로를 갖지 않게 하세요. packet capture와 firewall log로 확인해야 합니다.
  • 접근 제어와 감사. 누가 어떤 문서를 검색할 수 있는지, 어떤 tool을 호출할 수 있는지, 모든 요청이 어떤 trace ID로 남는지 정의해야 합니다. 닫힌 루프는 accountability가 있을 때 의미가 있습니다.
  • 데이터 분류 선행. 무엇을 RAG index에 넣을 수 있고, 무엇은 넣으면 안 되는지 먼저 정하세요. 애매한 질문이 들어온 뒤에 분류하면 이미 늦습니다.

실행 환경 조합

vLLM, CUDA, NVIDIA driver, container image 버전이 어긋나면 폐쇄망 안에서 원인을 확인하는 시간이 길어집니다. 버전 조합을 manifest에 고정하고 같은 구성으로 staging 시험을 마칩니다. 다중 GPU 구성은 tensor parallel 설정과 실제 노출된 GPU 수도 함께 확인합니다.

다국어 검색 품질

한국어와 영어 장비 용어가 섞인 문서를 영어 중심 embedding으로 색인하면 기대 문서가 top-k에서 빠질 수 있습니다. bge-m3 같은 다국어 모델을 후보로 두고, 승인된 대표 질문에서 검색 문서 ID와 개정을 직접 비교합니다. 모델 이름만으로 적합성을 판정하지 않습니다.

이 series가 이어지는 방향
