Tabletop 장면: owner가 다른 ticket

09:10, maintenance queue에 담당 module이 다른 ticket 한 건이 보입니다. 본문에는 현재 SOP처럼 보이는 절차가 인용됐지만 revision 표시는 없고 approval ID도 비어 있습니다. 답변만 고치면 화면은 깨끗해지지만 잘못된 source, 넓은 permission, 승인 없는 tool call 중 무엇이 원인인지는 사라집니다. 사고 대응 훈련을 위한 합성 장면이며 실제 조직의 사건이나 시각이 아닙니다.

바로 적용할 기준

사고 대응 표의 첫 열에 "멈출 것, 보존할 것, 알릴 사람"을 적으세요. Tool route, source revision, approval state, known safe state가 한 화면에 없으면 원인 추정은 보류하고 관련 route부터 read-only로 낮춥니다.

AI agent가 tool을 호출하는 순간, 잘못된 답변은 시스템 상태와 작업 순서에 흔적을 남깁니다. 잘못된 draft, 다른 owner에게 전달된 ticket, 과도한 permission, 폐기된 절차를 근거로 한 추천도 대응 대상입니다. 기술적 권고는 prompt 수정 전에 확산을 막고 재현 artifact를 보존하는 것입니다. 다만 이 런북은 조직의 법무·보안·안전 incident 절차를 대신하지 않습니다.

초심자용 그림

처음에는 prompt를 고치지 않습니다. 관련 tool 경로를 막고, prompt·source·권한·승인 상태를 보존한 뒤 담당 owner에게 알립니다. 원본을 남기지 않은 수정은 원인 비교와 재현을 어렵게 만듭니다.

무엇을 incident로 볼 것인가

피해가 확인된 경우만 incident로 잡으면 승인 누락이나 오래된 source 추천 같은 near miss가 빠집니다. Unauthorized retrieval, 잘못된 tool 실행, 누락된 승인, 근거 없는 citation, 반복된 refusal 실패, 개인정보 노출, 그리고 잘못된 출력에 따라 사용자가 조치를 시작한 경우를 보고 대상으로 정합니다.

조직별 영향도·법적 의무·기존 incident 체계에 맞춰 조정하는 Agent SEV 분류 템플릿.
LevelDefinitionResponse
SEV-3잘못된 답변이나 draft가 있었지만 action은 없음.Log, tag, evaluate, prompt/source 수정.
SEV-2잘못된 ticket, workflow, recommendation이 human queue에 도달.해당 tool path disable, owner 통지, trace 보존.
SEV-1State-changing action, data exposure, production-impacting recommendation.즉시 격리, escalation, formal incident review 시작.

첫 30분

첫 대응에서는 원인 결론보다 영향 확산을 먼저 막습니다. 관련 tool route를 멈추고 trace를 보존하며 영향받은 사용자를 확인합니다. 경계가 불명확하면 agent 전체를 read-only mode로 낮춥니다. 원본 artifact를 복사하기 전에 prompt나 index를 수정하면 변경 전후 비교가 불가능해집니다.

FAB 엔지니어 시점

Agent에서는 prompt version, policy version, retrieved source, user role, tool route를 한 시점으로 맞춰 봅니다. 여기에 원본 업무 시스템의 변경 이력과 known safe state를 붙여야 합니다. 이 항목이 빠지면 회고에서 서로 다른 실행을 놓고 원인을 추정하게 됩니다.

  1. Classify: severity와 affected domain을 지정합니다.
  2. Contain: 관련 tool path, role, index build를 비활성화합니다.
  3. Preserve: prompt, source, model version, user role, tool request, approval state, output을 복사합니다.
  4. Notify: process owner와 platform owner를 기본으로 호출합니다. 데이터·권한 영향은 security, 물리 안전 영향은 safety/EHS, 품질 판정 영향은 quality owner, 생산·라인 영향은 operations 또는 line owner에게 즉시 escalation합니다.
  5. Authorize recovery: 재가동은 영향 영역의 책임자가 승인합니다. 안전 기능은 safety/EHS와 설비 안전 절차, 제품 disposition은 quality, 라인 복귀는 operations/line owner의 기존 승인 체계를 따릅니다.
  6. Communicate: 영향 받은 사용자에게 무엇을 무시하고, 다시 하고, 수동 검증해야 하는지 알립니다.
작은 실습: 15분 tabletop

0분: 위 가상 ticket을 발견했다고 읽어줍니다. 5분: 참가자는 멈출 route, 보존할 artifact, 통지 owner를 각각 하나씩 고릅니다. 10분: "trace 저장소가 비어 있다"는 조건을 추가하고 영향 범위를 어디까지 잠정 표시할지 정합니다. 15분: 복구 허용 조건과 회귀 test를 한 줄로 남깁니다. 중지 권한자나 trace 위치를 찾지 못하면 자동화 범위를 넓히지 않고 그 공백을 런북 action item으로 등록합니다.

Trace 보존

Trace 항목은 사고가 난 뒤 추가할 수 없습니다. User role snapshot, policy·prompt·model version, retrieved source ID, tool schema와 argument, approval ID, 실행 결과, 최종 응답을 한 trace ID로 묶습니다. 민감한 argument는 그대로 적재하지 말고 조직의 보존·마스킹 정책을 적용해야 하며, 로그만으로 외부 시스템의 실제 상태까지 증명된다고 보아서는 안 됩니다.

아래 JSON은 특정 회사의 실제 incident가 아니라, 설명을 위해 단순화한 가상 사례의 예시 trace입니다. E-17과 SOP revision, incident ID도 모두 가상 식별자입니다.

Corrective action

"사용자가 검증한다"는 문구만으로 incident를 닫으면 같은 경로가 남습니다. 폐기 문서 차단, tool 승인 조건 추가, refusal test와 evaluation set 보강, permission sync 수정, read-only tool과 state-changing tool 분리를 검토합니다. 최종 리뷰에는 이번 징후를 사용자 도달 전에 검출할 회귀 test를 하나 이상 남기고, 배포 전 그 test가 실제로 실패했다가 수정 후 통과하는지 확인합니다.

단순화한 가상 사례

설명을 위해 만든 사례입니다. Maintenance triage agent가 가상의 E-17 설비에 대해 폐기된 rev-07 SOP를 읽고, ticket을 다른 module owner queue로 보냈다고 가정합니다. 물리적 사고는 없지만 잘못된 작업 지시가 queue에 남았습니다. 해당 tool path를 중지하고 trace를 보존한 뒤 rev-07을 superseded로 표시합니다. 이어 E-17 ownership 회귀 질문을 평가셋에 넣고, SOP 기반 ticket 생성에는 human approval을 요구합니다. 이 조치만으로 이미 ticket을 읽은 사용자의 후속 행동까지 확인되지는 않으므로 별도 영향 조사가 필요합니다.

런북은 배포 승인 때 확인할 운영 조건이 됩니다. Tool별 중지 방법, trace 보존 위치, 통지 owner, rollback 절차가 실제로 시험된 범위까지만 자동화 권한을 엽니다. Incident가 나기 전 역할별 허용 action을 정리하려면 다음으로 AI 정책·통제 매트릭스를 읽어 control owner와 증거 항목을 연결하세요.

가상 운영 시나리오와 기술적 검토를 바탕으로 작성했으며, 실제 조직의 절차에 맞춰 조정해야 합니다.
