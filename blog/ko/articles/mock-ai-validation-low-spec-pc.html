GPU 없이 AI 에이전트 운영 경계를 검증하는 Mock 랩

AE AI Edu Archive 응용 AI, 로보틱스, 제조 시스템
이 글의 순서
  1. 검증 가능한 범위
  2. 폴더와 실행 순서
  3. 8개 통제 테스트
  4. 결과 읽는 법
  5. 실제 모델로 교체하기

PC 사양이 낮다고 검증까지 멈출 필요는 없습니다. GPU가 없으면 로컬 LLM의 답변 품질과 속도는 확인하기 어렵습니다. 그렇다고 검색 권한, 도구 입력, 승인 범위, 감사 로그, 반복 종료 조건까지 전부 미룰 이유는 없습니다. 이 글에서는 모델을 정해진 응답만 내놓는 Mock 객체로 바꾸고, 운영 사고를 막는 경계부터 저사양 PC에서 검사합니다.

가상 사례: 장비가 안 되니 시험도 안 된다고 생각한 경우

팀 PC에서 목표 모델이 뜨지 않자 에이전트 검증 전체를 보류했다고 가정해 보겠습니다. 여기에는 서로 다른 두 문제가 묶여 있습니다. 그 PC로 생산 수준의 답변 품질은 측정하기 어렵지만, 상태 전이와 권한, 승인 재사용 차단, 실패 시 중단 동작은 충분히 확인할 수 있습니다. 이 사례는 설명을 위한 가상 상황이고, 아래의 테스트 결과는 이 저장소에서 실제 실행한 값입니다.

이 랩으로 확인할 수 있는 것과 없는 것

FAB에 비유하면 제품 웨이퍼를 넣기 전에 더미 웨이퍼와 인터록 체크로 밸브 순서, 퍼미시브, 비상 정지를 확인하는 단계에 가깝습니다. 인터록 시험이 통과했다고 막질이 합격이라는 뜻은 아니지요. 이 Mock 랩도 애플리케이션 통제를 검증할 뿐, 언어 모델의 정확도를 증명하지는 않습니다.

통제 검증과 모델 평가는 분리해서 기록합니다.
이 PC에서 검증 가능실제 모델 또는 목표 장비가 필요
최신 개정본·사용자 역할 필터의미 검색 재현율과 임베딩 품질
허용한 도구 이름과 인자의 정확한 계약모델이 알맞은 도구를 고르는 비율
승인 범위·만료·재사용 차단환각, 지시 이행, 거절 품질
감사 로그 장애와 반복 한도 처리지연 시간, 처리량, VRAM, 동시 사용자

따라서 Mock 테스트 통과의 의미는 정확히 한정해야 합니다. “주어진 입력에서 애플리케이션이 사양대로 멈췄다”는 증거이지, “이 에이전트가 현장 질문에 정확하고 안전하게 답한다”는 증거는 아닙니다.

폴더 구성과 첫 실행

구현은 labs/mock-ai-validation/에 있습니다. Python 3.10 이상이면 되고 표준 라이브러리만 사용합니다. 가상환경, 네트워크, 모델 다운로드, pip install이 필요하지 않습니다.

데모부터 보지 말고 거절 테스트부터 돌리는 편이 좋습니다. 정상 동작 한 번보다, 틀린 승인과 감사 로그 장애에서 정말 멈추는지가 운영에는 더 중요합니다.

처음 실행할 때의 합격 기준

통과: Python 3.10 이상이고 8개 테스트가 모두 ok이며, 데모가 status: escalated로 끝나고 artifacts/latest_run.json이 생성됩니다. 중단: 거절 테스트가 실패했는데 “Mock이니까 괜찮다”고 넘기지 마세요. 통제 코드나 테스트 입력을 먼저 고쳐야 합니다.

브라우저에서도 랩 설명서, 데모 실행 파일, 전체 테스트 코드를 확인할 수 있습니다.

8개 테스트를 운영 계약처럼 읽기

tests/test_controls.py에 실제 구현한 테스트입니다.
테스트눈으로 확인할 합격 조건막으려는 실패
최신 문서 + 권한 검색엔지니어에게 개정 8만 반환폐기 문서나 권한 밖 문서가 문맥에 섞임
근거가 없으면 보류권한 없는 문서만 맞으면 빈 목록 반환근거 공백을 편한 문장으로 채움
정확한 도구 계약선언하지 않은 sql 인자를 거부제한된 조회 도구가 임의 질의 창구로 변함
스레드 바인딩스레드 17의 승인을 99에서 사용하면 실패한 작업의 승인을 다른 작업에 전용
감사 로그 Fail-closed쓰기 핸들러 호출 횟수가 0기록을 남길 수 없는데 상태를 변경
일회용 승인두 번째 사용은 거부되고 실행 횟수는 1중복 티켓·중복 변경
반복 한도답변 없이 escalated 반환마지막 도구 출력을 최종 답처럼 표시
승인된 정상 경로실행 1회와 감사 영수증 1개 보존정상 경로만 통제를 우회

승인은 버튼 한 번이 아니라 대기 중인 작업 전체에 묶습니다

토큰은 단순히 “티켓 생성 허용”만 뜻하지 않습니다. 워크플로 스레드, 인터럽트, 요청자, 대상 자원, 정확한 작업, payload 해시, 검토자, 만료 시각을 함께 서명합니다. 그중 하나라도 달라지면 검증이 실패하고, 성공한 토큰도 한 번 사용하면 폐기합니다.

감사 시스템이 영수증을 만들지 못하면 상태 변경 핸들러를 호출하기 전에 멈춥니다. 같은 승인 토큰을 다시 보내도 실행 횟수는 한 번을 넘지 않습니다.

데모가 일부러 에스컬레이션으로 끝나는 이유

데모는 위험하지 않은 read_alarm 요청을 세 번 반복합니다. 한도에 닿으면 마지막 조회 결과를 해결책처럼 포장하지 않고, 아래처럼 중단 이유를 그대로 남깁니다.

화려한 답변은 아니지만 디버깅에는 이쪽이 훨씬 낫습니다. 반복된 관찰값, 실행 단계 수, 중단 사유가 남으니 어느 경계에서 막혔는지 바로 찾을 수 있습니다.

나중에 실제 모델로 바꾸는 순서

  1. 검색 fixture, 도구 레지스트리, 승인 서비스, 감사 로그 동작, 테스트 assertion은 그대로 둡니다.
  2. ScriptedModel.next()만 Ollama, vLLM 또는 회사가 승인한 원격 엔드포인트 어댑터로 교체합니다.
  3. 먼저 결정론적 통제 테스트를 다시 통과시킵니다. 그다음 별도 평가 세트로 도구 선택, 근거 답변, 거절, 지연 시간, 자원 사용량을 측정합니다.
  4. 모델 ID, 파일 해시, 런타임, 프롬프트 버전, 하드웨어, 컨텍스트 길이, 동시 실행 수를 결과와 함께 남깁니다.

실모델 시험으로 진행: 기존 통제가 모두 통과하고 목표 환경의 담당자와 롤백 경로가 정해졌을 때입니다. 중단: 모델 연결 때문에 스키마를 느슨하게 하거나 승인·감사 로그를 건너뛰거나 반복 한도 사건을 숨겨야 할 때입니다. 한계: 실모델 시험도 그 모델과 설정에 대한 증거일 뿐, 다음 버전까지 자동 보증하지는 않습니다.

하드웨어가 준비되면 소비자 GPU 온프렘 LLM 실습으로 이어가고, 상태를 바꾸는 도구를 열기 전에는 HITL 통제 설계를 먼저 확인하세요.
