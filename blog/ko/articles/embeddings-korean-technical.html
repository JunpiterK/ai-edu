합성 비교 장면

설명을 위해 단순화한 가상 평가입니다. 평가 화면 왼쪽에는 “TOOL_A RF reflected power 알람 후 재가동 조건”이 있고, 오른쪽 top-5에는 장비명이 같은 점검표 네 건과 다른 장비의 재가동 절 한 건이 떠 있다고 합시다. 정확한 식별자는 잡았지만 한국어 질문의 의도인 “재가동 조건”은 놓친 결과입니다. 이때 모델 이름만 바꾸지 말고 질문 원문, 기대 절, 검색 청크, 적용된 metadata filter를 한 행에 놓아 식별자 매칭과 의미 매칭을 따로 판정합니다.

기술 권고

후보 모델을 받으면 승인된 질문 30개와 청크 스냅샷부터 고정하세요. 각 질문에 기대 문서·절을 붙이고 top-k 결과를 저장하면 모델 차이와 문서 분할 문제를 구분할 수 있습니다. 출처·개정·권한이 빠진 청크는 점수가 높아도 운영 후보에서 제외하며, 한 모델에만 유리하도록 질의 문구를 다듬지 않습니다.

이 글의 권고는 특정 모델의 우승을 선언하는 것이 아니라 폐쇄망 안에서 재현 가능한 비교를 만드는 것입니다. 한국어, 영어, 부품 번호, 챔버 ID, 내부 약어가 한 문장에 섞였을 때 기대 절이 top-k 안에 드는지 보고, 처리 시간·가중치 반입 조건·라이선스도 함께 기록합니다. GPU가 필수인 것은 아니지만 문서량과 갱신 주기에 따른 CPU 처리 시간과 배치 크기는 직접 측정해야 합니다.

임베딩 모델이 RAG 품질을 결정하는 이유

임베딩 모델은 텍스트를 벡터로 바꾸고, 의미가 비슷한 텍스트가 가까운 위치에 놓이도록 만듭니다. 검색은 결국 이 벡터 공간에서 가까운 이웃을 찾는 일입니다. 모델이 한국어 질문과 그 답을 담은 영어 문서를 멀리 떨어뜨려 놓으면, 이후 LLM 프롬프트를 아무리 잘 써도 누락된 청크를 복구할 수 없습니다. 온프레미스 RAG가 답답하게 느껴질 때 가장 먼저 봐야 할 곳이 임베딩입니다.

LLM을 바꿔도 검색 단계에서 빠진 문서는 돌아오지 않습니다. 임베딩 평가는 답변 문체보다 먼저 끝내야 합니다.

두 후보: bge-m3와 multilingual-e5

완전 오프라인으로 실행해야 하는 한국어-영어 혼합 문서에서는 두 오픈 웨이트 계열이 실용적인 출발점입니다. 둘 다 현장 PoC에서 다루기 쉬운 크기이고, 단일 GPU 또는 CPU 보조 환경에서도 임베딩 배치를 운영할 수 있습니다. 다만 라이선스와 재배포 조건은 회사 반입 절차에서 별도로 확인해야 합니다.

  • bge-m3(BAAI): 약 560M 파라미터, 1024차원 dense vector, 최대 8192 토큰 컨텍스트, 100개 이상 언어를 지원합니다. query instruction prefix가 필요 없어 질문과 문서를 같은 방식으로 임베딩합니다.
  • multilingual-e5-large(intfloat): 1024차원이며 다국어 성능이 강하지만 prefix가 필요합니다. 모든 질문은 query:로, 저장 문서는 passage:로 시작해야 합니다. prefix를 빼먹으면 유사도 품질이 조용히 떨어집니다.

공개 벤치마크는 후보를 줄이는 자료로만 씁니다. 첫 비교에서는 bge-m3와 multilingual-e5를 같은 청크, 같은 질문, 같은 top-k 조건으로 실행하고 한국어·영어 혼합 질의를 따로 집계하는 편이 낫습니다. 긴 컨텍스트 지원이 곧 긴 SOP 전체를 한 청크로 넣어도 된다는 뜻은 아닙니다. 최종 선택은 내부 질문 세트의 적중률과 지연 시간, 반입·운영 조건을 함께 보고 정합니다.

한국어-영어 혼합 문서와 기술 용어

일반 다국어 모델은 일상적인 이중언어 텍스트를 잘 처리합니다. 팹에서 어려운 부분은 모델이 거의 보지 못했을 용어입니다. PECVD-03B 같은 챔버명, 결함 코드, 내부 약어는 subword tokenizer에서 의미 없는 조각으로 쪼개질 수 있습니다. 엔지니어가 보면 분명 관련 있는 문서도 벡터 공간에서는 멀어질 수 있습니다. 실용적인 완화책은 세 가지입니다.

  • 식별자를 청크 텍스트에 남깁니다. 전처리 중 부품 번호, 챔버 ID, 결함 코드를 제거하지 마세요. 질문에 정확한 이름이 들어올 때 강한 검색 앵커가 됩니다.
  • 하이브리드 키워드 채널을 추가합니다. dense vector를 BM25 같은 sparse 채널과 함께 써서 lot ID처럼 정확히 일치해야 하는 용어를 항상 찾을 수 있게 합니다.
  • 평가가 필요하다고 보여줄 때만 미세조정을 고려합니다. 도메인 미세조정은 실제 운영 부담이 크고 폐쇄망에서는 유지관리 비용도 큽니다. 시작점이 아니라 마지막 수단으로 다루세요.

차원, 정규화, 유사도

두 모델 모두 1024차원 벡터를 출력합니다. 하지만 차원 수 자체보다 다음 두 설정이 더 중요합니다.

  • 벡터를 정규화합니다. normalize_embeddings=True를 설정하세요. 모든 벡터가 단위 길이가 되면 dot product와 cosine similarity가 같아지고, 많은 벡터 스토어에서 빠른 inner-product index를 쓸 수 있습니다.
  • 일관성을 유지합니다. 인덱싱과 질의에 같은 metric을 사용하세요. 임베딩 모델을 바꾸면 전체 코퍼스를 다시 임베딩해야 합니다. 예전 벡터 공간과 새 벡터 공간은 비교할 수 없습니다.

로컬 임베딩 모델을 오프라인으로 로드하기

LangChain에서 Hugging Face 임베딩 모델을 실행할 때는 전용 langchain-huggingface 패키지를 사용하는 것이 현재 방식입니다. 폐쇄망 안으로 반입한 로컬 가중치 디렉터리를 지정하세요. 런타임에는 네트워크 호출이 없어야 하며, 모델 파일 해시와 반입 승인 이력도 함께 남겨두는 편이 좋습니다.

bge-m3는 query 또는 passage prefix가 필요 없으므로 embed_query와 embed_documents에 원문 텍스트를 그대로 넣을 수 있습니다. multilingual-e5를 선택했다면 임베딩 전에 query:와 passage: prefix를 직접 붙여야 합니다. 이 prefix를 빼먹는 것은 조용히 품질을 떨어뜨리는 실패 모드입니다.

이미 Ollama 기반 로컬 모델 서빙을 표준화했다면 langchain-ollama를 통해 임베딩 모델에 접근할 수도 있습니다. 오래된 community import 대신 전용 패키지를 사용하세요.

그다음 선택한 임베더를 로컬 벡터 스토어에 연결합니다. Chroma는 이제 langchain-chroma 패키지에 있으며, 예전 langchain_community.vectorstores import는 더 이상 권장되지 않습니다.

도입 전에 자체 데이터로 평가하기

엔지니어가 실제로 쓸 법한 질문 30~50개를 원문 그대로 모으고, 질문마다 정답 SOP의 문서 ID와 절 ID를 기록합니다. 예를 들어 TOOL_A RF reflected power 알람 후 재가동 조건처럼 한글 문장에 가상 장비 식별자와 영문 알람명이 섞인 형태도 남깁니다. 후보별 top-k 적중 여부뿐 아니라 틀린 결과가 같은 문서의 다른 절인지, 전혀 다른 장비 문서인지도 분류하세요. 전자는 chunking 조정으로 나아질 수 있고, 후자는 임베딩이나 필터를 다시 봐야 합니다.

후보 비교표 만들기

스프레드시트에 질문, 기대 절, 후보 A 순위, 후보 B 순위, 첫 오답 유형, 질의 지연 여섯 열을 만드세요. 첫 오답 유형은 같은 문서·다른 절, 다른 장비, 식별자 누락, 권한 필터 중 하나로 표시합니다. 세 문항만 채워도 “어느 모델이 더 좋아 보인다”가 아니라 “혼합 질의에서 어느 실패가 반복된다”로 대화가 바뀝니다. 소규모 결과는 통계적 우위를 증명하지 않으므로 최종 선정 근거로 단독 사용하지 않습니다.

모델을 고른 뒤에는 모델 경로, 가중치 해시, 정규화 설정, prefix 규칙과 평가 결과를 한 묶음으로 버전 관리합니다. 다음 모델 교체 때 같은 질문 세트를 다시 실행해야 개선인지 회귀인지 판단할 수 있습니다. 이 기록을 들고 폐쇄망 내부 문서 RAG 구축으로 넘어가면, 선택한 임베더를 로더·벡터 저장소·로컬 LLM 사이에 어디에 고정해야 하는지 이어서 확인할 수 있습니다.
