LangChain의 사전 구축된 루프 create_agent. 모든 수입이 검증되었습니다 현재에 반대 랭체인 1.x 라인 — 더 이상 사용되지 않음 initialize_agent, 아니요 LLMChain. 사용도구·에이전트를 최신 API 관점로 사용합니다.

모델에 tool이 필요한 이유

온프레미스 모델에게 "A1234 로트의 현재 상태는 어떻습니까?"라고 물어보세요. 그리고, 시스템에 액세스할 수 없으면 그럴듯해 보이는 답변이 나올 것입니다. 즉 공장에서 쓸모없는 것보다 더 나쁩니다. 도구 호출 반전하여 이 문제를 해결합니다. 제어: 최종 답변을 생성하는 모델 대신에 구조화된 요청 - "부르다 get_lot_status ~와 함께 lot_id="A1234"” — 그리고 귀하의 코드는 실제 기능과 손을 실행합니다. 결과가 돌아왔습니다. 모델은 데이터베이스에 직접적으로 접근하지 않습니다. 단지 묻습니다.

그 간접적인 방법은 바로 폐쇄된 네트워크 원한다. 매 도구는 당신이 작성하고 감사한 코드입니다. 모델은 당신이 도달하지 않은 어떤 것에도 도달할 수 없습니다 전달하면 각 통화는 게이트, 속도 제한 또는 제어할 수 있는 개별적이고 기록 가능한 이벤트입니다. 인간에게 가는 길. 모델은 도구를 요청만 하고, 실제 조회는 검증된 코드가 수행합니다 — 어떤 데이터도 망 밖으로 나가지 않습니다.

도구 호출은 작업이 아니라 요청입니다. 모델이 묻습니다. 감사된 코드 답변 여부와 방법을 결정합니다. — 에이전트를 실행 가능하게 만드는 안전 속성

@tool로 tool 정의하기

그것이하는 일. 그만큼 @tool 장식가가 평범하게 변하다 로 기능하다 BaseTool: 모델이 발견하고 호출할 수 있는 호출 가능 항목입니다. LangChain은 함수에서 세 가지 내용을 읽고 이를 모델에 전달합니다. JSON 스키마 — 이름, 독스트링 (이것은 모델이 이유를 설명하는 설명이 됩니다) 힌트 입력 (인수 스키마가 됨) 이 세 가지를 잘 작성하면 모델이 도구를 올바르게 사용하십시오. 모호하게 쓰면 그렇지 않을 것입니다.

장식가가 살고 있는 곳 langchain_core.tools. 인수를 입력하세요. 모델이 받는 유일한 명령인 것처럼 독스트링을 작성합니다. 그렇습니다.

팹 컨텍스트. 도구로 다음을 가리킵니다. 읽기 복제본, 라이브 MES가 아니며 반환 값을 모델이 인용할 수 있는 압축 문자열로 유지합니다. 독스트링은 프롬프트 엔지니어링 표면입니다. 인수 이름 지정 lot_id 그리고 예를 들면 ("A1234") 눈에 띄게 낮아짐 잘못된 전화. Tool의 도크스트링이 곧 접수됩니다 — 명함 이름과 예시가 인증을 받은 사람이 승인을 받습니다.

모델이 호출하게 하기: bind_tools

그것이하는 일. bind_tools 도구 스키마를 첨부합니다 매 턴마다 모델이 일반 텍스트로 응답할 수 있도록 채팅 모델에 연결 또는 하나 이상의 도구 호출. 아무것도 실행되지 않습니다. 모델에 옵션만 제공하고 모델에서 수행하는 모든 요청을 반환합니다. 메시지 tool_calls 필드.

모든 도구 호출 채팅 모델이 작동합니다. 온프레미스, 즉 ChatOllama 에서 는 langchain-ollama 지원하는 로컬 모델을 가리키는 패키지 함수 호출(예: 최근 Llama 또는 Qwen 빌드) 온프레미스 하나에 주목하세요 주의 사항: 일부 Ollama 모델은 무시합니다. tool_choice, 그래서 항상 그럴 수는 없습니다 힘 전화 — 당신은 그것을 조종하기 위해 독스트링에 의존합니다.

응답이 전달됩니다. tool_calls 완성된 답변 대신 당신의 이제 코드는 요청된 함수를 실행하고 결과를 반환하는 일을 담당합니다. — 이것이 다음에 만드는 루프입니다. 모델은 답을 만들지 않고 도구 호출을 요청했을 뿐입니다.

reason-act loop 직접 이해하기

도구 호출은 루프를 닫을 때만 유용합니다. 요청된 도구를 실행하고, 결과를 다음과 같이 추가합니다. ToolMessage, 그리고 모델을 다시 호출하여 그 결과를 읽고 다른 도구를 호출하거나 최종 답변을 작성할 수 있습니다. 이 은 반응 패턴 — 추론하고, 행동하고, 관찰하고, 반복하고 — 그리고 그것을 수행합니다 에이전트가 실제로 무엇인지 확인하는 가장 좋은 방법은 직접 한 번 보는 것입니다.

전체를 통과 call 에게 지시하다 tool.invoke() 은 현재 관용어: 도구는 ToolMessage 이미 태그가 지정되었습니다. 매칭 tool_call_id, 모델은 각 결과를 다음과 같이 정렬할 수 있습니다. 요청. 그만큼 range(5) 공장에서는 캡이 중요합니다. 이는 비용과 혼란스러운 모델이 반복되는 것을 중지합니다. 반복 상한은 폐쇄망 운영에서 필수입니다.

create_agent: 미리 만들어진 loop

프로덕션 환경에서는 해당 루프를 직접 작성하는 경우가 거의 없습니다. LangChain 1.x는 다음과 같이 제공합니다. create_agent (에서 langchain.agents), 이는 LangGraph 위에 완전한 이유-행동 그래프를 제공하고 스트리밍, 체크포인팅, 무료로 지속적인 실행이 가능합니다. 그것은 오래된 것을 대체합니다 langgraph.prebuilt.create_react_agent — 그리고 둘 다 오랫동안 사용되지 않음 initialize_agent. 모델, 도구 목록 및 시스템 프롬프트; 실행 가능한 그래프를 얻습니다.

입력 및 출력은 메시지 목록입니다. 이제 사용된 핸드 루프와 동일한 모양입니다. 당신을 위해 관리했습니다. 추론이 일어나는 대로 보려면 스트리밍하세요. 각 단계가 도착합니다. 청크로 표시되므로 운영자는 마지막 줄 전에 도구 호출과 그 결과를 볼 수 있습니다. 땅.

온프레미스 guardrail

패턴이 작습니다. 그것을 둘러싼 규율이 팹에서 그것을 안전하게 만드는 것입니다. 폐쇄된 네트워크에서 유지되는 몇 가지 규칙은 다음과 같습니다.

  • 도구는 달리 입증될 때까지 읽기 전용입니다. 조회로 시작 (get_lot_status, 사양 가져오기). 도구는 쓴다 — 릴리스 보류, 레시피 변경 — 모델이 아닌 사람의 확인 단계가 필요합니다. 말하자면. ENT Tool은 사람의 후속 조치입니다.
  • 루프를 막습니다. 손으로 쓰든, 직접 쓰든 create_agent, 반복 횟수를 제한하여 혼란스러운 모델 회전할 수 없습니다. 사전 빌드된 에이전트를 사용하면 재귀 제한을 설정할 수 있습니다.
  • 모든 도구 호출을 기록하십시오. 각 통화는 구조화된 이벤트입니다. 이름, 인수, 결과. 그 추적은 감사 기록이자 디버깅 영역입니다.
  • 모든 것이 온프레미스에 유지됩니다. 모델은 다음과 같습니다 ChatOllama 로컬 GPU 노드에서; 도구가 내부 복제본에 도달했습니다. 프롬프트 없음, 로트 ID 없음, 없음 결과는 네트워크를 떠나게 됩니다.

이것이 전체 사다리입니다. @tool 기능을 노출하고, bind_tools 모델이 요청하면 이유-행동 루프가 닫힙니다. 원, 그리고 create_agent 생산 준비가 완료된 선박. 다음으로 시작 하나의 읽기 전용 조회 도구와 제한 루프, 모든 통화를 기록하고 에이전트가 있습니다. 추측하는 대신 실시간 팹 데이터에서 응답합니다. 단 한 바이트도 남기지 않습니다. 식물. 아래 자료는 에이전트 스택을 끝에서 끝까지 안내합니다.
