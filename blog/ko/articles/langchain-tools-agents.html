한 번의 조회가 드러낸 도구 계약

설명을 위해 단순화한 가상 사례입니다. “A1234 로트가 어느 공정에 있는가”라는 입력에 에이전트가 get_lot_status를 골랐지만, 사용자가 적은 A-1234를 그대로 넘겨 빈 결과를 받았다고 해보겠습니다. 모델은 “로트가 존재하지 않는다”고 판단했지만 조회 복제본에는 하이픈을 제거한 ID로 저장돼 있었습니다. 도구 앞단에 ID 형식 검증을 넣고, 빈 결과와 조회 장애를 서로 다른 상태로 반환하며, 도구는 읽기 전용으로 제한했습니다. 이 조치는 잘못된 단정을 줄이지만 원천 MES의 지연이나 누락까지 해결하지는 못합니다.

도구 등록 전 한 줄

기술 권고: 새 도구마다 “누가 어떤 입력으로 무엇을 읽거나 바꾸며, 실패와 승인을 어디에 기록하는가”를 한 문장으로 적으세요. 읽기·쓰기 경계와 책임자를 설명할 수 없다면 에이전트 목록에 아직 등록하지 않는 편이 맞습니다. 이는 특정 회사의 도구 등록 절차가 아니라 최소 설계 기준입니다.

언어 모델은 공정 사양을 설명할 수 있지만, 혼자서 MES를 읽을 수는 없습니다. 로트 추적기, 레시피 데이터베이스, 최근 챔버 압력 같은 값은 모델 안에 들어 있지 않습니다. 도구는 이 간격을 메우는 감사 가능한 Python 함수입니다. 모델은 추측으로 답하는 대신 정해진 도구 호출을 요청하고, 코드는 실제 시스템에서 값을 읽어 모델에게 돌려줍니다. 에이전트는 어떤 도구가 필요한지 판단하고, 호출 결과를 읽은 뒤 다음 행동을 정하는 반복 루프입니다. 이 글에서는 @tool, bind_tools, 직접 작성한 reason-act loop, 그리고 LangChain의 create_agent를 최신 API 기준으로 설명합니다. 오래된 initialize_agent나 LLMChain 패턴은 쓰지 않습니다.

모델에 tool이 필요한 이유

온프레미스 모델은 “A1234 로트의 현재 상태는?”이라는 질문만으로 최신 MES 값을 알 수 없습니다. 도구 호출을 사용하면 모델은 답을 추측하는 대신 get_lot_status를 lot_id="A1234"로 호출하라는 구조화된 요청을 만듭니다. 실제 조회와 입력 검증은 사람이 검토한 코드가 수행하며, 모델은 데이터베이스 접속 권한을 직접 갖지 않습니다.

이 간접 구조는 폐쇄망의 통제 지점을 분명하게 만듭니다. 도구마다 권한 확인, 호출 횟수 제한, 사람 승인, 감사 로그를 적용할 수 있고 모델은 공개된 함수 밖으로 나갈 수 없습니다. 프롬프트와 로트 ID, 조회 결과도 내부 네트워크에 남겨야 합니다.

도구 호출은 실행이 아니라 요청입니다. 모델은 묻고, 감사된 코드는 그 요청을 받아들일지와 어떤 방식으로 처리할지를 결정합니다. — 에이전트를 실행 가능하게 만드는 안전 속성

@tool로 tool 정의하기

@tool 데코레이터는 Python 함수를 모델이 발견하고 호출할 수 있는 BaseTool로 바꿉니다. LangChain은 함수 이름, docstring, type hint를 읽어 도구 schema를 만듭니다. 이름은 선택 기준, docstring은 사용 조건, type hint는 인자 형식을 전달하므로 모호한 표현을 남기면 잘못된 호출로 이어집니다.

데코레이터는 langchain_core.tools에 있습니다. 인자 이름은 MES 화면과 작업표준서에서 쓰는 용어에 맞추고, docstring에는 조회 대상·허용 형식·반환 상태·사용 금지 조건을 적으세요. 이 설명은 개발 문서인 동시에 모델이 읽는 인터페이스입니다.

팹 적용 메모. 첫 도구는 운영 MES가 아닌 읽기 복제본을 조회하게 하세요. 반환값에는 상태 값뿐 아니라 조회 시각과 원천 시스템을 포함해 모델이 근거를 표시할 수 있게 합니다. lot_id처럼 현장 용어와 맞는 인자 이름과 "A1234" 같은 예시는 형식 오류를 줄이는 데 도움이 됩니다.

모델이 호출하게 하기: bind_tools

bind_tools는 도구 스키마를 chat model에 붙입니다. 이후 모델은 매 턴마다 일반 텍스트로 답하거나, 하나 이상의 도구 호출 요청을 반환할 수 있습니다. 이 단계에서 도구가 실행되는 것은 아닙니다. 모델에게 "이런 도구를 요청할 수 있다"는 선택지를 알려 주고, 모델이 요청한 내용은 메시지의 tool_calls 필드에 담깁니다.

도구 호출을 지원하는 chat model이면 같은 패턴을 사용할 수 있습니다. 온프레미스에서는 langchain-ollama의 ChatOllama로 함수 호출을 지원하는 Llama, Qwen 계열 모델을 로컬 서버에 연결할 수 있습니다. 다만 일부 Ollama 모델은 tool_choice를 완전히 따르지 않을 수 있습니다. 그래서 도구 이름과 독스트링을 더 분명하게 써야 합니다.

이 응답은 완성된 답변이 아니라 tool_calls입니다. 이제 우리 코드가 요청된 함수를 실행하고 결과를 모델에게 돌려줘야 합니다. 모델이 한 일은 답변 생성이 아니라 도구 호출 요청까지입니다. 다음 단계에서 이 요청과 실행 결과를 루프로 이어 줍니다.

reason-act loop 직접 이해하기

도구 호출 요청만으로는 답변이 완성되지 않습니다. 코드가 요청을 검증해 실행하고, 결과를 ToolMessage로 메시지 목록에 추가한 뒤 모델을 다시 호출해야 합니다. 모델은 결과를 보고 다른 도구가 필요한지 또는 답변할 수 있는지 판단합니다. 이 reason-act loop를 직접 한 번 실행해 보면 요청, 실행, 관찰, 다음 판단이 어디에서 나뉘는지 확인할 수 있습니다.

tool.invoke(call)에 전체 호출 객체를 넘기면 도구는 요청과 맞는 tool_call_id가 붙은 ToolMessage를 반환합니다. 모델은 이 ID로 어떤 결과가 어떤 요청에 대응하는지 알 수 있습니다. range(5)처럼 반복 횟수를 제한하는 것도 중요합니다. 혼란스러운 모델이 같은 도구를 끝없이 부르지 못하게 막는 안전장치입니다.

create_agent: 미리 만들어진 loop

운영 코드에서 이 루프를 매번 직접 작성하는 경우는 많지 않습니다. LangChain 1.x의 create_agent는 LangGraph 위에 reason-act 그래프를 구성해 주고, 스트리밍과 체크포인트 같은 실행 기능도 함께 제공합니다. 오래된 initialize_agent 패턴 대신 모델, 도구 목록, 시스템 프롬프트를 넘겨 실행 가능한 에이전트 그래프를 얻는 방식으로 생각하면 됩니다.

입력과 출력은 메시지 목록입니다. 앞에서 직접 만든 루프와 모양은 같지만 반복 실행과 도구 결과 연결을 프레임워크가 관리합니다. 추론 과정을 운영자가 보고 싶다면 스트리밍을 사용하세요. 모델 단계, 도구 단계, 최종 답변이 순서대로 도착하므로 마지막 문장만 기다리지 않고 중간 도구 호출도 확인할 수 있습니다.

온프레미스 guardrail

구현 코드는 짧지만 팹에서 운영하려면 도구 주변의 안전 장치가 더 많은 일을 맡습니다. 폐쇄망 에이전트에는 최소한 아래 기준이 필요합니다.

  • 도구는 기본적으로 읽기 전용입니다. get_lot_status나 스펙 조회처럼 상태를 바꾸지 않는 도구부터 시작하세요. hold release, 레시피 변경, 티켓 종료처럼 쓰기 작업을 하는 도구는 사람 승인 뒤에 있어야 합니다.
  • 루프에는 상한을 둡니다. 직접 루프를 만들든 create_agent를 쓰든 반복 횟수와 호출 가능한 도구를 제한해야 합니다. 모델이 같은 도구를 반복 호출하며 회전하는 상황을 막아야 합니다.
  • 모든 도구 호출을 기록합니다. 도구 이름, 인자, 결과, 호출자, 시간, 승인 여부를 구조화된 이벤트로 남기세요. 이 기록은 감사 로그이자 디버깅 자료입니다.
  • 전체 흐름을 온프레미스로 유지합니다. 모델은 로컬 GPU 노드의 ChatOllama로 호출하고, 도구는 내부 읽기 복제본이나 승인된 API만 접근합니다. 프롬프트, 로트 ID, 조회 결과가 외부 네트워크로 나가면 안 됩니다.

빈 결과를 실패로 바꾸는 도구 계약 점검

설명을 위한 합성 입력입니다. get_lot_status("A-1234")에 대해 아래 네 건을 직접 호출해 보세요. 정상 ID, 형식이 틀린 ID, 존재하지 않는 ID, timeout을 일으키는 시험 double을 각각 한 번씩 넣습니다.

입력·상황도구가 반환할 상태에이전트가 해서는 안 되는 말
정상 IDok + 조회 시각 + source근거 없이 다음 공정까지 예측
형식 오류invalid_inputMES에 로트가 없다고 단정
조회 결과 없음not_found조회 장애라고 단정
timeoutunavailable + 재시도 가능 여부로트 상태가 정상이라고 답변

네 상태가 로그와 최종 문장에서 구분되면 첫 계약 검사는 통과입니다. 동일 요청을 반복했을 때 호출 상한을 넘거나, 입력 오류가 tool call 전에 걸러지지 않거나, timeout을 빈 결과로 바꾸면 에이전트 공개를 멈춥니다. 이 검사는 원천 MES 데이터의 최신성과 정확성까지 보증하지 않습니다.

@tool로 검증된 함수를 노출하고, bind_tools로 모델이 호출을 요청하게 한 뒤, reason-act loop나 create_agent가 결과를 다시 전달합니다. 첫 적용은 읽기 전용 조회 도구 하나와 제한된 반복 횟수로 충분합니다. 이 계약이 안정되면 다음 글인 LangGraph workflow에서 재시도·승인 상태를 명시하고, 여러 내부 도구를 공유해야 할 때 MCP와 MES 연결 글로 넘어가세요. 데이터가 공장 밖으로 나가지 않는 것과 잘못된 조치를 실행하지 않는 것은 별도의 검증 항목입니다.
