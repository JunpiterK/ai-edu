RAG stack 구성요소별 설명

RAG는 기능 하나가 아닙니다. private document를 model이 답하기 전에 useful context로 바꾸는 작은 pipeline입니다. 구성요소 하나가 약하면 전체 system이 불안정하게 느껴집니다.

역할을 나누면 stack이 쉬워집니다. loader는 source material을 읽습니다. splitter는 chunk를 만듭니다. embedding model은 chunk를 vector로 바꿉니다. vector store는 그것을 index합니다. retriever는 질문에 맞는 chunk를 고릅니다. LCEL은 context, prompt, model, parser를 연결합니다.

1. DocumentLoader: 문서를 가져오기

무엇을 하는가. Loader는 PDF, HTML, CSV, Markdown, text, database export 같은 source file을 읽고 page_content와 metadata가 있는 LangChain Document object를 반환합니다.

언제 쓰는가. source material이 Python string 밖에 있으면 loader를 씁니다. 실제 RAG에서 metadata는 선택이 아닙니다. source path, page, section, equipment, document version, permission scope가 나중에 중요해집니다.

구현 note. PDF부터 시작하지 말고 plain text나 Markdown으로 먼저 시작하세요. PDF extraction 품질은 편차가 크고, extraction이 나쁘면 retrieval 문제가 model 문제처럼 보입니다.

2. TextSplitter: chunk로 자르기

무엇을 하는가. Splitter는 긴 document를 retrieve 가능하고 prompt에 넣을 수 있는 chunk로 나눕니다. 단순히 글자 수를 세는 것이 아니라 의미를 보존해야 합니다.

언제 쓰는가. 거의 모든 RAG system에서 splitter를 씁니다. manual 전체는 너무 크고, 너무 작은 fragment는 context를 잃습니다. 좋은 splitter는 heading, procedure step, table을 설명 text 가까이에 둡니다.

청킹은 검색 품질을 가장 많이 변경하는 하나의 손잡이입니다. 너무 크고 블러 삽입; 너무 작으면 단계가 주변 레시피를 잃습니다. — 블로그 기본값이 아닌 자신의 매뉴얼에 맞춰 Chunk_size를 조정하세요.

구현 note. RecursiveCharacterTextSplitter가 안전한 기본값입니다. 기술 문서에서는 실제 retrieved chunk를 본 뒤 chunk_size와 overlap을 조정하세요.

3. Embeddings: text를 vector로 바꾸기

무엇을 하는가. Embedding model은 text를 vector로 mapping해 semantic similarity를 검색할 수 있게 합니다. 의미가 비슷한 query와 chunk는 vector space에서 가까워야 합니다.

언제 쓰는가. keyword search가 너무 brittle할 때 embedding을 씁니다. paraphrase, synonym, 한국어/영어가 섞인 기술 표현에 강합니다. 하지만 exact fact, part number, relationship을 보장하지는 않습니다.

구현 note. 가정용 PC에서는 local embedding model부터 쓰세요. chunking을 평가하는 동안 embedding model을 고정해야 어떤 변화가 효과가 있었는지 알 수 있습니다.

4. VectorStore: vector 저장하기

무엇을 하는가. Vector store는 embedding과 metadata를 저장하고 similarity search를 수행합니다. Chroma는 prototype에 쉽고, 여러 app이 같은 index를 공유하면 Qdrant, Postgres/pgvector 같은 server-backed store가 더 적합합니다.

언제 쓰는가. document를 반복해서 검색해야 하면 vector store를 씁니다. one-off summary라면 필요 없을 수도 있지만 knowledge base라면 필요합니다.

구현 note. index를 persist하고 collection name을 versioning하세요. source corpus가 바뀌었는데 index name이 그대로면 debugging이 괴로워집니다.

5. Retriever: 맞는 chunk 가져오기

무엇을 하는가. Retriever는 application이 보는 search interface입니다. 나머지 chain은 retriever.invoke(question)을 호출해 Document를 받을 뿐 backend가 Chroma인지 Qdrant인지 BM25인지 hybrid search인지 몰라도 됩니다.

언제 쓰는가. model이 답하기 전에 evidence가 필요하면 retriever를 씁니다. 특정 tool, date range, product, document type, permission scope를 묻는 경우 metadata filter를 추가하세요.

구현 note. retrieved chunk를 항상 직접 확인하세요. 많은 RAG 실패는 fluent answer를 입은 retrieval failure입니다.

LCEL로 조립하기

최종 chain은 평범한 LCEL입니다. context를 retrieve하고, question을 보존하고, prompt를 formatting하고, local model을 호출하고, answer를 parse합니다. model은 context에서만 답하고 가능하면 source를 cite해야 합니다.

핵심 습관은 RAG를 component by component로 debug하는 것입니다. 답이 나쁘면 LLM을 탓하기 전에 loading, splitting, embedding, indexing, retrieval, prompting 중 어디가 실패했는지 확인하세요.

이 글은 현장 노트입니다. 인터랙티브 deck은 local RAG design을 diagram과 구현 예제로 slide by slide 설명합니다.
