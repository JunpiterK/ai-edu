검색 결과에서 원문으로 거슬러 올라가기

설명을 위해 단순화한 가상 사례입니다. 펌프 교체 후 체결 토크를 묻는 질문에 RAG가 숫자는 맞췄지만 경고 조건을 빠뜨렸다고 가정해 보겠습니다. 입력 PDF의 표는 두 단으로 추출됐고, 검색된 top-k에는 토크 행만 있고 바로 아래의 “신품 가스켓 사용 시” 조건은 없었습니다. 모델 문제로 판단하기 전에 추출 text와 chunk 경계를 확인하자 loader와 splitter 사이에서 의미가 갈라진 것이 보였습니다. 표 제목과 주의 문구를 같은 chunk로 묶고 문서 개정 metadata를 추가했습니다. 다만 스캔 품질이 낮은 도면과 손글씨는 OCR 자체를 별도로 평가해야 합니다.

기술 권고: 원문 추출, 청크 경계, 상위 검색 결과, 프롬프트 맥락, 최종 답변 순으로 한 건씩 저장합니다. 필요한 문장이 검색 결과에 없다면 모델을 바꿔도 같은 근거 부족이 반복됩니다. 이는 특정 현장의 성능 결과가 아니라 검색 오답을 분류하기 위한 일반 절차입니다.

RAG는 내부 문서를 모델이 답변 근거로 사용할 수 있는 context로 바꾸는 파이프라인입니다. 답이 틀렸다는 최종 현상은 같아도 원인은 원문 추출, 분할, embedding, 색인, 검색, prompt 중 어느 곳에나 있을 수 있습니다.

Loader는 원본을 Document로 읽고, splitter는 검색 단위로 나눕니다. embedding model과 vector store는 의미 기반 검색을 준비하며, retriever는 질문에 맞는 chunk를 고릅니다. LCEL은 선택된 context를 prompt와 모델, parser로 전달합니다. 이 역할 구분은 설명용 목록이 아니라 디버깅 순서입니다.

1. DocumentLoader: 문서를 가져오기

Loader는 PDF, HTML, CSV, Markdown, text, database export를 읽어 page_content와 metadata가 담긴 LangChain Document를 반환합니다. 이때 페이지 순서, 표, 머리말이 어떻게 추출되는지 샘플 몇 건을 먼저 확인해야 합니다.

원본이 Python 문자열 밖에 있으면 loader를 사용합니다. 운영 RAG에서는 source path, page, section, equipment, document version, permission scope를 metadata에 남기세요. 답변 인용뿐 아니라 폐기 문서 제외와 접근 권한 필터에 쓰입니다.

첫 연결 시험은 plain text나 Markdown으로 하는 편이 좋습니다. 그 상태에서 질문과 검색 흐름을 확인한 뒤 PDF를 넣으면, 새 오답이 추출 문제인지 검색 설정 문제인지 구분할 수 있습니다.

2. TextSplitter: chunk로 자르기

Splitter는 긴 문서를 검색과 prompt 주입에 적합한 chunk로 나눕니다. 크기 제한은 필요하지만 글자 수만 맞추면 절차 단계와 조건문, 표 제목과 값이 떨어질 수 있습니다.

매뉴얼 전체는 context window에 넣기 어렵고, 지나치게 작은 조각은 적용 조건을 잃습니다. 제목과 절차 단계, 표와 설명 문구가 함께 검색되도록 문서 구조를 우선 보존하세요. chunk 크기는 고정된 정답이 아니라 평가 질문에서 필요한 근거가 함께 들어오는지 보고 정합니다.

청킹은 검색 품질을 크게 바꾸는 손잡이입니다. chunk가 너무 크면 검색 결과가 흐려지고, 너무 작으면 절차 단계가 주변 경고와 레시피 맥락을 잃습니다. — 블로그 기본값이 아니라 자신의 매뉴얼에 맞춰 chunk_size를 조정하세요.

RecursiveCharacterTextSplitter는 시작점으로 쓸 수 있습니다. 기술 문서에서는 숫자만 조정하지 말고, 실패 질문의 retrieved chunk를 열어 경고 문구와 단위가 함께 남는지 확인하세요.

3. Embeddings: text를 vector로 바꾸기

Embedding model은 text를 vector로 바꿔 의미가 비슷한 query와 chunk를 가까운 위치에 놓습니다. “배기 불량”과 “exhaust pressure low”처럼 표현이 달라도 관련 문서를 찾는 데 유용합니다.

반면 정확한 part number, 레시피 개정, 권한 조건은 semantic similarity만으로 보장되지 않습니다. 이런 필드는 keyword 검색이나 metadata filter와 결합하고, 한국어·영어 혼용 질문을 평가셋에 포함해야 합니다.

폐쇄망이나 개인 PC에서는 local embedding model로 시작할 수 있습니다. chunking을 비교하는 동안 모델과 평가 질문을 고정해야 어느 변경이 검색 결과를 바꿨는지 판단할 수 있습니다.

4. VectorStore: vector 저장하기

Vector store는 embedding과 metadata를 저장하고 similarity search를 수행합니다. Chroma는 prototype에 쉽고, 여러 app이 같은 index를 공유해야 한다면 Qdrant, Postgres/pgvector 같은 server-backed store가 더 적합합니다.

같은 문서 집합을 반복 검색한다면 vector store가 필요합니다. 한 번의 요약처럼 문서 전체를 바로 넣을 수 있는 작업에는 색인을 만드는 비용이 더 클 수 있습니다.

색인은 지속 저장하고 collection name에 corpus version을 붙이세요. 원문은 바뀌었는데 같은 이름의 이전 색인을 조회하면 오답 재현 기록까지 뒤섞입니다.

5. Retriever: 맞는 chunk 가져오기

Retriever는 application이 보는 search interface입니다. 나머지 chain은 retriever.invoke(question)을 호출해 Document를 받을 뿐 backend가 Chroma인지 Qdrant인지 BM25인지 hybrid search인지 몰라도 됩니다.

모델이 답하기 전에 근거 문서를 골라야 할 때 retriever를 사용합니다. 특정 장비, 기간, 제품, 문서 유형, 권한 범위가 질문에 포함되면 metadata filter로 검색 대상을 먼저 좁히세요.

평가할 때는 답변과 함께 retrieved chunk도 보관하세요. 근거가 없는 유창한 답은 생성 실패로만 기록할 것이 아니라 retrieval failure로 분류해야 다음 조치가 맞아집니다.

LCEL로 조립하기

최종 chain은 평범한 LCEL입니다. context를 retrieve하고, question을 보존하고, prompt를 formatting하고, local model을 호출하고, answer를 parse합니다. model은 context에서만 답하고 가능하면 source를 cite해야 합니다.

RAG 오답은 구성요소별로 분류해야 합니다. 원문 누락, 의미가 끊긴 chunk, 잘못된 top-k, 오래된 index, 조건이 빠진 prompt를 구분하면 수정 뒤 같은 평가 질문으로 회귀를 확인할 수 있습니다. 모델 교체는 그 기록에서 생성 단계의 결함이 남았을 때 검토해도 늦지 않습니다.

검색 오답 한 건 판독하기

특정 회사의 실제 수치가 아닌 작은 실습입니다. 원문에 “신품 가스켓 사용 시 체결 토크 24 N·m, 재사용 금지”가 있고 질문은 “교체 후 토크는?”이라고 가정합니다. 검색 결과를 아래 순서로 열어 판정하세요.

  1. 추출 text에 24 N·m와 재사용 금지가 모두 있는지 봅니다. 하나라도 없으면 loader/OCR 단계에서 중단합니다.
  2. 두 문구가 같은 chunk 또는 서로 연결된 구조 안에 있는지 확인합니다. 갈라졌다면 splitter 실패로 기록합니다.
  3. top-k 안에 조건을 포함한 chunk가 있는지 확인합니다. 없다면 embedding·metadata filter·질문 표현을 비교합니다.
  4. context에는 조건이 있는데 답에서 빠졌다면 그때 prompt 또는 생성 단계 실패로 분류합니다.

완료 결과는 “정답/오답” 한 칸이 아니라 failure_stage, 빠진 근거, 검색된 문서 revision, 수정 전후 결과로 남깁니다. 이 실습은 텍스트형 SOP를 전제로 하며 병합 셀 표, 도면 기호, 흐린 스캔에는 별도의 구조 추출·OCR 평가가 필요합니다.

오답의 실패 단계를 나눴다면 다음에는 ‘의미를 찢지 않고 기술 문서 chunking하기’에서 splitter 경계를 더 깊게 다루세요. 검색 근거가 안정된 뒤 ‘폐쇄망 내부 문서 위에 RAG 구축하기’로 넘어가면 배포·개정 관리까지 같은 기록을 이어갈 수 있습니다. 연결된 인터랙티브 deck에서는 local RAG design을 diagram과 구현 예제로 단계별로 설명합니다.
