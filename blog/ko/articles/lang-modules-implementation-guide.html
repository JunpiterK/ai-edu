Lang ecosystem에서 헷갈리는 부분은 API 하나가 아닙니다. 어떤 package가 어떤 일을 맡는지 아는 것이 핵심입니다. 이 감각이 잡히면 stack은 평범한 Python처럼 보입니다. model이 들어오고, prompt가 들어오고, parser가 나오고, document가 들어오고, retriever가 나오고, tool이 들어오고, graph가 workflow를 만듭니다.

Lang 모듈은 LLM application을 위한 building block입니다. 단순 model call은 블록 하나일 뿐입니다. 진짜 가치는 prompt, parser, retrieval, tool, graph state를 조합해 파일에서 답하고, local function을 호출하고, approval을 요청하고, error에서 복구하는 workflow를 만들 때 나옵니다.

모듈 지도

코드를 쓰기 전에 이 표를 mental model로 사용하세요.

설치 세트

가정용 PC 또는 local LLM tutorial이라면 Ollama-compatible local serving으로 시작하세요. 나중에 provider를 바꾸더라도 app 전체를 다시 쓸 필요는 없습니다.

예상 output입니다. package 설치가 끝나고 virtual environment가 활성화된 prompt가 보여야 합니다.

Chat model + prompt + parser

가장 작은 Lang app입니다. chat model, prompt, output parser로 구성됩니다. tool이나 retrieval 없이 deterministic text transformation, summarization, simple assistant response가 필요할 때 씁니다.

이 pattern은 email rewriting, 한국어/영어 기술 번역, 회의록 요약, code explanation, glossary generation, report drafting처럼 model이 user input과 instruction만으로 처리할 수 있는 작업에 적합합니다.

private document, 현재 database 값, 정확한 계산, file write나 test 실행 같은 action이 필요한 질문에는 이 pattern만 쓰면 안 됩니다. 그런 경우 retrieval 또는 tool이 필요합니다.

각 모듈이 한 일입니다. ChatOllama는 local model을 호출했습니다. ChatPromptTemplate은 변수를 chat message로 바꿨습니다. StrOutputParser는 model message를 plain text로 바꿨습니다. pipe operator는 runnable chain을 만들었습니다.

실전 variation입니다. text 대신 JSON object가 필요하면 StrOutputParser를 structured-output parser로 바꾸거나 schema-capable interface를 통해 model에 요청하세요. risk classification이나 blog metadata generation이 대표 예입니다.

예상 output입니다.

Load, split, embed, retrieve

RAG는 네 종류의 모듈 family를 사용합니다. loader는 file을 Document object로 바꿉니다. splitter는 chunk를 만듭니다. embedding은 chunk를 vector로 바꿉니다. vector store는 vector를 저장하고 검색합니다. retriever는 query-to-documents interface를 제공합니다.

답이 근거에 기반해야 할 때 RAG를 사용합니다. policy가 무엇을 말하는지, 어떤 SOP가 alarm을 언급하는지, claim의 출처가 어디인지 묻는 경우 model은 먼저 source text를 retrieve하고 그 text에서 답해야 합니다.

각 RAG 모듈의 용도입니다.

  • Document loaders: 외부 자료를 page_content와 metadata가 있는 표준 Document 형태로 읽습니다.
  • Text splitters: 긴 document를 retrieve 가능한 chunk로 자릅니다. chunk가 나쁘면 좋은 model도 어리석어 보입니다.
  • Embeddings: text를 vector로 바꿔 비슷한 의미를 검색할 수 있게 합니다.
  • Vector stores: vector를 저장하고 similarity search를 수행합니다.
  • Retrievers: storage detail을 invoke(question) 뒤에 숨깁니다.

예상 output입니다.

각 piece를 언제 쓰는지입니다. source가 외부에 있거나 지저분하면 loader를 씁니다. document가 retrieval target보다 길면 splitter를 씁니다. semantic search가 중요하면 embedding과 vector store를 씁니다. app의 나머지 부분이 search 구현을 몰라도 되게 하려면 retriever를 씁니다.

실전 선택지입니다. 개인 note나 prototype에는 작은 persistent vector store가 좋습니다. 여러 app이나 user가 같은 index를 써야 하면 Qdrant, Postgres pgvector, server-backed store를 고려하세요. document type, date, equipment, product, author, permission scope가 질문에 포함되면 metadata filter를 사용합니다.

Tool과 structured call

Tool은 schema가 있는 function입니다. model이 system에게 계산, 검색, file lookup, internal API call, MCP server query 같은 일을 부탁해야 할 때 사용합니다. 첫 tool은 지루하고 안전하게 시작하세요.

language만으로 부족할 때 tool을 사용합니다. model은 계산기나 database client가 아니며, shell user로 두면 위험합니다. tool은 wrapper입니다. model이 estimate_weight_memory를 요청하면 Python code가 계산하고 model이 결과를 설명합니다.

좋은 첫 tool입니다.

  • Calculator: 정확한 arithmetic, unit conversion, GPU memory estimate.
  • Search: local index 또는 controlled file list query.
  • Lookup: 알려진 config, policy, glossary entry 읽기.
  • Test runner: disposable workspace 안에서만 pytest 실행.
  • Ticket draft: live external ticket이 아니라 draft object 생성.

예상 output입니다.

설계 rule입니다. tool docstring은 장식이 아닙니다. model이 읽는 interface의 일부입니다. 새벽 2시에 피곤한 junior engineer가 쓴다고 생각하고 tool name, argument name, docstring을 쓰세요.

위험 rule입니다. 첫날부터 general shell tool을 노출하지 마세요. narrow tool부터 시작하세요. run_tests()가 run_command(command)보다 안전하고, read_policy(name)가 read_file(path)보다 안전합니다. tool이 general해질수록 validation과 approval이 더 필요합니다.

LangGraph workflow

LCEL은 straight-line chain에 좋습니다. LangGraph는 classify, retrieve, answer, approval request, retry, stop처럼 state가 있는 workflow를 위한 것입니다. 핵심은 StateGraph입니다. 각 node는 current state를 읽고 partial update를 반환합니다.

workflow flag를 쓰기 시작했다면 LangGraph를 사용할 때입니다. needs_retrieval, tool_failed, approval_required, retry_count, final_answer 같은 변수가 code에 있다면 이미 머릿속으로 graph를 만들고 있는 것입니다. LangGraph는 그 state를 명시적으로 만듭니다.

실전 graph pattern입니다.

  • Router graph: question을 chat, RAG, tool, refusal path로 보냅니다.
  • RAG retry graph: retrieve, evidence grade, query rewrite, retrieve again.
  • Human approval graph: action을 draft하고 pause한 뒤 approval 후에만 resume합니다.
  • Coding agent graph: repo inspect, patch propose, test run, failure fix, diff summarize.
  • Incident assistant graph: alarm classify, SOP retrieve, read-only diagnostics call, next steps 생성.

예상 output입니다.

LangGraph는 application이 더 이상 single pipe가 아닐 때 사용합니다. conditional branch, human approval, retry, memory, checkpointing, multiple agents가 필요하면 giant function보다 graph가 보통 더 명확합니다.

실전 결합 흐름

실제 local assistant에서 모듈들이 어떻게 맞물리는지 보면 다음과 같습니다.

  1. User가 묻습니다: 12GB GPU로 local RAG용 7B model을 돌릴 수 있나?
  2. LangGraph는 GPU memory 언급을 보고 tool path로 route합니다.
  3. Tool이 raw model memory를 추정하고 KV cache overhead를 경고합니다.
  4. Retriever가 safe model setting에 대한 local lab policy를 가져옵니다.
  5. Prompt가 tool result와 retrieved policy를 결합합니다.
  6. Model이 recommendation과 source note로 답합니다.

Mini project layout

예제가 커지면 책임별로 나누세요. 이 layout은 가정용 PC lab에는 충분히 작고, portfolio project로 발전시키기에도 충분히 진지합니다.

기억할 모양은 이것입니다. core abstraction, provider call, retrieval module, tool, graph orchestration입니다. 이름은 커 보이지만 맡은 일은 작습니다.

추천 학습 순서

  1. Plain chat chain: prompt, model, parser.
  2. Structured output: model이 predictable shape를 반환하게 강제합니다.
  3. Tiny RAG: text file 하나, retriever 하나, source가 있는 answer 하나.
  4. One safe tool: 정확한 calculation 또는 read-only lookup.
  5. Router graph: chat vs RAG vs tool 결정.
  6. Agent workflow: inspect, plan, call tool, summarize.
  7. Approval and logging: 반복 사용해도 안전하게 만듭니다.

이 순서대로 가면 각 Lang module이 왜 존재하는지 이해한 뒤 의존하게 됩니다. full agent template부터 시작해서 어느 부분이 깨졌는지 헤매는 것보다 훨씬 낫습니다.

Security note. model output은 untrusted로 다루세요. model이 생성한 string을 file path, SQL filter, shell command, deserializer에 바로 넣지 마세요. tool이 늘어날수록 지루한 validation이 중요해집니다.
