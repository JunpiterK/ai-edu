용어가 엇갈리는 순간

합성 장면: 회의에서 말이 엇갈릴 때

오전 검토 회의 화면에 FAB agent pilot이라는 한 줄이 떠 있습니다. 문서 담당자는 SOP를 찾아주는 검색창을 떠올리고, 설비 담당자는 alarm을 읽어 ticket까지 만드는 흐름으로 받아들입니다. 안전 담당자가 "그 agent가 상태도 바꿉니까?"라고 묻고 나서야 세 사람이 서로 다른 시스템을 말하고 있었다는 사실이 드러납니다. 현장에서 자주 보는 대화를 합친 장면이며, 특정 회사의 실제 회의가 아닙니다.

30초 용어 확인법

회의 자료의 용어 옆에 읽는 것 / 바꾸는 것 / 멈추는 사람 세 칸을 그어 보세요. 예를 들어 SOP만 읽고 답을 초안하며 process owner가 중지한다면, 검색·생성·승인 책임을 한 문장에서 분리할 수 있습니다. 세 칸 중 하나가 비면 아직 제품 이름만 있고 운영 정의는 없는 상태입니다.

용어를 외우기 전에 그 용어가 만지는 데이터, 남기는 결과, 멈추는 사람을 적어보는 편이 훨씬 빠릅니다. RAG는 근거 문서를 찾아 모델에 건네는 흐름이고, agent는 정해진 도구를 선택해 호출 결과를 다음 판단에 반영하는 흐름입니다. 같은 화면에 들어 있어도 하나는 정보 조회에서 끝날 수 있고 다른 하나는 ticket 생성이나 상태 변경까지 이어질 수 있습니다. 제품 이름이 아니라 실제 권한과 데이터 흐름을 봐야 합니다.

한 문장 지도

RAG는 출처를 함께 보여주는 문서 조회, evaluation은 변경 전후 성능 확인, guardrail은 금지 행동을 막는 규칙, audit log는 실행과 승인 이력을 되짚는 기록입니다.

검색 결과 1등이 나왔다고 답이 맞는 것은 아닙니다. 예를 들어 문서 저장소에 동일한 알람 코드의 지침이 세 개 있고 그중 하나만 현재 승인본이라면, 의미 유사도만 높은 폐기본이 먼저 나올 수 있습니다. 그래서 chunk에는 본문뿐 아니라 document_id, revision, 승인 상태, 유효일, 설비군, 접근 등급이 따라다녀야 합니다. 식별자와 알람 코드는 vector 검색만 믿지 않고 keyword 검색을 함께 쓰는 편이 안전합니다.

현장형 실패 패턴: 검색은 성공했는데 답은 폐기본

처음 가설: 모델이 옛 절차를 기억해 hallucination을 냈다고 생각합니다. 확인: 답변의 인용 문서를 열어보니 모델이 만든 문장이 아니라 검색기가 superseded revision을 정상적으로 가져왔습니다. 수정: 승인 상태와 유효일을 검색 전 필터로 넣고, 활성 문서가 0건이면 “근거 없음”으로 답을 보류합니다. 남은 한계: 원천 문서 시스템의 승인 상태가 틀리면 RAG도 틀립니다. AI보다 먼저 문서 기준정보의 owner가 필요합니다.

Agent와 tool: 읽기, 제안, 쓰기를 나눕니다

Agent는 요청을 읽고 필요한 도구를 고른 뒤, 허용 범위 안에서 호출 결과를 다음 판단에 반영하는 흐름입니다. 위험도는 "읽고 설명"에서 "상태를 변경"하는 순간 크게 달라집니다. 조회 도구와 쓰기 도구를 권한, 승인 절차, 로그 정책까지 분리해야 하는 이유입니다.

FAB 엔지니어 시점

알람 이력을 요약하는 assistant까지는 부담이 작습니다. 그런데 같은 계정으로 corrective action을 닫고 owner까지 바꿀 수 있다면 이야기가 달라집니다. 처음에는 “어차피 사람이 확인 버튼을 누르니 괜찮겠지” 싶지만, 버튼 앞에 실제 변경 payload와 근거가 보이지 않으면 승인은 거의 반사 동작이 됩니다. 읽기 계정, 임시 초안 queue, 최종 쓰기 계정을 분리하고 마지막 단계만 명명된 owner가 승인하도록 설계해야 합니다.

평가와 통제: 시험 성적과 안전장치는 다릅니다

Evaluation은 "무언가를 바꾼 뒤에도 시스템이 여전히 맞게 동작하는가?"를 묻습니다. Control은 "시스템이 잘못된 행동을 하지 못하게 무엇이 막는가?"를 묻습니다. 초심자는 이 둘을 자주 섞지만, 둘은 함께 작동할 뿐 같은 개념은 아닙니다.

  • Evaluation: 고정 질문, 기대 근거, metric, 합격선을 version과 함께 관리합니다.
  • Control: 역할 권한, 승인 gate, tool allowlist, 입력 범위 검사를 둡니다.
  • Audit: 사용자, source와 model version, tool payload, 승인자, 시간을 구조화해 남깁니다.
  • Monitoring: 배포 뒤 검색 실패, 답변 보류, 승인 거절, 비정상 호출을 추세로 봅니다.

NIST AI RMF는 AI 위험관리를 Govern, Map, Measure, Manage의 지속 활동으로 다루며, 배포 환경과 비슷한 조건에서 test set과 metric을 문서화하고 변경 후 다시 평가할 것을 요구합니다. 즉 “정답률이 높다”는 말과 “쓰기 권한이 통제된다”는 말은 서로를 대신하지 못합니다. 하나는 성능 증거이고 다른 하나는 위험 억제 장치입니다.

수치로 보는 두 사례

사례 A: 최신 SOP를 top 5 안에 찾는가

설명용으로 알람 질문 40개를 만들고 각 질문마다 승인된 기대 문서를 지정했다고 합시다. 시스템이 34개 질문에서 기대 문서를 top 5 안에 찾았다면 Recall@5 = 34 / 40 = 85%입니다. 그런데 실패 6건 중 4건이 안전 관련 알람이면 전체 85%만 보고 배포하면 안 됩니다. 안전 관련 slice의 합격선을 별도로 두고 4건을 모두 원인 분류해야 합니다.

사례 B: 읽기 도구가 너무 많은 정보를 반환하는가

get_alarm_history가 30일치 전체 raw record를 반환하면 답변에 필요 없는 작업자 ID와 자유서술 메모까지 모델 context에 들어갈 수 있습니다. 처음에는 “쓰기 기능이 아니니 저위험”이라고 판단하기 쉽습니다. 하지만 최소권한은 읽기에도 적용됩니다. 요청 설비, 24시간, 승인된 필드 7개로 응답 schema를 제한하고, 권한 밖 설비를 섞은 20개 음성 test에서 반환 0건인지 확인합니다.

이 사례의 pass / stop / limit

Pass: 허용 설비 20건은 필요한 7개 필드만 반환하고, 비허용 설비 20건은 데이터 0건과 구조화된 거절 code를 남깁니다. Stop: 다른 설비 record나 불필요한 개인정보 필드가 한 건이라도 반환되면 pilot을 멈춥니다. Limit: 이 test는 접근 범위만 확인합니다. 답변 정확도, prompt injection, 원천 DB 권한 오류는 별도 시험이 필요합니다.

배포 전 용어 카드 한 장

회의에서 새 용어가 나오면 아래 일곱 칸을 채우세요. 이 카드가 비어 있는 상태에서 제품 비교를 시작하면, 사람마다 다른 시스템을 상상한 채 GPU와 모델 가격부터 논의하게 됩니다. 이런 순서로 시작한 회의는 길어지고 요구사항은 더 흐려지기 쉽습니다.

현재 검토 중인 AI 기능 하나를 골라 이 표를 채워 보세요. 세 칸 이상이 “추후 결정”이면 agent pilot보다 read-only 검색 pilot부터 시작하는 편이 낫습니다. 특히 책임자와 중단 조건이 비어 있다면 기술 문제가 아니라 운영 설계가 아직 끝나지 않은 것입니다.

근거 자료와 읽는 법

아래 자료는 특정 FAB 절차를 대신하는 규정이 아니라, 이 글의 평가·권한·도구 통제 원칙을 잡는 데 사용한 공식 문헌입니다.

  • NIST AI RMF Core: test set과 metric 문서화, 배포 조건과 유사한 평가, 지속 모니터링의 근거.
  • NIST SP 800-53 Rev. 5.1: 최소권한과 감사 기록을 시스템 통제로 해석할 때 참고.
  • Model Context Protocol specification: tool을 임의 코드 실행처럼 주의해 다루고 동의·권한·데이터 보호를 구현해야 한다는 공식 명세.

용어 카드에서 원본 데이터 칸이 자꾸 비어 있다면 다음에는 FAB 설비 데이터 101로 넘어가 source와 join key를 정리해 보세요. 반대로 tool과 승인 칸이 문제라면 Agent 사고 대응 런북이 중지·보존 기준을 이어서 보여줍니다.

FAB 운영 용어와 기술 문서를 기준으로 작성하고, 명확성과 기술적 정확성을 검토했습니다.
