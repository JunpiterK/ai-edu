폐쇄망 에이전트의 사람 검증과 MLOps 운영

승인 대기 화면

설명을 위해 단순화한 가상 상황입니다. 에이전트는 lot hold를 제안했고 담당 엔지니어는 최근 정비 이력을 보고 거부했습니다. 이때 화면에 승인 버튼만 있었다면 거부 이유, 제안 당시 상태, 이후 재개 경로가 남지 않습니다. Human-in-the-loop(HITL)는 버튼의 유무보다 누가 어떤 근거로 결정했고 프로세스 재시작 뒤에도 그 결정을 재현할 수 있는지를 다루는 설계입니다.

설계 시작점

기술 권고: 상태를 바꾸는 tool보다 승인자, 검토 payload, 감사 이벤트, 거부 후 경로를 먼저 정의하세요. 어느 항목이 비어 있으면 쓰기 도구는 연결하지 않고 조회·초안 작성 범위에 둡니다. 아래 제안 아키텍처를 운영 승인으로 오해해서는 안 됩니다.

팹의 도구 호출은 레시피 파라미터 쓰기, MES lot hold, 정비 티켓 발행처럼 실제 상태 변경으로 이어질 수 있습니다. 이 글은 그런 쓰기 경로를 검토하기 위한 제안 아키텍처를 설명합니다. 아래 코드는 승인 중단과 상태 저장의 연결 방식을 보여주는 비운영 예제이며, 실제 MES·설비·레시피 시스템에 연결되어 있지 않습니다. 운영 적용 전에는 설비·공정·IT/OT·보안 담당자의 승인과 별도 위험 분석이 필요합니다.

에이전트가 실행하기 전에 사람 승인 게이트가 필요한 이유

폐쇄망 안에 있다는 사실만으로 잘못된 실행이 안전해지지는 않습니다. 모델은 허용 범위를 벗어난 setpoint를 제안하거나 서로 다른 revision의 SOP 절차를 이어 붙일 수 있습니다. 실행 전 검토 화면에는 제안 값만 보여주지 말고 현재 상태, 근거 문서 revision, 대상 장비와 되돌리기 방법을 함께 표시해야 합니다.

구체적으로 에이전트의 도구를 두 종류로 나눕니다.

  • 읽기 도구 — MES 조회, 레시피 읽기, 장비 로그 가져오기. 라인을 변경하지 않으므로 비교적 자유롭게 실행할 수 있습니다.
  • 쓰기 도구 — 파라미터 설정, lot hold, 작업 지시 발행. 반드시 승인 게이트를 통과해야 하며, 지정된 사람이 resume하기 전에는 실행되지 않아야 합니다.
조회 오류는 정정할 수 있지만 쓰기 오류는 설비와 lot 상태를 바꿉니다. 쓰기 경로에는 중단, 실명 승인, 재개 기록이 필요합니다.

interrupt()로 만드는 승인 게이트

LangGraph에서는 노드 안의 interrupt(payload)로 실행을 멈추고 검토 payload를 호출자에게 돌려줄 수 있습니다. checkpointer를 함께 설정하면 graph state가 저장되고, Command(resume=...)로 전달한 결정이 interrupt()의 반환값이 됩니다. 서버 재시작 뒤에도 대기 상태를 유지하려면 뒤에서 설명하는 영속 checkpointer가 필요합니다.

운영 환경에서 특히 중요한 동작이 있습니다. resume하면 LangGraph는 노드를 interrupt() 다음 줄에서 이어서 실행하지 않고, 노드 처음부터 다시 실행합니다. 그래서 실제 side effect, 특히 실제 쓰기 작업은 반드시 interrupt가 반환된 뒤에 두어야 합니다. interrupt 위쪽 코드는 여러 번 실행되어도 안전해야 합니다.

approval_service는 예제용 경계 이름입니다. 운영 구현은 서버 인증에서 principal을 얻고, 승인 역할과 대상 thread/action 권한을 확인하며, exact action hash·만료 시각·일회성 approval ID가 묶인 서명 token만 발급해야 합니다. interrupt()로 중단한 실행을 나중에 재개하려면 graph compile 시 checkpointer도 연결합니다. 메모리 기반 저장소는 예제 실행에는 충분하지만 프로세스가 종료되면 상태가 사라집니다.

지속 상태: MemorySaver가 아니라 PostgresSaver가 필요한 이유

많은 튜토리얼은 InMemorySaver를 사용합니다. notebook 데모에는 간단하지만 승인 대기 중 프로세스가 재시작되면 메모리에 있던 그래프 상태가 사라집니다. 서비스 재배포가 있는 환경에서는 검토 중인 제안과 재개 위치를 데이터베이스에 보존해야 합니다.

제안 아키텍처에서는 온프레미스 Postgres와 PostgresSaver를 사용합니다. 처음 한 번 .setup()으로 checkpoint table을 만들고, 작업마다 고유한 thread_id를 부여해 중단 상태를 다시 찾습니다. 데이터베이스를 쓴다는 이유만으로 운영 요건이 충족되는 것은 아닙니다. 연결 정보는 secret store에서 읽고, 고가용성·백업 복구·보존 기간·승인자 접근 권한을 별도로 설계해야 합니다.

감사 로그와 승인 이력

Checkpoint는 실행 재개를 위한 상태이고 감사 로그는 사후 검증을 위한 기록이므로 같은 것으로 취급하면 안 됩니다. 쓰기 도구가 실행되기 전에 제안 내용과 결정 payload를 변조 방지 정책이 적용된 별도 로그 저장소에 남깁니다. 기록에는 제안된 tool call, 검토자 ID, 승인 또는 거부, timestamp, 전체 실행 이력과 연결되는 thread_id가 필요합니다.

이 예제에서 principal은 브라우저가 보낸 이름이 아니라 사내 SSO나 인증 미들웨어가 서버에서 확인한 주체입니다. 클라이언트가 principal_id를 임의로 넣지 못하게 해야 합니다. 또한 실제로 승인자와 timestamp가 보존된다고 말하려면 로그 수집 실패 시 쓰기 차단, 시간 동기화, 접근 통제, 보존 기간, 무결성 검증까지 시험해야 합니다.

재학습, 드리프트 감시, MLOps 루프

승인과 거부 이력은 평가 후보 데이터가 됩니다. 다만 모든 거부가 모델 오류는 아닙니다. 생산 우선순위, 승인자별 판단 차이, 화면 정보 부족도 원인이 될 수 있으므로 거부 사유를 분류한 뒤 tool별·shift별 추세를 봅니다. recipe family나 SOP가 바뀐 시점과 함께 보면 재평가가 필요한 구간을 찾을 수 있습니다.

격리 환경을 요구하는 팹이라면 MLOps loop도 승인된 내부 경계 안에 설계합니다. telemetry의 무반출은 배치 위치만으로 가정하지 않고 방화벽·DNS·packet capture와 공급망 검토로 확인하며, 외부 managed retraining service 사용 여부는 조직의 데이터 분류와 보안 승인을 따릅니다. 실용적인 내부 루프는 다음과 같습니다.

  • 관찰 — 도구별·shift별 거부율과 read-tool 검색 품질을 대시보드로 보고, 임계값을 넘으면 알림을 냅니다.
  • 수집 — 거부된 작업의 감사 로그와 엔지니어가 내린 올바른 결정을 모읍니다.
  • 갱신 — 먼저 RAG corpus를 다시 색인화합니다. 비용이 낮고 대부분의 drift를 고칠 수 있기 때문입니다. 검색만으로 부족할 때만 local model fine-tuning을 검토합니다.
  • 릴리스 게이트 — 새 model version도 상태 변경 작업처럼 다룹니다. shadow 평가를 거치고, 과거 승인 이력과 비교한 뒤, 승인 후 서비스에 투입합니다.

이 흐름은 온프레미스 구성만으로도 만들 수 있습니다. 다만 reject rate 하나로 성능을 판단하지 말고 승인 지연, 사유별 거부율, 잘못 승인된 작업, 실행 실패와 rollback 결과를 함께 봅니다.

승인 경로 장애 주입 체크리스트

특정 설비에 연결하지 않는 합성 tabletop입니다. 시험용 write adapter를 실제 변경 대신 DRY_RUN 이벤트만 남기도록 두고 아래 순서로 장애를 넣습니다.

  1. 승인 대기 중 애플리케이션을 재시작합니다. 같은 thread에서 제안값·근거 revision·승인자가 복원되는지 확인합니다.
  2. 만료된 approval token과 권한 없는 principal로 resume합니다. write adapter 호출은 0건이어야 합니다.
  3. 승인 직후 응답만 timeout으로 만듭니다. idempotency key로 재호출해 DRY_RUN 이벤트가 한 번만 남는지 봅니다.
  4. 감사 저장소를 일시적으로 사용할 수 없게 합니다. 기록 없이 실행하지 않고 전체 경로가 차단되는지 확인합니다.
  5. 거부 후 수정본을 다시 제출합니다. 이전 제안과 새 제안, 반려 사유가 하나의 이력으로 연결되는지 확인합니다.

중복 이벤트, 익명 승인, 사라진 반려 사유, 감사 실패 후 실행 중 하나라도 나오면 쓰기 release를 중단합니다. 이 체크리스트는 application 흐름만 다루며 PLC interlock, 설비 안전 로직, MES transaction rollback을 대신 검증하지 않습니다.

제안 구조의 검증 순서는 조회 도구, 쓰기 제안, 승인 중단, 영속 상태, 감사 이벤트, 거부·재개 시험입니다. 이 글의 예시만으로 실제 라인에 연결해서는 안 됩니다. MCP·MES 글에서 tool catalog와 권한 경계를 먼저 확인하지 않았다면 그 글로 돌아가 owner와 실패 상태를 채우고, 이후에만 이 승인 경로를 붙이세요. 실패 주입과 권한 시험, MES·설비 인터페이스별 rollback 검증을 마친 범위에서 담당 조직이 운영 여부를 승인해야 합니다.
