폐쇄망 에이전트의 사람 검증과 MLOps 운영

도구를 호출할 수 있는 에이전트는 공장을 바꿀 수 있습니다. 팹에서 도구 호출은 추상적인 개념이 아닙니다. 레시피 파라미터를 쓰고, MES에 lot hold를 걸고, 증착 장비의 정비 티켓을 열 수 있습니다. LLM이 이런 작업을 스스로 실행할 수 있는 순간, 질문은 더 이상 답변이 좋은지가 아니라 실행 전에 누가 승인했는지가 됩니다. 이 글의 운영 원칙은 단순합니다. 온프레미스 에이전트는 모든 상태 변경 작업 전에 사람 승인을 위해 멈춰야 하며, 그 멈춤은 프로세스 재시작 이후에도 유지되어야 합니다.

에이전트가 실행하기 전에 사람 승인 게이트가 필요한 이유

공개 인터넷에서 잘못된 에이전트 행동은 보통 낭비된 API 호출로 끝납니다. 하지만 폐쇄된 팹 네트워크 안에서는 영향 범위가 물리적입니다. LLM은 그럴듯하지만 틀린 setpoint를 만들 수 있고, 모호한 SOP를 잘못 읽을 수 있으며, 각각은 맞는 두 단계를 위험한 세 번째 단계로 연결할 수 있습니다. 이런 문제는 정확도 지표가 아니라 챔버를 아는 엔지니어가 잡아냅니다. 따라서 설계 목표는 더 똑똑한 모델만이 아닙니다. 실행 전에 사람이 전체 맥락을 보고 제안된 작업을 검토하는 신뢰할 수 있는 제어 지점이 필요합니다.

구체적으로 에이전트의 도구를 두 종류로 나눕니다.

  • 읽기 도구 — MES 조회, 레시피 읽기, 장비 로그 가져오기. 라인을 변경하지 않으므로 자유롭게 실행할 수 있습니다.
  • 쓰기 도구 — 파라미터 설정, lot hold, 작업 지시 발행. 반드시 승인 게이트를 통과해야 하며, 지정된 사람이 resume하기 전에는 실행되지 않습니다.
읽기 에이전트는 정확해야 합니다. 쓰기 에이전트는 멈출 수 있어야 하며, 모든 멈춤에는 사람의 이름이 붙어야 합니다.

로 만드는 승인 게이트interrupt()

LangGraph는 이 패턴에 필요한 primitive를 제공합니다. 노드 안에서 interrupt(payload)를 호출하면 런타임이 전체 graph state를 저장하고, payload를 호출자에게 보여준 뒤 실행을 멈춥니다. 이후 노드는 실행되지 않습니다. 그래프는 몇 분이 지나도, 서버가 재시작되어도, 누군가 Command(resume=...)로 재개하기 전까지 멈춘 상태로 남습니다. resume 값은 interrupt()의 반환값이 되므로 사람의 결정이 그대로 노드 로직으로 돌아옵니다.

운영 환경에서 특히 중요한 동작이 있습니다. resume하면 LangGraph는 노드를 interrupt() 다음 줄에서 이어서 실행하지 않고, 노드 처음부터 다시 실행합니다. 그래서 실제 side effect, 특히 실제 쓰기 작업은 반드시 interrupt가 반환된 뒤에 두어야 합니다. interrupt 위쪽 코드는 여러 번 실행되어도 안전해야 합니다.

interrupt를 쓰려면 graph compile 시 checkpointer가 필요합니다. checkpointer가 없으면 멈춘 상태를 저장할 곳이 없습니다. 이 checkpointer가 신뢰성의 핵심이고, 온프레미스 선택이 가장 중요해지는 지점입니다.

지속 상태: MemorySaver가 아니라 PostgresSaver가 필요한 이유

많은 튜토리얼은 InMemorySaver, 예전 이름으로 MemorySaver를 사용합니다. 체크포인트를 Python dictionary에 저장하므로 notebook 데모에는 좋지만 운영에는 맞지 않습니다. 승인 대기 중 프로세스가 재시작되면 멈춰 있던 그래프가 사라집니다. 엔지니어가 검토하던 hold도 사라지고, 그런 일이 있었다는 기록도 남지 않습니다. 24시간 돌아가고 뒤에서 서비스가 재배포되는 팹에서는 인메모리 상태가 곧 데이터 손실 버그가 됩니다.

온프레미스 Postgres와 함께 PostgresSaver를 사용합니다. 멈춘 상태는 이제 우리 네트워크 안의 데이터베이스 row가 됩니다. 재시작 후에도 살아 있고, 운영 데이터와 함께 백업되며, 화요일에 보류된 승인은 수요일 아침에도 그대로 남아 있습니다. 처음 한 번 .setup()을 실행해 checkpoint table을 만들고, thread_id를 키로 사용해 같은 대화를 정확히 멈춘 지점에서 다시 불러옵니다.

감사 로그와 승인 이력

모든 pause가 durable checkpoint가 되면 별도의 control plane 없이도 감사 이력을 만들 수 있습니다. checkpoint는 제안된 작업과 state를 기록하고, resume payload는 누가 어떤 결정을 내렸는지 기록합니다. write tool이 실행되기 전에 resume 순간 append-only log에 둘 다 남기세요. 규제 대상 라인이라면 각 기록에는 제안된 tool call, 검토자 ID, 승인 또는 거부, timestamp, 전체 에이전트 transcript와 연결되는 thread_id가 들어가야 합니다.

이제 라인의 모든 쓰기 작업 뒤에는 이름과 timestamp가 남고, 그 기록은 작업 자체와 같은 네트워크 안에 저장됩니다. 감사자나 불량 lot 이후의 post-mortem이 실제로 필요로 하는 기록이 바로 이것입니다.

재학습, 드리프트 감시, MLOps 루프

승인 게이트는 가장 좋은 데이터 소스이기도 합니다. 모든 거부는 사람이 잡아낸 모델 오류의 labeled example입니다. tool별, shift별로 reject rate를 시간에 따라 추적하세요. 거부율 상승은 에이전트가 실제 라인 운영 방식에서 벗어나기 시작했다는 가장 이른 신호일 때가 많습니다. 새 레시피 family, 바뀐 SOP, 재구성된 chamber가 원인일 수 있습니다.

팹은 air-gapped 환경이므로 MLOps loop도 전부 온프레미스에 남아야 합니다. telemetry는 네트워크 밖으로 나가지 않고, managed retraining service도 필요하지 않습니다. 실용적인 루프는 다음과 같습니다.

  • 관찰 — 도구별·shift별 거부율과 read-tool 검색 품질을 대시보드로 보고, 임계값을 넘으면 알림을 냅니다.
  • 수집 — 거부된 작업의 감사 로그와 엔지니어가 내린 올바른 결정을 모읍니다.
  • 갱신 — 먼저 RAG corpus를 다시 색인화합니다. 비용이 낮고 대부분의 drift를 고칠 수 있기 때문입니다. 검색만으로 부족할 때만 local model fine-tuning을 검토합니다.
  • 릴리스 게이트 — 새 model version도 상태 변경 작업처럼 다룹니다. shadow 평가를 거치고, 과거 승인 이력과 비교한 뒤, 승인 후 서비스에 투입합니다.

이 흐름에 cloud MLOps 플랫폼은 필요하지 않습니다. 필요한 것은 durable checkpoint store, 솔직한 audit log, 그리고 현장의 사람이 이미 이해하는 metric인 reject rate입니다.

패턴은 작지만 핵심은 분명합니다. 읽기는 자유롭게, 쓰기는 게이트 뒤에서, pause는 Postgres에 저장하고, 누가 결정했는지 기록합니다. 이렇게 만든 에이전트는 정확하기만 한 시스템이 아닙니다. 책임 추적이 가능하고, 재시작에도 안전하며, 실제 라인 가까이에 둘 수 있을 만큼 통제 가능한 시스템입니다. 아래 심화 deck은 이 글이 전제로 삼는 tool wiring까지 포함해 전체 agent-and-MCP 구성을 slide by slide로 설명합니다.
