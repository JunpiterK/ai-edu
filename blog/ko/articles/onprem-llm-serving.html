팹 네트워크를 떠나지 않는 모델은 여전히 ​​어딘가에서 실행되어야 합니다. 제조 작업 현장에서는 어떤 개방형 모델이 가장 똑똑한지 제약이 거의 없습니다. 해당 모델을 에어 갭 뒤의 GPU 상자에 배치하는 방법입니다. 실제 동시성을 구현하고 그것이 건강하다는 것을 증명합니다. 이것은 귀하가 사용하는 두 서버에 대한 현장 가이드입니다. 실제로는 — vLLM 그리고 올라마 — 그리고 모델이 카드에 맞는지 여부를 결정하는 메모리 연산입니다. 작은 망(air-gapped)으로 오픈 모델을 GPU 서버에 작용할 전 이야기입니다.

vLLM 또는 Ollama?

두 도구는 서로 다른 규모에서 동일한 문제를 해결합니다. 올라마 은 짧은 경로: 하나의 바이너리, GGUF 양자화된 가중치를 로컬 저장소로 가져오고 서버를 가동합니다. 초 안에. 단일 워크스테이션 GPU 또는 심지어 CPU에서도 원활하게 실행되며 다음과 같은 작업에 이상적입니다. 파일럿, 벤치 테스트 또는 하나의 공유 상자를 치는 소규모 팀 등이 있습니다. vLLM 은 처리량 엔진: 페이징 어텐션 KV 캐싱 및 연속 일괄 처리를 사용하여 많은 서비스를 제공합니다. 데이터센터 GPU에 대한 동시 요청을 수행하며 OpenAI 호환 HTTP API를 외부로 노출합니다. 상자. 파일럿·단일 GPU는 Ollama, 다수 요청·처리량은 vLLM입니다.

팹에 대한 유용한 경험 법칙: 소수 미만의 엔지니어가 대화형으로 모델링하려면 Ollama부터 시작하세요. RAG 서비스, 에이전트 또는 수십 개가 있는 순간 분석가가 동일한 서버를 공유하면 요청 대기열이 가장 먼저 중단됩니다. vLLM의 지속적인 일괄 처리가 무엇을 위해 만들어졌는지.

오늘 답변을 얻으려면 Ollama를 선택하세요. 모델이 아닌 대기열에 병목 현상이 발생하면 vLLM으로 이동하세요. — 폐쇄망 서빙 원칙

Ollama는 모델을 지역 Blob 저장소로 가져와서 제공합니다. 공기가 차단된 호스트에서는 스테이징 머신을 한 번 당기고 모델 디렉토리를 복사한 다음 OLLAMA_MODELS at it — 추론 시 출구가 없습니다.

vLLM은 단일 명령으로 Hugging Face 모델 디렉터리를 제공합니다. 온프레미스에 가중치를 적용하고, 그런 다음 시작하십시오 - --gpu-memory-utilization KV 캐시가 요구할 수 있는 VRAM의 양을 제한합니다. 그리고 --max-model-len 캐시 수학이 예측 가능하도록 컨텍스트 창의 경계를 지정합니다. --tensor-parallel-size 하나의 대형 모델을 여러 GPU에 걸쳐 분할합니다.

하나의 OpenAI-compatible endpoint

두 서버의 보상은 동일합니다. OpenAI 호환 HTTP 표면 /v1. RAG 서비스, 에이전트 및 노트북은 모두 하나의 프로토콜을 사용하므로 애플리케이션 코드를 건드리지 않고도 Ollama를 vLLM으로 교체하거나 한 모델을 다른 모델로 교체할 수 있습니다. LangChain에서 이는 단일 클라이언트 클래스를 의미합니다. vLLM의 경우 다음을 사용합니다. ChatOpenAI ~에서 langchain_openai 온프레미스를 지적했습니다 base_url; API 키는 네트워크를 떠나는 것이 없기 때문에 자리 표시자입니다.

Ollama는 자체적으로 최고 수준의 LangChain 통합을 보유하고 있습니다. ChatOllama ~에서 langchain_ollama — 기본 Ollama API를 사용합니다. 도구 호출 및 num_ctx 컨텍스트 창의 손잡이입니다. 같은 세 줄, 다른 호스트.

GPU memory sizing

무엇이든 다운로드하기 전에 산술을 수행하십시오. VRAM 수요는 두 부분으로 나뉩니다. 무게, 고정되어 있으며 KV 캐시, 이는 컨텍스트 길이 x 동시성. 가중치에 대한 유용한 봉투 뒷면 추정치:

  • 가중치 ≒ 매개변수 × 매개변수당 바이트. FP16은 2바이트이고, 8비트는 ~1바이트이고, 4비트는 ~0.5바이트입니다. 따라서 8B 모델은 대략 8 × 2 = 16 GB FP16에서, ~8 GB 8비트에서, 그리고 ~4 GB 4비트에서. 배열 수 × 복잡한 바이트.
  • KV 캐시 스윙 팩터입니다. 모델 레이어, 컨텍스트에 따라 확장됩니다. 길이, 진행 중인 요청 수. 이를 위해 카드 VRAM의 20~40%를 추가로 확보하세요. 그리고 그것을 양자화(--kv-cache-dtype fp8 vLLM에서) 해당 설치 공간을 대략 절반으로 줄입니다. 더 긴 컨텍스트나 더 많은 동시성이 필요할 때.
  • 간접비. CUDA 컨텍스트, 활성화 및 단편화 – 이것이 바로 이유입니다 --gpu-memory-utilization 0.90 정상적인 기본값입니다. 아니 1.0.

실제 예: 4비트의 70B 모델은 약 70 × 0.5 = 35 GB 무게의. 24GB 카드 한 장은 맞지 않고, 적당한 맥락에서는 48GB 카드 한 장이 딱 맞습니다. 두 개의 GPU를 원합니다(--tensor-parallel-size 2) 실제 KV 헤드룸을 추가하면. 양자화 계층은 모델을 "로드되지 않음"에서 "제공할 공간에 적합"으로 이동하는 레버입니다.

Batching과 concurrency

단일 요청은 최신 GPU에 거의 세금을 부과하지 않습니다. 공유 서비스는 처리 방법에 따라 생존하거나 죽습니다. 는 둘째, 열번째, 오십번째 요청이 중간 세대에 도착합니다. 이곳은 둘이서 도구가 급격히 갈라집니다. Ollama는 기본적으로 요청을 대부분 순차적으로 처리합니다. 소수의 사용자, 부하가 걸리는 대기열 증가. vLLM 지속적인 일괄 처리 인터리브 실행 중인 배치 토큰에 대한 새 요청을 토큰별로 처리하므로 처리량이 동시성에 따라 확장됩니다. 그 밑으로 무너지는 것.

vLLM 측에서는 두 가지 레버가 중요합니다. --max-num-seqs 실행되는 시퀀스 수를 제한합니다. 동시에 그리고 --max-model-len 시퀀스별 컨텍스트 경계. 동시성 높이기 처리량을 위해 KV 캐시 메모리를 교환합니다. 이것이 바로 위의 크기 산술이 나오는 이유입니다. 먼저. 클라이언트의 부탁 llm.batch([...]) Python 루프를 통해 요청이 서버를 함께 사용하면 Batcher가 이를 포장할 수 있습니다.

Health check와 운영

에어 갭(air-gapped) 서비스에는 이를 감시하는 공급업체 대시보드가 ​​없습니다. 즉, 귀하는 활성 상태를 소유합니다. 두 서버 모두 제공 값싼 신호: vLLM은 노출합니다. /health (활성), /v1/models (이것은 모델이 예상대로 로드되고 이름이 지정됨) 및 Prometheus /metrics 대기열 깊이 및 대기시간; 올라마는 아무렇지 않게 대답한다. GET / 로드된 모델을 나열합니다. /api/tags. 가장 간단한 것을 오케스트레이터의 준비 상태 프로브에 연결하세요.

선택한 모델은 주변 규율보다 중요하지 않습니다. 즉, 오프라인에서 가중치를 단계화하고 OpenAI 호환 엔드포인트가 앞에 있고, 매개변수-시간-바이트 규칙을 사용하여 카드 크기를 조정합니다. 커밋 및 조사 /health 당신이 말하는 것처럼. 워크플로을 증명하기 위해 Ollama에서 시작, 졸업 동시성이 제약인 경우 vLLM에 연결합니다. 두 경우 모두 지식이 네트워크를 떠나지 않습니다. 온프렘 서빙은 모델 선택보다 운영 규율이 핵심입니다.
