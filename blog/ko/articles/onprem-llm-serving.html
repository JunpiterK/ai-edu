합성 사례: 교대 전 부하 시험

현장에서 자주 보는 장면을 합쳐 예를 들면, 파일럿 서버의 채팅 화면에서는 첫 질문이 바로 끝나지만 네 명이 긴 SOP를 동시에 넣자 두 요청의 진행 표시가 멈춥니다. GPU 대시보드는 메모리 상한에 붙고 server log에는 새 요청의 queue time이 길어진 흔적이 남습니다.

처음에는 모델 자체가 느리다고 판단하기 쉽습니다. 그러나 짧은 입력 한 건은 정상이고 긴 입력이 겹칠 때만 지연이 커진다면, 먼저 볼 대상은 모델 순위가 아니라 KV cache, context 상한, batching 설정입니다. 같은 weight라도 입력·출력 길이와 동시 요청 수가 달라지면 병목 위치가 달라집니다.

이 사례는 특정 사업장의 측정값이 아닙니다. 기술 권고는 동일한 prompt trace로 동시성을 단계적으로 올리고 첫 토큰 지연, 전체 시간, 처리량, peak VRAM, 오류를 함께 기록하라는 것입니다. 한 번 답했다는 화면만으로 운영 준비를 판정할 수 없습니다.

작은 부하 실습

대표 프롬프트를 짧은 입력과 긴 입력으로 나누고 동시 요청을 1, 2, 4, 8처럼 올려 보세요. 각 구간에서 첫 토큰 지연, 전체 응답 시간, 처리량, GPU 메모리, 오류율을 함께 기록하면 증설과 설정 변경의 근거가 남습니다. 숫자 구간은 초기 예시값이며 실제 모델과 GPU에 맞춰 다시 정해야 합니다.

팹 네트워크를 떠나지 않는 모델도 결국 어딘가에서 안정적으로 실행되어야 합니다. 제조 현장에서 중요한 질문은 "어떤 오픈 모델이 가장 똑똑한가"에서 끝나지 않습니다. 그 모델을 폐쇄망 안의 GPU 서버에 어떻게 올릴지, 실제 동시 요청을 어떻게 받을지, 서비스가 살아 있다는 증거를 어떻게 남길지가 더 중요해집니다. 이 글은 현장에서 자주 쓰는 두 경로, vLLM과 Ollama, 그리고 모델이 GPU에 맞는지 판단하는 메모리 계산을 다룹니다.

vLLM 또는 Ollama?

두 도구는 서로 다른 규모에서 같은 문제를 풉니다. Ollama는 짧은 경로입니다. 하나의 daemon으로 GGUF 양자화 모델을 로컬 저장소에서 읽고, 비교적 쉽게 OpenAI 호환 API를 열 수 있습니다. 단일 워크스테이션 GPU, 작은 파일럿, 소수 엔지니어가 쓰는 공용 실험 서버에 잘 맞습니다. vLLM은 처리량 엔진에 가깝습니다. paged attention, KV cache 관리, continuous batching으로 여러 동시 요청을 더 안정적으로 처리하고, 데이터센터 GPU에서 OpenAI 호환 HTTP API를 제공합니다.

도구 선택은 사용자 수만으로 정하기 어렵습니다. 단일 GPU의 소규모 파일럿에서는 Ollama가 운영 항목을 줄여주지만, 긴 RAG 입력이나 배치 작업이 겹치면 소수 사용자 환경에서도 대기열이 생길 수 있습니다. 반대로 짧은 요청이 드문 서비스라면 vLLM의 처리량 이점이 크지 않을 수 있습니다. 동일한 모델·GPU·요청 묶음으로 두 서버를 시험한 뒤 p95 지연과 오류율을 비교하는 편이 정확합니다.

빠른 파일럿에는 구성 단순성을, 공유 서비스에는 부하 시험 결과를 우선합니다. — 서빙 선택 기준

이 주제의 전체 과정 보기

Ollama는 모델을 로컬 blob 저장소에 두고 제공합니다. 폐쇄망에서는 인터넷이 되는 staging machine에서 모델을 한 번 받은 뒤, 검증한 모델 디렉터리를 내부 서버로 옮기고 OLLAMA_MODELS가 그 경로를 보게 만듭니다. 추론 시점에는 외부로 나갈 이유가 없어야 합니다.

vLLM으로 처리량 확보하기

vLLM은 로컬에 반입한 Hugging Face 형식의 모델 디렉터리를 하나의 명령으로 서빙할 수 있습니다. --gpu-memory-utilization은 KV cache와 런타임이 사용할 수 있는 VRAM 상한을 잡고, --max-model-len은 컨텍스트 길이를 제한해 메모리 계산을 예측 가능하게 만듭니다. 여러 GPU에 모델을 나눠 올릴 때는 --tensor-parallel-size를 사용합니다.

하나의 OpenAI-compatible endpoint

두 서버를 쓰는 가장 큰 보상은 OpenAI 호환 HTTP 표면, 즉 /v1입니다. RAG 서비스, agent, notebook이 같은 프로토콜을 쓰면 애플리케이션 코드를 크게 바꾸지 않고 Ollama에서 vLLM으로, 또는 한 모델에서 다른 모델로 교체할 수 있습니다. LangChain에서는 vLLM을 langchain_openai의 ChatOpenAI로 호출하고, base_url만 내부 서버 주소로 지정하면 됩니다. API key는 요청 인증의 한 요소일 뿐입니다. 0.0.0.0에 bind한 서비스는 방화벽·망 분리·reverse proxy·TLS와 함께 접근 범위를 제한해야 합니다.

Ollama는 langchain_ollama의 ChatOllama 통합을 쓰면 됩니다. 기본 Ollama API를 사용하며, num_ctx로 컨텍스트 창을 조정할 수 있습니다. 호출 방식은 비슷하고, 달라지는 것은 내부 host와 모델 이름입니다.

GPU memory sizing

모델을 반입하기 전에 VRAM 예산부터 계산합니다. 수요는 고정 비용인 모델 가중치, 컨텍스트 길이와 동시 요청에 따라 변하는 KV cache, 런타임 오버헤드로 나뉩니다. 아래 계산은 후보를 거르는 거친 추정이며 실제 할당량은 엔진과 모델 구조에 따라 달라집니다.

  • 가중치 ≒ 파라미터 수 × 파라미터당 바이트. FP16은 약 2바이트, 8비트는 약 1바이트, 4비트는 약 0.5바이트입니다. 따라서 8B 모델은 FP16에서 약 8 × 2 = 16 GB, 8비트에서 약 8 GB, 4비트에서 약 4 GB가 필요합니다.
  • KV cache는 모델 layer, 컨텍스트 길이, 동시에 처리 중인 요청 수에 따라 커집니다. 카드 VRAM의 20~40%를 여유로 두는 방식은 초기 예산 예시일 뿐 고정 규칙이 아닙니다. 실제 모델의 cache 규격과 부하 시험 중 피크 할당량으로 보정하고, 필요하면 vLLM의 --kv-cache-dtype fp8 같은 옵션을 검토합니다.
  • 런타임 오버헤드도 남겨야 합니다. CUDA context, activation, 메모리 단편화가 있기 때문에 --gpu-memory-utilization 0.90 같은 값이 보통 더 안전합니다. 1.0은 실서비스에서 너무 공격적입니다.

예를 들어 70B 모델을 4비트로 올려도 가중치만 대략 70 × 0.5 = 35 GB입니다. 24GB GPU 한 장에는 맞지 않고, 48GB 한 장도 긴 컨텍스트와 동시 요청까지 고려하면 빠듯할 수 있습니다. 이때는 더 작은 모델, 더 짧은 컨텍스트, 더 낮은 동시성, 또는 --tensor-parallel-size 2 같은 다중 GPU 구성을 검토해야 합니다. 양자화는 만능이 아니라 선택지를 넓히는 레버입니다.

Batching과 concurrency

공유 서비스의 시험은 생성 중인 요청 사이로 새 요청이 들어올 때 시작됩니다. 설명을 위해 단순화한 초기 부하 예시로 동시성 1·2·4·8, 입력 1K·4K·8K token을 조합하고, 각 조건을 예열 뒤 반복 실행할 수 있습니다. 모델, GPU, 양자화, 출력 길이가 바뀌면 결과도 바뀌므로 이 숫자를 용량 보증으로 사용해서는 안 됩니다. Ollama와 vLLM도 같은 요청 trace로 비교해야 합니다.

vLLM에서는 두 가지 레버가 중요합니다. --max-num-seqs는 동시에 처리할 sequence 수를 제한하고, --max-model-len은 sequence별 컨텍스트 상한을 정합니다. 동시성을 높이면 처리량은 좋아질 수 있지만 KV cache 메모리를 더 씁니다. 그래서 sizing 계산이 먼저입니다. 클라이언트에서도 개별 Python loop보다 llm.batch([...])처럼 묶어서 보내면 서버 batcher가 더 잘 일할 수 있습니다.

Health check와 운영

폐쇄망 서비스에는 공급업체 대시보드가 자동으로 붙어 있지 않습니다. 살아 있는지, 어떤 모델이 올라와 있는지, 대기열과 지연 시간이 어떤지는 우리가 직접 봐야 합니다. vLLM은 /health, /v1/models, Prometheus /metrics를 제공합니다. Ollama는 GET /로 daemon 응답을, /api/tags로 설치된 모델을, /api/ps로 현재 메모리에 올라온 모델을 구분해 확인합니다. 최소한의 endpoint라도 orchestrator readiness probe와 운영 대시보드에 연결하세요.

운영 판정에는 모델 이름보다 시험 조건표가 더 오래 남습니다. 가중치를 오프라인으로 반입하고, OpenAI 호환 endpoint를 열고, VRAM 계산으로 후보를 줄인 다음 대표 요청으로 부하를 걸어야 합니다. /health가 정상이어도 p95 지연, 대기열, OOM, 재시작 후 로딩 시간이 기준을 넘으면 아직 운영 준비가 끝난 것이 아닙니다. 지식과 로그의 네트워크 경계는 별도 보안 시험으로 확인합니다.

서버의 실패 조건을 잡았다면 다음에는 답변이 어느 문서에서 왔는지 검증해야 합니다. 폐쇄망 문서 RAG 글은 이 endpoint 위에 SOP와 설비 로그를 연결하고, 인용 문서 ID와 개정을 평가하는 단계로 이어집니다.
