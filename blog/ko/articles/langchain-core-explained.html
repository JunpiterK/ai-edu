LangChain core 설명: LLM, prompt, LCEL pipeline

형식 오류를 따라간 기록

설명을 위해 단순화한 가상 사례입니다. CVD-07 알람 문자열을 받아 위험도·계통·다음 확인 항목을 만드는 작은 분류기를 생각해 보겠습니다. 로컬 7B 모델에서는 동작했지만 내부 API 모델로 바꾸자 parser가 간헐적으로 실패했습니다. 모델 성능 저하로 보였으나 로그를 비교하니 한쪽은 AIMessage, 다른 쪽은 이미 문자열로 변환된 JSON을 넘기고 있었습니다. 입력과 출력 형식을 단계별로 기록한 뒤 prompt, model, parser의 경계를 분리해 문제를 고쳤습니다. 다만 공통 Runnable 인터페이스가 모델별 tool calling이나 structured output 지원 차이까지 없애 주지는 않습니다.

기술 권고. prompt | model | parser를 붙이기 전에 각 단계를 한 번씩 invoke하고 반환 타입과 샘플 값을 남깁니다. 모델 교체 뒤 오류가 생기면 어느 경계의 계약이 달라졌는지 바로 확인할 수 있습니다. 이 권고는 위 합성 사례에서 끌어낸 일반 진단 절차이며 특정 회사의 검증 규정은 아닙니다.

LangChain core는 prompt, model, parser, retriever, tool을 공통 방식으로 호출하고 조합하는 인터페이스 묶음입니다. 각 단계가 무엇을 받고 무엇을 돌려주는지 알면 로컬 LLM 애플리케이션을 데이터가 흐르는 파이프라인으로 읽을 수 있습니다.

LangChain이 실제로 해결하는 것

현장에서 필요한 것은 구성요소를 통제된 범위에서 교체하는 능력입니다. 검색 로직은 유지한 채 prompt를 수정하거나, 문자열 parser를 typed object로 바꾸거나, 체인 전체를 다시 쓰지 않고 streaming을 붙여야 할 때 공통 인터페이스가 역할을 합니다.

온프레미스 환경에서는 모델의 위치가 자주 달라집니다. 개인 GPU의 7B 모델로 시작해 더 큰 양자화 모델로 옮기거나, 운영 단계에서 내부 API 뒤에 배치할 수 있습니다. endpoint 변경이 prompt와 후처리 코드까지 연쇄 수정하게 만들면 검증 범위가 불필요하게 커집니다. LangChain core는 이 결합을 줄이지만, 교체 후 회귀 평가는 여전히 필요합니다.

LangChain의 가치는 특정 클래스 하나가 아니라 공통 인터페이스에 있습니다. 프롬프트, 모델, 파서, 리트리버를 같은 방식으로 이어 붙일 수 있기 때문에 유닉스 파이프처럼 조합하고 교체할 수 있습니다. — LCEL의 요점

ChatModel: ChatOllama로 on-prem model 연결하기

ChatModel은 모델 호출을 감싸는 추상화입니다. 로컬 실습에서는 ChatOllama로 Ollama 서버에 연결하고, 모델 이름·temperature·생성 토큰 수 등을 설정할 수 있습니다. 이후 invoke, stream, batch를 같은 Runnable 규격으로 호출합니다.

모델은 단순 문자열보다 역할이 구분된 message 목록을 입력으로 받는다고 이해하면 됩니다. system message에는 정책과 역할, human message에는 현재 요청, AI message에는 이전 응답이 들어갑니다. LangChain은 provider별 JSON payload를 애플리케이션 곳곳에서 직접 만들지 않도록 이 구조를 유지합니다.

Prompt template은 실행 시점의 변수를 message로 변환합니다. ChatPromptTemplate.from_messages로 system instruction을 한 번 정의하고 장비명, 검색 문맥, 사용자 질문 같은 필드를 주입합니다. dictionary를 넣으면 모델에 보낼 prompt value가 나오는 Runnable이 됩니다.

  • invoke(x) — 하나의 입력, 하나의 전체 답변. 완료될 때까지 차단합니다.
  • stream(x) — 라이브 UI에 토큰을 조금씩 흘려보낼 때 씁니다.
  • batch([x1, x2, …]) — 여러 입력을 한 번에 처리합니다. 로그 묶음을 분류하거나 여러 질문을 동시에 평가할 때 좋습니다.

Prompt에는 역할을 섞지 않는 편이 좋습니다. 변하지 않는 동작 규칙은 system message에, 현재 요청은 human message에, 검색된 근거는 이름이 드러나는 field에 둡니다. 오답이 나왔을 때 모델, prompt, 검색 근거 중 어느 입력이 달라졌는지 비교하기 쉬워집니다.

Prompt: ChatPromptTemplate.from_messages

Model은 message를 반환합니다. 하지만 application은 string, JSON, Pydantic object, list, routing label이 필요할 때가 많습니다. Output parser는 느슨한 model output을 프로그램의 나머지 부분이 기대하는 shape로 바꿉니다. StrOutputParser가 가장 단순하고, format이 중요하면 structured parser가 다음 단계입니다.

블로그 작업에서는 parser가 title, summary, slug, tags를 반환할 수 있습니다. FAB 분류 작업이라면 risk_level, reason, required_approval 같은 필드가 필요합니다. 뒤쪽 코드는 자연스러운 문장보다 검증 가능한 계약에 의존해야 하며, 필수 필드가 없으면 조용히 기본값을 쓰지 말고 실패로 처리하는 편이 안전합니다.

LCEL, 즉 LangChain Expression Language는 prompt | model | parser 형태의 pipe syntax입니다. 각 piece는 Runnable입니다. 그래서 전체 chain도 invoke, batch, stream, composition이라는 공통 interface를 갖습니다.

Output parser: 느슨한 text에서 typed object까지

LCEL 문법의 실용적인 장점은 체인의 각 단계를 따로 관찰하고 교체하고 시험할 수 있다는 점입니다. parser만 바꿔도 prompt와 모델 호출부는 같은 규격으로 남습니다.

RunnableParallel과 RunnablePassthrough는 단순 체인에 검색 문맥을 더할 때 사용합니다. Passthrough는 사용자 질문을 그대로 보존하고, Parallel은 같은 입력에서 질문과 검색 context 같은 여러 필드를 만듭니다. prompt는 이렇게 정리된 dictionary를 받습니다.

전형적인 RAG chain은 이렇게 생겼습니다. question을 받고, document를 retrieve해서 context로 만들고, 원래 question을 유지하고, 둘을 prompt에 넣고, model을 호출하고, answer를 parse합니다. 특별한 RAG 마법은 없습니다. retrieval branch가 하나 추가된 Runnable composition입니다.

LCEL과 Runnable: pipeline을 묶는 pipe

검색·formatting·prompt·모델 호출·parse·logging을 한 함수에 넣으면 알람 분류가 틀렸을 때 어느 단계부터 재현해야 할지 알기 어렵습니다. LCEL에서는 입력과 출력이 분명한 작은 step으로 나누고, 각 step에 같은 테스트 입력을 다시 넣을 수 있습니다.

Chain에 conditional routing, retry, human approval, durable state가 필요해지면 LangChain core에서 LangGraph로 넘어갈 때입니다. Core는 composable step을 주고, LangGraph는 다음에 어떤 step이 실행될지 결정하는 state machine을 줍니다.

RunnableParallel과 RunnablePassthrough로 context 주입하기

학습과 구축은 plain chat chain에서 시작해 structured output, 작은 RAG, 읽기 전용 tool, router graph, 승인 workflow 순으로 넓히면 원인을 추적하기 쉽습니다. 새 계층은 기능 이름이 멋져서가 아니라 기존 체인이 처리하지 못한 입력이 확인됐을 때 추가하세요.

  • RunnableParallel — 같은 입력에서 여러 Runnable을 동시에 실행하고 결과 dictionary를 반환합니다. 체인의 Y자 분기라고 보면 됩니다.
  • RunnablePassthrough — 입력을 바꾸지 않고 그대로 넘깁니다. 덕분에 원래 질문이 새로 검색된 context와 함께 살아남습니다.

15분 경계 타입 추적 실습

위 체인을 한 번에 실행하기 전에 같은 입력을 세 단계로 나눠 호출해 보세요. 아래 결과는 특정 모델의 보장값이 아니라 무엇을 기록할지 보여주는 예시입니다.

  1. prompt.invoke({"language": "Korean", "context": "SOP-CVD-07: purge 30s before vent.", "question": "How long is the purge?"})의 반환 클래스와 최종 message를 기록합니다.
  2. 그 결과를 model.invoke(...)에 넣고 반환 클래스, content 타입, 빈 응답 여부를 확인합니다.
  3. 마지막으로 parser에 모델 결과를 넣고 애플리케이션이 요구하는 문자열 또는 typed object인지 검사합니다.

판정 예. 구조화 출력 예제에서 prompt와 model은 통과했는데 parser가 AlarmTriage의 필수 필드인 severity, subsystem, action 중 하나를 만들지 못하면 parser 계약 실패입니다. model 호출 자체가 timeout이면 출력 형식 문제로 분류하지 않습니다. 세 번씩 반복했을 때 타입이나 필수 필드가 흔들리면 LCEL로 묶기 전에 멈추고 schema·model 설정을 고칩니다.

이 실습은 경계 오류를 찾는 데 유용하지만 답의 공정 타당성은 판정하지 못합니다. 타입이 모두 맞아도 알람 해석은 틀릴 수 있으므로 별도의 도메인 평가셋이 필요합니다.

ChatModel, PromptTemplate, parser, Runnable, LCEL은 이후 RAG와 tool, MCP, agent가 공유하는 실행 단위입니다. 경계 타입이 안정되었다면 다음에는 RAG 구성요소 글에서 문서가 context로 들어오는 경로를 점검하세요. 분기와 승인 대기가 생긴 뒤에야 LangGraph의 state로 넘어가는 편이 고장 위치를 보존하기 쉽습니다.

다음 단계: chain에서 agent로

이 글은 현장 노트입니다. 연결된 인터랙티브 deck에서는 local LLM과 RAG stack을 diagram과 구현 예제로 단계별로 설명합니다.

물리학 박사(IGZO / graphene 반도체 물성)이며 증착 공정을 다루는 반도체 공정 엔지니어입니다. 팹을 위한 air-gapped LLM, RAG, agent 시스템과 강화학습·이상탐지 도구를 구축합니다. 이곳에서는 공정 know-how를 네트워크 밖으로 내보내지 않고 온프레미스 지식 자산으로 바꾸는 방법을 씁니다.
