LangChain core 설명: LLM, prompt, LCEL pipeline

LangChain core는 LLM 주변의 마법이 아닙니다. prompt, model, parser, retriever, tool을 같은 방식으로 조합하게 해주는 작은 interface 집합입니다. 이 interface를 이해하면 local LLM app은 callback 더미가 아니라 data pipeline처럼 보이기 시작합니다.

LangChain이 실제로 해결하는 것

LangChain이 해결하는 실전 문제는 교체 가능성입니다. local Ollama model을 다른 provider로 바꾸고, retrieval을 건드리지 않고 prompt를 바꾸고, string parser를 JSON parser로 교체하고, chain 전체를 다시 쓰지 않고 stream을 붙이고 싶을 때 core abstraction이 필요합니다.

On-prem 작업에서는 이것이 중요합니다. prototype은 gaming GPU의 7B model에서 시작했다가 더 큰 quantized model로 옮겨가고, 나중에는 internal API 뒤에서 돌 수 있습니다. model endpoint가 바뀔 때마다 application code가 무너지면 안 됩니다.

LangChain의 가치는 어느 하나의 클래스가 아닙니다. 프롬프트, 모델, 파서와 리트리버는 모두 동일한 인터페이스를 사용하므로 구성합니다. 유닉스 파이프처럼요. — LCEL의 요점

ChatModel: ChatOllama로 on-prem model 연결하기

ChatModel은 model-facing abstraction입니다. local lab에서는 ChatOllama가 가장 단순한 출발점입니다. Ollama server와 통신하고 chat message를 반환합니다. model name, temperature, context length 등을 설정한 뒤 같은 Runnable interface로 invoke, stream, batch를 호출합니다.

핵심 mental model은 model이 raw string이 아니라 message를 받는다는 점입니다. system message는 policy와 role을 담고, human message는 user task를 담고, AI message는 이전 응답입니다. LangChain은 provider마다 JSON payload를 직접 만드는 대신 이 구조를 명시적으로 유지합니다.

Prompt template은 variable을 message로 바꿉니다. ChatPromptTemplate.from_messages가 핵심 도구인 이유는 system instruction을 한 번 쓰고 runtime에 user-specific field를 주입할 수 있기 때문입니다. 결과는 reusable Runnable입니다. dictionary를 넣으면 prompt value가 나옵니다.

  • invoke(x) — 하나의 입력, 하나의 전체 답변. 완료될 때까지 차단합니다.
  • stream(x) — 라이브 UI에 대해 생성된 토큰을 생성합니다.
  • batch([x1, x2, …]) — 한 번에 많은 입력이 내장되어 있습니다. 동시성. 로그 스택에 레이블을 지정하는 데 이상적입니다.

Prompt는 지루하고 test 가능하게 유지하세요. 안정적인 behavior는 system message에, user content는 human message에, retrieved context는 명확한 field에 넣습니다. 그래야 잘못된 답이 model 때문인지, prompt 때문인지, retrieved evidence 때문인지 구분하기 쉽습니다.

Prompt: ChatPromptTemplate.from_messages

Model은 message를 반환합니다. 하지만 application은 string, JSON, Pydantic object, list, routing label이 필요할 때가 많습니다. Output parser는 느슨한 model output을 program의 나머지 부분이 기대하는 shape로 바꿉니다. StrOutputParser가 가장 단순하고, format이 중요하면 structured parser가 다음 단계입니다.

Blog workflow라면 parser가 title, summary, slug, tags를 반환할 수 있습니다. Fab workflow라면 risk_level, reason, required_approval을 반환할 수 있습니다. 핵심은 같습니다. downstream code는 그럴듯한 paragraph가 아니라 contract에 의존해야 합니다.

LCEL, LangChain Expression Language는 prompt | model | parser 형태의 pipe syntax입니다. 각 piece는 Runnable입니다. 그래서 전체 chain은 invoke, batch, stream, composition이라는 공통 interface를 갖습니다.

Output parser: 느슨한 text에서 typed object까지

이 syntax를 배울 가치가 있는 이유는 단순히 예쁜 function composition이 아니기 때문입니다. chain을 inspect, swap, test하기 쉬워집니다. 한 piece가 바뀌어도 주변 pipeline은 대체로 그대로 유지됩니다.

RunnableParallel과 RunnablePassthrough는 simple chain에서 RAG-style chain으로 넘어가는 다리입니다. RunnablePassthrough는 user question을 그대로 유지합니다. RunnableParallel은 question과 context처럼 여러 field를 동시에 계산합니다. 그러면 prompt는 정돈된 dictionary를 받습니다.

Typical RAG chain은 이렇게 생겼습니다. question을 받고, document를 retrieve해서 context로 만들고, 원래 question을 유지하고, 둘을 prompt에 넣고, model을 호출하고, answer를 parse합니다. 특별한 RAG 마법은 없습니다. retrieval branch가 하나 추가된 Runnable composition입니다.

LCEL과 Runnable: pipeline을 묶는 pipe

흔한 실수는 하나의 function 안에 너무 많은 것을 숨기는 것입니다. retrieve, format, prompt, model call, parse, log를 한 function이 모두 하면 각 부분을 test하기 어렵습니다. LCEL은 작고 이름 있는 step을 만들게 합니다.

Chain에 conditional routing, retry, human approval, durable state가 필요해지면 LangChain core에서 LangGraph로 넘어갈 때입니다. Core는 composable step을 주고, LangGraph는 다음에 어떤 step이 실행될지 결정하는 state machine을 줍니다.

RunnableParallel과 RunnablePassthrough로 context 주입하기

깔끔한 진행 순서는 plain chat chain, structured output, tiny RAG, one safe tool, router graph, approval workflow입니다. 각 layer는 단일 prompt에 complexity를 쌓는 대신 이유 있는 capability를 하나씩 추가합니다.

  • RunnableParallel — 동일한 입력에서 여러 Runnable을 실행합니다. 동시에 출력의 사전을 반환합니다. 체인의 Y-포크.
  • RunnablePassthrough — 입력을 변경하지 않고 전달하므로 원시 질문은 새로 검색된 컨텍스트와 함께 살아남습니다.

Local/on-prem portfolio에서는 이 글이 foundation입니다. ChatModel, PromptTemplate, Parser, Runnable, LCEL을 명확히 설명할 수 있으면 RAG, tool, MCP, agent는 자연스러운 확장이 됩니다.
