H100 없이 가정용 PC에서 로컬 LLM, MCP, Agent 실험실 만들기

온프레미스 LLM 엔지니어링을 배우기 위해 H100이 꼭 필요한 것은 아닙니다. 필요한 것은 평범한 PC, 작은 instruct 모델, 반복 가능한 설치 절차, 그리고 기능을 하나씩 추가하는 인내심입니다. 이 시리즈는 바로 그 순서로 진행합니다.

공개. 이 시리즈의 향후 버전에는 도서나 하드웨어 제휴 링크가 포함될 수 있습니다. 원칙은 단순합니다. 추천은 기술적으로 관련 있어야 하며, 어떤 글도 무언가를 구매하지 않아도 유용해야 합니다.

약속: 평범한 PC 한 대로 시작하기

우리는 실제 온프레미스 배포에서 쓰이는 시스템의 축소판을 만듭니다. 로컬 모델 서버, Python 클라이언트, 문서 인덱스, MCP 도구 서버, 가드레일이 있는 에이전트 루프가 순서대로 등장합니다. 목표는 벤치마크 자랑이 아니라 시스템을 손으로 이해하는 것입니다.

각 단계에는 세 가지 확인점이 있습니다. 무엇을 실행할지, 성공했을 때 어떤 출력이 보일지, 실패하면 무엇을 시도할지입니다. 출력 확인을 건너뛰지 마세요. 로컬 스택이 실제로 살아 있는지 확인하는 방법입니다.

실용적인 스택은 다음과 같습니다.

  • Ollama: 가장 쉬운 로컬 모델 서버와 OpenAI 호환 API.
  • Hugging Face Transformers: 직접 모델 로딩과 4비트 실험.
  • bitsandbytes: GPU 메모리가 부족할 때 쓰는 4비트/8비트 양자화 추론.
  • MCP: 로컬 도구, 파일, 워크플로를 표준 방식으로 에이전트에 노출.
  • Goose: Claude Code / Codex와 비슷한 계층. 터미널 에이전트, 확장, 파일 작업, 로컬 자동화.
  • Python: 검색, 도구, 로깅, 안전 확인을 이어 붙이는 언어.

하드웨어 단계별 기준

이 목록은 구매 명령이 아니라 학습용 지도입니다. 모델 계열은 빠르게 바뀌지만 메모리 제약은 쉽게 사라지지 않습니다.

  • CPU only, RAM 32 GB: 1B-3B 소형 모델은 느리지만 API와 프롬프트 학습에는 유용합니다.
  • VRAM 8 GB: 3B-7B 양자화 모델, 짧은 컨텍스트, 한 번에 한 사용자.
  • VRAM 12 GB: 7B-8B 양자화 모델을 비교적 편하게 실행. 작은 RAG 데모가 실제처럼 느껴지기 시작합니다.
  • VRAM 16 GB: 8B-14B 양자화 모델, 더 긴 컨텍스트, 더 나은 에이전트 실험.
  • VRAM 24 GB: 14B급 양자화 모델과 더 매끄러운 다중 도구 워크플로.
이론적 FLOPS를 구매하기 전에 VRAM을 구매하세요. 홈 온프레미스 LLM 작업의 경우 메모리는 프로젝트가 일반적으로 벽에 부딪히는 부분입니다.

시리즈 로드맵

  1. 로컬 LLM 서버. Ollama를 설치하고 작은 instruct 모델을 받은 뒤 Python에서 호출합니다.
  2. Hugging Face 직접 경로. Transformers로 모델을 로드하고, chat template을 적용하고, 4비트 양자화를 시도합니다.
  3. 로컬 RAG. PDF 또는 Markdown 파일 폴더를 색인화하고 인용과 함께 답합니다.
  4. 첫 MCP 서버. 계산기, 파일 검색, 시스템 노트처럼 안전한 로컬 도구를 노출합니다.
  5. Goose 코딩 에이전트. 로컬 모델 provider를 연결하고 작은 저장소를 검사, 편집, 테스트하게 합니다.
  6. 에이전트 루프. 모델이 도구를 선택하게 하되 구조화된 출력과 승인 게이트를 요구합니다.
  7. 운영. 로그, 평가 질문, GPU 모니터링, 백업, 실행 금지 명령 규칙을 둡니다.

1부: Ollama로 로컬 모델 서버 열기

먼저 지루하지만 변수가 적은 경로로 시작합니다. Ollama를 설치하고, 작은 모델을 받고, 로컬 OpenAI 호환 엔드포인트를 호출합니다. 대부분의 환경에서 서버는 http://localhost:11434에서 대기합니다.

1단계. 작은 instruct 모델 받기

처음에는 3B-7B 모델을 고르세요. 더 큰 모델은 나중에 해도 됩니다. 첫 승리는 내 PC가 모델을 다운로드하고, 로드하고, 로컬에서 답할 수 있음을 확인하는 것입니다.

예상 출력. 정확한 답변은 달라질 수 있지만, 모델 다운로드는 success로 끝나야 하고 모델은 프롬프트에 답해야 합니다.

성공 체크포인트. 여기서 일관된 답변을 얻었다면 로컬 모델 서버가 살아 있습니다. 느려도 괜찮습니다. 지금은 속도가 아니라 아키텍처를 먼저 확인하는 단계입니다.

2단계. Python에서 같은 모델 호출하기

이제 Python에서 로컬 엔드포인트를 호출합니다. RAG 라이브러리, 에이전트 프레임워크, Goose 스타일 도구는 보통 이런 API 형태를 통해 통합되기 때문에 중요합니다.

예상 출력. python local_chat.py를 실행하면 Ollama 터미널 UI가 아니라 Python 스크립트가 출력한 한 문단을 볼 수 있어야 합니다.

실패하는 경우. 연결 오류가 보이면 Ollama가 실행 중인지 확인하세요. model not found가 보이면 Python 코드의 model 값이 실제로 받은 모델 이름과 정확히 일치하는지 확인하세요.

여기까지 동작하면 첫 번째 벽돌을 얻은 것입니다. 많은 도구가 이미 호출할 줄 아는 로컬 모델 엔드포인트입니다. 이 단계가 지루해질 때까지 RAG, MCP, 에이전트를 추가하지 마세요.

2부: Hugging Face 4비트 추론

Hugging Face 경로는 더 많은 제어권을 줍니다. 동시에 설정을 잘못할 방법도 많아집니다. 첫 스크립트는 작게 유지하세요. 핵심은 토크나이저의 chat template을 사용하고, device_map="auto"를 쓰며, 일반 로딩을 이해한 뒤 4비트를 시도하는 것입니다.

3단계. 깨끗한 Python 환경 만들기

새 가상환경을 사용하세요. 로컬 LLM 작업은 의존성에 민감하므로 깨끗한 환경에서 시작하면 오류를 이해하기 쉽습니다.

예상 출력. 설치 과정에서 많은 패키지가 다운로드될 수 있습니다. 성공하면 설치된 패키지 목록이 출력되고 활성화된 프롬프트로 돌아옵니다.

4단계. 4비트 Hugging Face 채팅 스크립트 실행하기

예상 출력. 첫 실행은 모델 다운로드 때문에 느릴 수 있습니다. 로딩이 끝나면 짧은 답변이 출력되어야 합니다.

실패하는 경우. 많은 가정용 설정이 여기서 처음 막힙니다. CUDA 버전이 맞지 않거나, 모델이 너무 크거나, chat template이 없거나, 컨텍스트 길이를 너무 크게 잡았을 수 있습니다. 한 번에 하나의 변수만 바꾸세요. 패키지 다섯 개를 동시에 바꾸기 전에 더 작은 모델을 먼저 시도하세요.

3부: 첫 번째 MCP 도구 만들기

로컬 추론이 동작하면 도구를 추가합니다. “에이전트가 내 컴퓨터를 제어하게 하자”로 시작하지 마세요. 읽기 전용이거나 해가 없는 도구부터 시작하세요. MCP는 이런 도구를 표준 형태로 노출할 수 있어서 유용합니다.

5단계. 안전한 MCP 서버 실행하기

이 서버는 파일을 건드리지 않고, 셸 명령을 실행하지 않으며, 네트워크도 호출하지 않습니다. 안전한 도구 호출 두 개에만 응답합니다. 그래서 첫 MCP 확장으로 적합합니다.

예상 출력. 사용하는 전송 방식에 따라 서버는 조용히 클라이언트를 기다릴 수 있습니다. 조용해도 괜찮습니다. 성공한 MCP 서버는 종료되는 명령이 아니라 계속 실행되는 프로세스처럼 보일 때가 많습니다.

성공 체크포인트. 다음 목표는 강력하게 만드는 것이 아닙니다. 클라이언트를 연결하고 두 도구 이름이 발견되는지 확인하는 것입니다.

이 작은 서버는 화려하지 않습니다. 그래서 좋습니다. 안전한 에이전트는 거대한 권한 하나가 아니라 작고 지루한 기능들의 조합으로 만듭니다.

4부: Goose를 로컬 코딩 에이전트로 쓰기

로컬 추론과 MCP 도구가 동작하면 사람들이 실제로 원하는 것을 만들 수 있습니다. Claude Code나 Codex에 가까운 느낌의 로컬 코딩 도우미입니다. 이 시리즈에서 먼저 다룰 오픈소스 도구는 Goose입니다. Goose는 작은 코드베이스를 읽고, 편집을 제안하고, 테스트를 실행하고, MCP 도구를 호출하는 계층이 됩니다.

첫 번째 Goose 목표는 의도적으로 작습니다.

  • 모델은 Ollama 또는 다른 로컬 provider를 사용합니다.
  • 기본 작업 폴더가 아니라 장난감 Python 저장소를 엽니다.
  • 편집 전에 Goose에게 코드 설명을 먼저 요청합니다.
  • diff를 검토한 뒤에만 파일 편집을 허용합니다.
  • test, lint, format처럼 작은 허용 목록의 명령만 허용합니다.

예상 출력. Goose가 무엇을 건드리기 전에 기준선 테스트가 통과해야 합니다. 기준선이 실패한다면 에이전트 문제가 아니라 프로젝트 설정 문제입니다.

이제 Goose에게 프로젝트를 편집하지 말고 검사만 하라고 요청합니다. 좋은 첫 응답은 소스 파일, 테스트 파일, 그리고 프로젝트가 간단한 add 함수를 노출한다는 사실을 언급해야 합니다.

성공 체크포인트. 이것이 첫 Claude Code / Codex식 순간입니다. 에이전트가 저장소를 읽고 이해한 뒤 아무것도 바꾸지 않고 보고했습니다. 이 절제가 중요합니다. 편집은 검사 뒤에 옵니다.

두 번째 단계는 Goose를 MCP 서버에 연결하는 것입니다. 여기서 실습이 현실감을 얻기 시작합니다. 코딩 에이전트가 거대한 프롬프트에만 의존하지 않고 프로젝트 파일, 로컬 도구, 구조화된 기능을 사용할 수 있기 때문입니다. 안전 규칙은 그대로입니다. 먼저 읽기 전용, 그다음 작은 편집, 테스트, 더 넓은 도구 순서입니다.

처음부터 로컬 에이전트에게 전체 컴퓨터를 맡기지 마세요. 일회용 저장소를 사용하고, 백업을 준비하고, 비밀 정보는 폴더 밖에 두세요. 목적은 소중한 것을 맡기기 전에 에이전트 루프를 배우는 것입니다.

다음 단계

다음 글에서는 이 실습을 로컬 RAG로 확장합니다. 문서 폴더 하나, 로컬 임베딩, 벡터 인덱스, 인용이 붙은 답변을 만듭니다. 그다음 MCP 서버에 문서 검색 도구를 붙입니다. 마지막으로 Goose가 그 도구를 호출하고, 코드베이스를 검사하고, 패치를 만들고, 통제된 작업 공간 안에서 테스트를 실행하는 사용자-facing 에이전트가 됩니다.
