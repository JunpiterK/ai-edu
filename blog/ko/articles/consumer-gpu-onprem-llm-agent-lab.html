합성 사례: 첫 실행 15분

현장에서 자주 보는 장면을 합쳐 예를 들면, 로컬 모델 명령을 실행한 terminal은 다운로드 완료를 표시하지만 채팅 창에는 답이 나오지 않습니다. 작업 관리자의 GPU 메모리는 거의 변하지 않고, server log의 마지막 줄은 model load가 아니라 port bind 오류에 머물러 있습니다.

처음에는 GPU가 작아서 멈춘 것으로 보기 쉽습니다. 하지만 이 장면에서는 같은 port를 쓰는 이전 process가 남아 있다는 단서가 먼저입니다. process와 bind 주소를 확인해 충돌을 해소한 뒤에야 GPU 사용량과 driver·CUDA 조합을 판단할 수 있습니다.

그 다음 agent가 실습 폴더 밖의 파일을 열려고 하면 성공 여부와 관계없이 권한 범위를 닫습니다. 이 합성 사례의 결론은 설치를 반복하는 것이 아니라, server·GPU·tool 권한을 한 단계씩 분리해 흔적을 남기는 것입니다. 개인 PC에서 통과한 결과는 기능 검증일 뿐 공장망 반입 승인이나 운영 안정성을 대신하지 않습니다.

바로 적용할 기준

설치 명령, 정확한 model tag, package version, port, 예시 출력, 실패 log 위치를 한 장에 적으세요. 각 단계가 재실행될 때까지 RAG나 agent 기능을 더하지 않습니다.

가정용 PC에서도 local model API, 4-bit loading, MCP tool 발견, 제한된 agent 작업을 확인할 수 있습니다. 처리량과 긴 context, 다중 사용자 가용성은 production GPU server와 다르므로 이 글은 학습용 lab의 성공 조건만 다룹니다.

공개. 이 시리즈의 향후 버전에는 도서나 하드웨어 제휴 링크가 포함될 수 있습니다. 원칙은 단순합니다. 추천은 기술적으로 관련 있어야 하며, 어떤 글도 무언가를 구매하지 않아도 유용해야 합니다.

약속: 평범한 PC 한 대로 시작하기

Local model server, Python client, 문서 index, MCP server, 승인 gate가 있는 agent loop를 순서대로 연결합니다. 한 단계의 입력·출력·log를 확인한 뒤 다음 구성요소를 추가해야 실패 원인을 좁힐 수 있습니다.

아래 terminal block은 특정 PC에서 측정한 기록이 아니라 설명을 위해 단순화한 예시 출력입니다. Package version, OS, GPU, model revision에 따라 문구·속도·메모리 사용량이 달라집니다.

실용적인 스택은 다음과 같습니다.

  • Ollama: 가장 쉬운 로컬 모델 서버와 OpenAI 호환 API.
  • Hugging Face Transformers: 직접 모델 로딩과 4비트 실험.
  • bitsandbytes: GPU 메모리가 부족할 때 쓰는 4비트/8비트 양자화 추론.
  • MCP: 로컬 도구, 파일, 워크플로를 표준 방식으로 에이전트에 노출.
  • Goose: Claude Code / Codex와 비슷한 계층. 터미널 에이전트, 확장, 파일 작업, 로컬 자동화.
  • Python: 검색, 도구, 로깅, 안전 확인을 이어 붙이는 언어.

하드웨어 단계별 기준

이 목록은 2026년 7월 작성 시점의 학습용 출발점이며 구매 기준이 아닙니다. 실제 적재 가능 크기는 model architecture, quantization format, runtime overhead, context, KV cache, batch·동시성에 따라 달라집니다.

  • CPU only, RAM 32 GB: 1B-3B 소형 모델은 느리지만 API와 프롬프트 학습에는 유용합니다.
  • VRAM 8 GB: 3B-7B 양자화 모델, 짧은 컨텍스트, 한 번에 한 사용자.
  • VRAM 12 GB: 7B-8B 양자화 모델을 비교적 편하게 실행. 작은 RAG 데모가 실제처럼 느껴지기 시작합니다.
  • VRAM 16 GB: 8B-14B 양자화 모델, 더 긴 컨텍스트, 더 나은 에이전트 실험.
  • VRAM 24 GB: 14B급 양자화 모델과 더 매끄러운 다중 도구 워크플로.
후보 GPU에서는 peak VRAM, tokens/s, prompt 길이, 동시 요청 수를 함께 측정합니다. VRAM 용량 하나만으로 체감 성능이나 안정성을 확정하지 않습니다.

시리즈 로드맵

  1. 로컬 LLM 서버. Ollama를 설치하고 작은 instruct 모델을 받은 뒤 Python에서 호출합니다.
  2. Hugging Face 직접 경로. Transformers로 모델을 로드하고, chat template을 적용하고, 4비트 양자화를 시도합니다.
  3. 로컬 RAG. PDF 또는 Markdown 파일 폴더를 색인화하고 인용과 함께 답합니다.
  4. 첫 MCP 서버. 계산기, 파일 검색, 시스템 노트처럼 안전한 로컬 도구를 노출합니다.
  5. Goose 코딩 에이전트. 로컬 모델 provider를 연결하고 작은 저장소를 검사, 편집, 테스트하게 합니다.
  6. 에이전트 루프. 모델이 도구를 선택하게 하되 구조화된 출력과 승인 게이트를 요구합니다.
  7. 운영. 로그, 평가 질문, GPU 모니터링, 백업, 실행 금지 명령 규칙을 둡니다.

1부: Ollama로 로컬 모델 서버 열기

Ollama를 설치하고 작은 model을 받은 뒤 local OpenAI-compatible endpoint를 호출합니다. 기본 주소로 http://localhost:11434를 쓰는 구성이 많지만 설치 버전과 설정에서 실제 bind 주소를 확인합니다. 외부 interface에 노출한다면 별도 인증과 firewall 통제를 적용해야 합니다.

1단계. 작은 instruct 모델 받기

처음에는 3B-7B 모델을 고르세요. 더 큰 모델은 나중에 해도 됩니다. 첫 승리는 내 PC가 모델을 다운로드하고, 로드하고, 로컬에서 답할 수 있음을 확인하는 것입니다.

설명을 위해 단순화한 예시 출력입니다. Digest와 진행 표시, 답변 문장은 설치 시점의 model tag와 runtime에 따라 달라집니다.

종료 코드, model tag, 응답 생성 여부를 기록합니다. 한 번 답했다고 안정성이 확인되는 것은 아니므로 같은 prompt 반복과 긴 입력에서 OOM·timeout이 없는지도 다음 단계에서 측정합니다.

2단계. Python에서 같은 모델 호출하기

이제 Python에서 로컬 엔드포인트를 호출합니다. RAG 라이브러리, 에이전트 프레임워크, Goose 스타일 도구는 보통 이런 API 형태를 통해 통합되기 때문에 중요합니다.

설명을 위해 단순화한 예시 출력입니다. python local_chat.py를 실행했을 때 Python client가 응답 본문을 받는 흐름만 보여줍니다.

실패하는 경우. 연결 오류가 보이면 Ollama가 실행 중인지 확인하세요. model not found가 보이면 Python 코드의 model 값이 실제로 받은 모델 이름과 정확히 일치하는지 확인하세요.

여기까지 동작하면 local endpoint와 client의 연결은 확인됐습니다. RAG나 MCP를 붙이기 전에 model not found, server down, timeout을 각각 재현하고 어느 log에서 구분되는지 적습니다.

2부: Hugging Face 4비트 추론

Hugging Face 경로에서는 tokenizer, quantization, device placement를 직접 지정합니다. 첫 script는 tokenizer의 chat template과 실제 model revision을 확인하고, 작은 model에서 4-bit load와 생성이 끝나는지만 봅니다. device_map="auto"가 선택한 배치를 그대로 신뢰하지 말고 GPU·CPU memory 배치를 출력해 확인합니다.

3단계. 깨끗한 Python 환경 만들기

새 가상환경을 사용하세요. 로컬 LLM 작업은 의존성에 민감하므로 깨끗한 환경에서 시작하면 오류를 이해하기 쉽습니다.

설명을 위해 단순화한 예시 출력입니다. 실제 package 이름과 version, 설치 경로는 OS와 package index에 따라 달라집니다.

4단계. 4비트 Hugging Face 채팅 스크립트 실행하기

설명을 위해 단순화한 예시 출력입니다. Shard 수, 시간, 답변은 model revision, cache, GPU에 따라 달라지며 8초는 성능 기준이 아닙니다.

실패하는 경우. 많은 가정용 설정이 여기서 처음 막힙니다. CUDA 버전이 맞지 않거나, 모델이 너무 크거나, chat template이 없거나, 컨텍스트 길이를 너무 크게 잡았을 수 있습니다. 한 번에 하나의 변수만 바꾸세요. 패키지 다섯 개를 동시에 바꾸기 전에 더 작은 모델을 먼저 시도하세요.

3부: 첫 번째 MCP 도구 만들기

Local inference를 확인한 뒤 side effect가 없는 tool을 추가합니다. 첫 MCP server에는 계산과 고정된 lab policy 조회만 두고 file·shell·network 권한은 주지 않습니다.

5단계. 권한을 제한한 MCP 서버 실행하기

이 참조 코드는 file, shell, network를 호출하지 않는 두 함수를 노출합니다. 기능 범위가 작다는 뜻이지 운영 보안이 완성됐다는 뜻은 아닙니다.

설명을 위해 단순화한 예시 출력입니다. SDK version과 transport에 따라 이 문구가 표시되지 않을 수 있으며, 실제 상태는 client의 tool discovery와 server log로 확인합니다.

Client를 연결해 두 tool 이름과 schema가 발견되고 예상 밖 인자가 거부되는지 확인합니다.

이 코드는 local lab용 참조 예시입니다. 다른 host나 운영망에 배포하기 전에는 사용자·서비스 인증, TLS, timeout, rate limit, idempotency, 변조 방지 audit, 승인과 rollback을 설계하고 장애 test를 통과해야 합니다.

4부: Goose를 로컬 코딩 에이전트로 쓰기

로컬 추론과 MCP 도구가 동작하면 사람들이 실제로 원하는 것을 만들 수 있습니다. Claude Code나 Codex에 가까운 느낌의 로컬 코딩 도우미입니다. 이 시리즈에서 먼저 다룰 오픈소스 도구는 Goose입니다. Goose는 작은 코드베이스를 읽고, 편집을 제안하고, 테스트를 실행하고, MCP 도구를 호출하는 계층이 됩니다.

첫 번째 Goose 목표는 의도적으로 작습니다.

  • 모델은 Ollama 또는 다른 로컬 provider를 사용합니다.
  • 기본 작업 폴더가 아니라 장난감 Python 저장소를 엽니다.
  • 편집 전에 Goose에게 코드 설명을 먼저 요청합니다.
  • diff를 검토한 뒤에만 파일 편집을 허용합니다.
  • test, lint, format처럼 작은 허용 목록의 명령만 허용합니다.

설명을 위해 단순화한 예시 출력입니다. Goose 실행 전 기준선 test가 통과한 상태를 기록하며, 시간과 경로는 환경마다 달라집니다.

이제 Goose에 편집 권한을 주지 않고 repository 설명을 요청합니다. 응답에 source, test, add 함수가 포함되는지 확인하되, agent 문장 자체를 권한 준수 증거로 보지 말고 filesystem diff도 확인합니다. 아래 응답은 설명용 예시 출력입니다.

Goose를 시작할 때는 설치한 버전의 권한 설정에서 파일 쓰기와 shell tool을 허용하지 않고, 위 읽기 전용 prompt만 전달합니다. 이후 git status --porcelain이 빈 출력이고 git diff --exit-code와 pytest가 모두 0으로 끝나야 통과합니다. 다음에는 disposable repository에서 작은 편집을 허용하고 diff 검토와 test를 approval gate로 둡니다.

Goose를 MCP server에 연결할 때도 발견된 tool을 모두 허용하지 않습니다. Tool별 input, side effect, log를 확인하고 read-only에서 작은 편집 순서로 범위를 넓힙니다.

처음부터 로컬 에이전트에게 전체 컴퓨터를 맡기지 마세요. 일회용 저장소를 사용하고, 백업을 준비하고, 비밀 정보는 폴더 밖에 두세요. 목적은 소중한 것을 맡기기 전에 에이전트 루프를 배우는 것입니다.

다음 단계

다음 단계에서는 별도 실습 문서 폴더를 local embedding과 vector index로 연결하고 citation을 검사합니다. 그 뒤 MCP 검색 tool과 disposable repository를 붙여 조회, patch 초안, 승인, test, rollback 순서를 검증합니다. 이 lab 결과를 운영망에 옮기려면 조직의 반입·인증·감사·복구 절차를 다시 통과해야 합니다.

한 PC에서 API 호출이 재현됐다면 이제 “돌아간다”를 “몇 명이 써도 버틴다”로 바꿔야 합니다. 같은 모델을 Ollama와 vLLM에 올려 동시 요청, KV cache, OOM 조건을 비교하는 순서는 온프레미스 LLM 서빙 가이드에서 이어집니다.
