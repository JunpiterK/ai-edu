자동차 제조 AI vs FAB AI: 같은 공장이라는 말, 다른 물리

합성 장면: 같은 양식에 넣다가 사라진 정보

과제 검토 화면의 열이 tool / chamber / recipe / trace로만 잡혀 있습니다. 자동차 담당자가 용접 robot log를 넣자 VIN과 station 이력은 비고 칸으로 밀리고, takt와 safety state는 첨부 메모가 됩니다. 모델 입력은 만들어져도 어느 차량과 품질 gate가 영향을 받았는지 추적할 수 없습니다. 두 제조 환경에서 흔히 생길 수 있는 설계 충돌을 합친 가상 장면입니다.

바로 적용할 기준

과제명 아래에 "이 판단이 어느 VIN, station, robot cell, 품질 gate에 영향을 주는가"를 한 줄로 적어보세요. 네 항목 중 하나라도 특정되지 않으면 모델 평가를 미루고 genealogy, station event, 안전 승인 흐름을 먼저 정리합니다.

기술적 권고는 공통 AI 플랫폼보다 먼저 물리적 제어 단위를 분리하라는 것입니다. 자동차 라인은 VIN·part·station·takt가 판단 축이고, FAB은 wafer·lot·chamber·recipe context가 중심입니다. 안전 승인과 영향 범위를 계산하는 방식도 달라서 평가셋과 허용 action을 같은 템플릿으로 묶기 어렵습니다.

1분 모델

FAB 엔지니어는 "어느 chamber, 어느 recipe step, 어떤 trace shape, 어느 wafer, 어떤 metrology가 바뀌었나?"를 묻습니다. 자동차 제조 엔지니어는 "어느 VIN, 어느 part, 어느 station, 어느 robot cell, 어떤 torque result, 어떤 vision image, 어떤 작업자 보정, 어느 takt window가 바뀌었나?"를 묻습니다. AI 단어는 같아도 공장 문법이 다릅니다.

짧은 결론

FAB AI는 공정 제어와 장비 데이터 축으로, 자동차 AI는 제품 흐름·로봇셀·안전·품질 시스템 축으로 설계합니다. RAG, 온프레미스 LLM, MCP, 지식그래프, 평가 harness는 공통 인프라로 쓸 수 있지만 source ID와 승인 owner, 허용 action은 각 현장 단위에 맞춰 분리합니다.

왜 콘텐츠를 분리해야 하는가.
질문FAB AI자동차 제조 AI
제어 단위Wafer, lot, chamber, recipe step.VIN, part, station, robot cell, takt window.
핵심 리스크Yield loss, excursion, chamber drift, scrap.Line stop, wrong part, torque escape, unsafe motion, recall risk.
데이터 중심FDC trace, SPC, metrology, APC, equipment event.PLC tag, robot log, vision image, torque record, Andon, QMS, EOL test.
첫 AI 과제Fault detection, virtual metrology, recipe/context retrieval.Vision inspection, robot-cell anomaly review, diagnostic RAG, takt-aware escalation.

제어 단위

FAB에서는 보통 생각의 출발점이 장비입니다. chamber가 recipe를 수행하고, wafer가 공정 step을 지나며, metrology가 나중에 숨은 공정 결과를 보여줍니다. 그래서 trace alignment, golden run, chamber matching, recipe revision, process context가 아주 중요합니다.

자동차 조립에서는 제품 흐름에서 시작합니다. 차체, 모듈, 부품이 station을 지나며 로봇은 용접·도포·이송·체결·검사를 수행합니다. 한 station의 cycle time이 takt를 넘기면 downstream 대기와 buffer 상태가 바뀌므로 이상 판정의 시각과 대응 기한을 함께 기록해야 합니다.

현장 말투로 말하면

FAB에서는 장비 겉보기는 멀쩡한데 trace shape이 조용히 drift할 때 신경이 곤두섭니다. 자동차 라인에서는 station은 pass를 찍었는데 cycle time, torque pattern, vision confidence, 작업자 보정 이력이 서로 다른 이야기를 할 때 불안해집니다.

데이터 모양

FAB 데이터는 recipe context가 붙은 고밀도 time-series가 많고, 자동차 데이터는 제품 하나를 따라가는 event chain이 중심입니다. VIN 하나의 station 진입 시각, robot program, weld signal, rework 확인, torque result, EOL test를 시간순으로 맞춰야 서로 다른 시스템의 판정을 비교할 수 있습니다.

자동차 AI에서는 model input table을 만들기 전에 VIN과 station을 기준으로 증거 timeline을 구성합니다. 시스템 clock이 어긋나거나 rework가 수기로 남으면 이 연결만으로는 충분하지 않으므로 데이터 품질 표시도 함께 둡니다.

  • VIN genealogy: 한 차량에 어떤 부품, supplier lot, station, tool, test result가 붙었는지 묶는 이력입니다.
  • Station event: start, complete, blocked, bypassed, reworked, held, released 같은 흐름 이벤트입니다.
  • Robot-cell evidence: program ID, path segment, servo load, weld current, gun wear, fixture state, safety state입니다.
  • Quality evidence: inspection image, defect class, control plan item, FMEA linkage, 8D action, containment status입니다.

로봇이 문제를 바꾼다

로봇셀에는 motion, zone, fixture, sensor, 작업자, maintenance mode가 함께 있습니다. AI의 이상 판정이 robot motion이나 line stop으로 이어지기 전에 safety PLC와 interlock을 우회하지 않는지, line owner·quality owner·safety owner 중 누가 승인하는지 정해야 합니다. Maintenance mode에서도 사람의 cell 진입과 재가동 확인 절차는 기존 안전 규정을 따릅니다.

설명을 위해 단순화한 가상 사례를 보겠습니다. Body shop 용접 로봇의 servo load가 일정 기간 상승하고 weld defect 건수도 늘었다고 가정합니다. 두 신호의 동시 상승만으로 원인을 확정할 수는 없습니다. Gun wear, fixture 상태, 최근 maintenance, robot program revision, 영향 VIN을 확인한 뒤 containment 범위와 planned stop 점검 항목을 정합니다. Servo load 단위와 경계값은 로봇·축·작업 조건에 따라 달라지므로 이 예시에는 실제 수치를 넣지 않았습니다.

Agent 설계 차이

FAB Agent와 자동차 Agent는 MCP, RAG, approval gate, audit log를 공유할 수 있습니다. 하지만 권한 설계는 달라야 합니다.

Agent 권한 설계 예시.
Agent actionFAB 관점자동차 관점
증거 읽기최근 alarm, trace, recipe revision, metrology.VIN route, station event, robot log, vision result, Andon history.
조치 제안Chamber review, lot hold, PM state 확인.Station escalation, 영향 VIN 검사, containment 초안.
상태 변경Process owner와 MES control gate 필요.Line owner, quality owner, safety owner 승인 필요.
위험 구간Recipe, dispatch, hold/release, engineering change.Robot motion, safety bypass, line stop/restart, quality disposition.

Takt-aware 설계에서는 답변마다 사용 기한을 붙입니다. Safety interlock 이상은 즉시 정해진 escalation 경로로 보내고, 반복 cycle time 증가는 다음 shift handover 항목으로, 원인 비교는 생산 안정화 이후 engineering investigation으로 분리할 수 있습니다. 이 우선순위와 시간은 모델이 임의로 정하지 않고 line owner와 safety·quality owner가 승인한 규칙에서 가져옵니다.

작은 실습: 과제 분류 4칸

검토 중인 AI 과제 하나를 골라 대상 ID / 물리적 위치 / 판단 기한 / 승인 owner를 채웁니다. 예를 들어 "용접 이상 검토"라면 VIN, weld cell, 현재 shift, quality owner·maintenance owner가 들어갑니다. FAB 과제라면 wafer·lot, chamber, metrology 도착 전후, process owner로 바뀝니다. 두 환경의 칸이 섞이거나 owner가 비어 있으면 공통 모델을 논의하기 전에 과제 경계를 다시 나누는 편이 안전합니다.

시리즈 로드맵

자동차 시리즈는 개념에서 시작해서 실제 산출물로 이어지는 편이 좋습니다.

  1. 자동차 제조 데이터 101: PLC tag, robot log, vision image, torque record, QMS, PLM, MES, EOL test.
  2. VIN genealogy vs wafer genealogy: traceability를 retrieval과 품질 조사 backbone으로 보는 법.
  3. Takt-aware Agent: 라인 cadence와 approval gate를 기준으로 Agent 응답을 설계하는 법.
  4. Robot cell AI architecture: camera, robot controller, PLC, safety PLC, MES, historian, audit log 연결.
  5. Vision inspection: golden sample, false alarm, defect taxonomy, rework loop, model drift.
  6. Safety and quality checklist: robot safety, functional safety, IATF 16949, FMEA, control plan, OT security.

이 순서라면 각 글의 산출물이 분명해집니다. FAB 쪽은 recipe context와 장비 trace 평가표를, 자동차 쪽은 VIN evidence chain과 robot-cell 승인표를 남깁니다. 공통 인프라는 공유하되 평가셋과 운영 권한은 두 환경을 섞지 않습니다. 먼저 영향 차량의 포함·제외 근거가 필요하다면 VIN genealogy vs wafer genealogy를, 대응 기한과 승인 흐름이 문제라면 Takt-aware AI Agent를 다음 글로 고르세요.

제조 시스템의 차이를 기술적으로 비교했으며, 가상 사례는 실제 사업장 데이터가 아닙니다.
