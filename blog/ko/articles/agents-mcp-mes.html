MCP로 에이전트를 사내 시스템에 연결하기: MES, 설비 API, 권한과 감사

에이전트는 자신이 실제로 접근할 수 있는 시스템만큼만 유용합니다. 수율 이상을 추론할 수 있어도 MES 로트 이력, 마지막 챔버 클린 시각, 보류 처리 상태를 읽지 못한다면 값비싼 자동완성에 가깝습니다. 에이전트 프로젝트에서 어려운 부분은 모델보다 연결 방식입니다. 모델이 이해할 수 있는 스키마와 넘을 수 없는 경계를 두고 사내 시스템을 안전하게 호출하게 만드는 일이 핵심입니다. Model Context Protocol(MCP)은 바로 이 연결 문제를 표준화하기 위한 프로토콜입니다. 이 글은 온프레미스 MES 또는 설비 API를 MCP 서버로 노출하고, 에이전트에 연결하며, 모든 호출에 권한과 감사 체계를 두는 방법을 폐쇄망 팹 환경을 기준으로 설명합니다.

왜 에이전트에는 임시 연결 코드보다 프로토콜이 필요한가

프로토콜이 없으면 모든 통합은 일회성 코드가 됩니다. MES REST 호출을 감싸는 Python 함수, 설비 히스토리언용 함수, 레시피 관리 시스템용 함수를 따로 만들고, 각 함수가 모델에 노출할 JSON 스키마를 직접 작성해야 합니다. 실제 API가 바뀔 때마다 스키마도 맞춰야 하고, 다음 에이전트 프레임워크를 시도할 때 같은 작업을 반복합니다. 한 번은 동작하지만 빠르게 관리하기 어려워집니다.

MCP는 이 문제를 클라이언트-서버 계약으로 바꿉니다. 내부 시스템은 자신의 기능을 표준 방식으로 설명하는 MCP 서버 뒤에 놓입니다. 에이전트, IDE, 채팅 앱 같은 MCP 클라이언트는 아래쪽 시스템의 구현을 몰라도 기능을 발견하고 호출할 수 있습니다. 팹에서 중요한 가치는 분리입니다. MES를 담당하는 팀이 하나의 MCP 서버를 안정적으로 게시하면, 공장 안의 여러 에이전트가 같은 서버를 재사용할 수 있습니다.

MCP는 OS에 대한 장치 드라이버와 마찬가지로 에이전트에 대한 것입니다. 장비팀에서 작성합니다. 운전자는 한 번; 그 위의 모든 애플리케이션은 표준 인터페이스를 호출합니다. — 유지해야 할 정신 모델

MCP의 기본 구조: 서버, 도구, 스키마

MCP 서버는 세 가지 기본 요소를 노출합니다. 도구는 에이전트가 호출할 수 있는 함수입니다. 예를 들어 query_lot_status나 place_hold 같은 작업이 여기에 해당합니다. 리소스는 SOP 파일이나 스펙 테이블처럼 컨텍스트로 읽어 올 수 있는 읽기 전용 데이터입니다. 프롬프트는 서버가 제공하는 재사용 가능한 템플릿입니다. 시스템 통합에서는 주로 도구가 핵심 역할을 맡습니다.

각 도구에는 이름, 설명, 타입이 지정된 파라미터로 구성된 스키마가 붙습니다. 에이전트는 이 스키마를 읽고 언제 어떤 인자로 도구를 호출할지 판단합니다. 그래서 스키마는 계약이며, 모호하지 않아야 합니다. place_hold라는 도구가 “로트에 엔지니어링 hold를 건다”라고 설명되고 lot_id 파라미터를 요구한다면, 모델은 자신이 무엇을 할 수 있고 무엇을 제공해야 하는지 알 수 있습니다. 설명이 정확하면 도구 호출도 정확해지고, 설명이 흐릿하면 모델은 추측하게 됩니다.

MES와 설비 API를 MCP 서버로 노출하기

Python에서 가장 적은 보일러플레이트로 서버를 만드는 방법은 공식 MCP SDK의 일부인 FastMCP를 사용하는 것입니다. pip install "mcp[cli]"로 설치한 뒤 일반 함수에 @mcp.tool()을 붙이면 타입 힌트와 독스트링이 에이전트가 보는 스키마가 됩니다. JSON Schema를 손으로 작성하지 않아도 됩니다. 아래 예시는 이미 공장 네트워크 안에 있는 내부 MES API를 도구로 감싸는 형태입니다.

두 함수와 그 독스트링만으로도 검색 가능한 MCP 서버가 됩니다. query_lot_status와 last_chamber_clean은 모두 읽기 전용입니다. 처음에는 이런 읽기 전용 도구로 시작하는 것이 좋습니다. 공장 상태를 바꾸는 도구는 더 강한 권한 확인과 승인 절차가 필요하며, 다음 섹션에서 다룹니다.

langchain-mcp-adapters로 에이전트에 연결하기

에이전트 쪽에서는 langchain-mcp-adapters를 사용합니다. pip install langchain-mcp-adapters로 설치하면 MCP 서버의 도구를 LangChain 도구로 변환할 수 있습니다. MultiServerMCPClient는 하나 이상의 서버에 연결하고, get_tools()는 서버가 공개한 스키마를 가져오며, 최신 create_agent는 이 도구들을 온프레미스 모델에 바인딩합니다. 아래 코드는 LangChain 1.x와 LangGraph 계열의 최신 API를 기준으로 하며 오래된 LLMChain이나 initialize_agent 패턴을 쓰지 않습니다.

이제 에이전트는 직접 하드코딩하지 않은 두 도구를 사용할 수 있습니다. 사용자가 두 부분으로 된 질문을 던지면 에이전트는 query_lot_status와 last_chamber_clean을 차례로 호출하고, 결과를 합쳐 답변합니다. 클라이언트를 레시피 시스템이나 불량 데이터베이스를 감싼 두 번째 MCP 서버에 연결하면 새 도구가 자동으로 발견됩니다. 서버를 추가하면 에이전트가 사용할 수 있는 도구도 확장됩니다.

권한, 감사, 가드레일

읽기 전용 도구는 비교적 위험이 낮습니다. 하지만 도구가 로트 hold, dispatch, 레시피 편집처럼 공장 상태를 바꾸는 순간 스키마만으로는 충분하지 않습니다. 주변에 세 가지 장치가 필요합니다.

  • 권한 확인(authz). 무엇을 누가 호출할 수 있는지는 모델이 아니라 MCP 서버가 결정합니다. 호출자의 역할을 도구 내부에서 검증해야 하며, 모델 프롬프트는 보안 경계가 아닙니다.
  • 감사 로그. 모든 호출은 호출자, 인자, 결과, 타임스탬프와 함께 기록되어야 합니다. 그래야 에이전트가 수행한 작업도 사람이 수행한 작업처럼 추적할 수 있습니다.
  • Human-in-the-loop. 상태를 바꾸는 도구는 즉시 실행하지 않고 확인 단계를 반환해야 합니다. 현장에서 실제 변경이 일어나기 전에 엔지니어가 승인할 수 있어야 합니다.

아래 place_hold 도구는 호출자의 역할을 확인하고, 감사 기록을 남기며, 권한이 없을 때 추측하지 않고 거부합니다. 가드레일은 서버 안에 있어야 합니다. 그래야 프롬프트 조작으로 우회할 수 없습니다.

역할 정보는 전송 계층에서 오고, 감사 로그는 상태 변경 전에 기록됩니다. 따라서 아무리 영리한 프롬프트라도 도구가 자신의 권한 범위를 넘어 행동하게 만들 수 없습니다. 모델은 요청할 수 있지만, 최종 결정은 서버가 합니다. 이 분리가 실제 MES 작업을 에이전트에 연결할 때의 안전성을 만듭니다.

전체 루프를 온프레미스로 유지하기

MCP가 팹 환경에 잘 맞는 이유는 이 설계가 클라우드를 전제로 하지 않기 때문입니다. MCP 서버는 공장 내부 호스트에서 실행되고 내부 주소에 바인딩됩니다. 전송은 LAN 위의 HTTP이며, 모델은 Ollama나 vLLM으로 로컬 서빙됩니다. 에이전트, 클라이언트, 가드레일, 내부 API가 모두 같은 폐쇄망 경계 안에 있습니다. 토큰도, 도구 호출도, 로트 ID도 그 경계를 넘지 않습니다.

작고 읽기 전용인 형태로 시작하세요. MCP 서버 하나, 조회 도구 두세 개, 실제로 확인하는 감사 로그면 충분합니다. 에이전트가 현장의 실제 질문에 정확히 답하는지 먼저 증명한 뒤, 상태를 바꾸는 도구를 넘겨야 합니다. 그때는 역할 확인과 사람 승인 절차로 감싸야 합니다. 프로토콜은 깔끔한 연결 지점을 제공하지만, 운영 규율은 별도로 설계해야 합니다.

MCP는 “에이전트를 우리 시스템에 연결하자”라는 막연한 일을 재사용 가능하고, 타입이 지정되고, 감사 가능한 서버 몇 개로 바꿉니다. 한 팀이 좋은 MES 서버를 게시하면 공장 안의 모든 에이전트가 그 기반을 공유할 수 있습니다. 아래 심화 과정은 계획, 도구, 메모리, 가드레일까지 전체 에이전트 스택을 단계별로 다룹니다.
