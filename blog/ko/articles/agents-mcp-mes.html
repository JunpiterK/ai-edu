데모 뒤에 남는 질문

설계를 설명하기 위해 만든 완전한 가상 장면입니다. 주간 검토 화면에서 agent가 MES 조회 tool을 한 번 호출해 로트 상태를 정상 표시했다고 가정하겠습니다. 시연은 통과한 듯 보이지만, 권한 없는 시험 계정에서도 결과가 노출되고 timeout 재시도 기록에 원 요청 ID가 없다면 검토 순서를 바꿔야 합니다. 성공 화면보다 권한 거부, 중복 실행, audit 연결을 먼저 시험합니다. MCP는 tool schema를 표준화하지만 이런 통제를 대신 구현하지 않습니다.

바로 적용할 기준

기술 권고: MCP server를 만들기 전에 tool catalog부터 적으세요. 읽기 전용, draft 생성, 상태 변경 tool을 분리하고 각 항목에 owner, 인증 방식, approval, timeout, audit, rollback을 붙입니다. 이 칸이 비어 있는 tool은 운영 catalog에 올리지 않습니다.

수율 질문에 답하려면 MES 로트 이력, 마지막 chamber clean 시각, hold 상태처럼 현재 데이터를 읽어야 합니다. 이때 모델에 DB 계정이나 내부 API를 직접 주지 않고, 허용한 기능과 입력 schema만 tool로 노출하는 연결 계층이 필요합니다. Model Context Protocol(MCP)은 그 tool 계약과 발견 방식을 표준화합니다. 이 글의 코드는 폐쇄망 FAB에서 MES·설비 API를 감싸는 흐름을 설명하는 참조 아키텍처이며, 운영 배포본은 아닙니다.

왜 에이전트에는 임시 연결 코드보다 프로토콜이 필요한가

MES REST 호출, historian 조회, recipe 관리 함수를 agent마다 따로 만들면 API 변경 때 schema와 권한 코드를 여러 곳에서 고쳐야 합니다. 어느 client가 어떤 버전을 쓰는지도 추적하기 어렵습니다. MCP server에 tool 계약을 모으면 client는 내부 구현 대신 공개된 이름, 설명, 입력 schema를 기준으로 호출할 수 있습니다.

MES 담당 팀이 server와 schema version을 관리하고 여러 client가 재사용하는 구조가 가능해집니다. 다만 server 하나에 권한이 집중되므로 인증서 교체, schema 호환성, rate limit, 장애 격리의 owner도 함께 정해야 합니다. 표준 protocol을 쓴다는 사실만으로 이 운영 책임이 사라지지는 않습니다.

MCP server는 내부 API를 agent마다 다시 연결하지 않도록 tool 계약을 한곳에 둡니다. 계약의 재사용성과 실제 시스템의 접근 통제는 별도로 검증합니다. — 설계 검토 기준

MCP의 기본 구조: 서버, 도구, 스키마

MCP server는 tool, resource, prompt를 노출할 수 있습니다. Tool은 query_lot_status처럼 호출 가능한 함수이고, resource는 SOP나 spec table처럼 읽는 데이터입니다. Prompt는 server가 제공하는 재사용 템플릿입니다. MES 통합에서는 상태를 조회하거나 변경하는 tool의 경계를 먼저 검토합니다.

각 tool에는 이름, 설명, typed parameter schema가 붙습니다. place_hold라면 대상 lot ID, 허용 reason 형식, 변경되는 상태, 승인 필요 여부를 설명에 포함합니다. Schema는 모델의 잘못된 인자를 줄이는 장치이지 보안 경계가 아닙니다. Server는 값 범위, 현재 상태, 호출자 권한을 다시 검증해야 합니다.

MES와 설비 API를 MCP 서버로 노출하기

아래 예시는 FastMCP로 내부 MES API의 읽기 함수를 감싸는 최소 구성입니다. Type hint와 docstring에서 tool schema를 만들 수 있어 구조를 확인하기에는 편합니다. 예시의 HTTP 주소, 고정 timeout, 0.0.0.0 bind는 설명용 값이며 운영 설정으로 복사해서는 안 됩니다.

두 함수는 tool 발견과 호출 흐름을 보여주는 참조 코드입니다. 읽기 전용이라도 lot ID 열람 권한, 사용자·서비스 인증, TLS, 입력 검증, timeout·retry, 응답 마스킹, audit를 추가해야 합니다. 운영 전에는 실제 MES의 부하와 장애 모드에서 이 조건을 시험하고, 허용 network interface만 bind합니다.

langchain-mcp-adapters로 에이전트에 연결하기

Client 예시는 langchain-mcp-adapters로 server schema를 가져와 온프레미스 모델에 tool을 연결합니다. 패키지 API와 transport 설정은 버전에 따라 바뀔 수 있으므로 설치 시점의 공식 문서와 lock file을 기준으로 확인해야 합니다. 아래 코드는 호출 관계를 설명하기 위한 것이며 인증 header, TLS trust store, 연결 timeout, circuit breaker는 생략했습니다.

Client는 두 tool의 schema를 발견하고 질문에 따라 호출을 요청할 수 있습니다. 실제 실행에서는 호출 순서와 결과 결합이 항상 같다고 가정하지 말고, 허용 tool 목록과 최대 호출 횟수, 실패 시 답변 형식을 정합니다. Server를 추가했다고 곧바로 agent 권한에 포함하지 말고 schema 검토와 evaluation을 통과한 tool만 catalog에 공개합니다.

권한, 감사, 가드레일

읽기 전용 tool도 생산정보 노출 위험이 있습니다. Lot hold, dispatch, recipe 편집처럼 상태를 바꾸는 tool은 여기에 중복 실행과 rollback 실패 위험까지 더해집니다. 최소한 다음 통제를 server에서 집행합니다.

  • 권한 확인(authz). 무엇을 누가 호출할 수 있는지는 모델이 아니라 MCP 서버가 결정합니다. 호출자의 역할을 도구 내부에서 검증해야 하며, 모델 프롬프트는 보안 경계가 아닙니다.
  • 감사 로그. 모든 호출은 호출자, 인자, 결과, 타임스탬프와 함께 기록되어야 합니다. 그래야 에이전트가 수행한 작업도 사람이 수행한 작업처럼 추적할 수 있습니다.
  • Human-in-the-loop. 상태를 바꾸는 도구는 즉시 실행하지 않고 확인 단계를 반환해야 합니다. 현장에서 실제 변경이 일어나기 전에 엔지니어가 승인할 수 있어야 합니다.

아래 place_hold는 권한 확인과 pending confirmation 형태를 보여주는 human-in-the-loop 보안 경계 의사코드입니다. 인증 미들웨어가 토큰 또는 mTLS identity를 검증한 뒤 만든 principal을 server-side adapter가 주입한다고 가정하며, MCP client가 tool argument나 header로 보낸 role 값은 신뢰하지 않습니다. 실제 hold를 적용하지 않으며 완전한 승인 workflow도 아닙니다.

운영 구현에서는 client가 제공한 role 문자열을 신뢰하지 않고 인증 미들웨어가 검증한 사용자·서비스 identity의 principal 또는 claims에서 권한을 계산합니다. TLS, approval token의 만료와 1회성 사용, idempotency key, timeout 뒤 상태 조회, 변조 방지 audit, 승인 취소, MES rollback 또는 보상 절차를 함께 시험해야 합니다. Prompt 우회 test뿐 아니라 승인 API 장애와 중복 요청도 검증한 뒤 상태 변경 권한을 엽니다.

첫 MCP tool catalog 채우기

아래는 특정 사업장의 운영표가 아닌 worked example입니다. 첫 서버에 조회 도구 하나만 올린다고 가정하고 빈칸 없이 한 행을 작성합니다.

항목예시 기록release를 막는 조건
도구·ownerquery_lot_status · MES API 담당장애·schema 변경 owner가 없음
입력·권한정규화한 lot ID · lot 조회 권한client가 보낸 role을 그대로 신뢰함

감사·민감정보request ID, principal, 호출 시각 · 응답 field 마스킹한 호출을 audit에서 재구성할 수 없음
timeout·재시도서버 상한, 읽기 요청 1회 재시도중복 요청 상한이나 circuit breaker가 없음

작성 뒤 정상 계정과 권한 없는 계정으로 같은 ID를 호출하고, timeout을 주입한 뒤 세 이벤트가 request ID로 이어지는지 확인하세요. 한 칸이라도 “추후 결정”이면 catalog 공개를 보류합니다. 조회 도구를 통과했다고 상태 변경 도구도 안전하다는 뜻은 아니며, 쓰기 경로에는 별도의 승인·멱등성·rollback 시험이 필요합니다.

전체 루프를 온프레미스로 유지하기

MCP server, local model, client, 내부 API를 같은 폐쇄망에 배치할 수는 있습니다. 그러나 로컬 배치만으로 egress 차단이 증명되지는 않습니다. Package update 경로, DNS·proxy 설정, telemetry, log sink를 network audit로 확인하고 model request와 tool argument가 허용 경계 밖으로 나가지 않는지 기록합니다.

첫 범위는 server 하나와 조회 tool 두세 개로 제한합니다. 실제 계정별 권한 거부, timeout, 잘못된 ID, MES 장애, audit replay를 확인한 뒤 draft tool로 넓힙니다. 상태 변경은 인증·TLS·idempotency·human approval·rollback tabletop을 통과한 경우에만 별도 release gate에서 승인합니다.

MCP는 내부 기능을 재사용 가능한 typed tool 계약으로 묶습니다. MES server를 공유할 때는 schema version, owner, 허용 client, audit 위치, 장애 시 차단 방법도 함께 게시해야 합니다. 조회 catalog를 검증한 다음에는 ‘폐쇄망 에이전트의 사람 검증과 MLOps 운영’에서 쓰기 승인과 감사 이력을 설계하세요. 아래 심화 과정은 계획, tool, memory, guardrail을 더 넓은 agent stack에서 다룹니다.
