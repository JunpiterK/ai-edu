Opening artifact: failed fixture

expected_revision=C, retrieved_revision=B, and the required phrase stop and notify facilities is missing. That three-line test result tells us more than an index count ever will: the parser ran, the chunks were stored, and the useful clause still did not survive retrieval.

Working rule

Do not approve a splitter by looking at average chunk length. Approve it when named questions retrieve the required clause together with its source, revision, heading, and operating condition.

Plant documents are full of dependencies that ordinary prose does not have. A warning belongs to a step; a unit belongs to a value; a table row belongs to its header. Chunking is where those dependencies either become retrievable evidence or turn into plausible fragments.

Constructed boundary trace

The embedding model initially took the blame for missing the pressure question. The retrieval trace pointed elsewhere: revision B ranked first because the heading-only split had detached revision C's warning from "Chamber clean." Keeping the revision, step label, and warning in one section fixed the fixture. This does not show that the same rule will work across the full corpus; it shows which boundary failed in this constructed case.

Why chunking decides your RAG quality

A RAG system embeds each chunk into a vector and, at query time, returns the chunks nearest to the question. If a chunk separates a warning from the step it warns about, or separates a torque value from the fastener it applies to, the retriever can return the right neighborhood while still giving the model an incomplete or unsafe fragment. The failure is quiet: the answer sounds fluent, but the engineer who knows the procedure spots the missing clause.

Technical documents are harder than prose because meaning lives in structure: a step number, a table row, a sub-clause under a heading. Good chunking preserves that structure instead of cutting across it.

A chunk should be the smallest span a person can read in isolation and still act on correctly.

Reproducible fixture and acceptance test

The following single-file fixture uses constructed maintenance text and a deterministic lexical scorer so the chunk boundary can be tested without a model, embedding service, or network. It checks that the current revision and its exception remain together and rank first. This is a pipeline smoke test, not evidence of semantic-retrieval quality.

Run this test in the same locked environment as the splitter. For a real retrieval acceptance test, replace the lexical scorer with the deployed embedder and index, pin artifact hashes, and record at least query ID, expected source and revision, required clause, top-k, retrieved chunk IDs, and pass/fail. A site reviewer should approve the fixture before it becomes a release gate.

Splitting on document structure

Recursive splitting is content-blind. It does not know a heading from an ordinary sentence. When the source has real structure, such as Markdown headings from a converted manual or section numbers in an SOP, split on that structure first and then apply the recursive splitter inside each section. This two-pass pattern keeps chunks anchored to headings and lets you carry those headings as metadata.

Code and structured configuration deserve their own separators. The recursive splitter ships language profiles so it breaks Python on def and class boundaries rather than mid-function. That matters when manuals contain recipe scripts, PLC snippets, or configuration examples.

Size and overlap trade-offs

There is no universal chunk size. You tune it to your corpus and embedder. Small chunks embed tightly and retrieve precisely, but they can lose surrounding context. Large chunks keep context together, but the embedding becomes an average of too many ideas and the exact value you need may sink into noise.

  • Overlap can reduce boundary loss. The illustrative 10-20% range is a starting experiment, not a universal target; measure duplicate hits and retrieval recall on the fixture.
  • Match size to the embedder. A model trained on short passages will not reward 2,000-token chunks. Keep chunks near the length the embedder was tuned for.
  • Evaluate instead of guessing. Use real fab questions and measure whether the correct clause appears in top-k. Chunk size is a knob you tune, not a default you copy.

Tables, formulas, and code blocks

Character-count splitting does the most damage here. A table split between its header row and data rows produces numbers without column names. Formulas and tolerances can become useless if the operator, unit, or limit is separated from the condition. Three rules keep these structures intact.

  • Tables: treat each table as an atomic unit. Extract it during document conversion, serialize it as Markdown or a short natural-language summary, and keep header rows attached to every emitted row group.
  • Formulas and tolerances: do not split inside expressions. Add formula delimiters to the separator list so the splitter prefers to cut around them, not through them.
  • Code and recipe snippets: use a language-aware splitter so functions, classes, and recipe blocks stay whole.

The practical move is to parse structure during conversion, not after the document has become one flat string. If your PDF-to-text step already tags tables, code fences, and formulas, the splitter can route each type to the right strategy instead of treating the whole document as plain prose.

Metadata makes chunks retrievable

A chunk without metadata is an orphan. Source file, section title, revision, page, and character offset turn nearest-neighbor search into something you can filter, cite, and audit. In a regulated fab, that is the difference between an answer people trust and an answer nobody can use.

Choose chunking from retrieval evidence. Preserve structure, protect tables and formulas, attach source metadata, and rerun the versioned fixture after every parser, splitter, embedder, or index change. Passing covers only the cases represented in that fixture. Once the boundaries are stable, continue with the embedding-model evaluation; otherwise model comparisons will be measuring broken chunks.
