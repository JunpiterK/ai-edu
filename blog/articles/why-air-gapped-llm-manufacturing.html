Choosing an isolation boundary for manufacturing LLMs

AE AI Edu Archive Semiconductor process engineer · Air-gapped LLM / RAG / Agents
On this page

  2. Choose the boundary
  3. Capability and operational tradeoffs
  4. Decision and verification checklist

Start with a blank data-flow worksheet. Give every row six fields: payload, classification, source, destination, transfer mechanism, and evidence owner. Include prompts, retrieved passages, outputs, logs, model artifacts, software packages, telemetry, identities, and administrator access. The network pattern follows the completed sheet; “air-gapped” is not a synonym for every local deployment.

Illustrative scenario: local was not the boundary

A rack-mounted GPU can make the deployment feel self-contained. The offline-start test is where that confidence slips: the embedding component attempts DNS resolution for a missing model artifact. No protected payload is shown leaving in this scenario, but the architecture claim is now unsupported. The release stops while the team inventories the missing artifact and reviews the attempted route. This is a constructed example, not a reported plant event.

Worksheet rule

If a component, payload, route, or owner is missing from the sheet, the boundary is not ready for approval. A deployment location is a fact; an isolation claim needs test evidence.

Decision framework

Use an air gap when policy or hazard analysis requires physical or equivalent isolation and controlled import/export. Use no-egress on-prem when the workload can stay on a routed internal network with outbound traffic denied. Use controlled transfer when approved artifacts or reviewed outputs cross a boundary through a managed gate. Use private cloud when contracts, region, tenant controls, identity, logging, and data handling satisfy the classification. Security, system, data, and business owners approve the choice.

What actually leaves when you paste into a cloud LLM

A request to an externally hosted service is an outbound transfer. The payload can include more than the visible question: retrieved chunks, conversation history, system prompts, tool results, attachments, identifiers, and diagnostic logs. Review the actual provider, contract, region, retention, training-use settings, subprocessors, and client telemetry rather than inferring them from the chat interface.

  • Prompts can contain protected know-how. Recipe context, tool identifiers, process windows, and troubleshooting notes may carry the information needed for a useful answer.
  • Yield & defect data is competitive intelligence. A defect-rate table or an excursion timeline tells a competitor precisely where process is weak. Apply the site's classification and approved-use rules.
  • Contractual and regulatory duties depend on context. Legal, export-control, privacy, and customer obligations should be assessed by the responsible specialists for the specific data and service.
  • Location and handling need evidence. Confirm processing region, storage, retention, access, deletion, and incident terms for the selected service.

An unapproved transfer does not require malicious intent. Usable controls include approved tools, clear data-handling rules, technical blocking where appropriate, and review paths that do not force engineers to improvise.

Common mistake 1: "It runs locally, so no data can leave."

A local inference process does not establish the boundary for embedding clients, vector stores, package managers, observability, DNS, authentication, or administration. Inventory the complete stack, deny unapproved routes, and collect firewall, proxy, DNS, and configuration evidence under representative startup and load conditions. Packet capture can supplement those controls when the network owner approves it.

The data that improves a prompt may also be the data that policy does not allow you to send.

Choose the boundary

The four patterns below solve different assurance and operating problems. Select one from data classification, required integrations, update path, support model, recovery objective, and accepted residual risk.

For the air-gap pattern shown below, the internal stack has three functional layers:

  • On-prem serving. An open-weight model (Llama, Qwen, gpt-oss, and peers) running on your own GPUs behind a serving stack like vLLM or Ollama. The weights are files you copied in through an approved import process; runtime egress is blocked and verified.
  • Air-gapped RAG. Approved SOPs, spec sheets, and equipment logs are embedded and indexed locally. The design intends to keep document content inside the boundary; network, storage, logging, and transfer evidence must verify that claim.
  • Internal wiring. The model connects to in-house systems (MES, historians) through scoped tools. Read and write permissions, audit coverage, and approval gates are verified separately.

Common mistake 2: "We can just run pip install and pull the model."

Automatic model and package downloads fail when no route exists and can create unreviewed supply-chain paths when a route does exist. Build a versioned bill of materials, verify licenses and hashes, scan artifacts, include tokenizers and runtime dependencies, test offline startup, and import through the approved mechanism. Settings such as HF_HUB_OFFLINE=1 are useful application controls, not proof of network isolation.

In this pattern, model artifacts enter through the controlled import path while prompts, retrieved context, responses, and operational logs remain inside the defined boundary. The diagram is a design claim. Release evidence comes from the manifest, network controls, transfer records, identity tests, and observed runtime behavior.

Illustrative boundary test

Example procedure, not a plant test result: start the service with an empty local tokenizer cache and the intended network policy. Record attempted DNS and outbound connections from every component. Import missing artifacts through the approved manifest, repeat under the declared load, and let the network and security owners disposition any connection attempt before release.

Capability and operational tradeoffs

Isolation does not determine answer quality by itself. Model family, quantization, context length, serving configuration, retrieval quality, language coverage, and evaluation set all matter. Open-weight families can be evaluated locally, but suitability is a measured use-case decision rather than a general claim about on-prem or hosted models.

RAG can give a local model access to approved internal evidence. That grounding may improve a document-based task when retrieval returns the right current source, but it does not guarantee a correct answer. Compare candidate architectures on the same approved questions, sources, roles, latency conditions, and refusal cases.

Common mistake 3: "The card has 24 GB of VRAM, so the model fits."

Weight size alone omits KV cache, runtime overhead, context length, batch shape, and concurrency. Test the intended prompt and output distributions under load, record peak memory and failures, and leave measured operating headroom. Cache quantization or shorter context can reduce memory use where the selected runtime supports them, with quality and latency re-evaluated.

Common mistake 4: "Four-bit quantization is automatically better because it is smaller."

Lower precision trades memory for task-dependent quality and performance. Treat formats such as Q4_K_M as candidates, then measure them on the approved retrieval and answer set. Do not infer a release threshold from format size alone.

Decision and verification checklist

Record the accountable owner and acceptance evidence for each item before pilot release:

  • Boundary and transfers. Name the selected pattern, allowed flows, denied flows, import/export process, and recovery path.
  • Artifacts and licenses. Keep a bill of materials, hashes, scan results, provenance, and specialist review of applicable license terms.
  • Capacity. Declare model, quantization, GPU, context, concurrency, latency objective, and load-test result.
  • Identity and retrieval authorization. Test who can query, which sources each role can retrieve, administrator access, and audit retention.
  • Data governance. Record allowed classifications, source owners, retention, deletion, current-revision rules, and incident handling.
Boundary tabletop exercise

Select one prompt and trace every copy: client memory, gateway log, retrieval query, source chunk, model request, response, audit store, backup, and administrator view. Pass when each copy has an approved location, retention rule, access owner, and observed route consistent with the worksheet. Stop on an unknown sink, unapproved DNS or HTTP attempt, missing import record, or a write-capable plant tool without its named approving owner. Limit: a tabletop plus packet capture samples known paths; it does not prove the absence of every covert or future path.

Common mistake 5: "Everything is installed, so why will it not start?" CUDA and driver mismatch.

Pin and validate the driver, accelerator runtime, serving engine, and model format together before import. Test cold start and multi-GPU visibility on representative hardware, and retain the previous compatible bundle for rollback.

Common mistake 6: RAG is running, but retrieval misses because the embedding language is wrong.

An English-centric embedder may miss Korean technical language. Evaluate multilingual candidates, such as bge-m3, against approved site documents and questions. A named model is an evaluation candidate, not a universal recommendation.
