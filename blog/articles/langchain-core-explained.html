LangChain core, explained: LLMs, prompts, and LCEL pipelines

LangChain core is not magic around an LLM. It is a small set of interfaces that make prompts, models, parsers, retrievers, and tools compose in the same way. Once you understand those interfaces, a local LLM app stops looking like a pile of callbacks and starts looking like a data pipeline.

What LangChain actually solves

The practical problem LangChain solves is replacement. You want to swap a local Ollama model for another provider, change a prompt without touching retrieval, replace a string parser with a JSON parser, or stream a chain without rewriting the chain. Core abstractions make those pieces interchangeable.

For on-prem work, that matters because the stack changes often. A prototype may start with a 7B model on a gaming GPU, move to a larger quantized model, and later run behind an internal API. The application code should not collapse every time the model endpoint changes.

LangChain's value is not any one class. It is that the prompt, the model, the parser, and the retriever all speak the same interface — so you compose them like Unix pipes. — the whole point of LCEL

The ChatModel: connecting an on-prem model with ChatOllama

The ChatModel is the model-facing abstraction. In a local lab, ChatOllama is usually the simplest starting point: it talks to an Ollama server and returns chat messages. You configure model name, temperature, and sometimes context length, then call invoke, stream, or batch through the same Runnable interface.

The important mental model is that the model receives messages, not raw strings. System messages carry policy and role. Human messages carry the user's task. AI messages are prior responses. LangChain keeps that structure explicit instead of forcing you to hand-build JSON payloads for every provider.

A prompt template turns variables into messages. ChatPromptTemplate.from_messages is the workhorse because it lets you write a system instruction once and inject user-specific fields at runtime. The result is a reusable Runnable: pass a dictionary in, get a prompt value out.

  • invoke(x) — one input, one full answer. Blocks until done.
  • stream(x) — yields tokens as they are produced, for live UIs.
  • batch([x1, x2, …]) — many inputs at once, with built-in concurrency. Ideal for labeling a stack of logs.

Keep prompts boring and testable. Put stable behavior in the system message, user content in human messages, and retrieved context in a clearly named field. That makes it easier to see whether a wrong answer came from the model, the prompt, or the retrieved evidence.

Prompts: ChatPromptTemplate.from_messages

Models return messages. Applications often need strings, JSON, Pydantic objects, lists, or routing labels. Output parsers turn loose model output into the shape the rest of your program expects. StrOutputParser is the simplest case; structured parsers are the next step when format matters.

For a blog workflow, a parser might return title, summary, slug, and tags. For a fab workflow, it might return risk_level, reason, and required_approval. The point is the same: downstream code should depend on a contract, not on a paragraph that merely looks right.

LCEL, the LangChain Expression Language, is the pipe syntax: prompt | model | parser. Each piece is a Runnable. That gives the whole chain a shared interface: invoke for one input, batch for many inputs, stream for partial output, and composition for larger systems.

Output parsers: from loose text to typed objects

This is why the syntax is worth learning. It is not just prettier function composition. It makes chains inspectable, swappable, and testable. When one piece changes, the surrounding pipeline usually stays the same.

RunnableParallel and RunnablePassthrough are the bridge from simple chains to RAG-style chains. RunnablePassthrough keeps the user's question available. RunnableParallel lets you compute several fields at once, such as question and context. Then the prompt receives a tidy dictionary.

A typical RAG chain looks like this: take the question, retrieve documents into context, keep the original question, format both into a prompt, call the model, parse the answer. No special RAG magic is required; it is the same Runnable composition with one branch doing retrieval.

LCEL and Runnable: the pipe that ties it together

The common mistake is hiding too much inside one function. If a function retrieves, formats, prompts, calls the model, parses, and logs, you cannot easily test the pieces. LCEL encourages small named steps.

Once a chain needs conditional routing, retries, human approval, or durable state, it is time to move from LangChain core to LangGraph. Core gives you composable steps. LangGraph gives you a state machine for deciding which step happens next.

Injecting context with RunnableParallel and RunnablePassthrough

The clean progression is: plain chat chain, structured output, tiny RAG, one safe tool, router graph, approval workflow. Each layer adds a reasoned capability instead of piling complexity onto a single prompt.

  • RunnableParallel — runs several Runnables on the same input concurrently and returns a dict of their outputs. A Y-fork in the chain.
  • RunnablePassthrough — forwards the input unchanged, so the raw question survives alongside the freshly retrieved context.

For local/on-prem portfolios, this article is the foundation. If you can explain ChatModel, PromptTemplate, Parser, Runnable, and LCEL clearly, then RAG, tools, MCP, and agents become straightforward extensions.
