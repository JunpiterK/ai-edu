Contract check

A chain receives a dictionary and returns an object. Before reading the pipe operator, write down the type at each boundary: input variables, prompt messages, model message, parser result, and failure. That ledger is the working model for this guide.

Illustrative boundary trace

A malformed final answer initially puts the local model under suspicion. The captured prompt messages are correct, however, and the parser receives an AIMessage where a downstream function expects a string. That boundary artifact clears the model and points straight to the missing parser.

LangChain core gives prompts, chat models, parsers, and other Runnables a shared way to connect. The pipe is compact; the values crossing it still deserve ordinary interface discipline. For each component below, ask four questions: what enters, what leaves, what can vary, and how does failure reach the caller?

Debug shortcut

Capture the value and Python type on both sides of the failing pipe before changing the prompt or model. Most wiring faults become pleasantly boring once the boundary is visible.

What LangChain actually solves

The practical problem is controlled substitution. A model, parser, or retriever can be replaced when the new component satisfies the same input and output contract. Provider-specific configuration, message support, streaming behavior, token limits, and exceptions can still differ, so substitution needs contract tests.

For a local stack, record the package version, model identifier, endpoint, prompt revision, parser schema, timeout, and representative test cases. Those artifacts make a component change reviewable.

For every pipe boundary, specify the accepted value, returned value, timeout, and error behavior. Shared Runnable methods do not make component semantics identical.

The ChatModel: connecting an on-prem model with ChatOllama

Contract. A chat model accepts a supported prompt or message sequence and returns an AI message. ChatOllama adapts that contract to an Ollama endpoint. Verify timeout, context limit, tool-call support, and response metadata for the selected model and package version.

invoke, stream, and batch share a Runnable vocabulary, but they have different operational behavior. Test partial-stream failures, batch concurrency, cancellation, and retry policy at the adapter boundary.

  • invoke(x) — one input, one full answer. Blocks until done.
  • stream(x) — yields tokens as they are produced, for live UIs.
  • batch([x1, x2, …]) — many inputs at once, with built-in concurrency. Ideal for labeling a stack of logs.

Test boundary. Assert the returned type, empty-response handling, maximum latency under the stated host load, and behavior when the endpoint is unavailable. Content-quality evaluation is separate; an AI message can satisfy the type contract and still be wrong.

Prompts: ChatPromptTemplate.from_messages

Contract. ChatPromptTemplate accepts a mapping containing every required variable and returns a prompt value containing messages. Missing variables should fail before the model call. Separate system instructions, retrieved context, and user text so tests can inspect their positions.

Prompt text is instruction, not an authorization control. Fields such as required_approval may describe a proposed route, but application code must enforce identity and policy outside the prompt.

Test boundary. Invoke the template with a complete mapping and assert message roles and variable placement. Then test missing variables, braces in untrusted text, oversized context, and a context field containing instruction-like content.

Output parsers: from loose text to typed objects

Contract. An output parser accepts a model result and returns the application type. StrOutputParser returns text. with_structured_output() is a model wrapper, not a standalone parser: it asks the provider or tool-calling adapter for the declared shape and returns a schema instance only when generation and validation succeed. Callers must handle validation errors, missing fields, invalid enum values, and provider behavior that does not support the requested schema.

Typed output constrains shape, not factual accuracy. Validate domain values and authorization separately before downstream use.

Test boundary. Test StrOutputParser directly with fixed model-result fixtures. Test triage_llm through a stub chat model or recorded provider responses that exercise valid output, malformed output, and schema-validation failure; it does not expose a separate parser to call directly. Assert the returned Python type and exact exception path before adding a live-model integration test.

LCEL and Runnable: the pipe that ties it together

Contract. LCEL composes Runnables when the upstream output is accepted by the downstream input. The composed chain exposes Runnable methods, but timeout, retry, streaming, and concurrency behavior still depend on its components and configuration.

Test boundary. Stub each Runnable and assert the values crossing every edge. Add integration tests for endpoint timeout, parser failure, and batch limits. Conditional routing, human review, and durable state introduce workflow contracts beyond this straight-line chain.

Injecting context with RunnableParallel and RunnablePassthrough

Contract. RunnableParallel returns a mapping of branch outputs. RunnablePassthrough forwards its input. In this RAG shape, the prompt must accept both context and question, and the formatter must accept the retriever's document list.

  • RunnableParallel — runs several Runnables on the same input concurrently and returns a dict of their outputs. A Y-fork in the chain.
  • RunnablePassthrough — forwards the input unchanged, so the raw question survives alongside the freshly retrieved context.

Worked boundary ledger

Simplified exercise. Run the purge-time question with a stub retriever that returns one known Document. Record four rows: input dictionary, rendered message roles, model-response type, and parser-return type.

  • Pass: every required variable is present, context remains separate from the question, the model adapter returns an AI message, and the parser returns the type expected by the caller.
  • Stop: a missing variable reaches the model, instruction-like context changes message roles, an endpoint error becomes empty text, or the parser failure is swallowed.
  • Limit: a clean type ledger proves composition, not factual accuracy, source authority, latency under load, or permission to act.
