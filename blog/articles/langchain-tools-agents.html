Pre-use review

The proposed tool has one function: read a lot status from an MES replica. Before a model receives its schema, the review team asks what identity reaches the replica, how stale the result may be, what a timeout returns, and which log proves the lookup occurred.

Illustrative message trace

No visible lookup result makes it seem that the model refused to call get_lot_status. The saved assistant message tells a different story: it contains a valid tool request, while the application printed content and ignored tool_calls. The snag is in the loop, not the model. Inspecting the whole message changes the next action from prompt tuning to implementing the tool-result turn.

A language model does not have an MES session. A LangChain tool exposes a bounded application function that the model may request. The application still owns authentication, authorization, validation, timeout, logging, and error handling. The examples below put the request and response messages on the bench; they are instructional sketches, not a qualified MES connector.

Trace shortcut

Log the assistant message, validated tool-call payload, connector result, and matching tool_call_id as one trace. If one link is missing, stop before asking the model for a final answer.

Why a model needs tools

Ask a model for the current status of lot A1234 without supplying a trusted lookup and it may produce an unsupported answer. Tool calling changes the interface: instead of the model producing the final answer, it can produce a structured request — “call get_lot_status with lot_id="A1234"” — and your code runs the real function and hands the result back. The model never touches the database directly; it only asks.

The model request must pass through an allowlisted tool registry and server-side policy. A tool call is a structured event that can be rejected, rate-limited, logged, or escalated. Network isolation and data-flow restrictions must be verified in the environment; binding a tool does not provide them.

Treat model output as an untrusted request. Application code decides whether the authenticated principal may execute the named read operation.

Defining a tool with @tool

What it does. The @tool decorator turns an ordinary function into a BaseTool: a callable the model can discover and invoke. LangChain reads three things off your function and ships them to the model as a JSON schema — the name, the docstring (which becomes the description the model reasons over), and the type hints (which become the argument schema). Clear names reduce ambiguity, but call selection and argument accuracy still require model-specific tests.

The decorator lives in langchain_core.tools. Type your arguments, and write the docstring as if it were the only instruction the model gets — because it is.

Boundary. A replica protects the transactional database from this query path, but it introduces lag. A qualified implementation should return a structured status with source, schema version, and observed_at; the compact string above keeps the tool mechanics visible and omits those required controls.

Letting the model call it: bind_tools

What it does. bind_tools attaches your tool schemas to a chat model, so that on each turn the model may respond with normal text or with one or more tool calls. It does not run anything — it only gives the model the option and returns any requests it makes on the message's tool_calls field.

A compatible tool-calling chat model is required. This example uses ChatOllama from the langchain-ollama package, pointed at a local model that supports function calling (for example a recent Llama or Qwen build). Note one on-prem caveat: some Ollama models ignore tool_choice, so you cannot always force a call — you rely on the docstring to steer it.

The response carries tool_calls instead of a finished answer. Your code is now responsible for running the requested function and returning the result — which is the loop we build next. The model has requested a tool call; it has not produced the answer yet.

The reason-act loop, by hand

Tool calling only becomes useful when you close the loop: run the requested tool, append its result as a ToolMessage, and invoke the model again so it can read that result and either call another tool or write the final answer. This is the ReAct pattern — reason, act, observe, repeat — and doing it once by hand is the best way to see what an agent actually is.

Passing the whole call dict to tool.invoke() is the current idiom: the tool returns a ToolMessage already tagged with the matching tool_call_id, so the model can line up each result with its request. The range(5) cap bounds this example's loop. Select the runtime limit from latency and load tests, and return an explicit escalation result when it is reached.

create_agent: the loop, prebuilt

LangChain provides create_agent in langchain.agents to assemble the reason-act graph on top of LangGraph. Streaming, checkpointing, and durable execution require explicit configuration and testing. It supersedes the older langgraph.prebuilt.create_react_agent — and both replace the long-deprecated initialize_agent. Pass a model, a list of tools, and a system prompt; you get a runnable graph.

The input and output are message lists — the same shape the hand loop used, now managed for you. To watch the reasoning as it happens, stream it: each step arrives as a chunk, so an operator sees the tool call and its result before the final line lands.

On-prem notes and guardrails

The loop is small; its operational controls determine whether it is acceptable for the bounded read-only use case. Review these controls with the system owners:

  • Keep this integration read-only. Start with lookups (get_lot_status, spec fetches). A tool that writes — releases a hold, changes a recipe — should require a human confirmation step, not a model's say-so. A human click alone is not sufficient; a separate authorized system workflow must enforce any write.
  • Cap the loop. Whether hand-written or via create_agent, bound the number of iterations so a confused model cannot spin. Pass a tested recursion_limit to the prebuilt agent and convert GraphRecursionError into an explicit unavailable or escalation outcome.
  • Reconcile tool and connector logs. A model-side call record alone is incomplete. Match request ID, principal, policy decision, connector outcome, source, and timestamps.
  • Verify data flow. Test model endpoints, telemetry, dependencies, logs, and error reporting against the approved network and retention design.

Replay a four-event tool trace

Simplified exercise. Use a stub MES replica and save four events for lot A1234: user question, assistant tool request, tool result with observed_at, and final assistant answer. Then replay a timeout and a stale result.

  • Pass: the tool name is allowlisted, the lot ID validates, request and result IDs match, source age is visible, and timeout or stale data produces an unavailable answer rather than a current-status claim.
  • Stop: the model can name an unregistered tool, a result lacks source time, retries exceed the declared cap, or connector and agent events cannot be reconciled.
  • Limit: the stub does not establish MES truth, production load behavior, identity propagation, or authorization.

@tool exposes the schema, bind_tools lets the model request it, and the loop returns the result. Once that trace is understandable, continue to the MCP article to see how the same tool contract can be published and governed across clients.
