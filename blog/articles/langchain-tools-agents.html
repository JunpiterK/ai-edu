Tools and agents in LangChain: @tool, bind_tools, and create_agent

AE AI Edu Archive Semiconductor process engineer · Air-gapped LLM / RAG / Agents
On this page

  2. Defining a tool with @tool
  3. Letting the model call it: bind_tools
  4. The reason-act loop, by hand
  5. create_agent: the loop, prebuilt
  6. On-prem notes and guardrails

A language model can reason about a spec, but it cannot read the MES. It has no live access to your lot tracker, your recipe database, or the last hour of chamber pressure. A tool closes that gap: it is a plain Python function the model is allowed to call, so the model can fetch a real number instead of hallucinating one. An agent is the loop that lets the model decide which tool to call, read the result, and decide again. This article builds both from scratch, then replaces the hand-written loop with LangChain's prebuilt create_agent. Every import is verified against the current LangChain 1.x line — no deprecated initialize_agent, no LLMChain. 폐쇄망에서 쓰는 도구·에이전트를 최신 API로만.

Why a model needs tools

Ask an on-prem model “what is the current status of lot A1234?” and, with no access to your systems, it will invent a plausible-looking answer. That is worse than useless in a fab. Tool calling fixes this by inverting control: instead of the model producing the final answer, it produces a structured request — “call get_lot_status with lot_id="A1234"” — and your code runs the real function and hands the result back. The model never touches the database directly; it only asks.

That indirection is exactly what a closed network wants. Every tool is code you wrote and audited. The model cannot reach anything you did not hand it, and each call is a discrete, loggable event you can gate, rate-limit, or route to a human. 모델은 도구를 요청만 하고, 실제 조회는 검증된 코드가 수행합니다 — 어떤 데이터도 망 밖으로 나가지 않습니다.

A tool call is a request, not an action. The model asks; your audited code decides whether — and how — to answer. — the safety property that makes agents fab-viable

Defining a tool with @tool

What it does. The @tool decorator turns an ordinary function into a BaseTool: a callable the model can discover and invoke. LangChain reads three things off your function and ships them to the model as a JSON schema — the name, the docstring (which becomes the description the model reasons over), and the type hints (which become the argument schema). Write those three well and the model calls the tool correctly; write them vaguely and it will not.

The decorator lives in langchain_core.tools. Type your arguments, and write the docstring as if it were the only instruction the model gets — because it is.

Fab context. Point the tool at a read-replica, not the live MES, and keep the return value a compact string the model can quote. The docstring is your prompt-engineering surface: naming the argument lot_id and giving an example ("A1234") measurably lowers malformed calls. 도구의 도크스트링이 곧 프롬프트입니다 — 인자 이름과 예시가 정확할수록 호출이 정확해집니다.

Letting the model call it: bind_tools

What it does. bind_tools attaches your tool schemas to a chat model, so that on each turn the model may respond with normal text or with one or more tool calls. It does not run anything — it only gives the model the option and returns any requests it makes on the message's tool_calls field.

Any tool-calling chat model works. On-prem, that is ChatOllama from the langchain-ollama package, pointed at a local model that supports function calling (for example a recent Llama or Qwen build). Note one on-prem caveat: some Ollama models ignore tool_choice, so you cannot always force a call — you rely on the docstring to steer it.

The response carries tool_calls instead of a finished answer. Your code is now responsible for running the requested function and returning the result — which is the loop we build next. 모델은 답을 만들지 않고 도구 호출을 요청했을 뿐입니다.

The reason-act loop, by hand

Tool calling only becomes useful when you close the loop: run the requested tool, append its result as a ToolMessage, and invoke the model again so it can read that result and either call another tool or write the final answer. This is the ReAct pattern — reason, act, observe, repeat — and doing it once by hand is the best way to see what an agent actually is.

Passing the whole call dict to tool.invoke() is the current idiom: the tool returns a ToolMessage already tagged with the matching tool_call_id, so the model can line up each result with its request. The range(5) cap matters in a plant — it bounds cost and stops a confused model from looping. 반복 상한은 폐쇄망 운영에서 필수입니다.

create_agent: the loop, prebuilt

You rarely hand-write that loop in production. LangChain 1.x ships it as create_agent (from langchain.agents), which builds the full reason–act graph on top of LangGraph and gives you streaming, checkpointing, and durable execution for free. It supersedes the older langgraph.prebuilt.create_react_agent — and both replace the long-deprecated initialize_agent. Pass a model, a list of tools, and a system prompt; you get a runnable graph.

The input and output are message lists — the same shape the hand loop used, now managed for you. To watch the reasoning as it happens, stream it: each step arrives as a chunk, so an operator sees the tool call and its result before the final line lands.

On-prem notes and guardrails

The pattern is small; the discipline around it is what makes it safe in a fab. A few rules that hold up in a closed network:

  • Tools are read-only until proven otherwise. Start with lookups (get_lot_status, spec fetches). A tool that writes — releases a hold, changes a recipe — should require a human confirmation step, not a model's say-so. 쓰기 도구는 사람 승인 뒤에.
  • Cap the loop. Whether hand-written or via create_agent, bound the number of iterations so a confused model cannot spin. The prebuilt agent lets you set a recursion limit.
  • Log every tool call. Each call is a structured event — name, args, result. That trail is your audit record and your debugging surface.
  • Everything stays on-prem. The model is ChatOllama on a local GPU node; the tools hit internal replicas. No prompt, no lot ID, no result ever leaves the network.

That is the whole ladder: @tool exposes a function, bind_tools lets the model request it, the reason–act loop closes the circle, and create_agent ships that loop production-ready. Start with one read-only lookup tool and a capped loop, log every call, and you have an agent that answers from live fab data instead of guessing — without a single byte leaving the plant.
