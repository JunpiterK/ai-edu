Field experience

My starting artifact was a plain four-line handoff: input file, transformation rule, expected output, and acceptance check. I did not enter manufacturing as a programmer. I joined with a doctorate in science and worked on process improvement and new manufacturing methods. I began experimenting with generative AI in 2024; from early 2025, ChatGPT, Gemini, Grok, and Claude became regular tools for code work. The useful loop was to run the output, inspect the failure, reduce it to an approved reproducible example, and try again.

The short version

Generative AI shortened the distance between a process problem I understood and code I could test. The durable habits were decomposing the problem, specifying inputs and expected outputs, returning complete errors from approved synthetic reproductions, and following the current data and software-transfer authorization.

Starting without a coding background

I began with small scripts: merge CSV files, normalize columns, calculate daily summaries, and draw charts for a review. I already knew which tools belonged together, which lots must be excluded, and why an outlier could not simply be deleted. The useful prompt was a testable specification: input columns, exclusion rules, expected table, and an acceptance check.

Observed workflow

“Write a good program” left too many decisions unstated. Supplying the runtime, sample input, expected output, edge cases, and verification checks produced code that was easier to inspect and correct.

Controlled security case study

Non-transferable authorization boundary

This section records one author's approved inbound-only exception at that organization and time. It is not a policy template or permission for another employee, site, or company. In this case, generic code and package files created or obtained outside could enter only through the route approved by the security team; internal code and information could not leave. The author used an LTE tablet, Acode, and an approved email text-transfer route within that narrow workflow. Personal devices, personal email, public AI, removable media, and any inbound software may be prohibited elsewhere. Obtain written authorization from the relevant security and system owners before transferring anything.

My actual method was manual. I found each required module on PyPI, saved the files in a folder approved for import to the company PC, and installed them one by one. I gave Claude or GPT the exact environment, such as 64-bit Windows with CPython 3.10.1, and requested direct and transitive dependencies, compatible ranges, proposed pins, wheel filenames, and installation order. That answer was a checklist, not authority; every item was checked against PyPI.

Package search: Open the official PyPI site (pypi.org)

Finding the right PyPI file, screen by screen

Step 0: record the target PC first

Record the Python implementation and version, OS, CPU architecture, and Python bitness. CPython 3.10.1 normally maps to the cp310 tag. The strongest local evidence is the compatible-tag list from python -m pip debug --verbose.

python --version
python -c "import sys, platform, struct; print(sys.implementation.name); print(platform.system()); print(platform.machine()); print(struct.calcsize('P') * 8)"
python -m pip debug --verbose

Step 1. Search the package name on the PyPI home page

Open pypi.org, type the exact module name into the central search box, and press Enter or the magnifier.

Step 2. Select the exact project, not a similar name

Results such as pandas, quick-pandas, and random-pandas are different distributions. Match the exact name, summary, and release activity before opening the card.

Step 3. Check the project page and open Release history

The header shows the current default release. If the company Python is older, the newest package may not support it. Use Release history before downloading files.

Step 4. Choose a compatible, non-withdrawn release

Do not choose solely by the largest version number. Avoid releases marked withdrawn and treat pre-releases as test candidates. Check Python support and the versions already validated with the rest of the stack. This example deliberately uses pandas 2.0.3 because it provides CPython 3.10 wheels.

Step 5. Verify the selected version page

Confirm the header says pandas 2.0.3 and the pinned command says pip install pandas==2.0.3, then open Download files.

Step 6. Distinguish source archives from built wheels

A .tar.gz source distribution may require local compilers and headers. In a sealed Windows environment, prefer a matching built .whl when one exists. You need one matching wheel, not every file on the page.

Step 7. Filter to the target interpreter and platform

For CPython 3.10.1 on 64-bit x86 Windows, select cp310 and win_amd64. Re-read the gray description: it should say CPython 3.10 and Windows x86-64.

Advanced reading from the supplied screenshot: what cp314t means

The supplied pandas 3.0.3 screen separates Windows ARM64 and x86-64, musl Linux, glibc Linux, Apple Silicon, and Intel macOS builds. Every visible filename contains cp314-cp314t: cp314 identifies the CPython 3.14 interpreter, while cp314t is the ABI tag for a free-threaded CPython 3.14 build. The trailing t matters. An ordinary CPython 3.14 build uses a different ABI wheel such as cp314-cp314.

The decision

For an ordinary CPython 3.10.1 64-bit Windows PC, every file visible in this screenshot is the wrong choice. Even cp314t-win_amd64 matches only the OS and CPU; its Python and ABI tags do not match. Interpreter, ABI, and platform must all agree.

python -VV
python -c "import sysconfig; print(sysconfig.get_config_var('Py_GIL_DISABLED'))"
# 1 indicates a free-threaded-capable build.

win_arm64 is Windows ARM, while win_amd64 is Windows x86-64. musllinux_1_2 targets musl libc 1.2 or later. manylinux_*_x86_64 and manylinux_*_aarch64 target glibc-based Linux on different CPU architectures. macosx_11_0_arm64 is Apple Silicon with macOS 11 or later, while macosx_10_15_x86_64 is Intel macOS 10.15 or later. A filename containing multiple manylinux tags declares multiple platform compatibility tags; let the target machine's pip compatible-tag list make the final decision.

Every inbound package followed the approved software-import route, including malware, license, and hash checks where required. I kept equipment names, recipe values, yield, alarms, filenames, and internal documents out of external prompts.

Verified first-person incident

I did not begin with a perfect understanding of this distinction. The security department flagged an instance involving code leaving the company, and I signed a non-recurrence pledge. I then asked the security department to clarify the permitted direction instead of assuming. The rule confirmed for my situation at that time was that internal code and information could not be sent outside, while generic code created externally could be brought in through the approved import process. From then on, I sent no internal code or data to external AI services. I created only generic code and synthetic examples containing no company information outside the network, then brought them in through the route that had been confirmed.

Case limitation

The inbound-only route above was an approved exception for this author in that setting, not transferable policy. A technically possible transfer remains unauthorized unless the applicable organization explicitly approves the device, account, data class, direction, scanning, licensing, and import record.

The LTE tablet and Acode loop

Under that approved case-specific route, I drafted generic, non-sensitive code on an LTE tablet when external AI was inaccessible from the corporate PC. Chat alone was awkward for long files, so I used the Android editor Acode to inspect line numbers, indentation, syntax highlighting, and the complete file.

  1. Create a minimal example with no company, tool, product, or real process identifiers.
  2. Give the AI the goal, sample input, expected output, and runtime environment.
  3. Review the complete file in an editor.
  4. Use only the explicitly approved text or software-import route. Do not generalize this case or bypass attachment controls.
  5. Run first against synthetic data and capture the complete error and line number.
  6. Do not send an internal traceback outside. Reproduce the failure with approved synthetic content before requesting a correction.
Simplified example: a weak repair handoff

“It doesn't work” leaves the model guessing. A useful repair handoff includes the Python version, command, full traceback from an approved synthetic reproduction, synthetic failing input, and expected output. Stop if the failure cannot be reproduced without internal data; that is a review problem, not a prompt-writing problem.

A reproducible workflow

1. Define

One input, one output, one acceptance condition.

2. Authorize

Get the current data, device, account, network, and transfer route approved.

3. Minimize

Use generic code and synthetic examples only where the authorization permits external work.

4. Specify

State OS, Python, package versions, and network constraints.

5. Verify

Check row counts, totals, edge cases, and repeatable tests inside the approved environment.

6. Operate

Record approval route, licenses, hashes, dependencies, and rollback.

Worked handoff check

Use a synthetic CSV with 12 rows: 10 valid records, one missing timestamp, and one excluded recipe label. Pass when the script reports 12 input rows, explains both excluded rows, emits 10 rows, and reproduces the expected grouped totals. Stop if a row disappears without a reason, the environment cannot be reconstructed, or the correction would require sending internal data outside the approved boundary. Limit: this fixture checks transformation logic, not suitability for production data or process decisions.

This workflow shortened the path from a defined problem to testable code in my case. Its security step is organization-specific: current written authorization controls, not this article or the historical exception described above.
