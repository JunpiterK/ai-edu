Opening artifact: identity map

MES chamber=A-02, historian asset=CH_A2, and maintenance location=Bay7/A2 are three labels that may or may not describe one physical chamber. The graph should not merge them until a named source owner approves the mapping and its effective interval.

The useful graph work begins before RDF syntax. Start from system-of-record rows: lot history, chamber events, recipe revisions, alarms, and metrology observations. Then make identifiers and relationships explicit while keeping the source record close enough for reconciliation.

Constructed identity check

Several lots were absent from a particle-excursion query, putting the SPARQL pattern under suspicion. The reconciliation report showed a more basic snag: two source aliases had been mapped to separate chamber nodes. After the source owner resolved the identity mapping, the query could be evaluated. The IDs and observations in this article are constructed, not plant records.

Choosing SQL, RDF, or both

Relational databases are a strong fit for controlled transactions, fixed constraints, aggregates, and well-understood joins. RDF is useful when multiple systems identify the same entities differently, relationships evolve, or a query must traverse several domains while retaining semantic labels and provenance. SQL engines can also execute multi-table joins well; RDF does not make an expensive or poorly indexed traversal disappear.

A practical split is to keep MES and historian records in their governed relational stores, publish approved snapshots or change events, and build an RDF integration layer for cross-domain investigation. Use SQL for source reconciliation and quantitative calculations. Use SPARQL for relationship patterns and provenance traversal. Measure both against representative queries before choosing storage and indexing technology.

The design question is which representation preserves the required constraints, query performance, identifiers, and source trace for this workload.

Triples: subject-predicate-object

RDF statements use a subject, predicate, and object. In this illustrative scenario, Chamber-A ran Lot-4471 and Lot-4471 usedRecipe Etch-Std-7. Shared identifiers make the statements traversable. The identifiers still require mapping rules and source controls; matching labels alone is not entity resolution.

Notice that ex:Lot-4471 appears as an object in one triple and a subject in the next. That shared identifier is the hinge that turns isolated facts into a traversable graph.

TBox and ABox: schema vs. facts

A working ontology has two layers. The TBox, or terminology box, defines classes and relationships: Chamber is a kind of Equipment, and ranLot connects a Chamber to a Lot. The ABox holds instances and assertions. RDFS or OWL domains can infer a type; they do not, by themselves, reject bad input. Validation needs a constraint layer.

  • TBox — fab:Chamber rdfs:subClassOf fab:Equipment. Defined once, changes rarely.
  • ABox — ex:Chamber-A rdf:type fab:Chamber. Grows every shift as new lots, recipes, and reports arrive.

SHACL and source controls

Use SHACL shapes to test required fields, cardinality, datatype, unit, and identifier patterns before publishing an ingest batch. For example, a process observation may require exactly one observed lot, one recipe revision, one source record, and one event timestamp. A failed shape goes to quarantine; it should not be silently repaired by the LLM or merged into the query graph.

  • Source allowlist: accept only registered exports or APIs with a named system owner.
  • Batch manifest: record source revision, extraction window, checksum, parser version, row count, and ingest run ID.
  • Identity and ACL mapping: resolve source principals and permission scopes before exposing graph results.
  • Reconciliation: compare accepted, rejected, and duplicate counts with the source snapshot, then retain the SHACL report.

SHACL reports establish conformance to the declared shapes, not truth or regulatory acceptance. The source owner must resolve semantic conflicts and approve mappings.

Reuse standard ontologies

Reuse established vocabulary where its semantics fit the source data and review requirements. Reuse can reduce mapping work, but an ontology name does not establish compliance or interoperability by itself. A few vocabularies are worth evaluating:

  • rdf: / rdfs: — the base layer: typing with rdf:type, subclassing, labels, and comments.
  • PROV-O (prov:) — the W3C provenance ontology: who or what generated a fact, from what, and when.
  • QUDT or OM — units and quantities, so 5.0 is unambiguously 5.0 mTorr, not 5.0 anything.
  • SOSA/SSN — sensors and observations, a natural fit for equipment telemetry and metrology.

Define fab-specific terms such as Chamber, Recipe, Lot, ProcessStep, and ToolGroup only after checking the source semantics. Document every mapping decision and its owner so later consumers know where two terms are equivalent, narrower, or merely similar.

PROV-O: observation-level provenance

Attach provenance to the observation or assertion record, not only to a lot or recipe node. The statement that Lot-4471 used Etch-Std-7 may come from one MES event at a specific time; another statement about the same lot may come from metrology. Giving each observation its own identifier lets a query return the source record, extraction activity, responsible service, event time, and assertion time without implying that one source supports every fact about the entity.

Loading triples with rdflib

rdflib is enough for a local instructional graph. Bind namespaces, add triples, and create a separate observation entity for the provenance of the recipe-use assertion. Persistence, concurrency, authorization, backups, and query performance require separate evaluation.

You can also parse existing RDF with g.parse("export.ttl", format="turtle"). Do so only after source registration, SHACL validation, and identifier reconciliation; parsing success is not an ingestion approval.

Querying with SPARQL

SPARQL matches graph patterns; SQL evaluates relational queries. Either can return joined data and provenance when the schema stores them. Here SPARQL is convenient because the investigation follows RDF relationships and returns the observation source in the same pattern.

You can pass namespaces programmatically with g.query(q, initNs={"fab": FAB}) or pre-bind values with initBindings. The required observation patterns exclude recipe links that lack a matching observation; the optional source exposes incomplete provenance separately. Decide whether to reject, quarantine, or display those results with the source owner.

Implement one source-to-observation mapping, its SHACL shape, and a reconciliation report, then compare the same investigation in SQL and SPARQL. A valid, sourced graph can still carry a wrong source observation or mapping. When the graph is ready to serve retrieval, continue with the vector-plus-graph pattern and keep assertion provenance visible through fusion and reranking.
