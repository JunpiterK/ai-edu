A knowledge graph for process know-how

A process spec is a web of relationships, not a list. An etch recipe belongs to a chamber; that chamber sits in an equipment group; a lot ran on it during a specific shift; a deviation report references both the lot and the recipe. Flatten that into a spreadsheet and the connections vanish. A knowledge graph keeps those relationships explicit and lets you ask questions that cross equipment, recipe, lot, report, and provenance boundaries.

Why a graph, not a table

Relational tables are excellent when the schema is fixed and the joins are known in advance. Fab know-how is rarely that tidy. New equipment types, failure modes, process relationships, and exception cases appear constantly. The interesting questions, such as which lots touched any chamber that later showed particle excursions, are exactly the multi-hop joins that tables handle poorly. A graph stores each fact as an edge, so adding a new relationship does not require altering a central schema. Because every node and edge has a stable identifier, the same entity can be described by many sources without collision.

A table answers the questions you designed it for. A graph answers the ones you have not thought of yet.

Triples: subject-predicate-object

Everything in the graph is a triple: a subject, a predicate, and an object. Chamber-A ran Lot-4471. Lot-4471 usedRecipe Etch-Std-7. Each triple is one atomic fact, and facts compose into a graph by sharing subjects and objects. Read as sentences, a handful of triples already describes real fab structure.

Notice that ex:Lot-4471 appears as an object in one triple and a subject in the next. That shared identifier is the hinge that turns isolated facts into a traversable graph.

TBox and ABox: schema vs. facts

A working ontology has two layers. The TBox, or terminology box, defines the types and relationships that are allowed: Chamber is a kind of Equipment, and ranLot connects a Chamber to a Lot. The ABox, or assertion box, holds the actual instances: Chamber-A is a Chamber, and it ran Lot-4471. Keeping them separate means every fact can be validated against the schema, and a reasoner can infer new facts. If ranLot has domain Equipment, then anything that ranLot something is automatically classified as equipment.

  • TBox — fab:Chamber rdfs:subClassOf fab:Equipment. Defined once, changes rarely.
  • ABox — ex:Chamber-A rdf:type fab:Chamber. Grows every shift as new lots, recipes, and reports arrive.

Reuse standard ontologies

Do not invent vocabulary for concepts the world has already modelled. Reusing standard ontologies makes your graph interoperable, saves design effort, and, for a regulated fab, aligns provenance and measurement terms with language auditors already recognise. A few are worth knowing:

  • rdf: / rdfs: — the base layer: typing with rdf:type, subclassing, labels, and comments.
  • PROV-O (prov:) — the W3C provenance ontology: who or what generated a fact, from what, and when.
  • QUDT or OM — units and quantities, so 5.0 is unambiguously 5.0 mTorr, not 5.0 anything.
  • SOSA/SSN — sensors and observations, a natural fit for equipment telemetry and metrology.

Define only what is genuinely fab-specific, such as Chamber, Recipe, Lot, ProcessStep, and ToolGroup, then reuse standard vocabularies for the rest. This keeps the model smaller, easier to audit, and easier to connect to other systems later.

PROV-O: provenance for audit

In a regulated fab, a fact without a source is a liability. When the graph says a lot ran a particular recipe, an auditor will ask who says so, which record it came from, and when it was asserted. PROV-O answers exactly this with three core ideas: prov:Entity for the thing, prov:Activity for the process that produced it, and prov:Agent for the person or system responsible. Add predicates such as prov:wasGeneratedBy, prov:wasDerivedFrom, and prov:wasAttributedTo, and the graph stops being a claim and becomes evidence.

Loading triples with rdflib

On an air-gapped host, rdflib is enough to build, store, and query a graph in pure Python. Bind your namespaces once, add triples with graph.add(...) as (subject, predicate, object) tuples, and attach PROV-O provenance to the facts you assert.

You can also ingest existing RDF instead of building every triple by hand. Point g.parse("export.ttl", format="turtle") at an MES export that already emits RDF, and the triples merge into the same graph.

Querying with SPARQL

SPARQL is to a graph what SQL is to a table. You describe a pattern of triples with variables, and the engine returns every binding that matches. Because facts and provenance live in the same graph, one query can return both the answer and its source, which is what makes the result auditable.

You can pass namespaces programmatically with g.query(q, initNs={"fab": FAB}) instead of inline PREFIX lines, or pre-bind values with initBindings to parameterise reusable queries. The OPTIONAL block means lots without recorded provenance still appear. You can then flag them as ungoverned, which is precisely the gap an audit should surface.

Start small: model three entity types, Chamber, Recipe, and Lot; reuse PROV-O for provenance from day one; and grow the ABox as data arrives. A graph built this way becomes the backbone for GraphRAG, where an LLM retrieves connected facts with their sources rather than loose text chunks.
