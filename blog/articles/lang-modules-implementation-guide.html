Lang modules implementation guide

The confusing part of the Lang ecosystem is not one API. It is knowing which package owns which job. Once that clicks, the stack becomes ordinary Python: model in, prompt in, parser out, documents in, retriever out, tools in, graph out.

Think of the Lang modules as building blocks for LLM applications. A plain model call is only one block. The real value appears when you combine prompts, parsers, retrieval, tools, and graph state into workflows that can answer from your files, call local functions, ask for approval, and recover from errors.

The module map

Use this table as the mental model before writing code.

ModuleWhat it doesUse it when
langchain-coreBase abstractions: messages, prompts, output parsers, runnables, tools, documents.You are composing chains or defining tools. This is the common base of most Lang applications.
langchain-openai, langchain-ollama, etc.Provider-specific chat models and embeddings.You need to call a model provider or a local model server.
langchain-communityCommunity integrations: loaders, vector stores, tools, utilities.You need an integration that is not in core.
langchain-text-splittersChunking utilities for documents.You are building RAG and need stable retrieval units.
langgraphStateful, multi-step, cyclic agent workflows.You need memory, branching, approval, retry, or long-running state.

Rule of thumb. If it is a reusable interface, it probably lives in langchain-core. If it talks to a vendor or external system, it probably lives in a provider or community package. If the workflow has state and branches, reach for LangGraph.

The same pattern repeats: model alone is good for language work, RAG is good when answers must come from documents, tools are good when the model must ask Python to do something, and LangGraph is good when the workflow needs state, routing, retry, or approval.

Install set

For a home PC or local LLM tutorial, start with Ollama-compatible local serving. You can swap providers later without rewriting the whole app.

Expected output. You should end with installed packages and an activated shell prompt.

Chat model + prompt + parser

This is the smallest Lang app: a chat model, a prompt, and an output parser. Use it when you want deterministic text transformation, summarization, or a simple assistant response without tools or retrieval.

Use this pattern for: email rewriting, Korean/English technical translation, meeting-note summarization, code explanation, glossary generation, report drafting, and any task where the model only needs the user's input and the instructions you provide.

Do not use only this pattern for: questions that require private documents, current database values, exact calculations, or actions like writing files and running tests. Those need retrieval or tools.

What each module did. ChatOllama called the local model. ChatPromptTemplate turned variables into chat messages. StrOutputParser converted the model message into plain text. The pipe operator created a runnable chain.

Practical variation. If you need a JSON object instead of text, replace StrOutputParser with a structured-output parser or ask the model through a schema-capable interface. Example outputs include risk classification or blog metadata generation.

Expected output.

Load, split, embed, retrieve

RAG uses four families of modules. Loaders turn files into Document objects. Splitters make chunks. Embeddings turn chunks into vectors. Vector stores save and search those vectors. Retrievers expose a simple query-to-documents interface.

Use RAG when the answer must be grounded. If the user asks what a policy says, which SOP mentions an alarm, or where a claim came from, the model should retrieve source text first and answer from that text.

What each RAG module is for:

  • Document loaders: read external material into a standard Document shape with page_content and metadata.
  • Text splitters: cut long documents into retrievable chunks. Bad chunks make good models look foolish.
  • Embeddings: convert text into vectors so similar meaning can be searched.
  • Vector stores: save vectors and perform similarity search.
  • Retrievers: hide storage details behind invoke(question).

Expected output.

When to use each piece. Use loaders when the source is external or messy. Use splitters whenever documents are longer than your retrieval target. Use embeddings and vector stores when semantic search matters. Use retrievers when the rest of the app should not care how search is implemented.

Common practical choices. Use a small persistent vector store for personal notes or a prototype. Use Qdrant, Postgres with pgvector, or another server-backed store when multiple apps or users need the same index. Use metadata filters when the question includes document type, date, equipment, product, author, or permission scope.

Tools and structured calls

Tools are functions with schemas. Use them when the model should ask the system to do something: calculate, search, look up a file, call an internal API, or query an MCP server. Keep the first tools boring and safe.

Use tools when language is not enough. Models are bad calculators, unreliable database clients, and dangerous shell users unless capabilities are wrapped carefully. A tool is the wrapper. The model asks for estimate_weight_memory; your Python code performs the calculation; the model explains the result.

Good first tools:

  • Calculator: exact arithmetic, unit conversion, GPU memory estimates.
  • Search: query a local index or controlled file list.
  • Lookup: read a known config, policy, or glossary entry.
  • Test runner: run pytest only inside a disposable workspace.
  • Ticket draft: create a draft object, not a live external ticket at first.

Expected output.

Design rule. A tool docstring is not decoration. It is part of the interface the model reads. Write tool names, argument names, and docstrings as if a tired junior engineer will use them at 2 a.m.

Danger rule. Do not expose a general shell tool on day one. Start with narrow tools. run_tests() is safer than run_command(command). read_policy(name) is safer than read_file(path). The more general the tool, the more validation and approval you need.

LangGraph workflow

LCEL is great for straight-line chains. LangGraph is for workflows with state: classify, retrieve, answer, ask for approval, retry, or stop. The core idea is a StateGraph: each node reads the current state and returns a partial update.

Use LangGraph when you catch yourself writing workflow flags. If your code has variables like needs_retrieval, tool_failed, approval_required, retry_count, or final_answer, you are already building a graph mentally. LangGraph makes that state explicit.

Practical graph patterns:

  • Router graph: send questions to chat, RAG, tool, or refusal path.
  • RAG retry graph: retrieve, grade evidence, rewrite the query, retrieve again.
  • Human approval graph: draft an action, pause, resume only after approval.
  • Coding agent graph: inspect repo, propose patch, run tests, fix failures, summarize diff.
  • Incident assistant graph: classify alarm, retrieve SOP, call read-only diagnostics, produce next steps.

Expected output.

When to use LangGraph. Use it when the application is no longer a single pipe. If you need conditional branches, human approval, retries, memory, checkpointing, or multiple agents, a graph is usually clearer than a giant function.

A combined practical flow

Here is how the modules fit together in a real local assistant:

  1. User asks: Can my 12GB GPU run a 7B model for local RAG?
  2. LangGraph routes the question to the tool path because it mentions GPU memory.
  3. A tool estimates raw model memory and warns about KV cache overhead.
  4. The retriever fetches the local lab policy about safe model settings.
  5. The prompt combines tool result and retrieved policy.
  6. The model answers with a recommendation and source note.

How to choose the next module

When building a real app, do not add everything at once. Let the problem pull in the module.

If the problem is...Add thisWhy
The model writes the wrong format.Output parser / structured outputThe response needs a contract, not a better vibe.
The answer must come from our files.Loader + splitter + retrieverThe model needs evidence before generation.
The model needs exact math or lookup.@toolLet Python do exact work and let the model explain it.
The app has multiple possible paths.LangGraph routerBranching belongs in state, not prompt spaghetti.
A human must approve before action.LangGraph interrupt / approval stateApproval is a workflow step, not a prompt paragraph.
The same index powers several apps.Server-backed vector storeLocal in-memory prototypes stop scaling quickly.
The agent can edit files or run tests.Narrow tools + graph + logsActions need boundaries, audit, and rollback.

Recommended learning order

  1. Plain chat chain: prompt, model, parser.
  2. Structured output: force the model to return a predictable shape.
  3. Tiny RAG: one text file, one retriever, one answer with source.
  4. One safe tool: exact calculation or read-only lookup.
  5. Router graph: decide chat vs RAG vs tool.
  6. Agent workflow: inspect, plan, call tool, summarize.
  7. Approval and logging: make it safe enough to use repeatedly.

If you follow that order, you will understand why each Lang module exists before you depend on it. That is much better than starting with a full agent template and wondering which part broke.

Security note. Treat model output as untrusted. Do not pass arbitrary model-generated strings directly into file paths, SQL filters, shell commands, or deserializers. The more tools you add, the more boring validation matters.
