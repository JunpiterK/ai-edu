Repository brief

Build one command-line repository named lang-lab. It answers questions from a versioned local GPU/RAG policy file, estimates model-weight memory with one read-only calculation tool, and routes unsupported questions to a stop response. It does not call MES, edit files, execute shell commands, or control equipment.

Illustrative endpoint diagnosis

The initial dependency list assumes the repository needs every Lang package before the first prompt can run. Installation succeeds, and the first concrete failure is only an unreachable Ollama endpoint. That clue changes the build order: prove prompt -> model -> parser first, then add retrieval, one tool, and routing. A large dependency set would have hidden a small interface problem.

The package map is much easier to hold when every dependency serves this one bounded repository. langchain-core defines contracts, langchain-ollama adapts the local model, loaders and splitters prepare the policy file, Chroma stores candidate chunks, and LangGraph chooses retrieval, calculation, or stop.

Build shortcut

Keep one smoke test per new boundary. If the current boundary fails, stop there; another framework layer will not repair it.

The snippets are deliberately small enough to inspect. Treat them as parts on a bench, not a finished repository, until the shared configuration and acceptance tests at the end exist.

Use this table as the mental model before writing code.

Package ownership can change across releases, so pin and test the versions used by the repository. The table identifies the role each package plays here, not a universal selection rule.

Load, split, embed, retrieve

RAG uses four families of modules. Loaders turn files into Document objects. Splitters make chunks. Embeddings turn chunks into vectors. Vector stores save and search those vectors. Retrievers expose a simple query-to-documents interface.

Here RAG has one corpus and one purpose: retrieve the versioned local GPU/RAG policy and return source metadata with the answer.

What each RAG module is for:

  • Document loaders: read external material into a standard Document shape with page_content and metadata.
  • Text splitters: cut long documents into retrievable chunks. Bad chunks make good models look foolish.
  • Embeddings: convert text into vectors so similar meaning can be searched.
  • Vector stores: save vectors and perform similarity search.
  • Retrievers: hide storage details behind invoke(question).

Expected output.

Retain the source checksum, chunk configuration, embedding model ID, collection ID, and fixed retrieval results. This guide uses one local collection; shared multi-user storage is outside scope.

Tools and structured calls

The repository exposes one tool schema: estimate_weight_memory. Python calculates raw weight storage from parameter count and quantization bits. The result excludes KV cache, runtime buffers, framework overhead, and measured throughput.

LangGraph workflow

The graph stores the question, selected route, and answer. Its conditional edge permits only tool, rag, or stop. Route tests should use fixed questions and must confirm that the stop path calls neither tool nor retriever.

A possible next-stage repository flow

The code above is a routing-only smoke test: each selected node returns guidance text and then ends. It does not yet call a tool, retriever, or model. A later implementation could compose the following bounded path:

  1. User asks: Can my 12GB GPU run a 7B model for local RAG?
  2. LangGraph routes the question to a composed calculation-and-retrieval path.
  3. A tool estimates raw model memory and warns about KV cache overhead.
  4. The retriever fetches the local lab policy about tested model settings.
  5. The prompt combines tool result and retrieved policy.
  6. The model labels the estimate, reports assumptions, and includes a source note.

Repository layout

Place the snippets behind these module boundaries. Add pinned dependencies, a configuration file for model and index IDs, and tests for every acceptance row before treating the repository as complete.

run_chat.py, run_rag.py, and run_graph.py should import the same configuration and component factories. That prevents the examples from drifting into three unrelated applications.

Implementation order

  1. Plain chat chain: prompt, model, parser.
  2. Structured output: force the model to return a predictable shape.
  3. Tiny RAG: one text file, one retriever, one answer with source.
  4. One bounded tool: the weight-memory calculation.
  5. Router graph: decide chat vs RAG vs tool.
  6. Combined answer: label estimates and attach retrieved source metadata.
  7. Acceptance suite: route, contract, retrieval, invalid-input, and non-goal tests.

Twenty-minute repository acceptance card

Simplified exercise. Use three fixed questions: one policy lookup, one 7-billion-parameter/4-bit memory estimate, and one unrelated request. Record the route, returned artifact, source metadata, and any external call for each.

  • Pass: the policy question reaches the expected policy chunk, the calculation returns 3.5 GB as raw weights, and the unrelated request reaches stop without invoking the retriever or tool.
  • Stop: a route is ambiguous, source metadata is absent, invalid numeric input is accepted, or any unregistered capability is called.
  • Limit: this checks repository wiring only. It says nothing about answer quality across a real corpus, runtime memory overhead, throughput, or permission enforcement.

Once that card passes, the next useful read is the core-components article: it slows down the same prompt, model, parser, and Runnable boundaries so their tests are easier to write. A calculated estimate and a retrieved policy still do not prove that a model fits or performs on a particular GPU; that requires hardware testing under stated conditions.

Security note. Treat model output as untrusted. Do not pass arbitrary model-generated strings directly into file paths, SQL filters, shell commands, or deserializers. The more tools you add, the more boring validation matters.
