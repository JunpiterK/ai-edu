On-prem LLM, MCP, and agents on a gaming GPU

You do not need an H100 to learn on-prem LLM engineering. You need a normal PC, a small instruct model, a repeatable setup, and the patience to add one capability at a time. This series follows that order.

Disclosure. Future versions of this series may include affiliate links to books or hardware. The rule is simple: recommendations must be technically relevant, and every article must remain useful without buying anything.

The promise: one normal PC

We are building the same shape of system used in serious on-prem deployments, only scaled down: a local model server, a Python client, a document index, an MCP tool server, and an agent loop with guardrails. The goal is not benchmark glory. The goal is to understand the system with your own hands.

Each step has three checks: what to run, what success looks like, and what to try if it fails. Do not skip the output checks. They are how you know the local stack is actually alive.

The practical stack:

  • Ollama for the easiest local model server and OpenAI-compatible API.
  • Hugging Face Transformers for direct model loading and 4-bit experiments.
  • bitsandbytes for 4-bit / 8-bit quantized inference when your GPU is tight.
  • MCP for exposing local tools, files, and workflows to agents in a standard way.
  • Goose for the Claude Code / Codex-like layer: terminal agent, extensions, file work, and local automation.
  • Python for the glue: retrieval, tools, logging, and safety checks.

Hardware tiers

Use this as a learning map, not a shopping command. Model families change quickly, but memory pressure does not.

  • CPU-only, 32 GB RAM: slow with small 1B-3B models, but useful for learning APIs and prompts.
  • 8 GB VRAM: 3B-7B quantized models, short context, one user at a time.
  • 12 GB VRAM: comfortable quantized 7B-8B models; small RAG demos start to feel real.
  • 16 GB VRAM: 8B-14B quantized models, longer context, better agent experiments.
  • 24 GB VRAM: 14B-class quantized models and smoother multi-tool workflows.
Buy VRAM before you buy theoretical FLOPS. For home on-prem LLM work, memory is where the project usually hits the wall.

The series roadmap

  1. Local LLM server. Install Ollama, pull a small instruct model, and call it from Python.
  2. Hugging Face direct path. Load a model with Transformers, apply a chat template, and try 4-bit quantization.
  3. Local RAG. Index a small folder of PDFs or Markdown files and answer with citations.
  4. First MCP server. Expose a safe local tool such as a calculator, file search, or system note.
  5. Goose coding agent. Connect a local model provider and let the agent inspect, edit, and test a tiny repo.
  6. Agent loop. Let the model choose tools, but require structured output and approval gates.
  7. Operations. Logs, evaluation questions, GPU monitoring, backups, and “do not run this command” rules.

Part 1: local server with Ollama

Start with the boring path because it removes ten variables. Install Ollama, pull a small model, and call the local OpenAI-compatible endpoint. On most machines, the server listens at http://localhost:11434.

Step 1. Pull a small instruct model

Pick a 3B-7B model first. Larger models can wait. The first win is proving that your PC can download, load, and answer locally.

Expected output. The exact answer will vary, but the pull should end with success, and the model should answer the prompt.

Success checkpoint. If you get a consistent answer here, the local model server is alive. Slow is fine. We are testing architecture first and speed later.

Step 2. Call the same model from Python

Now call the local endpoint from Python. This matters because RAG libraries, agent frameworks, and Goose-style tools often integrate through this API shape.

Expected output. Run python local_chat.py. You should see one paragraph printed by Python, not only inside the Ollama terminal UI.

If it fails. If you see a connection error, confirm Ollama is running. If you see “model not found,” make sure the Python model value exactly matches the model name you pulled.

Once this works, you have the first brick: a local model endpoint that many tools already know how to call. Do not add RAG, MCP, or agents until this is boring.

Part 2: Hugging Face 4-bit inference

The Hugging Face path gives you more control. It also gives you more ways to misconfigure the machine, so keep the first script small. The important details are: use the tokenizer’s chat template, use device_map="auto", and try 4-bit only after you understand normal loading.

Step 3. Create a clean Python environment

Use a fresh virtual environment. Local LLM work is dependency-sensitive, and a clean environment makes errors easier to understand.

Expected output. The install may download a lot. A successful install ends with installed packages and returns to the activated prompt.

Step 4. Run a 4-bit Hugging Face chat script

Expected output. The first run may be slow because the model downloads. After loading, you should see a short answer.

If it fails. This is the first place many home setups fail: wrong CUDA version, model too large, missing chat template, or context length too ambitious. Change one variable at a time. Try a smaller model before changing five packages at once.

Part 3: first MCP tool

After local inference works, add tools. Do not start with “the agent controls my computer.” Start with a read-only or harmless tool. MCP is useful because it gives that tool a standard shape.

Step 5. Run a harmless MCP server

This server does not touch files, run shell commands, or call the network. It only answers two safe tool calls. That makes it a good first extension.

Expected output. Depending on the transport you run, the server may sit quietly and wait for a client. Quiet is fine. A successful MCP server often looks like a long-running process, not a command that exits.

Success checkpoint. Your next task is not to make it powerful. Your next task is to connect a client and confirm that the two tool names are discoverable.

That tiny server is not impressive, and that is why it is good. Safe agents are built from small, boring capabilities, not one giant permission grant.

Part 4: Goose as a local coding agent

Once local inference and MCP tools work, you can build what people actually want: a local coding assistant that feels closer to Claude Code or Codex. The open-source tool I would start with is Goose. In this series, Goose becomes the layer that can inspect a small codebase, propose edits, run tests, and call MCP tools.

The first Goose milestone is intentionally small.

  • Use Ollama or another local provider as the model.
  • Open a toy Python repo, not your main work folder.
  • Ask Goose to explain the code before it edits anything.
  • Allow file edits only after reviewing the diff.
  • Allow only a small command allowlist such as test, lint, and format.

Expected output. Before Goose touches anything, your baseline tests should pass. If the baseline fails, you do not have an agent problem; you have a project setup problem.

Now ask Goose to inspect the project without editing it. A good first response should mention the source file, the test file, and the fact that the project exposes one simple add function.

Success checkpoint. This is the first Claude Code / Codex-like moment: the agent read a repository, understood it, and reported back without changing anything. That restraint matters. Editing comes after inspection.

The second step is connecting Goose to the MCP server. That is where the lab starts to feel real: the coding agent can use project files, local tools, and structured capabilities instead of relying on one enormous prompt. The safety rule stays the same: read-only first, then small edits, tests, and wider tools later.

Do not start by giving a local agent your whole computer. Use a disposable repo. Keep secrets outside the folder. The point is to learn the agent loop before trusting it with anything valuable.

What comes next

The next article turns this into local RAG: one document folder, local embeddings, a vector index, and cited answers. After that, the MCP server gets a document-search tool. Then Goose becomes the user-facing agent that can call those tools, inspect a codebase, make patches, and run tests inside a controlled workspace.
