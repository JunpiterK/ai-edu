LangGraph: stateful, resumable agent workflows

Illustrative restart test

An SPC excursion produces a draft deviation report and a bundle of source references. A quick review window makes an in-memory pause seem sufficient, but a service restart clears that state before the reviewer returns. The failed resume changes the design to a durable checkpoint keyed by report ID. A process engineer may reject the draft with revision instructions or accept it into a review repository. The equipment ID, report ID, and decisions below are constructed examples, not plant events.

The workflow must survive the wait between draft and review. It also needs an authenticated reviewer, a durable checkpoint, an expiry rule, an idempotent commit, and an audit event for every decision. LangGraph can coordinate those transitions. It does not authenticate users, approve a deviation under a quality system, or confer regulatory approval. Those controls remain in the surrounding application and governed business process.

State shortcut

Write the restart point, resume identity, expiry, and external-effect key beside every interrupt. If any one is undefined, the graph is not ready for a durable pause.

Why a graph, not a chain

A linear chain is useful for a fixed A → B → C path. This deviation example needs more than that. It loops back when a reviewer rejects, it can skip a step when the workflow contract allows it, and it must survive the reviewer going home for the night and coming back the next shift. LangGraph models the workflow as a directed graph — nodes are units of work, edges are the transitions, and a checkpointer persists the state between them. Because the state is saved after every node, the graph is resumable: you can stop it mid-run, close the process, and resume the identified run at its saved transition.

Choose a graph here because the required contract includes branching, pause, expiry, and resume. A straight chain remains appropriate for fixed pipelines.

The example points to a local LLM stack and SQLite checkpoint file. Verify package downloads, model endpoints, telemetry, backups, file permissions, and audit destinations against the site's approved network design.

State, nodes, edges

The state is a typed dictionary that flows through the graph. Each node receives it, returns a partial update, and LangGraph merges that update in. Define it once with TypedDict; the add_messages reducer appends to a running message list instead of overwriting it.

A node is just a function: it takes the state and returns the fields it wants to change. Here the first node asks the on-prem model to draft a deviation report from process context handed in as messages.

Edges connect nodes. A plain edge always goes from one node to the next; the special constants START and END mark where the graph enters and exits. You add nodes and edges to a StateGraph builder, then compile() it into something runnable.

Conditional branching

The interesting part of a review process is the fork: approved exits, rejected loops back for a revision. That is a conditional edge — you give it a routing function that reads the state and returns the name of the next node.

Checkpoints make it resumable

A checkpointer stores graph state under a thread_id. This example uses SqliteSaver so a process restart does not erase the paused run. The service owner still needs file permissions, encryption where required, backup/restore tests, retention, concurrent-access tests, and a mapping between thread_id and the deviation record.

Human-in-the-loop with interrupt()

interrupt() pauses and returns a review payload to the application. The application authenticates the reviewer, checks role and expiry, and supplies a structured resume value. Do not trust a browser-supplied reviewer ID. Also keep side effects out of the pre-interrupt portion because the node may be replayed during resume.

The graph route requires a verdict and authenticated reviewer ID. The application additionally rejects expired or context-mismatched reviews, and the repository's unique commit key makes a repeated resume return the existing write rather than duplicate it. put_with_audit_outbox() is one database transaction: if either the report write or required outbox insert fails, both roll back. A protected dispatcher can deliver that durable event to the audit sink and reconcile its receipt without creating an unaudited commit. These are workflow controls, not evidence that the deviation has received regulatory or quality-system approval.

Running the review loop

The first invocation runs to the interrupt. A trusted server handler then derives reviewer_id from its authenticated principal, checks the role and deadline, writes the decision attempt to the audit sink, and resumes the same thread_id. The example rejects once to demonstrate that feedback reaches the next draft.

Test restart/resume, rejected revision, unauthorized reviewer, expired review, duplicate resume, unavailable audit sink, and repository failure before a controlled trial. The authenticated application principal, not the model or form payload, supplies reviewer identity. The repository and quality workflow remain the authority for the deviation record.

Transition-table drill

Simplified exercise. On paper, walk one report through draft -> review -> rejected -> draft -> review -> approved. At each arrow, record state before, authenticated actor, event, state after, checkpoint key, audit event, and possible external effect.

  • Pass: a restart can resume every waiting state, rejection feedback reaches the next draft, expired or unauthorized resumes are denied, and a duplicate approval resolves to the existing commit.
  • Stop: identity comes from the form, two transitions can write the same report without an idempotency key, or an audit outage still permits the commit.
  • Limit: the drill checks transition design. It does not validate evidence, prove exactly-once behavior in an external repository, or grant quality-system approval.

LangGraph supplies state transitions, pause, and resume for this constructed workflow. The next-reading bridge is the HITL/MLOps article, which moves the focus from graph mechanics to approval payloads, durable evidence, and release monitoring.
