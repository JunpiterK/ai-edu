LangGraph: stateful, resumable agent workflows

A chain runs start to finish. That is fine until a step needs a decision it cannot make alone — an SPC deviation report the model drafted, a recipe change it proposes, a maintenance ticket it wants to close. In a fab, those steps do not just run; they get reviewed and approved by an engineer before anything commits. LangGraph exists for exactly that shape of work: a workflow that can pause, wait for a human, and resume from where it stopped — with no cloud call and no lost state. 멈췄다가 사람의 승인을 받고 그 지점부터 다시 이어가는 워크플로를 코드로 모델링한다.

Why a graph, not a chain

A linear chain has one path: A → B → C. A real approval process is not linear. It loops back when a reviewer rejects, it skips a step when the change is trivial, and it must survive the reviewer going home for the night and coming back the next shift. LangGraph models the workflow as a directed graph — nodes are units of work, edges are the transitions, and a checkpointer persists the state between them. Because the state is saved after every node, the graph is resumable: you can stop it mid-run, close the process, and pick up on a different day at the exact node where you paused.

A chain answers a question. A graph runs a process — one that can wait for a human without losing its place. — the one line to remember

Everything below runs on the local air-gapped stack: an on-prem LLM through Ollama, weights on disk, state in a local checkpointer. Nothing about the graph reaches out to a network.

State, nodes, edges

The state is a typed dictionary that flows through the graph. Each node receives it, returns a partial update, and LangGraph merges that update in. Define it once with TypedDict; the add_messages reducer appends to a running message list instead of overwriting it.

A node is just a function: it takes the state and returns the fields it wants to change. Here the first node asks the on-prem model to draft a deviation report from process context handed in as messages.

Edges connect nodes. A plain edge always goes from one node to the next; the special constants START and END mark where the graph enters and exits. You add nodes and edges to a StateGraph builder, then compile() it into something runnable.

Conditional branching

The interesting part of a review process is the fork: approved exits, rejected loops back for a revision. That is a conditional edge — you give it a routing function that reads the state and returns the name of the next node.

Checkpoints make it resumable

A checkpointer saves the full state after every node. That one choice is what turns a script into a durable process: the run is keyed by a thread_id, so you can walk away and resume the same thread hours later at the node where it stopped. For development an in-memory saver is enough; for a real deployment you swap in the SQLite or Postgres checkpointer without touching the graph. On an isolated network, that store is a local file — no external database, no telemetry.

Human-in-the-loop with interrupt()

Here is the node that makes this a review process. Inside human_review, calling interrupt() stops the graph and hands a payload to whatever is driving it — a UI, a CLI, a review queue. The graph state is checkpointed at that exact point. When an engineer responds, you resume by re-invoking the graph with a Command(resume=...); the value you pass in becomes the return value of the interrupt() call, and the node continues as if it had never stopped.

Two things make this safe for a regulated floor. First, the model never commits anything on its own — the commit node is only reachable through an explicit human approved. Second, because the pause is a real checkpoint, the reviewer is under no clock: approve now, or next shift, and the graph resumes identically. 사람이 승인해야만 커밋에 도달하고, 언제 돌아와도 같은 지점에서 이어진다.

Running the review loop

Drive it in two phases. The first invocation runs until the interrupt and returns the payload. You show that to the engineer, collect a verdict, then resume with a Command. The thread_id ties both phases to one durable run.

That is the whole pattern: a state definition, a few node functions, one conditional edge, a checkpointer, and a single interrupt(). The same skeleton scales to any fab workflow that a person must sign off on — a recipe-change proposal, an alarm triage, a shift handover summary. The model does the drafting and the routing; the engineer keeps the authority; and because every step is checkpointed on the local network, the run survives interruptions without ever leaving the air gap.
