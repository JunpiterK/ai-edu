Hybrid retrieval: vectors + knowledge graph

AE AI Edu Archive Semiconductor process engineer · Air-gapped LLM / RAG / Agents
On this page

  2. What a knowledge graph adds
  3. The pattern: candidates → verify → expand

A vector store is a similarity engine, not a fact engine. Ask it which chambers share the RF generator model that failed on EV-03, and it will return paragraphs that sound like the question, not the chambers that actually share that part. On a fab network, where an answer can feed a maintenance decision, that gap matters. This article makes a simple claim: for relationship and exact-fact questions, pair vector search with a knowledge graph. Vectors find candidates; the graph verifies and expands.

Where vector search breaks

Dense retrieval embeds a query and returns the nearest chunks by cosine similarity. That is excellent for a semantic question such as explain the purge sequence for the load-lock. It is weak in three places that show up constantly in manufacturing knowledge:

  • Relationships — which recipes run on the same source as recipe A is a graph traversal, not a similarity score. Nearby text is not the same as a connected entity.
  • Exact facts — part numbers, torque values, and interlock IDs are brittle. An embedding can place PN-4471 and PN-4741 near each other in vector space, but on the floor they are different parts.
  • Multi-hop — which SOPs are affected if this valve is recalled needs valve → chamber → tool → SOP. Each hop is a join the vector index cannot make.

You can hide part of the problem with a larger k and a reranker, but you are still asking a similarity index to perform a lookup it was never built to do.

Vectors tell you what a document is about. A graph tells you what a thing is connected to. Fab questions usually need both.

What a knowledge graph adds

A knowledge graph stores entities as nodes and relationships as typed edges: (:Tool)-[:HAS_CHAMBER]->(:Chamber)-[:USES_PART]->(:Part). Querying it with Cypher is exact and deterministic: no embedding blur, no ranking guess, just the edges that exist. That is exactly the shape of the questions vectors struggle with. The trade-off is coverage. A graph only knows the entities and relationships you extracted into it, and it does not perform fuzzy semantic matching. Neither system is sufficient alone; together they cover each other's blind spots.

Both stores can live entirely inside the plant network. The embedding model runs from local weights, and a graph database such as Neo4j runs on-prem. Hybrid retrieval does not require external egress, which is the point for an air-gapped fab.

The pattern: candidates → verify → expand

The reliable pattern is not run both and hope. It is a pipeline:

  • Candidates — vector search over document chunks returns semantically relevant passages. This is the recall layer.
  • Verify and expand — pull the entities mentioned in those chunks, then query the graph for exact facts and neighbours: shared parts, affected SOPs, and the tool a chamber belongs to.
  • Merge and rerank — fuse both result sets and rerank so the top context is both semantically on-topic and factually connected.

Wiring it in LangChain

Everything below uses the current split packages: langchain_neo4j, langchain_huggingface, and langchain_core. No legacy langchain.embeddings or LLMChain. Start with the two stores, both running on-prem.

Next, build a small custom retriever that turns the graph into a LangChain retriever. It receives a query, finds exact entities and connected facts with parameterised Cypher, and returns them as Document objects. In production you would add a dedicated entity extraction step; here the query parameter keeps the example concrete and injection-safe.

With both retrievers speaking the same interface, EnsembleRetriever can fuse them. It merges the two result lists with Reciprocal Rank Fusion, so a document ranked highly by either retriever can survive. The weights let you lean more heavily on vectors for broad recall or on the graph for fact-heavy corpora.

Reranking the merged set

RRF fusion is rank-based, not content-aware. To push the genuinely relevant context to the top before it reaches the prompt, wrap the ensemble in a ContextualCompressionRetriever with a cross-encoder reranker. The reranker also runs from local weights, so no external API is required.

Feed context into your normal LCEL chain, prompt, on-prem LLM, and parser, exactly as you would with a single-retriever RAG. The hybrid retriever is a drop-in replacement for the plain retriever; nothing downstream has to know how the context was assembled.

Vector-only RAG is the right default for explanation questions and the wrong tool for relationship and exact-fact questions. Pairing it with a knowledge graph, vectors for recall, graph for verified structure, and a reranker to settle the order, closes that gap without leaving the network.
