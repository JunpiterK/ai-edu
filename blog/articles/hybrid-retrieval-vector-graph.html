Hybrid retrieval: vectors + knowledge graph

Opening artifact: retrieval trace

query_entity=EV-03; vector branch returned a maintenance paragraph at rank 1; graph branch returned no rows; entity resolver recorded unresolved. That is not a reranking problem. The trace says the relationship path never started.

A vector index ranks similar text; a graph query returns assertions stored in the graph. Neither result is automatically true, current, authorized, or complete. For relationship and exact-identifier questions, a hybrid retriever can place both document passages and graph assertions in front of the answer system, provided every result retains its branch, provenance, effective date, and access decision.

Where vector search breaks

Dense retrieval embeds a query and returns the nearest chunks by cosine similarity. That is excellent for a semantic question such as explain the purge sequence for the load-lock. It is weak in three places that show up constantly in manufacturing knowledge:

  • Relationships — which recipes run on the same source as recipe A is a graph traversal, not a similarity score. Nearby text is not the same as a connected entity.
  • Exact facts — part numbers, torque values, and interlock IDs are brittle. An embedding can place PN-4471 and PN-4741 near each other in vector space, but on the floor they are different parts.
  • Multi-hop — which SOPs are affected if this valve is recalled needs valve → chamber → tool → SOP. Each hop is a join the vector index cannot make.

You can hide part of the problem with a larger k and a reranker, but you are still asking a similarity index to perform a lookup it was never built to do.

Vectors rank passages by similarity. Graph queries return stored relationships. Both require provenance and review.

What a knowledge graph adds

A knowledge graph stores entities as nodes and relationships as typed edges: (:Tool)-[:HAS_CHAMBER]->(:Chamber)-[:USES_PART]->(:Part). A Cypher query deterministically returns matching stored nodes and edges under the current database state. That output is a stored assertion, not independent verification. It may be stale, incomplete, incorrectly extracted, or outside the requester's access scope.

At query time, filter assertions to the requester's authorization and the question's effective date. If provenance is missing, return the relationship as unverified context or omit it according to policy. A reviewer follows the source record before using the assertion for a maintenance or production decision.

Both stores can be deployed inside the plant network. Local model paths and an on-prem graph endpoint support that design, while firewall, DNS, package, update, plugin, and telemetry checks provide the evidence that the deployed system respects the intended boundary.

The pattern: retrieve in parallel, then qualify

The implementation below fans the query out to two retrievers and merges their ranked outputs. It does not use vector candidates to verify graph edges, and the graph branch does not prove the returned assertions. Treat it as a retrieval skeleton:

  • Vector branch: return document chunks ranked by semantic similarity.
  • Graph branch: resolve a canonical entity or identifier, then return connected stored assertions that pass authorization and effective-date filters.
  • Merge and rerank: fuse both result sets for relevance. Reranking changes order; it does not validate truth or freshness.

The diagram's "verify + expand" label means a graph traversal retrieves matching stored edges. It does not mean the graph has independently verified the underlying assertion.

With both retrievers speaking the same interface, EnsembleRetriever can fuse their independent result lists with Reciprocal Rank Fusion. A result ranked highly by either branch can survive. The example weights and k values are illustrative, not evaluation results; tune them on held-out queries.

Reranking the merged set

RRF fusion is rank-based, not content-aware. A ContextualCompressionRetriever with a local cross-encoder can reorder the merged candidates by estimated relevance. It cannot confirm that a graph assertion is true, current, or authorized; those checks occur before fusion and remain visible in result metadata.

Before feeding context to an LCEL chain, preserve the branch type, source record, revision, effective interval, access decision, and retrieval time in each document's metadata. The prompt and citation formatter must distinguish a quoted source passage from a graph assertion. Downstream components therefore do need to know how the context was assembled.

Use vector retrieval for semantic passages and a governed graph branch for stored relationships and exact identifiers. Evaluate entity resolution, authorization filtering, effective-date selection, provenance coverage, retrieval recall, and stale-assertion behavior separately. To build the relationship side with better source discipline, continue with the process knowledge-graph field note. Hybrid retrieval improves access to evidence; it does not turn stored assertions into truth.
