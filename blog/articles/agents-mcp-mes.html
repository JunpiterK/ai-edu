Wiring an agent to internal systems with MCP: MES, equipment APIs, permissions, and audit

An agent is only as useful as the systems it can touch. A model that can reason about a yield excursion but cannot read the MES lot history, query the last chamber-clean timestamp, or open a hold is a very expensive autocomplete. The hard part of an agent project is rarely the model. It is the wiring: how the model calls internal systems safely, with a schema it understands and a boundary it cannot cross. The Model Context Protocol (MCP) is a standard for exactly that wiring. This article shows how to expose an on-prem MES or equipment API as an MCP server, connect it to an agent, and put permissions and audit around every call inside an air-gapped fab.

Why an agent needs a protocol, not more glue

Without a protocol, every integration is bespoke. You write a Python function that wraps the MES REST call, another for the equipment historian, and a third for the recipe management system. Then you hand-write the JSON schema each one exposes to the model, keep it in sync with the real API, and repeat the whole thing for the next agent framework you try. It works once. It rots fast.

MCP replaces that with a client-server contract. Your internal systems sit behind MCP servers that describe their capabilities in a standard way. Any MCP client, including an agent, IDE, or chat app, can discover those capabilities and call them without knowing the system underneath. The value in a fab is decoupling: the team that owns the MES publishes one MCP server, and every agent in the plant can reuse it.

MCP is to agents what a device driver is to an OS. The equipment team writes the driver once; every application above it just calls the standard interface. — the mental model to keep

What MCP is: server, tools, schema

An MCP server exposes three kinds of primitive. Tools are functions the agent can call, such as query_lot_status or place_hold. Resources are read-only data the agent can pull in as context, such as an SOP file or spec table. Prompts are reusable templates the server offers. For system integration, tools do the heavy lifting.

Each tool carries a typed schema: a name, a description, and typed parameters. The agent reads that schema to decide when and how to call the tool, so the schema is the contract and it has to be honest. A tool named place_hold described as “puts an engineering hold on a lot” with a typed lot_id parameter tells the model precisely what it can do and what it must supply. Get the description right and the model uses the tool correctly; get it vague and it guesses.

Permissions, audit, and guardrails

Read-only tools are low risk. The moment a tool can change plant state, such as placing a hold, dispatching a lot, or editing a recipe, the schema alone is not enough. You need three controls around it.

  • Permission checks (authz). The MCP server, not the model, decides who may call what. Enforce it inside the tool against the caller’s role; a model prompt is never a security boundary.
  • Audit. Every call is logged with caller, arguments, result, and timestamp, so an action taken by the agent is as traceable as one taken by a person.
  • Human-in-the-loop. State-changing tools return a confirmation step instead of acting immediately, so an engineer approves before anything happens on the floor.

Because the role comes from the transport layer and the audit line is written before any state change, no clever prompt can talk the tool into acting outside its lane. The model can ask; the server decides. That separation is what makes an agent safe to point at real MES actions.

Keeping the whole loop on-prem

The reason MCP fits a fab so well is that nothing in this design assumes the cloud. The MCP servers run on plant hosts and bind to internal addresses. The transport is HTTP over the LAN. The model is served locally through Ollama or vLLM. The agent, the client, the guardrails, and every internal API live inside the same air-gap boundary. No token, tool call, or lot ID crosses it.

Start small and read-only: one MCP server, two or three lookup tools, and an audit log you actually watch. Prove the agent answers real floor questions correctly before you hand it a single tool that can change state. When you do, wrap that tool in role checks and human confirmation. The protocol gives you a clean interface; operational discipline still matters.

MCP turns “connect the agent to our systems” from a pile of bespoke glue into a small number of reusable, typed, audited servers.
