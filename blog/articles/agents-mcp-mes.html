Lab reference architecture

This article shows tool discovery and call flow. The sketch omits mTLS, a server-authenticated principal, server-side authorization, idempotency, durable protected audit, human approval, and tested rollback or compensation. Keep it in a lab until the relevant system, process, and IT/security owners define and test the controls required for the intended MES use.

Illustrative catalog review

A single general mes_query tool appears to reduce integration work. Its draft schema soon accumulates optional SQL-like filters, area names, and output modes that no reviewer can reason about consistently. That review finding leads to a catalog of narrow lookups with separate owners and permission rules. MCP made discovery reusable; it did not remove the need for a small operational contract.

Tool catalog gate

Before writing a server, classify each tool as read-only, draft-producing, or state-changing. Record its owner, authenticated principal, authorization rule, approval path, timeout behavior, idempotency key, audit destination, and rollback procedure. An incomplete entry stays out of the MES catalog.

Yield questions may require current MES lot history, chamber-clean timestamps, or hold state. Rather than giving a model database credentials, an integration layer can expose a limited set of typed tools. MCP standardizes the tool contract and discovery flow; it does not provide the plant authentication, authorization, approval, audit durability, or rollback controls by itself.

Why an agent needs a protocol, not more glue

Without a protocol, every integration is bespoke. You write a Python function that wraps the MES REST call, another for the equipment historian, and a third for the recipe management system. Then you hand-write the JSON schema each one exposes to the model, keep it in sync with the real API, and repeat the whole thing for the next agent framework you try. It works once. It rots fast.

MCP provides a client-server contract. An internal system can sit behind a server that describes approved capabilities, and compatible clients can discover their names, descriptions, and schemas. Reuse also concentrates risk: the server owner must manage schema versions, allowed clients, certificates, rate limits, failure isolation, and the emergency disable path.

An MCP server can centralize a reusable tool contract. Reuse of that contract and authorization to the underlying system are separate review decisions. Design review boundary

What MCP is: server, tools, schema

An MCP server exposes three kinds of primitive. Tools are functions the agent can call, such as query_lot_status or place_hold. Resources are read-only data the agent can pull in as context, such as an SOP file or spec table. Prompts are reusable templates the server offers. For system integration, tools do the heavy lifting.

Each tool carries a name, description, and typed parameter schema. For place_hold, the description should identify the target lot ID, reason constraints, state change, and approval requirement. Schema validation can reduce malformed calls, but it is not a security boundary; the server must validate values, current state, and the authenticated caller's authorization.

Permissions, audit, and guardrails

Read-only tools can still expose restricted production data. State-changing tools add replay, duplicate execution, partial failure, and rollback risk. The server must enforce controls independently of model instructions.

  • Permission checks (authz). The MCP server, not the model, decides who may call what. Enforce it inside the tool against the caller’s role; a model prompt is never a security boundary.
  • Audit. Every call is logged with caller, arguments, result, and timestamp, so an action taken by the agent is as traceable as one taken by a person.
  • Human-in-the-loop. State-changing tools return a confirmation step instead of acting immediately, so an engineer approves before anything happens on the floor.

The following place_hold function is security-boundary pseudocode. It assumes authentication middleware validates a token or mTLS identity and injects a server-created principal out of band. The client must not supply role as a tool argument or trusted header. This function does not apply a hold and is not a complete approval workflow.

A production implementation must derive authorization from the server-authenticated principal, then add mTLS, one-time approval tokens, idempotency keys, status reconciliation after timeout, durable tamper-evident audit, approval cancellation, and MES rollback or compensation. Prompt-injection tests are insufficient; test duplicate requests, approval-service failure, and partial MES failure before enabling writes.

Keeping the whole loop on-prem

MCP servers, local models, clients, and internal APIs can be deployed inside a sealed network. That topology does not prove containment. Audit package-update paths, DNS, proxies, telemetry, and log sinks, then retain firewall and network-trace evidence that model requests and tool arguments stay inside the approved boundary.

Limit the lab to one server and a few lookup tools. Test denied identities, invalid IDs, timeouts, MES unavailability, and audit replay. Keep state-changing MES access out of this lab; the site's owners must define and pass a separate release gate for identity, authorization, idempotency, protected audit, human approval, and recovery.

Tool-catalog decision record

Simplified exercise. Draft one row for a read-only get_lot_status tool: owner, schema version, authenticated principal, allowed lot scope, freshness field, timeout result, audit destination, and disable owner. Then submit an unknown lot, a denied principal, and a simulated timeout.

  • Pass: discovery exposes only the intended schema; all three tests return typed, non-sensitive outcomes; and server, connector, and audit records reconcile by request ID.
  • Stop: authorization depends on prompt text, the client supplies trusted identity, timeout can trigger an untracked retry, or the disable path has no named owner.
  • Limit: this record covers one read path. It does not qualify state-changing tools, prove network containment, or validate MES behavior under production load.

MCP can package internal capabilities as reusable typed contracts. Continue with the HITL/MLOps article before considering write tools; it adds payload-bound approval, durable workflow state, execution receipts, and release monitoring to the discussion.
