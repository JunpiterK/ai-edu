Illustrative scenario: the configured control failed review

The data-egress row looks ready to close because “no external access” is enabled. One reviewer asks for the denied-egress probe from the embedding worker, and the evidence packet comes up empty: nobody tested from that workload identity. The row stays open. Configuration is supporting evidence; the pass decision waits for a repeatable test and retained network records. This is a constructed review scenario, not a site incident.

Entry rule

A row is incomplete until it names the control objective, threat, test procedure, retained evidence, accountable owner, approved exceptions, and residual risk. A policy statement or configured feature by itself does not demonstrate control effectiveness.

Open the review with one row on the screen. Read it left to right: objective, threat, control, test, evidence, owner, exception, residual risk. Policy principles matter, but plant teams need to know which decision this row supports and what result blocks release.

Beginner picture

A policy is like saying "only qualified people may enter the cleanroom." A control is the badge reader, access list, gowning procedure, camera log, and periodic review that make the sentence true. AI policy works the same way. "Use AI safely" means very little until it becomes tool permissions, approval gates, audit logs, and tests.

Register rules

A statement like “protect confidential data” is not yet an operational control. In a plant AI system, it may map to network segmentation, role-based retrieval, prompt logging, document permission sync, redaction rules, and incident response. The selected controls depend on the architecture and risk review. The same translation is needed for “human oversight,” “accuracy,” and “safety.”

The register connects policy intent to a repeatable test. Record test frequency and the exact evidence location in the site copy. Failed tests create corrective actions; exceptions require an approving authority, compensating controls, an expiry date, and a review before renewal.

Where agents raise the stakes

A RAG assistant mostly speaks. An agent may read a ticket, call a tool, draft an update, open a workflow, or trigger a request. That difference changes the governance model. The control boundary must move from "is the answer good?" to "is the next action allowed, reversible, observable, and approved?"

Use a two-lane design. Read-only tools can be broad: search documents, summarize logs, compare revisions, retrieve equipment history. State-changing tools must be narrow: create a draft, request approval, file a ticket with human confirmation, or produce a runbook that someone signs off. Do not let the first version of an agent directly change setpoints, release holds, alter recipes, or close corrective actions. The cost of a bad first release is not technical debt; it is lost trust.

Illustrative boundary

A first release could read alarm history and draft a maintenance ticket for review. Direct ticket closure or dispatch-owner changes remain outside scope until authorization, approval, idempotency, audit, and rollback tests have accepted evidence.

Make governance routine

Control-row drill

Take the state-change row and run three cases in a non-production environment: an authorized dry run, an expired approval, and a duplicate retry. Pass when the first produces only the approved draft action and the latter two are denied with traceable reasons; confirm that the owner can find the approval, tool payload, idempotency record, and rollback result. Stop if a write executes without the required principal, approval, or target validation. Limit: three fixtures exercise known paths and do not establish complete resistance to prompt injection or misuse.

Review cadence should follow change and risk. Re-run affected controls after model, prompt, index, identity, tool, network, or approval-workflow changes. The owner records pass, fail, or exception; a failed high-risk control blocks release until the designated authority accepts a documented alternative.

The register is useful when a reviewer can locate the test result behind each claim. Configuration screenshots, policy text, and vendor documentation may support the record, but they do not replace execution evidence from the deployed environment. Once the rows have owners and tests, continue with the isolation-boundary decision; its data-flow worksheet supplies concrete payloads and routes for the first control rows.
