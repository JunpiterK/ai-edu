Serving LLMs on-prem: vLLM, Ollama, and GPU sizing

A model that never leaves the fab network still has to run somewhere. On a manufacturing shop floor the constraint is rarely which open model is smartest — it is how to put that model on a GPU box behind the air gap, keep it answering under real concurrency, and prove it is healthy. This is a field guide to the two servers you will actually reach for — vLLM and Ollama — and to the memory arithmetic that decides whether a model fits your card at all. 폐쇄망(air-gapped) 안에서 오픈 모델을 GPU 서버에 올리는 실전 이야기입니다.

vLLM or Ollama?

The two tools solve the same problem at different scales. Ollama is the short path: one binary, GGUF-quantized weights pulled into a local store, and a server up in seconds. It runs happily on a single workstation GPU — or even CPU — and is ideal for a pilot, a bench test, or a small team hitting one shared box. vLLM is the throughput engine: it uses paged-attention KV caching and continuous batching to serve many concurrent requests on datacenter GPUs, and it exposes an OpenAI-compatible HTTP API out of the box. 파일럿·단일 GPU는 Ollama, 다중 동시 요청·처리량은 vLLM.

A useful rule of thumb for the fab: if fewer than a handful of engineers will query the model interactively, start with Ollama. The moment a RAG service, an agent, or a dozen analysts share the same server, the request queue is what breaks first — and that is exactly what vLLM's continuous batching is built for.

Pick Ollama to get answers today; move to vLLM when the queue, not the model, is your bottleneck. — 폐쇄망 서빙 원칙

Ollama pulls a model into a local blob store and serves it. On an air-gapped host you pull once on a staging machine, copy the model directory over, and point OLLAMA_MODELS at it — no egress at inference time.

vLLM serves a Hugging Face model directory with a single command. Stage the weights on-prem, then launch — --gpu-memory-utilization caps how much VRAM the KV cache may claim, and --max-model-len bounds the context window so the cache math stays predictable. --tensor-parallel-size shards one large model across several GPUs.

One endpoint, OpenAI-compatible

The payoff of both servers is the same: an OpenAI-compatible HTTP surface at /v1. Your RAG service, your agent, and your notebooks all talk one protocol, so you can swap Ollama for vLLM — or one model for another — without touching application code. In LangChain this means a single client class. For vLLM, use ChatOpenAI from langchain_openai pointed at your on-prem base_url; the API key is a placeholder because nothing leaves the network.

Ollama has a first-class LangChain integration of its own — ChatOllama from langchain_ollama — which talks the native Ollama API and adds niceties like tool calling and a num_ctx knob for the context window. Same three lines, different host.

GPU memory sizing

Before you download anything, do the arithmetic. VRAM demand splits into two parts: weights, which are fixed, and the KV cache, which grows with context length times concurrency. A serviceable back-of-envelope estimate for the weights:

  • Weights ≈ parameters × bytes-per-parameter. FP16 is 2 bytes, 8-bit is ~1 byte, 4-bit is ~0.5 byte. So an 8B model is roughly 8 × 2 = 16 GB at FP16, ~8 GB at 8-bit, and ~4 GB at 4-bit. 파라미터 수 × 양자화 바이트.
  • KV cache is the swing factor — it scales with model layers, context length, and the number of requests in flight. Budget an extra 20–40% of card VRAM for it, and quantize it (--kv-cache-dtype fp8 in vLLM) to roughly halve that footprint when you need longer context or more concurrency.
  • Overhead. Leave 10–15% for the CUDA context, activations, and fragmentation — this is why --gpu-memory-utilization 0.90 is a sane default, not 1.0.

Worked example: a 70B model at 4-bit is about 70 × 0.5 = 35 GB of weights. That will not fit one 24 GB card, comfortably fits one 48 GB card with modest context, and wants two GPUs (--tensor-parallel-size 2) once you add real KV headroom. The quantization tier is the lever that moves a model from "won't load" to "fits with room to serve."

Batching and concurrency

A single request barely taxes a modern GPU; a shared service lives or dies on how it handles the second, tenth, and fiftieth request arriving mid-generation. This is where the two tools diverge sharply. Ollama processes requests largely sequentially by default — fine for a few users, a growing queue under load. vLLM's continuous batching interleaves new requests into the running batch token-by-token, so throughput scales with concurrency instead of collapsing under it.

Two levers matter on the vLLM side: --max-num-seqs caps how many sequences run concurrently, and --max-model-len bounds per-sequence context. Raising concurrency trades KV-cache memory for throughput — which is precisely why the sizing arithmetic above comes first. On the client, favor llm.batch([...]) over a Python loop so requests reach the server together and the batcher can pack them.

Health checks and operations

An air-gapped service has no vendor dashboard watching it — you own liveness. Both servers give you cheap signals: vLLM exposes /health (liveness), /v1/models (is the model loaded and named as expected), and Prometheus /metrics for queue depth and latency; Ollama answers a bare GET / and lists loaded models at /api/tags. Wire the simplest one into your orchestrator's readiness probe.

The model you choose matters less than the discipline around it: stage weights offline, put an OpenAI-compatible endpoint in front, size the card with the parameters-times-bytes rule before you commit, and probe /health like you mean it. Start on Ollama to prove the workflow, graduate to vLLM when concurrency is the constraint — and in both cases the knowledge never leaves the network. 온프렘 서빙은 모델 선택보다 운영 규율이 핵심입니다.
