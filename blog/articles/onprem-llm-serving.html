Serving LLMs on-prem: vLLM, Ollama, and GPU sizing

AE AI Edu Archive Semiconductor process engineer · Air-gapped LLM / RAG / Agents
On this page

  2. One endpoint, OpenAI-compatible
  3. GPU memory sizing
  4. Batching and concurrency
  5. Health checks and operations

Put a one-page workload card beside the GPU host. Record the model and quantization, GPU model and count, prompt and output token distributions, request arrival pattern, concurrent users, streaming mode, and latency objective. A throughput number without those conditions is hard to compare or repeat. This guide uses that card to test vLLM and Ollama while accounting for weights, KV cache, queueing, restart behavior, and health signals.

Illustrative scenario: the server looked slow

The model server gets blamed when time to first token rises during a test. The request log tells a less dramatic story: every client sent a long prompt at the same instant, and queue time accounts for most of the delay. The next run keeps the prompt distribution but uses the expected arrival pattern. Stop the product comparison until client timing and queue time are separated. This is a simplified diagnosis, not a measured result.

Workload-card rule

Do not accept “fast” or “fits” as a result. Record test conditions, queue time, time to first token, generation rate, failures, peak VRAM, and restart behavior. Change one server variable per comparison.

vLLM or Ollama?

Both tools can serve local models, but their packaging, model formats, scheduling behavior, hardware support, and operational surfaces differ. Ollama can be convenient for a compact workstation deployment and GGUF-based workflows. vLLM provides continuous batching and a throughput-oriented serving surface. Neither description is a selection rule.

Run the same candidate model, quantization, prompt set, output cap, warm-up sequence, and arrival pattern against each compatible server. Compare time to first token, inter-token latency, completed requests per minute, queue time, peak VRAM, error rate, cold-start time, and recovery after restart. Select the server that meets the declared service objective and can be operated by the assigned platform owner.

Illustrative load-test conditions

Example values, not plant measurements: one 24 GB GPU; one 8B 4-bit model; 2,048-token prompts; up to 512 generated tokens; streaming enabled; 10 warm-up requests; then 15 minutes each at 1, 4, 8, and 16 concurrent clients. Hold model files, sampler settings, client host, network path, and request corpus constant. Report percentile latency and failures with the result.

Stage and start Ollama

Ollama pulls a model into a local blob store and serves it. On an air-gapped host you pull once on a staging machine, copy the model directory over, and point OLLAMA_MODELS at it. Confirm the copied manifest and blobs are complete before the offline runtime starts.

Start vLLM from local weights

vLLM can serve a local Hugging Face model directory from a single command. Stage the complete artifact bundle on-prem, then launch. --gpu-memory-utilization influences how much device memory vLLM reserves for execution and KV cache, and --max-model-len bounds the context window so the cache math stays predictable. --tensor-parallel-size shards one large model across several GPUs.

One endpoint, OpenAI-compatible

Both servers can expose an OpenAI-compatible HTTP surface at /v1. A common protocol can reduce client changes when swapping servers or models, but model IDs, supported parameters, streaming details, tool calling, and error behavior still need compatibility tests. In LangChain this means a single client class. For vLLM, use ChatOpenAI from langchain_openai pointed at your on-prem base_url. The example points to an internal host; network policy and authentication still need to be enforced outside the client snippet.

Ollama has a first-class LangChain integration of its own — ChatOllama from langchain_ollama — which talks the native Ollama API and adds niceties like tool calling and a num_ctx knob for the context window. Same three lines, different host.

GPU memory sizing

Before you download anything, do the arithmetic. VRAM demand splits into two parts: weights, which are fixed, and the KV cache, which grows with context length times concurrency. A serviceable back-of-envelope estimate for the weights:

  • Weights ≈ parameters × bytes-per-parameter. FP16 is 2 bytes, 8-bit is ~1 byte, 4-bit is ~0.5 byte. So an 8B model is roughly 8 × 2 = 16 GB at FP16, ~8 GB at 8-bit, and ~4 GB at 4-bit: parameter count multiplied by bytes per quantized weight.
  • KV cache is the swing factor — it scales with model layers, context length, and the number of requests in flight. Budget an extra 20–40% of card VRAM for it, and quantize it (--kv-cache-dtype fp8 in vLLM) to roughly halve that footprint when you need longer context or more concurrency.
  • Overhead. Leave 10–15% for the CUDA context, activations, and fragmentation — this is why --gpu-memory-utilization 0.90 is a sane default, not 1.0.

Worked example: a 70B model at 4-bit is about 70 × 0.5 = 35 GB of weights. That will not fit one 24 GB card, comfortably fits one 48 GB card with modest context, and wants two GPUs (--tensor-parallel-size 2) once you add real KV headroom. The quantization tier is the lever that moves a model from "won't load" to "fits with room to serve."

Batching and concurrency

A single-request smoke test proves only that one request completed. For a shared service, measure what happens when requests arrive during generation. Scheduling behavior depends on server version, model backend, GPU, context mix, and configuration. vLLM's continuous batching is designed to combine active sequences; Ollama behavior should be measured with the intended model and concurrency settings rather than assumed from a single-user test.

Two levers matter on the vLLM side: --max-num-seqs caps how many sequences run concurrently, and --max-model-len bounds per-sequence context. Raising concurrency trades KV-cache memory for throughput — which is precisely why the sizing arithmetic above comes first. On the client, favor llm.batch([...]) over a Python loop so requests reach the server together and the batcher can pack them.

Health checks and operations

An air-gapped service has no vendor dashboard watching it — you own liveness. Both servers give you cheap signals: vLLM exposes /health (liveness), /v1/models (is the model loaded and named as expected), and Prometheus /metrics for queue depth and latency; Ollama answers a bare GET / and lists currently running models at /api/ps. Wire the simplest one into your orchestrator's readiness probe.

The strict probe below runs in Bash and requires curl and jq in the approved host image. Its dependency checks and every failed assertion exit nonzero; the orchestrator must preserve that exit status.

Release exercise: pass, hold, or stop

Start with this checklist:

  1. Freeze the workload card: model ID and hash, runtime version, GPU, context and output caps, arrival pattern, concurrency, and latency and error objectives.
  2. Run the strict readiness probe above and retain its exit code and response evidence. Stop immediately if either expected model is absent.
  3. Send ten warm-up requests, then run the fixed corpus at the declared concurrency while recording queue time, percentile latency, failures, and peak VRAM.
  4. Restart the service and repeat the readiness probe until it passes or the recovery objective expires.
  5. Record pass only when identity, load, memory, and recovery criteria all clear. Record hold when quality is acceptable but queueing or memory needs tuning. Record stop on a wrong model, out-of-memory termination, unapproved fetch attempt, masked probe failure, or missed recovery objective.

Limits: thresholds belong to the workload owner, and this checklist does not validate answer quality or network isolation by itself.

Keep the benchmark record beside the deployment manifest. Stage weights through the approved import path, size memory for the tested context and concurrency, and probe readiness as well as liveness. Re-run the declared workload after a model, driver, server, quantization, or prompt-envelope change. A result from one stack is evidence for that stack under those conditions, not a categorical rule about Ollama or vLLM. Once the service has a repeatable operating envelope, continue with RAG over sealed documents; retrieval tests should inherit the same endpoint version, prompt limits, and health record.
