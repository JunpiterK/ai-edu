From tribal knowledge to a plant knowledge asset

Illustrative scenario: more documents made retrieval worse

A review packet asks to add a second maintenance domain after the first corpus clears its gate. Broader coverage sounds like an easy win, until the fixed questions retrieve a superseded instruction from the new domain. Expansion stops while the source owner resolves revision metadata and retirement rules. This is a constructed gate review, not a reported rollout.

Packet rule

Every gate packet needs a source register, fixed evaluation set, permission snapshot, failure log, named owner, and disable path. More documents and users are scope changes, not routine content updates.

The opening artifact is a gate packet, not a knowledge slogan. Plant know-how appears in approved procedures, shift logs, ticket comments, and conversations that may never become records. Some is useful; some is obsolete, local, or disputed. Each gate therefore requires an owner, a metric collected under stated conditions, and a stop condition. Keeping models and data on-premises may address an egress requirement, but it does not establish source quality, authorization, or fitness for plant decisions.

The undocumented moat

Two plants with similar equipment can still develop different operating knowledge. Before indexing it, classify each source: controlled procedure, system-of-record export, operator note, or unverified recollection. Record the revision, effective date, owner, and permission scope. Retrieval should preserve those distinctions instead of presenting every passage with equal authority.

An on-prem LLM can lower the effort needed to search existing records. It does not convert an informal note into an approved instruction. The implementation starts with source inventory and evaluation, then adds users and domains only when governance and operations can support them.

At every review, inspect the source record, evaluation result, access decision, and rollback or disable path.

A four-gate review

The four gates are evidence reviews, not calendar promises. Example durations in the figure are planning assumptions and must be replaced with site estimates. An air-gapped design remains a requirement only where the site owner specifies it; network controls must be verified separately.

Owners, metrics, and stop conditions

Use the table as a review record. Metric values come from the approved test set and current baseline; they are not universal thresholds. A failed stop condition sends the work back for correction or closes it.

Table 1. Stage-by-stage roadmap for an on-prem knowledge asset.

1 · Evidence feasibility Knowledge-domain owner with an IT/RAG engineer Source register, fixed evaluation set, keyword baseline, cited retrieval results, and failure log for one narrow corpus. Grounded-answer and retrieval-hit rates against the baseline, with grader agreement reported. Stop if source ownership cannot be established, permissions cannot be represented, or the candidate does not beat the agreed baseline.
2 · Controlled pilot Process or equipment owner, with IT/security Named user group, ACL test report, freshness report, feedback queue, and disable procedure. Task completion, time-to-answer, citation correctness, stale-source rate, and access-control test pass rate. Stop on unauthorized retrieval, an unresolved critical source error, missing operational ownership, or no measurable benefit under the test protocol.
3 · Domain expansion Platform owner with each source-system owner Versioned ingestion manifests, load and recovery tests, permission mappings, per-domain evaluations, and capacity measurements. Per-domain retrieval quality, queue age, refresh completion, error budget, and compute cost under documented load. Stop adding domains when quality falls below its accepted baseline, permission mappings are incomplete, or recovery and capacity tests fail.
4 · Operational ownership Service owner and knowledge-domain owner Runbook, support roster, scheduled evaluation, source-retirement workflow, incident log, and quarterly scope review. Evaluation trend, source age, incident closure time, ownership coverage, and successful restore tests. Disable affected domains or return to pilot if evaluation lapses, the named owner is absent, restore tests fail, or stale sources exceed the approved limit.

Why the PoC stays deliberately small

The first gate answers a bounded question: does retrieval over the selected records outperform the current search method on the fixed test set? Gate 2 tests use under controlled conditions

The pilot adds identity, permission, freshness, feedback, and disable-path tests. The process or equipment owner decides whether retrieved material is suitable for the pilot task; IT/security owns identity and access controls. Keep the tool advisory and read-only unless a separately governed change process authorizes more.

Gate 2 review exercise

Start with this checklist:

  1. Register one current procedure and one superseded revision with document ID, revision, effective date, owner, and expected retirement state.
  2. Create two test users with intentionally different source permissions and capture the permission snapshot used for the run.
  3. Ask the same revision-sensitive question as each user, retaining retrieved source IDs, ranks, citations, and access decisions.
  4. Verify that each user sees only authorized current material and that every accepted citation identifies its revision and owner.
  5. Disable the pilot domain, repeat the baseline-service check, restore it from the documented procedure, and record the result.
  6. Pass only when retrieval, citation, permissions, disablement, and restore evidence all match expectations. Stop on unauthorized text, a superseded top result, a missing owner, an unexplained permission difference, or an unrehearsed disable path.

Limit: this small fixture does not measure broad task coverage; it tests revision, access, and disable-path handling.

Gate 3 tests each added domain

A larger corpus introduces duplicates, conflicting revisions, and permission joins. Do not average those failures into one plant-wide score. Report quality, freshness, and access results by domain, and stop expansion where the source owner cannot resolve them.

Making it stick

Gate 4 assigns recurring work: source refresh, evaluation, access review, incident response, restore testing, and source retirement. New notes should enter through a defined review path; routine ticket closure should not silently promote text into approved guidance.

The review outcome is one of four decisions: proceed, correct and repeat, reduce scope, or stop. Record the owner and evidence with that decision. Retrieval can make recorded knowledge easier to find, but it cannot prove that undocumented judgment has been captured or that retrieved guidance is correct for a new event.
