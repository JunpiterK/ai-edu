Engineering decision record

EDR template date: July 14, 2026. Decision scope: shortlist release-specific open-weight artifacts for an air-gapped evaluation. Status: proposed, not approved for production. The record begins with artifact identity and license, then moves to memory and task evidence. The repository links below are mutable discovery links, not frozen evidence. A completed EDR must capture the resolved commit SHA, the license and use-policy files with their hashes, and a SHA256 manifest for every imported artifact before legal or technical approval.

Decision rule

Record the exact repository, resolved commit SHA, artifact filenames and SHA256 values, license and use-policy hashes, runtime, quantization, hardware, and evaluation result. Family names and mutable branch links are too broad for approval because releases and derivative artifacts can carry different files and terms.

The opening artifact is not a leaderboard; it is a release-record template. A downloadable model may still be the wrong dependency for the intended use. Screen the captured revision and exact license bundle, deployment footprint, and task evidence in that order. This record is engineering input to procurement, security, and legal review, not legal advice or a license determination.

Illustrative scenario: license review ends the benchmark early

A team shortlists the highest-scoring 32B model and expects quantization to be the difficult part. The repository review changes the order of work: the linked release does not clear the intended commercial-use gate. Benchmark quality and VRAM tuning are now irrelevant for that candidate. The decision becomes hold, pending documented permission for the exact artifact. This scenario is constructed and does not report a plant selection.

BYO-model: what “open” actually buys you

Under a BYO-model design, the approved parameter files, tokenizer, configuration, runtime dependencies, and license bundle are transferred through the site's controlled import process and served on local hardware. Local serving can remove runtime model-API calls, but network controls and packet evidence are still required to verify the sealed boundary.

“Open weights” and “open license” answer different questions. Download access does not establish permission for a particular commercial use, modification, or redistribution path. Read the license, use policy, notices, and artifact revision as parts of the dependency bundle before spending time on a benchmark.

License first: the filter before benchmarks

For an air-gapped deployment, the review should answer several license and use-policy questions before technical evaluation.

  • Commercial use. Does counsel confirm that the exact artifact and intended use fit the license grant and related policies? A permissive license may simplify the review but does not eliminate it.
  • Redistribution and naming. If you fine-tune and share the result, must you rename it or carry an attribution notice? Meta’s license, for instance, requires derivative models to keep Llama in the name and ship a notice file.
  • User-count or field-of-use caps. Some licenses add thresholds, such as Llama’s 700M monthly active user clause, or use restrictions you must confirm you clear.
  • Offline auditability. The license text must travel with the weights and hold up in an on-prem review. Click-through terms fetched at runtime do not help when there is no runtime internet.

EXAONE 4.0: release-specific license boundary

The EXAONE 4.0 32B discovery page currently exposes the EXAONE AI Model License Agreement 1.2 - NC. Its text limits the grant to research and educational purposes and restricts commercial use unless separately agreed. Do not treat the mutable main link as evidence of what was reviewed: resolve and record the repository commit SHA, preserve the exact license bytes and hash, and bind them to the imported artifact manifest. The conclusion then applies only to that captured revision, not automatically to every past or future EXAONE artifact.

Do not infer permission from the ability to download or run the weights. Legal review must classify the proposed lab evaluation, internal business use, fine-tuning, output use, and any derivative or redistribution before the artifact enters the approved pipeline.

Sizing by GPU: what fits on what

Once a model family clears the license gate, the second filter is physical: how much VRAM you have and how much the model needs. The practical lever is quantization. Serving 4-bit (Q4) weights roughly halves 8-bit memory and cuts FP16 memory by about three quarters, usually with an acceptable quality tradeoff for grounded document Q&A.

Two planning footnotes matter. First, these numbers are weights only; leave headroom for the serving runtime and the KV cache, which grows with context length and concurrency. Second, Mixture-of-Experts (MoE) models change the math. gpt-oss-120b has 117B total parameters but activates only a smaller subset per token, so the serving footprint can be much lower than the headline size suggests. Size for real concurrency, not just parameter count.

A decision path for the fab

Put the filters in order and the shortlist becomes easier to defend. The pseudocode captures review sequence, not legal logic or a deployment recipe.

Five-minute decision record

Start with this checklist:

  1. Open one candidate discovery page and record the repository name plus the resolved commit SHA; do not record main as the revision.
  2. Export the exact license, use policy, notices, tokenizer, configuration, and weight-file list at that commit. Record a SHA256 value for each imported file or for a signed manifest that enumerates them.
  3. Write the intended use, users, modifications, output use, and distribution path, then name the required legal and security reviewers.
  4. Estimate weight plus runtime memory and name the target GPU, context length, concurrency, and fixed evaluation set owner.
  5. Pass to test only when the captured SHA and hashes reproduce the candidate, required reviewers clear the intended use, and the target host has measured headroom. Stop on a mutable-only reference, unresolved terms, a hash mismatch, missing files, an unreviewed derivative, or no representative evaluation owner.

Limit: this checklist narrows candidates; it does not grant rights or prove model quality.

For an initial evaluation, 14B and 32B dense artifacts are candidate size tiers, not recommendations. Measure task quality, peak VRAM, latency, throughput, restart behavior, and concurrency on the target runtime and hardware. Archive the artifact hash, configuration, license, notices, use policy, and legal approval with the evaluation record.

EDR outcome: advance only release-specific artifacts that clear the required legal, security, hardware, and task reviews. Public benchmarks can shape the test set but cannot substitute for it. Reopen this record when the artifact, license, runtime, quantization, hardware, use case, or distribution plan changes. With a candidate recorded, the natural next read is on-prem serving, where the question shifts from “may we evaluate it?” to “can we operate it under the declared load?”
