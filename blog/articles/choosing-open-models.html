Choosing an open model for air-gapped use

AE AI Edu Archive Semiconductor process engineer · Air-gapped LLM / RAG / Agents

The weights may be free; the license is not always free for your use case. Once you accept that the model has to run inside the air gap, the next question is which model to copy in. For a manufacturer, the honest answer is that this is a procurement and governance decision before it is a benchmark decision. You are choosing a weight file you will host, audit, and defend for years. The useful order is license, GPU footprint, then benchmark.

BYO-model: what “open” actually buys you

The organizing principle is BYO-model: bring your own model. An open-weight release ships the actual parameter files. You copy them across the air gap once, serve them on your own GPUs, and stop calling an external API. Capability becomes something you install, not something you rent by the token. That is the only shape that survives a sealed network.

But “open weights” and “open license” are different claims. A model can be fully downloadable and still forbid the one thing a for-profit plant needs: commercial operation. Before any benchmark table, read the license as if it were a production dependency, because it is.

License first: the filter before benchmarks

For an air-gapped deployment, only a few license questions matter. They all reduce to what you may do without asking anyone else.

  • Commercial use. Can a for-profit company run this in production? A permissive license such as Apache 2.0 or MIT says yes plainly. A community or NC license may not.
  • Redistribution and naming. If you fine-tune and share the result, must you rename it or carry an attribution notice? Meta’s license, for instance, requires derivative models to keep Llama in the name and ship a notice file.
  • User-count or field-of-use caps. Some licenses add thresholds, such as Llama’s 700M monthly active user clause, or use restrictions you must confirm you clear.
  • Offline auditability. The license text must travel with the weights and hold up in an on-prem review. Click-through terms fetched at runtime do not help when there is no runtime internet.
Table 1. Open-model licensing at a glance. Verify the exact release you download.
Family License Commercial? Notable conditions
Qwen3 Apache 2.0 Yes Permissive; attribution only. Dense 0.6B-32B plus MoE variants.
DeepSeek V3 / R1 MIT Yes Permissive; host, modify, and commercialize freely.
gpt-oss (OpenAI) Apache 2.0 Yes Permissive. 20B / 120B open-weight releases.
Llama 3.3 Llama Community Yes* *Derivative naming and notice requirements; 700M MAU threshold; acceptable use policy applies.
EXAONE (LG) EXAONE 1.x - NC No Non-commercial / research and education only unless a separate production agreement exists.
A downloadable weight file is not a usable production model until its license clears your legal team. For a fab, “non-commercial” is a disqualifier, not a footnote.

The EXAONE trap: open weights, non-commercial license

This deserves its own section because it is an easy mistake to make in a Korean plant. LG’s EXAONE models can be strong and downloadable from Hugging Face, but some releases are under the EXAONE AI Model License with an NC restriction. NC means non-commercial. Research, academic, and educational use may be allowed; commercial deployment requires a separate arrangement with LG AI Research.

For a self-learning pilot or a university lab, that may be fine. For a for-profit fab putting the model in front of process engineers to do real work, it is a disqualifier unless a separate commercial agreement exists. The trap is that nothing technical stops you from running it. The weights load, answers look good, and the issue appears later when an audit or customer security review checks the license. Read those two letters before you build.

Sizing by GPU: what fits on what

Once a model family clears the license gate, the second filter is physical: how much VRAM you have and how much the model needs. The practical lever is quantization. Serving 4-bit (Q4) weights roughly halves 8-bit memory and cuts FP16 memory by about three quarters, usually with an acceptable quality tradeoff for grounded document Q&A.

Table 2. Approximate VRAM by model size. Weights only; add serving and KV-cache overhead.
Size FP16 (approx.) Q4 (approx.) Runs on
7-8B ~16 GB ~5-6 GB Consumer GPU (16-24 GB)
14B ~28 GB ~10 GB RTX 3090 / 4080-class
32B ~65 GB ~19-20 GB 24 GB tight; 48 GB comfortable
70B ~140 GB ~40 GB 1x 80 GB (Q4) or multi-GPU
120B MoE - native MXFP4 ~1x 80 GB (gpt-oss-120b)

Two planning footnotes matter. First, these numbers are weights only; leave headroom for the serving runtime and the KV cache, which grows with context length and concurrency. Second, Mixture-of-Experts (MoE) models change the math. gpt-oss-120b has 117B total parameters but activates only a smaller subset per token, so the serving footprint can be much lower than the headline size suggests. Size for real concurrency, not just parameter count.

A decision path for the fab

Put the filters in order and most of the field narrows on its own. This is the concept, not a deployment recipe, but it captures the sequence a review will expect.

For a first air-gapped pilot, the practical sweet spot is usually a permissively licensed 14B-32B dense model at Q4 on a single data-center GPU. That is enough capability for grounded document Q&A while staying small enough to serve without a cluster. Move to 70B or MoE only when the task genuinely needs more reasoning and the VRAM budget supports it. Whatever you choose, copy the license text along with the weights; an air-gapped audit will ask for both.

Headline benchmarks should come last, not first. The license decides whether you may run the model at all. The GPU footprint decides whether you can run it. Only then should a leaderboard break the tie. Get that order right and the model you copy across the gap today is still defensible two years from now.
