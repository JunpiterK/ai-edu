The RAG stack, component by component

AE AI Edu Archive Semiconductor process engineer · Air-gapped LLM / RAG / Agents

RAG is not one feature. It is a small pipeline of components that turn private documents into useful context before the model answers. If one component is weak, the whole system feels unreliable.

The stack is easier to understand when you separate the jobs. A loader reads source material. A splitter creates chunks. An embedding model converts chunks into vectors. A vector store indexes them. A retriever chooses the chunks for a question. LCEL then connects context, prompt, model, and parser.

1. DocumentLoader: get the paper in

What it does. A loader reads a source file, PDF, HTML, CSV, Markdown, text, or a database export, and returns LangChain Document objects with page_content and metadata.

When to use it. Use loaders whenever your source material lives outside Python strings. Metadata is not optional in real RAG: source path, page, section, equipment, document version, and permission scope become essential later.

Implementation note. Start with plain text or Markdown before adding PDF. PDF extraction quality varies wildly, and bad extraction looks like a bad model even when retrieval is the real problem.

2. TextSplitter: cut it into chunks

What it does. A splitter turns long documents into chunks small enough to retrieve and fit into the prompt. It should preserve meaning, not merely count characters.

When to use it. Use splitters for almost every RAG system. Whole manuals are too large; tiny fragments lose context. A good splitter keeps headings, procedure steps, and tables close to the text they explain.

Chunking is the one knob that most changes retrieval quality. Too big and the embedding blurs; too small and a step loses its surrounding recipe. — tune chunk_size against your own manuals, not a blog default

Implementation note. RecursiveCharacterTextSplitter is the safe default. For technical documents, tune chunk_size and overlap after looking at actual retrieved chunks, not before.

3. Embeddings: turn text into vectors

What it does. An embedding model maps text into vectors so semantic similarity can be searched. A query and a chunk that mean similar things should land near each other in vector space.

When to use it. Use embeddings when keyword search is too brittle. They help with paraphrases, synonyms, and Korean/English mixed technical wording. They do not guarantee exact facts, part numbers, or relationships.

Implementation note. For a home PC, use a local embedding model first. Keep the model fixed while evaluating chunking, or you will not know which change helped.

4. VectorStore: keep the vectors

What it does. A vector store saves embeddings and metadata, then performs similarity search. Chroma is easy for prototypes. Qdrant, Postgres/pgvector, or another server-backed store is better when several apps share the same index.

When to use it. Use a vector store when documents need to be searched repeatedly. For one-off summaries, you may not need one. For a knowledge base, you do.

Implementation note. Persist the index and version your collection names. If the source corpus changes but the index name stays the same, debugging becomes painful.

5. Retriever: fetch the right ones

What it does. A retriever is the application-facing search interface. The rest of the chain calls retriever.invoke(question) and receives Documents, without caring whether the backend is Chroma, Qdrant, BM25, or hybrid search.

When to use it. Use retrievers whenever the model needs evidence before answering. Add metadata filters when users ask about a specific tool, date range, product, document type, or permission scope.

Implementation note. Always inspect retrieved chunks. Many RAG failures are retrieval failures wearing a fluent answer.

Assemble it with LCEL

The final chain is ordinary LCEL: retrieve context, preserve the question, format a prompt, call the local model, and parse the answer. The model should answer only from context and cite sources when possible.

This is the key habit: debug RAG component by component. If the answer is bad, ask whether loading, splitting, embedding, indexing, retrieval, or prompting failed before blaming the LLM.
