SOP intake

A document owner submits SOP-CVD-07 revision 12 for indexing. The intake package must show the approved file, effective date, superseded revision, permission group, extraction result, and acceptance tests. Ingestion does not begin with the embedding model; it begins with qualification of that package.

Illustrative extraction clue

The missing purge instruction looks like an embedding miss until a page-level extraction check shows the heading but not the table row that contains the duration. Re-embedding cannot recover text the loader never produced, so the candidate document goes back to extraction instead of moving downstream.

Think of this RAG stack as an ingestion traveler for a controlled document. Each station leaves an artifact: extracted text, inspected chunks, model identity, collection manifest, and retrieval results. A failed station quarantines the candidate with a reason. A pass means the declared ingestion checks passed; it does not approve the SOP or make every generated answer reliable.

Triage shortcut

When retrieval misses, inspect in pipeline order: source status, extracted text, chunk boundary, metadata, embedding, then search configuration. Do not tune the last knob first.

SOP ingestion qualification

Define the test protocol before running the loader. The document control owner approves source status. IT/security approves the permission mapping. The RAG owner runs extraction and retrieval tests. Keep the following artifacts with the index release:

Source manifestDocument ID, revision, effective date, owner, checksum, and ACL are present; status is approved for the intended audience.Quarantine the file and ask the document owner to resolve status or metadata.
Extraction reportPage count matches; required headings, procedure steps, warnings, and sampled tables are present in extracted text.Change parser/OCR settings or reject the format. Do not index partial extraction.

Thresholds must be set by the site for the selected corpus and task. Example values in code are starting configurations, not acceptance recommendations.

1. DocumentLoader: get the paper in

Artifact: extracted Document objects plus an extraction report. The loader reads PDF, HTML, CSV, Markdown, text, or a database export into page_content and metadata.

Pass/fail: reconcile file and page counts, sample required procedure elements, and verify document ID, revision, page, section, and permission metadata. Parser success alone is not a pass.

Store the extraction report and source checksum. If required text or tables are missing, quarantine the document and change the parser or OCR route before proceeding.

2. TextSplitter: cut it into chunks

Artifact: a versioned chunk set and chunking configuration. Splitting creates retrieval units small enough for the selected model context while attempting to preserve procedure structure.

Pass/fail: inspect boundary samples around headings, numbered steps, warnings, and tables. Fail when a step loses its condition, warning, unit, or document metadata.

Chunking is the one knob that most changes retrieval quality. Too big and the embedding blurs; too small and a step loses its surrounding recipe. — tune chunk_size against your own manuals, not a blog default

RecursiveCharacterTextSplitter is a starting implementation, not a qualified default for every SOP. Tune size and overlap against inspected chunks and the fixed retrieval set.

3. Embeddings: turn text into vectors

Artifact: the embedding model ID, model-file checksum, runtime configuration, and vectors associated with the candidate chunk set. Embeddings support similarity search; they do not establish exact facts, identifiers, or relationships.

Pass/fail: run the fixed retrieval questions, including exact part numbers, abbreviations, negation, and mixed Korean/English terms where relevant. Fail or add a complementary search method when required cases miss the expected source.

Change one qualified variable at a time. Keep the embedding model fixed while comparing chunking configurations, and retain results with the release manifest.

4. VectorStore: keep the vectors

Artifact: an immutable candidate collection, collection manifest, backup or rebuild instructions, and access configuration. A vector store saves embeddings and metadata and executes similarity search.

Pass/fail: verify chunk counts and metadata, deny out-of-scope principals in an ACL test, reopen the collection, and reproduce a sample query set. Fail on count mismatch, missing metadata, unauthorized results, or an unreproducible build.

Use a new collection or release identifier when the corpus, model, or chunking configuration changes. Keep the prior qualified collection as the rollback target until the release owner closes the change.

5. Retriever: fetch the right ones

Artifact: retriever configuration, fixed question set, expected sources, scored results, and failure analysis. The retriever returns Document objects without requiring the caller to know the search backend.

Pass/fail: evaluate expected-source recall and wrong-revision retrieval under the declared k, filters, and corpus. Permission filtering must occur in an enforced query path, not only in prompt text.

Inspect both passing and failing chunks. A fluent generated answer cannot compensate for retrieval of the wrong revision or an unauthorized source.

Assemble it with LCEL

Artifact: a versioned chain configuration and end-to-end evaluation report. LCEL retrieves context, preserves the question, formats the prompt, calls the selected local model, and parses text. Prompt instructions to use context or cite sources require evaluation; they are not enforcement.

Pass/fail: confirm that answers cite the expected document ID and revision, abstain under the agreed no-evidence cases, and do not expose sources outside the test principal's scope. Diagnose failures at the retained loader, chunk, embedding, index, retrieval, prompt, or model artifact.

Run one document through the traveler

Simplified exercise. Pick one approved training document, not a live operating instruction. Ask one exact step question, one abbreviation question, one superseded-revision trap, and one question with no supporting text. Retain the source checksum, extracted passage, winning chunks, and answer.

  • Pass: expected text survives extraction, step and unit stay together, the current revision is retrieved for the first two questions, the superseded revision is absent, and the no-evidence case abstains.
  • Stop: a required table cell disappears, ACL metadata is missing, an old revision wins, or an out-of-scope source appears. Quarantine the candidate collection rather than patching the final prompt.
  • Limit: four questions are a wiring check, not a release evaluation. Corpus coverage, multilingual queries, permission combinations, and answer quality need a larger owner-approved set.

The release owner should promote only the candidate index whose manifest and required tests pass, with a named rollback target. Once the ingestion traveler is working, the sealed-documents article is the natural next read because it expands from component checks to corpus access, citation, and release design. This procedure does not validate the underlying SOP content or make generated instructions suitable for unsupervised plant action.
