Opening artifact: candidate sheet

model_revision, tokenizer_revision, query_prefix, passage_prefix, vector_dimension, and corpus_snapshot belong on the comparison sheet before the first score does. Without them, a retrieval difference cannot be reproduced or assigned to the model.

The model decision starts with a retrieval test, not a reputation. For an isolated deployment, weights and runtime dependencies need to be staged inside the approved boundary, and network behavior still needs verification. The evaluation should look like the questions the service will receive: Korean and English prose, mixed units, part numbers, chamber IDs, and local abbreviations.

Constructed manifest check

A weak result on a Korean alarm query seemed to implicate multilingual-e5. The run manifest exposed a configuration mismatch instead: stored passages had no passage: prefix while queries used query:. After correcting both paths and rebuilding the index, the comparison could begin. That outcome is constructed; it demonstrates a configuration check, not a benchmark result.

How the embedding model affects retrieval quality

An embedding model maps text to a vector so that similar meaning can rank nearby. Retrieval is nearest-neighbor search over those vectors. If the relevant Korean or English passage does not enter the retrieved set, downstream prompting cannot cite it. Embedding choice is therefore one retrieval variable to test alongside chunking, metadata filters, exact-match search, and index settings.

An embedding model determines similarity ranking under a specific corpus, query, chunking policy, and index configuration. Measure that complete retrieval path.

The two candidates: bge-m3 and multilingual-e5

For a Korean-English mix intended for an offline boundary, bge-m3 and multilingual-e5-large are two candidates worth testing. Confirm license terms, model files, dependency sources, runtime network behavior, memory use, latency, and hardware fit before selection.

  • bge-m3 (BAAI): about 560M parameters, 1024-dimensional dense vectors, context up to 8192 tokens, and support for 100+ languages. It does not require a query instruction prefix, so queries and passages are embedded the same way.
  • multilingual-e5-large (intfloat): also 1024-dimensional and strongly multilingual, but it requires prefixes. Every query should start with query: and every stored passage with passage:. Forget the prefix and similarity quality degrades quietly.

Public benchmarks do not reproduce a site's vocabulary, chunking, or query mix. bge-m3 supports longer inputs than multilingual-e5-large, which may matter when a controlled document section cannot be split without losing context. Treat both as candidates until the site-specific evaluation below shows the trade-off.

Mixed Korean-English and technical jargon

General multilingual models handle ordinary bilingual text well. The fab-specific problem is vocabulary the model may never have seen: a chamber name such as PECVD-03B, a defect code, or an internal acronym. A subword tokenizer can fragment these terms, and documents that look obviously related to an engineer may land far apart in vector space. Three mitigations are practical, in order of effort.

  • Keep identifiers in the chunk text. Do not strip part numbers, chamber IDs, or defect codes during preprocessing. They are strong retrieval anchors when a query names them exactly.
  • Add a hybrid keyword channel. Pair dense vectors with BM25 or another sparse channel so exact-match terms such as lot IDs remain findable even when the dense model misses them.
  • Fine-tune only if evaluation proves you need it. Domain fine-tuning is real work and a maintenance burden inside a sealed network. Treat it as a last resort, not the starting point.

Dimension, normalization, and similarity

Both models output 1024-dimensional vectors. Two settings matter more than the number itself.

  • Normalize vectors. Set normalize_embeddings=True. When every vector has unit length, dot product and cosine similarity become equivalent, and many vector stores can use fast inner-product indexes.
  • Stay consistent. Use the same metric for indexing and querying. If you change the embedding model, re-embed the entire corpus; old and new vector spaces are not comparable.

Loading a local model offline

The current non-deprecated way to run Hugging Face embedding models through LangChain is the dedicated langchain-huggingface package. Point it at the local directory where you copied the weights. Runtime should not call the network.

bge-m3 does not require query or passage prefixes, so embed_query and embed_documents can receive raw text. If you choose multilingual-e5 instead, you must add prefixes yourself before embedding. Forgetting query: and passage: is a quiet failure mode.

If your shop has standardized on local models served by Ollama, embedding models can also be reached through langchain-ollama. Use the dedicated package rather than older community imports.

Then wire the chosen embedder into a local vector store. Chroma now lives in the langchain-chroma package; older langchain_community.vectorstores imports are deprecated.

Evaluate before you commit

Freeze the test conditions before comparing candidates. Use the same approved corpus snapshot, parser, chunk boundaries, metadata filters, vector metric, index parameters, and hardware. Record model revision, tokenizer revision, precision, batch size, query and document prefixes, and the date of the corpus snapshot. Rebuild the index for each model rather than mixing vector spaces.

  1. Build judgments: sample questions from the intended users and languages, then have a document owner mark every acceptable source section. Keep ambiguous or unanswerable questions and label them explicitly.
  2. Separate tuning from test: use one set to choose chunk size, overlap, k, and hybrid-search settings. Report final results on a held-out set that was not used for those choices.
  3. Run controlled comparisons: execute each query against each rebuilt index under the frozen conditions. Store query text, ranked chunk IDs, source revision, scores, and latency.
  4. Score retrieval: report recall at the operational k, mean reciprocal rank, no-answer behavior, and p50/p95 latency. Slice results by Korean, English, mixed-language, exact identifier, and long-section queries.
  5. Review failures: classify misses as embedding, chunking, stale source, access filter, exact-match, or labeling failures. Change one variable at a time and rerun the held-out set only after the design is fixed.
Example values, not plant measurements

A pilot might begin with 40 tuning questions and 60 held-out questions, retrieve k=5, and require the correct released section to appear in at least 54 of the 60 held-out results. Those values illustrate a procedure, not a recommended acceptance threshold. The retrieval owner and document owner should set the threshold from the cost of misses, false matches, and review time.

The decision record should name the selected model and revision, rejected candidates, test conditions, metric results, unresolved failure slices, and the owner who approves re-indexing. Repeat the evaluation after a model, parser, chunking, corpus, or index change.

An embedding model earns its place by retrieving the right released evidence under frozen conditions. Keep the manifest beside the results, rebuild each candidate's index, and retain the failure slices. With that choice documented, the natural next read is hybrid retrieval, where dense similarity is paired with exact identifiers and governed relationships.
