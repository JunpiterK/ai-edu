VIN genealogy vs wafer genealogy: traceability for automotive AI

AEAI Edu ArchiveApplied AI, robotics, manufacturing systems
On this page
  1. What genealogy means
  2. VIN vs wafer
  3. The evidence timeline
  4. Why RAG needs genealogy
  5. Starter manifest

A containment record needs two lists: units included by the current evidence and units excluded with a documented reason. Genealogy supplies that evidence by joining append-only, supersedable events around a VIN or wafer identity. The timeline can be corrected, but the original observation and the correction both need to remain reviewable.

Illustrative scenario: rework widens the containment boundary

A VIN list filtered by torque station and clock window looks tidy. Then a rework-route event shows that one vehicle passed the station twice, while a component-swap event links another VIN to the suspect part lot. Station time alone cannot define membership. The revised query includes route history, component installation, late-arriving events, and explicit exclusions. This is a constructed scenario, not a reported recall or quality event.

What genealogy means

Genealogy earns its keep during investigation. When a vehicle fails an EOL test or a wafer later shows a metrology excursion, engineers need the manufacturing facts that traveled with that unit, including rework and corrections.

Genealogy also gives RAG and agents a scope boundary: this vehicle or wafer, this supplier lot or carrier, this station or chamber window, this program or recipe revision, and this quality action. Retrieval can still be incomplete or wrong, but at least its membership claim is inspectable.

The evidence timeline

A useful automotive AI system should be able to reconstruct a VIN timeline. The timeline does not need to be perfect on day one. Start with high-value stations and expand. A practical first slice might cover body shop weld cells, final assembly torque tools, vision inspection, and EOL test.

  • Before station: VIN, model variant, build option, supplier lot, incoming inspection state.
  • During station: station start/end, robot program, PLC state, tool ID, operator ID or team, safety state.
  • Quality evidence: torque result, image result, defect code, rework reason, bypass reason.
  • After station: release, hold, containment, EOL test, warranty feedback, field symptom.

Event model

Store each observation as an event with event_id, subject_id, event_type, event time, ingestion time, station or operation, equipment and program version, result, source-system key, and supersession link. Component-install events connect a part serial or supplier lot to the VIN. Quality-action events record hold, release, containment, rework, and disposition without overwriting the event that triggered them.

Illustrative containment query

Example values, not plant measurements: quality identifies an EOL symptom on one VIN and suspects torque station 220, program revision 14, between 10:00 and 11:00. The query below returns candidate VINs; the quality owner reviews completeness and declares the containment boundary.

SELECT DISTINCT e.subject_id AS vin
FROM genealogy_event e
JOIN genealogy_event q ON q.subject_id = e.subject_id
WHERE e.event_type = 'torque_result'
  AND e.station_id = 'torque-220'
  AND e.program_revision = '14'
  AND e.event_time >= :window_start
  AND e.event_time <  :window_end
  AND q.event_type = 'eol_result'
  AND q.result_code = :suspect_symptom;

The result is a candidate set, not proof of causation. Missing events, clock skew, rework routes, component swaps, and late-arriving records can change membership. Preserve the query, parameters, source snapshot, exclusions, reviewer, and release decision with the containment record.

Why RAG needs genealogy

Automotive RAG is much stronger when retrieval is scoped by genealogy. A generic question like "Why did this EOL test fail?" is too wide. A genealogy-aware question is sharper: "For this VIN, retrieve the EOL failure, previous station events, related torque records, matching DTC notes, repair manual section, and prior 8D cases for the same symptom."

This is where vector search and structured filters should work together. Vector search finds similar descriptions. Structured genealogy keeps the answer grounded in the right vehicle, time window, model variant, and quality boundary.

Starter manifest

For a first pilot, define a small manifest. Do not attempt every station and every system at once.

vin_genealogy_manifest:
  scope:
    plant: "plant-a"
    line: "final-assembly-1"
    stations: ["torque-220", "vision-310", "eol-900"]
  entities:
    - VIN
    - station_event
    - tool_result
    - vision_result
    - eol_result
    - quality_action
  required_keys:
    - vin
    - timestamp
    - station_id
    - equipment_id
    - result_status
    - source_system
  first_questions:
    - "Which VINs share this failed EOL symptom?"
    - "Which station window is common?"
    - "Which torque or vision evidence changed before the failure?"
    - "Which rework or containment action already exists?"
Worked exercise: containment decision record

Run the sample query for one approved test window and record the parameters, source snapshot, candidate VIN count, unmatched event count, rework routes, component swaps, late arrivals, exclusions, reviewer, and decision time. Pass: every included and excluded VIN has a reviewable reason and rerunning against the frozen snapshot returns the same set. Stop: do not release or narrow containment when required sources are missing, event clocks are unresolved, or a superseding quality action is unprocessed. Limit: reproducible membership supports containment review; it does not prove the suspected station caused the symptom.

Once the timeline can support that record, continue with takt-aware AI agents to decide how quickly the evidence must be routed and which owner can act on it.

Reference event model and illustrative query; adapt keys and containment authority to the plant systems.
