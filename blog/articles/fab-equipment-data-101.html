FAB equipment data 101: what engineers must organize before AI

Open a blank source manifest before opening a model notebook. Its first row should name one system of record, its owner, event-time meaning, revision rule, access boundary, allowed use, and known gap. That small artifact tends to expose the hard part early: whether trace, event, recipe, lot, metrology, and maintenance records can answer one bounded question without hand repair.

Illustrative scenario: rows match, identities do not

The query returns rows when chamber ID and timestamp are used to connect an etch trace to the affected wafer. A shift-boundary review then finds reused local chamber names and collection timestamps several minutes after event time. The operational snag is not the model; it is ambiguous identity and clock semantics. The team stops the feature build, adds the host tool ID and run ID, documents both clocks, and keeps unmatched records visible. This is a constructed example, not a plant incident.

Why this comes before AI

A pilot cannot support a reproducible decision until the team answers basic data questions:

  • Which system is the source of truth?
  • Is this current revision or a copied local file?
  • Does the timestamp mean event time, collection time, upload time, or engineer edit time?
  • Can every user see this data, or does retrieval need permission filtering?
  • Does this field identify tool, chamber, recipe, lot, wafer, product, shift, and owner?
  • What data should never be indexed, summarized, or exported?

Record the answers in the source manifest and make missing context visible to the application. A demonstration may still run on copied files, but that result does not establish source currency, authorization, or repeatability.

Readiness check

Select one wafer or run and reproduce its context from governed sources. Pass: source row IDs, units, event times, revision IDs, and unmatched-key counts survive the join. Stop: do not use the joined set for model evaluation when identity or time meaning is unresolved, or when an undocumented spreadsheet repair changes membership. Limit: a clean join supports traceability; it does not prove that a correlated change caused the process result.

A hypothetical chamber join

Illustrative scenario: a dry etch chamber shows intermittent CD drift. The question is, "Which recorded changes overlap the affected runs?" This does not establish causality. It defines the records that an engineer would inspect in a stitched view:

  • Trace: pressure and RF power during the affected recipe steps.
  • Recipe: exact recipe revision and any setpoint change around that shift.
  • Lot context: product, route, operation, chamber, wafer order, hold history.
  • Metrology: CD measurement timing, tool, sampling plan, and units.
  • Alarms: warnings during the run and any acknowledged but unresolved events.
  • Maintenance: last PM, chamber clean, part replacement, leak check, or technician note.
  • Documents: current troubleshooting guide, known issue memo, control plan.

Build a source manifest first

A source manifest is a controlled inventory of what data exists and how it may be used. Create it before indexing or feature extraction, and assign an owner to every unresolved field.

Illustrative manifest

The YAML below demonstrates field shape only. Its names, access category, freshness interval, and dates are example values, not plant measurements or policy.

source: etch_chamber_trace
owner: equipment_engineering
system_of_record: historian
join_keys:
  - tool_id
  - chamber_id
  - lot_id
  - wafer_id
  - recipe_step
  - event_time
freshness_rule: "available within 15 minutes after run end"
access_level: "module_engineering_only"
allowed_use:
  - anomaly_review
  - virtual_metrology_features
  - engineer_copilot_context
blocked_use:
  - external_export
  - automated_recipe_change
known_weakness: "recipe_step labels changed during 2026 Q2 migration"

Make a do-not-index list

Teams often ask what should go into the AI system. First identify what must not go in yet. A do-not-index list excludes unsafe, stale, noisy, or restricted sources before a pilot begins.

  • Superseded SOPs: keep for audit, but do not retrieve as active guidance.
  • Personal notes with no owner: useful for interviews, risky as authoritative source.
  • Unapproved recipe copies: dangerous if a model treats them as released truth.
  • Restricted yield or customer data: use only with explicit access filtering and purpose control.
  • Low-quality OCR scans: fix extraction before indexing, especially tables and warning boxes.
  • Mixed-unit CSV files: block until units and calibration context are explicit.
Rule of thumb

If a junior engineer could get in trouble for using the source without asking, the AI should not silently retrieve it. Permission-aware retrieval is not an advanced feature. In a plant, it is part of basic hygiene.

What this unlocks

Once the data floor is mapped, the rest of the AI roadmap becomes more practical:

  • RAG: documents now have owners, revisions, freshness rules, and access levels.
  • FDC: trace signals can be aligned to recipe step, chamber, lot, wafer, and golden runs.
  • Virtual metrology: features can be selected from known process context, not random columns.
  • RCA: commonality analysis can join tool, chamber, recipe, lot, metrology, and maintenance facts.
  • Agents: tools can be split into safe read-only queries, drafts, queued actions, and approved execution.

The next action is to complete the manifest for one bounded join, run it on approved records, and report unmatched keys and time conflicts to the named source owners. Once that evidence chain is stable, continue with FDC from first principles to see how recipe-step alignment and baseline ownership turn trace data into review signals. The manifest still does not prove data quality or causality; it makes those claims testable.
