Approval packet

The reviewer should receive one inspectable object: exact tool name and arguments, payload hash, requester, source revisions, expected effect, expiry, recovery path, and approval policy. A button without that packet is only a pause in the UI.

Illustrative approval replay

A confirmation dialog after the model proposes a recipe-record change appears to provide the required human gate. During a tabletop replay, however, the proposal changes between display and execution because the approval is not bound to a payload hash. That evidence moves identity and payload verification into a trusted approval service, which rejects changed or expired proposals.

Human review is one control in an agent workflow, not a safety guarantee. A state-changing tool call can alter a recipe record, propose an MES lot hold, or open a maintenance ticket. Before enabling it, name the threat, authorization boundary, approval principal, durable state, audit evidence, recovery path, and owner.

Approval shortcut

Ask the reviewer to approve a hashable action packet, not a sentence generated for the screen. Execution should reject any mismatch in payload, principal, policy, or expiry.

Why a human gate before the agent acts

An LLM can propose an unsupported setpoint, misread an ambiguous procedure, or combine valid steps incorrectly. A reviewer may catch those errors, but reviewer presence does not replace authorization, deterministic limits, target-system controls, or safety engineering. The approval screen must show the exact action, arguments, source evidence, expected consequence, expiry, and rollback or recovery path before the reviewer decides.

Split tools by capability, then authorize every call. Read tools require authentication, source ACL enforcement, purpose and field limits, and an auditable requester identity. Write tools also require deterministic payload validation, a least-privilege execution identity, and named approval before execution.

  • Read tools: query MES, read a recipe, or pull an equipment log. They do not change plant state, but they can disclose restricted data and therefore do not run freely.
  • Write tools: set a parameter, place a hold, or dispatch a work order. They pass through validation and an approval gate before a named principal can resume execution.
Authorize every read. For every write, also require a validated payload, named approval, idempotent execution, and a recorded outcome.

The approval gate with interrupt()

LangGraph gives us the exact primitive for this pattern. Inside a node, call interrupt(payload): the runtime persists the whole graph state, surfaces the payload to the caller, and halts. Nothing downstream runs. The graph stays paused across minutes, or across a server restart, until someone resumes it with Command(resume=...). The resume value becomes the return value of interrupt(), so the human decision flows straight back into the node logic.

One behavior matters in production: when you resume, LangGraph reruns the node from the top. It does not continue from the line after interrupt(). Keep every real side effect, especially the actual write, after the interrupt returns. Everything above the interrupt must be safe to execute more than once.

This is a control sketch, not a production approval service. The approval service must derive the reviewer identity from trusted server-side authentication, authorize that principal for the named resource, and bind the decision to the exact action hash, thread ID, pending interrupt ID, requester, resource, expiry, and one-time approval ID. The resume handler below verifies that complete context before it addresses the interrupt. Do not expose direct graph-resume access to a browser or let client-supplied identity or thread fields establish trust.

Compiling the graph requires a checkpointer. Without one, interrupt() has nowhere to persist the paused state. The checkpointer supports recovery; the interrupt-specific resume map, signed context, required audit receipt, and idempotency key protect separate authorization, evidence, and replay boundaries.

Durable state: why PostgresSaver, not MemorySaver

Many tutorials reach for InMemorySaver, formerly MemorySaver. It stores checkpoints in a Python dictionary, which is perfect for a notebook demo and wrong for production. If the process restarts while an approval is pending, the paused graph disappears. The engineer's half-reviewed hold vanishes, and you have no record that it ever existed. In a fab that runs 24/7 and redeploys services behind the scenes, in-memory state is a data-loss bug waiting to happen.

Use PostgresSaver against an on-prem Postgres database. The paused state becomes a row in a database that lives on your network. It survives restarts, it is backed up with the rest of your operational data, and a pending approval from Tuesday is still there on Wednesday morning. Run .setup() once to create the checkpoint tables. Resume is keyed by thread_id, so the same conversation reloads exactly where it paused.

Audit logs and the approval trail

A durable checkpoint supports workflow recovery; it is not an immutable audit trail. Checkpoint rows may be updated, deleted, or accessed by application administrators. Append a separate structured decision event to required durable audit storage before resume; if that append has no receipt, do not address the interrupt or fire the write tool. The record should include the proposed tool call and payload hash, requester and reviewer identities, policy result, approve or reject decision, trusted timestamp, model/workflow version, source references, thread and interrupt IDs, and later execution receipt.

The sketch requires append_required() to return only after durable storage and to raise on timeout, rejection, or uncertain delivery. That contract makes decision-audit failure close the resume path; production deployment still needs restricted writers, retention policy, integrity or tamper-evidence controls, synchronized time, access review, and reconciliation between the audit receipt and target-system execution receipt.

Retraining, drift watch, and the MLOps loop

Approval outcomes are review data, not automatically ground-truth labels. A rejection can reflect model error, stale evidence, ambiguous policy, reviewer disagreement, or a changed operating condition. Track outcome rates by tool, workflow version, source revision, and reviewer context, then sample cases for adjudication before using them for retraining.

For an air-gapped design, the MLOps loop is intended to stay on-prem. Verify that claim against model endpoints, dependency update paths, DNS, proxies, telemetry, and log sinks; topology alone is not evidence of containment. A practical loop looks like this:

  • Watch: dashboard approval outcomes, read-tool authorization denials, retrieval quality, execution failures, and reconciliation gaps; investigate thresholds set by the relevant owners.
  • Collect: sample approval records for adjudication and separate model error from policy, data, interface, and reviewer causes.
  • Refresh: test whether a corpus or index update addresses the adjudicated failures before considering model tuning. Retrieval changes are usually easier to inspect, but they will not correct every failure class.
  • Release gate — treat a new model version as a state-changing action: shadow it, compare its would-be decisions against approved history, and require sign-off before it serves the line.

This loop can run on-prem, but it still needs explicit owners for identity, data access, application behavior, target-system writes, audit retention, and model release. No single outcome rate is sufficient to describe reliability.

Approval tabletop with a hard stop

Simplified exercise. Use a non-production stub action. Pause it for approval, restart the workflow service, resume with an authorized principal, then replay the same approval ID and alter one payload field.

  • Pass: the restart preserves the pending packet, the valid resume executes once, the replay is rejected or reconciled to the existing receipt, and the changed payload fails hash verification.
  • Stop: the browser supplies reviewer identity, an expired approval resumes, audit delivery failure still permits execution, or the target result cannot be reconciled.
  • Limit: a stub proves workflow behavior only. It does not qualify a plant write, establish functional safety, settle retention requirements, or replace owner approval.

The implementable pattern is to authorize reads, validate and approve writes, persist workflow state, execute idempotently, and reconcile a separate audit record with the target system. The LangGraph workflow article is the useful companion read when you need to inspect the pause, resume, rejection, and duplicate-transition mechanics in code.
