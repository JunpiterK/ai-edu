Human-in-the-loop and MLOps for on-prem agents

AE AI Edu Archive Semiconductor process engineer · Air-gapped LLM / RAG / Agents
On this page

  2. The approval gate with interrupt()
  3. Durable state: why PostgresSaver, not MemorySaver

An agent that can call tools can change the plant. In a fab, a tool call is not an abstraction: it can write a recipe parameter, file an MES lot hold, or open a maintenance ticket on a deposition chamber. The moment an LLM can trigger those actions on its own, the question stops being whether the answer is good and becomes who signed off before it happened. This article defends one operating rule: an on-prem agent must pause for human approval before any state-changing action, and that pause has to survive a process restart.

Why a human gate before the agent acts

On the open internet, a bad agent action is usually a wasted API call. Inside a sealed fab network, the blast radius is physical. An LLM can hallucinate a plausible but wrong setpoint, misread an ambiguous SOP, or chain two correct steps into a dangerous third. Accuracy metrics do not catch that. An engineer who knows the chamber does. The design goal is therefore not only a smarter model; it is a reliable point of control where a person reviews the proposed action, with full context, before it commits.

Concretely, split the agent's tools into two classes:

  • Read tools — query MES, read a recipe, pull an equipment log. These run freely because they cannot change the line.
  • Write tools — set a parameter, place a hold, dispatch a work order. Each one passes through an approval gate and cannot execute until a named human resumes it.
A read agent needs to be right. A write agent needs to be stoppable, and every stop needs a name attached to it.

The approval gate with interrupt()

LangGraph gives us the exact primitive for this pattern. Inside a node, call interrupt(payload): the runtime persists the whole graph state, surfaces the payload to the caller, and halts. Nothing downstream runs. The graph stays paused across minutes, or across a server restart, until someone resumes it with Command(resume=...). The resume value becomes the return value of interrupt(), so the human decision flows straight back into the node logic.

One behavior matters in production: when you resume, LangGraph reruns the node from the top. It does not continue from the line after interrupt(). Keep every real side effect, especially the actual write, after the interrupt returns. Everything above the interrupt must be safe to execute more than once.

Compiling the graph requires a checkpointer. Without one, interrupt() has nowhere to persist the paused state. That checkpointer is the whole reliability story, and it is where the on-prem decision matters most.

Durable state: why PostgresSaver, not MemorySaver

Many tutorials reach for InMemorySaver, formerly MemorySaver. It stores checkpoints in a Python dictionary, which is perfect for a notebook demo and wrong for production. If the process restarts while an approval is pending, the paused graph disappears. The engineer's half-reviewed hold vanishes, and you have no record that it ever existed. In a fab that runs 24/7 and redeploys services behind the scenes, in-memory state is a data-loss bug waiting to happen.

Use PostgresSaver against an on-prem Postgres database. The paused state becomes a row in a database that lives on your network. It survives restarts, it is backed up with the rest of your operational data, and a pending approval from Tuesday is still there on Wednesday morning. Run .setup() once to create the checkpoint tables. Resume is keyed by thread_id, so the same conversation reloads exactly where it paused.

Audit logs and the approval trail

Because every pause is a durable checkpoint, you can build an audit trail without adding a separate control plane. The checkpoint records the proposed action and state; the resume payload records who decided and what they decided. Write both to an append-only log at the moment of resume, before the write tool fires. For a regulated line, each record should include the proposed tool call, reviewer ID, approve or reject, timestamp, and thread_id that ties it back to the full agent transcript.

Now every write on the line has a name and a timestamp behind it, stored on the same network as the action itself. That is the record an auditor, or a post-mortem after a bad lot, actually needs.

Retraining, drift watch, and the MLOps loop

The approval gate is also your best data source. Every rejection is a labeled example of the model being wrong in a way a human caught. Track the reject rate over time, sliced by tool and shift. A rising reject rate is often the earliest signal that the agent has drifted away from how the line actually runs: a new recipe family, a changed SOP, or a retooled chamber.

Because the fab is air-gapped, the MLOps loop stays entirely on-prem. No telemetry leaves the network, and no managed retraining service is required. A practical loop looks like this:

  • Watch — dashboard the reject rate and read-tool retrieval quality; alert when either crosses a threshold.
  • Collect — mine the audit log for rejected actions and the corrected decision the engineer made instead.
  • Refresh — re-index the RAG corpus first, because it is cheap and fixes most drift; fine-tune the local model only when retrieval alone cannot close the gap.
  • Release gate — treat a new model version as a state-changing action: shadow it, compare its would-be decisions against approved history, and require sign-off before it serves the line.

None of this needs a cloud MLOps platform. It needs a durable checkpoint store, an honest audit log, and a metric, reject rate, that a human on the floor already understands.

The pattern is small, and it is the point: read freely, write behind a gate, persist the pause to Postgres, and log who decided. An agent built this way is not just accurate. It is accountable, restart-safe, and safe enough to bring near a running line.
