PROLOGUE · 왜 에이전트인가배경

"답"을 넘어 "현장을 도는 일"까지

RAG 챗봇은 묻는 말에 답합니다. 하지만 현장의 일은 — "이 알람이 왜 떴는지 진단하고, 절차를 확인하고, 이력을 모아, 보고·조치까지" — 여러 시스템을 오가는 다단계 작업입니다. 이것을 전용 에이전트들이 나눠 맡습니다.

알람 1건 처리 — 사람: SOP 문서고 · FDC · 모니터링 서비스 · MES 4곳을 직접 조회, 평균 ~25분  /  에이전트: 같은 4곳을 도구 4회 호출로 모아 초안 ~3분 + 사람은 검증만. ※ 수치는 이해를 돕는 예시
현실

도구가 흩어져 있다

SOP는 문서고, 센서는 FDC, 이상진단은 전용 모니터링 서비스, 이력은 MES. 사람이 다 돈다.

묶기

오케스트레이션

복합 상황이 오면 오케스트레이터가 여럿을 호출·종합해 하나의 답·행동으로.

전제모델·검색·도구·데이터가 전부 사내 폐쇄망 안. 외부 API·클라우드로 한 바이트도 나가지 않습니다 — 그래서 watsonx + gpt-oss-120b 온프레미스.
PROLOGUE · 에이전트란LLM + 도구 + 루프

에이전트 해부 — LLM이 도구 호출문을 "글로 써서" 시스템을 부린다

에이전트 = LLM(판단) + 도구(손) + 루프(반복). LLM은 시스템을 직접 만지지 못합니다. 대신 아래처럼 JSON 호출문을 쓰고, 시스템이 실행한 결과를 읽고, 다음 행동을 정합니다.

화살표를 따라 읽으면 됩니다 — 쓰고(Action) · 실행되고 · 읽고(Observation) · 다시 판단. 근거가 모이면 루프를 빠져나옵니다.
LLM이 '쓴' 도구 호출문 (Action) {"tool": "fdc_query", "args": {"eqp": "CHM-C2", "around": "10:30", "win": "±15m"}} 시스템이 돌려준 결과 (Observation) {"flag": "ABNORMAL", "sensor": "진공압 ↑스파이크", "valve_current": "변동"} → LLM은 이 JSON을 '읽고' 다음 행동을 결정한다. 분류가 필요하면 fdc_classify, 근거가 충분하면 답변으로 탈출.
챗봇과의 차이 — 챗봇은 한 번 답하고 끝. 에이전트는 결과를 보고 계속 행동을 고쳐 쓰는 루프가 있습니다. 그래서 폭주 방지(스텝 상한·타임아웃)가 반드시 함께 갑니다.

기억 전략

표를 외우지 말고 짝을 보세요

폐쇄망 슬라이드의 표는 "제약 ↔ 대응"이 1:1로 짝지어져 있습니다. "외부 X가 안 되면 → 사내에 X를 세운다"는 한 문장이 다섯 줄 전부입니다. 카탈로그도 마찬가지 — "한 에이전트 = 한 책임 + 한 도구 묶음" 하나로 여섯 카드가 정리됩니다.

남길 질문
  1. 폐쇄망에서 외부 LLM API 대신 무엇을 쓰나?
  2. "한 에이전트 = 한 책임"이 감사·디버깅에 왜 유리한가?
  3. 읽기와 쓰기의 경계는 어디인가?
정답 확인
  1. watsonx.ai에 gpt-oss-120b를 사내 서빙합니다. 임베딩·검색도 BGE·Vector DB로 전부 사내 — 데이터가 경계 밖으로 나가지 않습니다.
  2. 문제가 생겼을 때 어느 에이전트·어느 도구·어느 스텝인지 로그로 바로 좁혀집니다. 책임이 겹치면 추적 범위가 곱으로 늘어납니다.
  3. 조회·진단(READ)은 자율, 티켓·메일·설비조작(WRITE)은 사람 승인 게이트 뒤. 쓰기 도구는 액션 Agent에만 있습니다.

세 개가 다 답해지면 챕터 Ⅲ으로. 흔들리면 해당 슬라이드로 돌아가면 됩니다.

SOP Agent — 절차서를 근거로 READ-ONLY

하는 일

"이 상황 대응법?"에 SOP를 RAG 검색해 단계와 기준을 인용해 답. 읽기 전용.

도구

vector_search(BGE+Rerank) · sop_lookup(문서·조항 직접 조회).

가치

최신 개정본을 메타로 우선, 출처(문서·조항) 클릭 검증 → 절차 누락·구버전 적용 방지.

원칙근거 없으면 "해당 절차 없음" — 절차를 지어내지 않는다.
Q "PVD-A104 누설 시 즉시 조치?" A1 vector_search("PVD-A104 누설 즉시 조치") O1 [1] SOP-PVD-117 §2 / [2] §3.2 A ① 게이트밸브 차단 ② N₂ 퍼지 ③ 30분 추세 기록 ④ 시프트리더 보고 [1] 30분 내 3회↑ 반복이면 §4 점검 [2] 출처: SOP-PVD-117 v3.2 §2, §3.2 (클릭 검증)
CHAPTER III · 전문 에이전트② FDC Agent

FDC Agent — 설비 센서의 이상을 읽는다 READ-ONLY

FDC(Fault Detection & Classification) — 공정 설비의 센서 trace(진공압·온도·RF·유량…)를 감시해 이상을 감지하고 유형을 분류하는 시스템. 에이전트는 이 FDC를 도구로 조회합니다.

도구

fdc_query(설비·시각 구간 센서 trace·이상플래그) · fdc_classify(이상 유형 분류·점수).

하는 일

알람 시각 전후 어떤 센서가 튀었는지, 분류 코드(예: VAC-LEAK)와 신뢰도를 가져와 근거로 제시.

연계

이상진단·SOP 에이전트와 교차검증 — "센서가 가리키는 원인"과 "진단 요약"이 일치하는지.

Q "10:30 PVD-A104 시점 센서 이상?" A1 fdc_query(line=3, eqp="CHM-C2", around="10:30", win="±15m") O1 진공압 센서 ↑스파이크, 게이트밸브 전류 변동 — flag=ABNORMAL A2 fdc_classify(trace) O2 분류 "VAC-LEAK" · 신뢰도 0.91 A 10:30 진공압 스파이크, 누설 패턴 (VAC-LEAK 0.91)으로 분류됨.
CHAPTER III · 전문 에이전트③ 이상진단 Agent

이상진단 Agent — 펌프·로봇 모니터링 요약을 해석 READ-ONLY

핵심 아이디어 — 무거운 이상진단은 전용 서비스가 이미 합니다. 에이전트는 그 요약 진단(이상점수·추세·원인후보)을 받아 사람 언어로 설명하고, FDC·SOP와 교차검증하고, 조치까지 잇습니다.

① 무한 루프

③ 권한 초과

제조망 쓰기 시도

진단 중 "히터 설정을 바꾸면 되겠다"며 제어 명령을 시도.

④ 에이전트 핑퐁

서로 위임만 반복

FDC→이상진단→FDC… 서로 "그건 네 담당"이라며 결론 없이 공을 넘긴다.

가드레일 → 최대 위임 깊이 2 — 초과 시 오케스트레이터가 회수·종합.
