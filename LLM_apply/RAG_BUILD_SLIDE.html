PROLOGUE · 왜 사내 RAG인가목표

우리가 만들 것 — "근거를 대는 사내 AI"

엔지니어가 챗봇에 묻습니다. "라인3 진공 누설 알람(PVD-A104) 대응 절차랑, 지난주 반복 이력 알려줘." AI가 사내 SOP·MES·결함DB를 근거로, 출처를 인용하며, 필요하면 이슈까지 등록합니다.

CHAPTER II · 임베딩왜 BGE인가

임베딩 모듈로 BGE를 쓰는 이유

BGE(BAAI General Embedding) — 베이징 AI 연구원(BAAI)이 공개한 오픈 임베딩 모델. 그중 BGE-M3(2024)가 사내 RAG의 사실상 표준 후보입니다. FIELD · 현장 검증

Multilingual

다국어

100+ 언어, 한국어 강함. 사내 한·영 혼용 문서에 적합.

Multi-granularity

긴 입력

최대 8192 토큰까지 한 번에 — 긴 절차서도 통째로.

Multi-function

3-in-1

Dense(의미)+Sparse(키워드)+Multi-vector(ColBERT) 동시. ACAD · Multi-vector는 연구 단계

On-prem · Open

사내 설치

오픈 가중치 → 사내 GPU에서 실행, 데이터 외부 유출 0.

출력 차원
Dense 벡터 1024차원. 한 청크 = 1024개의 실수. 이 좌표 사이 거리가 "의미 유사도".
함께 쓰는 짝
검색 정밀화용 BGE-reranker-v2-m3(재순위). 임베딩(넓게)→reranker(정밀)의 2단 구조.

※ 대안 — e5-mistral Jina-v3 Qwen3-Embedding · 한국어·온프레미스·라이선스를 기준으로 선택. 외부 API(OpenAI text-embedding-3)는 데이터 유출 이슈로 사내에선 지양.

CHAPTER II · 임베딩텍스트 → 1024차원 벡터

한 청크가 1024개의 숫자가 되는 과정

입력 청크 "PVD-A104 알람은 진공도가 §3 허용 범위를 벗어났을 때 발생한다." ↓ BGE-M3 인코더 (Transformer) [ 0.021, -0.114, 0.067, ... ] ← 1024개 ↓ L2 정규화 (길이=1) 단위 벡터 = "의미 공간 위의 한 점"

정규화하면 코사인 유사도 = 내적(dot)이 되어 검색이 빠르고 안정적.

왜 "숫자"여야 하나

컴퓨터는 글자의 의미를 모릅니다. 의미를 좌표로 바꿔야 "가깝다/멀다"를 계산할 수 있습니다.

비슷한 의미 = 가까운 좌표

"균열"과 "크랙", "누설"과 "leak"은 글자가 달라도 벡터가 가깝게 학습돼 있음.

CHAPTER III · 인덱싱수집 → 적재

사내 문서를 벡터로 쌓는 5단계

01

수집

SOP·MES·PLM·메일. 권한·보안등급 메타 함께.

02

파싱

PDF·표·이미지 추출. 표는 행 보존, 그림은 캡션·OCR.

03

청킹

구조 인식 분할. 헤더·표 안 깨지게, 300~800자 + 오버랩.

04

임베딩

각 청크 → BGE → 1024차원. 배치로 대량 처리.

05

적재

벡터+메타+원문을 Vector DB에 upsert.

예 · 사내 SOP 한 청크의 적재 레코드 { "id": "SOP-PVD-117#c12", "vector": [0.021, -0.114, ...], # BGE 1024차원 "text": "PVD-A104 알람 발생 시 ① 게이트 밸브 즉시 차단 ...", "meta": { "source":"SOP-PVD-117 v3.2", "page":2, "dept":"PVD", "sec_level":"L2", "updated":"2026-03" } }

메타데이터가 핵심 — 출처 인용, 접근권한 필터(sec_level), 최신본 우선에 모두 쓰입니다.

CHAPTER III · 인덱싱청킹 전략

청킹 — 검색 품질의 8할

너무 작으면

한 문장이 여러 조각으로 → 맥락 깨짐, 청크 수·임베딩 비용 폭증, 표·헤더 분리.

너무 크면

한 청크에 잡음이 섞여 의미가 흐려짐 → 검색이 둔해지고 엉뚱한 회수.

실무300~800자에서 시작 + 10~15% 오버랩, 평가셋으로 튜닝.

구조 인식 분할

§ 헤더는 청크 시작에 유지, 표는 행 단위로 자르지 않기, 리스트는 묶어서. 문단 경계 우선.

오버랩(overlap)

인접 청크가 일부 겹치게 — 경계에 걸친 문장이 한쪽에서 잘려 검색 누락되는 걸 방지.

메타 주입
각 청크 앞에 "[SOP-PVD-117 §3] " 같은 헤더 경로를 붙이면 검색·인용 정확도↑.

ANN — 근사 최근접 이웃

전부 비교하면 수백만 번. HNSW(그래프 색인)로 "가까운 군집부터" 탐색해 밀리초에 top-K. 정확도 99%↑를 속도와 맞바꾼 근사 검색.

CHAPTER VIII · 운영평가 · 모니터링 · 재인덱싱

"잘 되고 있나"를 숫자로 — 평가와 운영

RAG 평가 (Ragas 등)

Faithfulness(근거 충실도) · Answer Relevancy(질문 적합) · Context Precision/Recall(검색 품질). 평가셋으로 정량 측정.

두 갈래로 분리 측정

검색이 틀렸나(엉뚱한 청크) vs 생성이 틀렸나(근거 무시·환각). 원인별로 고친다.

모니터링

지연시간·실패율·"자료 없음" 비율·사용자 피드백(👍/👎)·토큰 비용을 대시보드로.

재인덱싱 (CDC)

문서가 바뀌면 변경분만 다시 임베딩·upsert. 최신본 메타로 옛 청크 강등. 주기 배치 또는 변경 감지.

철칙평가셋 없이 배포 금지 — 측정 없으면 개선도 없다.

온프레미스

sec_level 메타로 검색 단계에서 필터 — 볼 수 있는 문서만 회수.

③

승인 게이트

쓰기·외부통신(메일·티켓)은 사람 승인 필수.

④

감사 로그

질의·검색·생성·행동을 출처와 함께 기록 → 재현·추적.

⑤

인젝션 방어

문서에 숨은 지시 차단, 출력 PII·금칙어 필터.

왜 watsonx + gpt-oss바로 이 다섯을 사내 경계 안에서 모두 충족하기 위함 — 오픈웨이트 모델을 사내 GPU에 올리는 구조의 핵심 이유.
EPILOGUE · 한 장으로

사내 문서를 검색 가능한 지능으로

BGE로 의미를 좌표화하고, Vector DB로 빠르게 회수하고, Rerank로 정밀하게 거르고, watsonx의 gpt-oss-120b로 근거 위에서 답하고, 에이전트 오케스트레이션으로 실행까지 — 전부 사내 경계 안에서.

  2. RAG가 재학습 없이 최신 사내 지식을 반영하는 방법은?
  3. 에이전트가 쓰기 작업(티켓·메일) 전에 반드시 거치는 것은?
정답 확인
  1. 질의 임베딩 → ANN 검색+권한 필터 → Hybrid 융합(RRF) → Rerank → top-5 컨텍스트 주입.
  2. 모델은 그대로 두고 검색 결과를 프롬프트에 주입 — 문서가 바뀌면 인덱스만 증분 갱신.
  3. 사람 승인 게이트(HITL) — 거부되면 쓰기 없이 보고서만 작성.
